# 5장.  RAG를 기능이 아니라 아키텍처로 이해하기

## 5.1 LLM은 왜 지식을 신뢰하기 어려운가

4장에서는 LLM 기반 시스템이 어떤 요소로 구성되는지를 살펴봤다. 입력이 들어오고, 모델이 처리하고, 출력이 나오는 파이프라인 구조였다. 그런데 그 파이프라인의 중심에 놓인 LLM 자체는 정보를 어떻게 보유하고 있을까? 그리고 그 방식이 실무에서 어떤 문제를 일으킬까?

이 절에서는 RAG가 등장하게 된 이유를 LLM 자체의 구조적 특성에서 찾는다. 먼저 LLM이 지식을 저장하는 방식을 정리하고, 거기서 비롯되는 세 가지 한계를 하나씩 살펴본다. 이 한계들을 이해하지 않으면, RAG를 왜 도입하는지, 그리고 왜 단순히 "검색 기능을 붙이는 것"으로는 부족한지를 설명하기 어렵다.

::: note 파라메트릭 메모리와 비파라메트릭 메모리
Lewis et al. (2020)은 LLM이 지식을 보유하는 방식을 두 가지로 구분한다.[^web_01]

- **파라메트릭 메모리(Parametric Memory)**: 모델의 가중치(파라미터) 안에 학습을 통해 인코딩된 지식이다. 모델이 실행될 때 외부 저장소를 참조하지 않고 가중치에서 직접 꺼내 쓰는 지식이다.
- **비파라메트릭 메모리(Non-Parametric Memory)**: 모델 외부의 저장소(데이터베이스, 문서 저장소 등)에 보관되며, 필요할 때 검색을 통해 가져오는 지식이다. 가중치를 변경하지 않아도 내용을 갱신할 수 있다.

이 구분이 이 장 전체 논의의 뼈대가 된다. RAG는 비파라메트릭 메모리를 파이프라인에 통합하는 방식이다.
:::

GPT 계열, Claude, Gemini와 같은 현재의 대형 언어 모델은 파라메트릭 메모리에 의존한다. 수천억 개의 파라미터에 방대한 텍스트 데이터로부터 추출한 패턴과 사실이 녹아 있다. 사용자가 질문을 던지면 모델은 그 파라미터를 활성화해 답변을 생성한다. 별도의 데이터베이스를 조회하거나 인터넷을 탐색하는 것이 아니다. 모델 안에 고정된 지식을 꺼내 쓰는 것이다. 이 구조는 추론 속도와 범용성이라는 강점을 가져다주지만, 동시에 세 가지 구조적 한계를 내재한다.

**그림 5-1. LLM 단독 시스템의 지식 구조적 한계 세 가지**

```
┌─────────────────────────────────────────────────────────┐
│                   LLM (파라메트릭 메모리)                   │
│                                                         │
│   ┌─────────────┐  ┌──────────────┐  ┌──────────────┐  │
│   │  지식 고착   │  │     환각      │  │ 폐쇄 도메인  │  │
│   │             │  │              │  │  지식 부재   │  │
│   │ 학습 완료   │  │ 모르는 것을  │  │ 내부 문서,  │  │
│   │ 시점에서    │  │ 그럴듯하게   │  │ 규제 문서는  │  │
│   │ 지식 동결   │  │ 채워서 출력  │  │ 학습 불포함  │  │
│   └─────────────┘  └──────────────┘  └──────────────┘  │
│                              │                          │
└──────────────────────────────┼──────────────────────────┘
                               ▼
              LLM 단독으로는 신뢰 가능한 정보를
              안정적으로 생성하기 어렵다
```

### 5.1.1 고착된 지식과 사라지는 정확도

LLM의 학습은 특정 시점에 수집된 데이터로 이루어진다. 그 시점 이후에 발생한 사건, 변경된 법령, 새로 발표된 제품 사양, 갱신된 기업 정책은 모델이 알 수 없다. 학습이 완료되는 순간 모델의 세계 인식은 그 시점에서 멈춘다. 이를 **지식 고착(Knowledge Cutoff)** 또는 **컷오프 문제**라고 부른다.

컷오프는 처음에는 그다지 심각해 보이지 않을 수 있다. 모델을 배포한 직후라면 컷오프와 현재 시점의 간극이 몇 달에 불과하기 때문이다. 그러나 시스템이 운영을 지속하면서 이 간극은 조용히 누적된다. 1년 뒤, 2년 뒤에도 같은 모델이 운영되고 있다면, 그 모델은 갈수록 오래된 세계를 기준으로 답변하고 있는 셈이다.

기업 내규 질의 시스템을 예로 들어보자. 인사팀이 LLM 기반 사내 Q&A 챗봇을 도입했다. 초기에는 직원 만족도가 높았다. 그런데 6개월 뒤 육아휴직 관련 내규가 개정되었다. 법령 변경에 따라 급여 지급 기준이 바뀌었고, 신청 절차도 달라졌다. 그러나 챗봇은 여전히 개정 전 내규를 기반으로 답변을 제공한다. 직원들은 잘못된 정보를 신뢰하고 그에 따라 행동한다. 문제는 챗봇이 이 오류를 틀렸다는 신호 없이 자신 있는 어조로 제공한다는 점이다.

모델을 주기적으로 재학습하면 해결되지 않을까? 이론적으로는 가능하지만, 실제로 수십억~수천억 개의 파라미터를 가진 대형 언어 모델을 재학습하는 데는 막대한 컴퓨팅 자원과 시간이 필요하다. 비용과 주기의 현실적 제약 때문에, 지식 고착은 운영 중인 시스템에서 구조적으로 발생하는 문제로 봐야 한다.

### 5.1.2 모른다고 말하지 않는 모델

지식 고착보다 더 까다로운 문제가 있다. LLM은 자신이 모른다는 사실을 명시적으로 인식하고 표현하는 구조가 아니다. 모델은 확률 기반의 토큰 생성 방식으로 작동한다. 다음에 올 가장 그럴듯한 단어를 예측하는 과정을 반복해 문장을 완성한다. 이 구조에서 "모른다"는 답변은 "관련 있는 토큰 시퀀스를 생성하지 않는다"는 결정을 필요로 하는데, 이 결정이 학습된 방식으로 일관되게 작동하지 않을 때 **환각(Hallucination)** 이 발생한다.

::: note 환각(Hallucination)
환각이란 LLM이 사실적으로 틀리거나 근거가 없는 내용을 생성하면서도, 표현상으로는 확신에 찬 어조를 유지하는 현상을 가리킨다.[^web_08] 잘못된 정보가 "잘못되었다"는 신호 없이 출력된다는 점이 핵심적인 위험이다. 환각은 완전히 없는 사실을 만들어내는 경우(사실 충돌형)와 실재하는 정보를 잘못 조합하거나 왜곡하는 경우(충실도 결여형)로 나뉜다.
:::

환각이 특히 위험한 이유는 출력의 형식이 정확한 정보와 구별되지 않기 때문이다. 모델이 확신 없이 말할 때와 틀린 내용을 말할 때의 언어 표현이 유사하다. 독자는 답변의 자신감 있는 어조를 정확성의 신호로 오인하기 쉽다.

2023년 미국에서 실제로 일어난 사례가 이 위험을 잘 보여준다. 한 변호사가 법원 서류를 작성하면서 LLM이 생성한 판례 목록을 그대로 인용했는데, 제출된 판례 중 다수가 실제로 존재하지 않는 것으로 확인되었다.[^web_08] 모델은 존재하지 않는 판례의 이름, 법원명, 판결 연도까지 구체적으로 생성했다. 법률 영역처럼 도메인 특화 지식이 중요한 분야에서 LLM 단독으로 판단을 내릴 때 어떤 결과가 발생할 수 있는지를 보여주는 사례다.

### 5.1.3 폐쇄 도메인 지식의 부재

세 번째 한계는 앞의 두 가지와 성격이 다르다. 지식 고착과 환각이 LLM의 작동 방식에서 비롯된 문제라면, 폐쇄 도메인 지식의 부재는 학습 데이터 자체의 구성 문제다.

LLM 사전 학습에 사용되는 데이터는 주로 공개된 인터넷 텍스트, 학술 문헌, 도서 등으로 구성된다. 기업 내부의 계약서, 제품 사양서, 내부 규정, 고객 상담 이력, 의료 기관의 임상 프로토콜, 금융 기관의 내부 감사 보고서 같은 자료는 사전 학습 데이터에 포함되지 않는다. 이런 자료들은 공개되어 있지 않거나, 설령 공개되어 있더라도 도메인 특화 맥락 없이는 의미 있게 학습하기 어렵다.

결과적으로, 기업이나 기관이 LLM에게 자신들의 내부 문서에 관한 질문을 던지면, 모델은 관련 지식이 없는 상태에서 답변을 생성해야 한다. 이때 앞서 살펴본 환각이 더 빈번하고 더 자신감 있게 나타나는 경향이 있다.[^web_11] 관련된 파라메트릭 메모리가 빈약할수록, 모델은 일반적인 언어 패턴에만 의존해 그럴듯한 내용을 채워 넣으려 하기 때문이다.

::: box 세 가지 한계가 결합될 때
세 한계가 동시에 작용하면 문제는 단순 합산이 아니라 상호 강화 형태로 나타난다. 기업 내부 문서를 다루는 시스템을 예로 들자. 학습 시점 이후 개정된 내규(지식 고착)를 다루는 질문에, 사전 학습 데이터에 없는 내용(폐쇄 도메인 지식 부재)을 묻는다면, 모델은 관련 정보가 전혀 없는 상태에서 그럴듯한 내용을 생성(환각)한다. 오류 하나하나가 눈에 띄지 않는 어조로 포장된 채 출력된다. 이 구조에서 LLM 단독 시스템은 신뢰 가능한 정보를 안정적으로 제공하기 어렵다.
:::

이 세 가지 한계는 모델의 품질 문제가 아니다. 파라메트릭 메모리에 의존하는 LLM의 구조적 속성에서 비롯된 것이다. 더 좋은 모델로 교체한다고 해서 사라지지 않는다. 따라서 해결책도 모델 내부가 아니라 시스템의 정보 흐름 구조에서 찾아야 한다.

여기서 자연스럽게 한 가지 질문이 따라온다. 이 한계를 인식한 팀들은 어떤 방식으로 대응했을까? 가장 흔한 접근은 "검색 기능을 붙이는 것"이었다. 관련 문서를 찾아서 프롬프트에 함께 넣으면 되지 않겠냐는 발상이다. 그런데 이 방식은 실제로 얼마나 충분할까? 다음 절에서는 RAG를 '기능'으로만 이해했을 때 어떤 일이 일어나는지를 구체적으로 살펴본다.

## 5.2 RAG를 기능으로 보면 무슨 일이 일어나는가

앞 절에서는 LLM이 왜 지식을 신뢰하기 어려운지를 구조적으로 살펴봤다. 지식 고착, 환각, 폐쇄 도메인 지식 부재라는 세 가지 한계가 결합되면, LLM 단독 시스템은 신뢰 가능한 정보를 안정적으로 제공하기 어렵다는 결론에 이르렀다. 이 인식에서 출발해 많은 팀이 자연스럽게 같은 방향으로 손을 뻗는다. "검색 기능을 붙이면 되지 않을까?" 관련 문서를 찾아 프롬프트에 함께 넣어주면 모델이 더 정확하게 답하리라는 발상이다.

실제로 이 접근은 효과가 있다. 아무것도 없는 것보다는 낫다. 그런데 문제는 RAG를 이렇게 "기능을 붙이는 것"으로만 이해했을 때 무슨 일이 일어나느냐에 있다. 이 절에서는 그 상황을 구체적으로 들여다본다.

::: note "RAG 품질 = 검색 정확도"라는 오해
RAG를 기능으로 이해하는 팀은 종종 "RAG 품질을 높이려면 검색 정확도를 높여야 한다"는 등식을 암묵적으로 가정한다. 이 오해가 이 절에서 교정하려는 핵심 인식이다. 검색 정확도는 RAG 품질의 필요 조건이지 충분 조건이 아니다. 검색이 잘 되더라도 컨텍스트 구성, 문서 청킹 전략, 생성 단계의 지시 방식 등이 결함을 가지면 출력 품질은 기대에 미치지 못한다.
:::

### 5.2.1 "검색이 이상하다"는 진단의 한계

RAG를 도입한 팀이 기대와 다른 답변을 받았을 때 가장 먼저 내리는 진단은 대개 두 가지 중 하나다. "검색이 이상하다"거나 "모델이 나쁘다"는 것이다. 그리고 실무 팀의 경험상 "모델이 나쁘다"는 진단은 교체 비용이 크기 때문에 쉽게 선택하기 어렵다. 결국 대부분의 문제는 "검색 쪽 문제"로 귀인된다.

이 귀인 구조에는 근본적인 결함이 있다. RAG 파이프라인에서 실패는 한 지점이 아니라 여러 단계에서 독립적으로 발생하기 때문이다. Barnett et al. (2024)은 실무 RAG 시스템에서 반복적으로 관찰되는 실패 지점을 체계적으로 분류했는데, 이 중 상당수가 검색 단계가 아니라 그 전후의 처리 단계에서 발생한다.[^web_03] "검색이 이상하다"는 진단 언어는 이 다양한 실패를 하나의 버킷에 쓸어 담아버린다.

구체적으로 세 가지 상황을 구별해볼 필요가 있다.

첫 번째는 검색 자체가 실패한 경우다. 질문에 관련된 문서가 실제로 없거나, 임베딩 품질 문제로 유사도 계산이 잘못되어 엉뚱한 문서가 올라오는 경우다. 이 경우에는 "검색이 이상하다"는 진단이 맞다.

두 번째는 검색은 성공했지만 컨텍스트 구성이 실패한 경우다. 관련 문서는 올바르게 찾아왔지만, 그 문서를 어떻게 자르고 어떤 순서로 프롬프트에 배치하느냐에 따라 모델이 그 정보를 활용하는 방식이 달라진다. 검색된 문서 세 개 중 가장 관련성이 높은 문서가 컨텍스트 끝에 배치되고 노이즈가 앞에 놓인다면, 모델은 앞쪽 정보에 더 많이 의존하는 경향을 보일 수 있다. 이 실패는 "검색이 이상하다"가 아니라 "컨텍스트 구성이 잘못되었다"는 언어로 진단해야 한다.

세 번째는 검색과 컨텍스트 구성이 모두 성공했지만 생성 단계에서 실패하는 경우다. 모델이 주어진 컨텍스트를 무시하거나, 컨텍스트와 파라메트릭 메모리 사이에서 어느 쪽을 우선할지 잘못 판단하는 경우가 여기에 해당한다. 이 역시 검색 문제가 아니다.

기능 관점의 진단 언어는 이 세 상황을 구별하지 못한다. 결과적으로 팀은 두 번째와 세 번째 유형의 실패를 첫 번째 유형으로 오인해 검색 모듈만 반복적으로 수정하는 루프에 빠진다. 고쳐도 나아지지 않는 이유가 애초에 진단 자체가 틀렸기 때문인데, 그 사실을 인식하기 위한 언어가 없는 것이다.

**그림 5-2. 기능으로 본 RAG — 실패 발생 시 어디를 봐야 할지 모르는 상황**

```
  사용자 질문
      │
      ▼
┌─────────────┐        실패 발생 시 진단 언어:
│  [검색]     │◄───── "검색이 이상하다" (검색 실패)
└──────┬──────┘        "모델이 나쁘다" (생성 실패)
       │                        │
       │                        └─ 이 두 범주만 존재하는 구조
       ▼
┌─────────────────────────────────────┐
│                                     │
│  ??? 컨텍스트 구성    ??? 청킹 전략  │◄── 기능 관점에서는
│  ??? 문서 품질        ??? 순서 배치  │    보이지 않는 영역
│                                     │
└──────┬──────────────────────────────┘
       │
       ▼
┌─────────────┐
│  [생성]     │
└──────┬──────┘
       │
       ▼
  최종 답변
```

### 5.2.2 Naive RAG가 만들어내는 새로운 실패 지점

기능 관점으로 RAG를 구현하면 자연스럽게 특정한 구조에 수렴한다. Gao et al. (2024)은 이를 **Naive RAG(단순 RAG)** 라고 부른다.[^web_02] 질문을 그대로 검색 쿼리로 전환하고, 유사 문서를 검색하고, 검색 결과를 그대로 컨텍스트에 붙여 생성 단계로 넘기는 구조다.

::: box Naive RAG의 전형적 구조
Gao et al. (2024)이 정의한 Naive RAG는 다음 네 단계로 구성된다.[^web_02]

1. **인덱싱(Indexing)**: 문서를 청크(chunk, 조각)로 분할하고 임베딩하여 벡터 저장소에 적재한다.
2. **검색(Retrieval)**: 사용자 질문을 임베딩하여 유사도가 높은 청크를 벡터 저장소에서 꺼낸다.
3. **컨텍스트 구성(Augmentation)**: 검색된 청크를 그대로 프롬프트에 붙인다.
4. **생성(Generation)**: 컨텍스트가 포함된 프롬프트를 LLM에 전달하여 답변을 생성한다.

각 단계에는 명시적인 설계 판단이 없다. 청크 크기는 임의로 정해지고, 컨텍스트 구성 방식은 단순 연결(concatenation)이며, 검색 결과의 품질을 별도로 검증하는 단계가 없다. 이 구조에서 각 단계는 설계 결정 지점이 아니라 고정된 처리 절차로 취급된다.
:::

Naive RAG의 문제는 단순함 자체가 아니다. 각 단계가 설계 판단 없이 연결되어 있다는 점, 그리고 그 결과로 새로운 종류의 실패가 조용히 발생한다는 점이다.

가장 주목해야 할 패턴은 검색된 컨텍스트가 잘못되었을 때 모델이 어떻게 반응하느냐다. RAG가 없을 때 모델은 파라메트릭 메모리에 의존해 답변을 생성한다. 그 답변이 틀릴 수 있지만, 적어도 모델이 활용하는 정보의 출처는 일정하다. 그런데 Naive RAG가 잘못된 컨텍스트를 제공하면, 모델은 그 잘못된 컨텍스트를 근거 삼아 더 확신에 찬 오답을 생성한다. 외부에서 제공된 정보는 모델에게 "신뢰할 수 있는 근거"처럼 작동하기 때문이다. 이 패턴은 환각보다 오히려 더 까다로운 문제를 만든다. 틀린 답이 더 그럴듯한 형태로, 더 자신감 있는 어조로 출력되기 때문이다.

RAGEC 프레임워크를 제시한 연구에 따르면, RAG 파이프라인에서 발생하는 오류 중 청킹 오류와 검색 오류가 환각보다 더 빈번하게 관찰된다.[^web_12] 이 사실은 중요한 함의를 가진다. RAG를 도입했을 때 남아 있는 품질 문제의 상당 부분이 모델 자체의 한계가 아니라 파이프라인 설계의 결함에서 비롯된다는 것이다.

Naive RAG의 또 다른 결함은 "검색 이후를 설계하지 않는다"는 암묵적 선택에 있다. 청킹 전략은 결과에 직접적인 영향을 미친다. 문서의 논리 단위를 무시하고 고정 크기로 자르면, 단일 문장의 앞부분과 뒷부분이 서로 다른 청크에 배치되거나, 하나의 논리적 주장이 여러 청크로 분산된다. 그 결과 검색된 청크 하나하나는 의미 있어 보이지만, 모델이 그것을 조합해 정합한 답변을 내놓기 어려운 상황이 만들어진다. 문서 품질, 컨텍스트에 배치되는 청크의 순서, 중복된 정보의 처리 방식도 마찬가지다. 이 모든 변수가 결과에 영향을 미치지만, 기능 관점에서는 이 변수들이 "설계 결정 지점"이 아니라 그냥 "디폴트값"으로 방치된다.

앞서 살펴봤듯이, 3장의 논지도 같은 구조를 가리킨다. 문제가 모델 자체에 있는 것이 아니라 시스템의 정보 흐름 구조에 있다는 것이다.[^web_03] RAG를 기능으로 보면 이 구조적 문제를 인식할 언어가 없다. "RAG를 붙였는데 왜 이상한 답이 나오는가"라는 질문은 "검색이 이상하다"와 "모델이 나쁘다" 사이에서 진단이 헛돌고, 정작 고쳐야 할 지점은 계속 보이지 않는다.

::: tip 실무 진단의 출발점
RAG 시스템에서 기대와 다른 출력이 나왔을 때, 가장 먼저 해야 할 질문은 "검색이 잘못된 것인가?"가 아니다. "파이프라인의 어느 단계에서 정보가 손상되었는가?"를 단계별로 확인하는 것이 유효하다. 검색된 문서가 실제로 관련이 있는지, 그 문서가 적절한 단위로 분할되어 있는지, 컨텍스트가 모델에 효과적으로 전달되었는지를 순서대로 점검하면 문제의 위치를 훨씬 빠르게 특정할 수 있다.
:::

지금까지 살펴본 내용은 결국 하나의 질문으로 수렴된다. RAG를 기능이 아니라 다른 방식으로 바라본다면, 지금 보이지 않는 것들이 보이게 될까? 다음 절에서는 RAG를 아키텍처로 재정의했을 때 무엇이 달라지는지를 구체적으로 살펴본다. "기능으로 보면 이것이 안 보인다"는 사실을 먼저 확인했으니, 이제 "아키텍처로 보면 이것이 보인다"는 대비 구도가 선명하게 드러날 것이다.

## 5.3 RAG를 아키텍처로 재정의하면 무엇이 보이는가

앞 절에서는 RAG를 기능으로 바라보면 무엇이 보이지 않는지를 살펴보았다. Naive RAG의 각 단계가 설계 판단 없이 연결될 때, 잘못된 청킹이나 부적절한 검색 결과가 파이프라인을 조용히 통과한다는 것, 그리고 그 결과로 발생하는 실패를 "검색이 이상하다"나 "모델이 나쁘다" 사이에서 진단이 헛도는 상황이 된다는 것을 확인했다. 이제 시각을 바꾸어 살펴볼 차례다. RAG를 아키텍처로 재정의했을 때 비로소 보이기 시작하는 것들이 있다.

::: note RAG의 아키텍처적 정의
RAG는 LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조다.

이 정의에서 핵심은 세 부분에 있다. 첫째, 통제의 주체가 시스템이다. 모델이 스스로 무엇을 참조할지 결정하는 것이 아니라, 시스템 설계가 그 경계를 정한다. 둘째, 통제 대상이 출처와 범위, 두 가지다. 어디서 정보를 가져올 것인가(출처)와 얼마나 어떤 정보를 가져올 것인가(범위)는 별개의 설계 판단이다. 셋째, 통제 지점이 하나가 아니다. 색인, 검색, 컨텍스트 구성, 생성, 후처리 각 단계에서 시스템이 개입한다.
:::

IBM의 공식 기술 문서는 RAG를 "아키텍처 패턴(architectural pattern)"으로 명시적으로 정의한다.[^web_07] 학술 논문이 아닌 산업 현장에서도 이미 같은 언어가 쓰이고 있다는 점은 의미가 있다. "기능을 추가한다"는 표현과 "아키텍처 패턴을 적용한다"는 표현은 설계자의 사고방식 전체를 다르게 만든다.

**표 5-1. 기능 관점 vs. 아키텍처 관점에서의 RAG 이해 비교**

| 비교 항목 | 기능 관점 | 아키텍처 관점 |
|---|---|---|
| RAG를 정의하는 언어 | "검색 기능을 LLM에 붙인다" | "정보 흐름의 출처와 범위를 시스템이 통제한다" |
| 설계 질문 | "어떤 벡터 DB를 쓸 것인가?" | "각 단계에서 어떤 정보가 통과해야 하는가?" |
| 실패 진단 방식 | "검색이 이상하다 / 모델이 나쁘다" | "파이프라인의 몇 번째 단계에서 정보가 손상되었는가?" |
| 확장성 인식 | 기능 단위로 교체·추가 | 통제 지점 단위로 재설계 |
| 실패 책임 소재 | 모호하거나 단일 컴포넌트에 귀결 | 각 통제 지점에 명시적으로 배분 가능 |

### 5.3.1 정보 흐름을 통제한다는 것의 의미

"검색을 추가한다"는 행위는 정보 흐름의 한 지점만을 건드린다. 사용자의 질문이 벡터 검색으로 연결되고, 그 결과가 프롬프트에 붙는 것. 여기서 설계자의 주의는 대체로 "검색이 잘 되는가"에 집중된다. 그런데 아키텍처 관점에서 보면 이 흐름 위에 훨씬 많은 판단이 있다.

"정보 흐름을 통제한다"는 것은 어떤 정보가 어느 단계에서 시스템에 진입하고, 어떻게 선택되며, 어떻게 구성되어 모델에 전달되는지를 전 구간에 걸쳐 설계하는 것이다. 같은 질문을 두 관점에서 각각 어떻게 표현하는지 비교해보면 차이가 선명해진다.

기능 관점의 설계자는 이런 질문을 던진다. "어떤 임베딩 모델을 쓸 것인가?", "상위 몇 개의 청크를 가져올 것인가?", "어떤 벡터 데이터베이스가 성능이 좋은가?" 반면 아키텍처 관점의 설계자는 다른 질문을 던진다. "문서를 어떤 단위로 분할해야 의미 단위가 보존되는가?", "검색된 결과 중 실제로 질문과 관련 있는 것을 어떻게 걸러낼 것인가?", "여러 청크를 어떤 순서로 모델에 전달할 때 모델이 정보를 더 잘 활용하는가?", "모델의 출력이 제공된 컨텍스트에 실제로 근거하고 있는지 확인하는 단계를 둘 것인가?" 이 두 질문 목록의 차이가 바로 관점 전환이 가져다주는 실질적인 차이다.

통제 지점이 여럿이라는 것은 설계 결정이 여럿이라는 뜻이다. 그리고 각 결정은 독립적으로 실패할 수 있다. Gao et al. (2024)이 제시한 Modular RAG 프레임워크는 이 사실을 구조적으로 명시한다. 각 모듈이 독립적인 설계 결정 지점이며, 재구성 가능한 단위로 취급될 수 있다는 것이다.[^web_04] 검색 모듈에서의 판단과 컨텍스트 구성 모듈에서의 판단은 서로 독립적이다. 어느 하나가 잘 작동해도 다른 하나가 실패하면 전체 출력이 손상된다.

**그림 5-1. RAG 아키텍처의 정보 흐름 통제 레이어 구조**

```
외부 문서 저장소
       │
       ▼
┌─────────────────┐
│   색인(Indexing) │◄── [통제 지점 1] 청킹 전략, 임베딩 방식 선택
│  청킹 · 임베딩  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  검색(Retrieval) │◄── [통제 지점 2] 유사도 기준, 결과 수, 필터링
│  벡터 유사도    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 컨텍스트 구성   │◄── [통제 지점 3] 배치 순서, 중복 제거, 관련성 재검증
│ (Augmentation)  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   생성(LLM)     │◄── [통제 지점 4] 프롬프트 구조, 출력 형식 제약
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  후처리(Post)   │◄── [통제 지점 5] 근거 확인, 출력 검증, 필터링
└────────┬────────┘
         │
         ▼
     최종 출력
```

::: box Naive → Advanced → Modular RAG 진화 계보
RAG의 구조적 진화는 "기능에서 아키텍처로"의 이동 궤적을 따른다.[^web_02][^web_04]

**Naive RAG**: 질문을 그대로 검색 쿼리로 전환하고, 유사 청크를 프롬프트에 연결한 후 생성한다. 각 단계는 고정된 처리 절차이며, 설계 판단의 여지가 거의 없다. "검색을 붙인다"는 기능 관점이 가장 자연스럽게 구현되는 형태다.

**Advanced RAG**: 청킹 전략, 하이브리드 검색(키워드 + 의미 기반), 재순위화(reranking), 쿼리 재작성 등 각 단계에 명시적인 설계 판단이 개입하기 시작한다. "기능"이었던 것들이 "조정 가능한 변수"가 된다. 그러나 전체 파이프라인의 구조는 여전히 고정되어 있다.

**Modular RAG**: 각 처리 단계를 독립적인 모듈로 분리하고, 필요에 따라 재구성하거나 교체할 수 있는 프레임워크로 진화한다. 라우팅, 적응적 검색(adaptive retrieval), 반복 검색(iterative retrieval) 등이 이 단계에서 등장한다. 각 모듈이 독립적인 아키텍처 선택 지점이 된다. "기능을 붙인다"는 언어는 이 수준에서 더 이상 유효하지 않다.

이 진화 계보는 도구의 발전이기도 하지만, 더 근본적으로는 RAG를 바라보는 인식의 변화다. 각 단계가 독립적으로 설계되어야 하며, 독립적으로 평가되어야 한다는 인식이 구조에 반영된 결과다.
:::

### 5.3.2 하네스와의 연결: 통제 구조는 외부에서 감싼다

RAG를 아키텍처로 보는 관점은 1장에서 도입한 하네스 개념과 자연스럽게 연결된다. 하네스(harness, 평가 하네스)는 실행과 평가를 반복 가능하게 수행하는 외부 구조다. 모델 내부를 건드리지 않고 시스템의 동작을 감시하고, 결과를 측정하고, 필요할 때 개입할 수 있는 바깥 레이어다.

RAG의 각 통제 지점, 즉 색인, 검색, 컨텍스트 구성, 생성, 후처리는 하네스가 감시하고 개입할 수 있는 경계이기도 하다. 검색 단계에서 가져온 청크가 질문과 실제로 관련이 있는지, 컨텍스트에 배치된 정보가 모델 입력 한계 안에서 의미 있게 구성되어 있는지, 생성된 출력이 제공된 컨텍스트에 근거를 두고 있는지를 각 경계에서 측정하고 검증하는 것이 가능하다. RAGAS와 같은 프레임워크가 검색, 컨텍스트 구성, 생성 각 단계를 독립적으로 평가하는 메트릭을 제시하는 것도 같은 이유에서다.[^web_06] "RAG를 아키텍처로 본다"는 것은 곧 각 통제 지점에서 하네스가 작동해야 한다는 인식과 연결된다.

::: tip 두 관점에서 같은 시스템 기술하기
동일한 RAG 시스템을 두 관점에서 다르게 기술할 수 있다.

**기능 관점**: "사용자의 질문을 받아 관련 문서를 검색하고, 검색 결과와 함께 LLM에 질의하여 답변을 생성하는 시스템이다."

**아키텍처 관점**: "색인 단계에서 문서의 의미 단위를 보존하는 청킹 전략을 적용하고, 검색 단계에서 의미 유사도와 키워드 기반 검색을 결합한 후 재순위화를 거쳐 관련성을 검증한다. 컨텍스트 구성 단계에서는 중복 제거와 배치 순서를 설계하며, 생성 이후에는 출력이 컨텍스트에 근거하고 있는지를 별도 단계에서 확인한다."

두 기술 방식은 같은 시스템을 설명하지만, 어디서 무엇을 개선해야 하는지, 어디서 실패가 발생했는지를 추적하는 언어는 전혀 다르다. 팀이 어떤 언어를 쓰느냐에 따라 설계 회의에서 논의될 수 있는 것과 논의되지 못하는 것이 달라진다.
:::

RAG를 아키텍처로 재정의하는 것은 단순히 용어를 바꾸는 일이 아니다. 설계 판단이 어디에 있는지, 실패가 어디서 비롯되는지, 그리고 시스템을 어떻게 개선해야 하는지를 이야기할 수 있는 언어를 갖추는 일이다. 이 언어가 없으면, 잘 작동하는 RAG 시스템을 만드는 것도, 작동하지 않는 RAG 시스템을 고치는 것도 어렵다.

여기까지 RAG의 아키텍처적 정의와 그 정의가 가져다주는 관점 전환을 살펴보았다. 이제 이 구조를 구성하는 각 단계를 하나씩 분해해볼 필요가 있다. 색인, 검색, 컨텍스트 구성, 생성, 후처리 각 단계가 정보 흐름 통제의 어느 지점을 담당하며, 어떤 설계 결정이 그 안에 들어 있는지를 이어지는 절에서 구체적으로 들여다본다.

## 5.4 RAG 파이프라인 단계별 역할과 통제 지점

앞 절에서 RAG를 "여러 통제 지점의 집합으로 이루어진 아키텍처"로 정의했다. 그 정의가 설득력을 가지려면, 각 통제 지점이 실제로 무엇을 통제하는지가 구체적으로 드러나야 한다. 이 절에서는 RAG 파이프라인을 구성하는 다섯 단계, 즉 문서 전처리·색인, 검색, 컨텍스트 구성, 생성, 출력 후처리를 하나씩 분해하여 각 단계가 담당하는 역할과 그 안에 들어 있는 설계 결정을 구체적으로 살펴본다.

**표 5-2. RAG 파이프라인 단계별 역할과 통제 대상**

| 단계 | 역할 | 통제 대상 | 주요 설계 변수 |
|------|------|-----------|---------------|
| 문서 전처리·색인 | 정보 진입 통제 | 어떤 정보를 검색 가능한 상태로 만들 것인가 | 청킹 전략, 메타데이터 구조, 색인 방식 |
| 검색 | 정보 선택 통제 | 어떤 정보를 꺼낼 것인가 | 쿼리 구성, 유사도 계산 방식, Top-K 수, 조건부 검색 여부 |
| 컨텍스트 구성 | 정보 구성 통제 | 꺼낸 정보를 어떻게 모델에 전달할 것인가 | 배치 순서, 중복 제거, 길이 제한, 출처 표시 |
| 생성 | 정보 활용 | 주어진 정보를 바탕으로 출력을 만든다 | 프롬프트 구조, 모델 선택, 출력 형식 제약 |
| 출력 후처리 | 출력 통제 | 생성된 결과를 검증·정제한다 | 출처 확인, 신뢰도 임계값, 포맷 검증 |

**그림 5-4. RAG 파이프라인의 단계별 정보 흐름과 통제 지점**

```

        │
        ▼

└────────┬─────────┘
         │ 검색 가능한 청크
         ▼

         ▼

└────────┬─────────┘
         │ 구성된 컨텍스트
         ▼
```

각 단계는 독립적인 평가 지점이기도 하다. RAGAS와 같은 프레임워크가 검색, 컨텍스트 구성, 생성 각 단계를 별도의 메트릭으로 측정하는 이유는, 단계별로 발생하는 실패가 서로 다른 원인을 가지고 있기 때문이다.[^web_06] 이 절에서 각 단계를 분리해서 설명하는 것도 같은 이유에서다.

### 5.4.1 정보 진입 통제: 색인이 결정하는 것들

RAG 파이프라인에서 가장 먼저 주목해야 할 단계는 문서 전처리와 색인이다. 이 단계는 파이프라인의 "상류(upstream)"에 해당한다. 여기서 내려진 결정은 이후 모든 단계에 영향을 미치며, 나중에 잘못을 발견하더라도 수정 비용이 크다. 검색 단계에서 아무리 정교한 알고리즘을 쓰더라도, 색인에 담기지 않은 정보는 꺼낼 수 없다. Gao et al. (2024)가 Naive RAG의 가장 근본적인 한계 중 하나로 청킹 전략의 취약성을 지목한 것도 이 때문이다.[^web_02]

#### 청킹 전략

문서를 어떤 단위로 분할할 것인가의 문제다. 청킹(chunking)은 원문 문서를 검색 가능한 단위로 쪼개는 작업을 가리킨다. 표면적으로는 기술적 전처리처럼 보이지만, 실제로는 "어떤 단위로 정보를 검색 가능하게 만들 것인가"라는 설계 선택이다.

::: box 청킹 전략은 왜 설계 결정인가
청킹은 세 가지 방식으로 구현될 수 있으며, 각각의 트레이드오프가 있다.

**문장 단위 청킹**: 문장 경계를 기준으로 분할한다. 청크가 작아서 정밀한 검색이 가능하지만, 단일 문장만으로는 맥락이 사라지는 경우가 많다. "규정 위반 시 해고될 수 있다"는 문장은 어떤 규정인지, 어떤 상황인지에 대한 정보 없이는 의미가 제한된다.

**단락 단위 청킹**: 단락이나 섹션 경계를 기준으로 분할한다. 맥락을 더 잘 보존하지만, 청크가 커질수록 하나의 청크 안에 관련 없는 정보가 섞일 확률이 높아진다. 검색된 청크 전체가 관련 있지 않은 상황에서도 모델에 전달될 수 있다.

**의미 단위 청킹(Semantic Chunking)**: 텍스트의 주제 경계나 의미 유사성을 기준으로 분할한다. 맥락 보존과 정밀도 사이의 균형을 추구하지만, 구현 복잡도가 높고 도메인에 따라 효과가 달라진다.

어느 전략이 "최선"인지는 문서 유형, 질문 유형, 검색 방식에 따라 달라진다. 청킹 전략은 한 번 결정하면 전체 색인을 재구축해야 변경할 수 있다는 점에서, 초기 설계 시점에 명시적으로 논의되어야 할 결정이다.
:::

#### 메타데이터와 색인 방식

청킹 못지않게 중요한 것이 메타데이터 설계다. 문서 출처, 생성 날짜, 버전 정보, 접근 권한 등의 메타데이터는 검색 단계에서 필터링 조건으로 활용될 수 있다. 그런데 이 정보가 색인 시점에 구조화되어 있지 않으면, 검색 단계에서는 그 정보를 활용할 방법이 없다. "2024년 이후에 갱신된 문서만 검색하라"는 조건은 색인에 날짜 필드가 존재해야 실행 가능하다.

색인 방식 자체도 설계 변수다. 벡터 색인은 의미 기반 검색에 강하지만 정확한 용어 매칭에 약하다. 키워드 기반 색인(BM25 등)은 반대의 특성을 가진다. 하이브리드 방식은 두 가지를 결합하지만 구성 비율 자체가 또 하나의 설계 결정이 된다.[^web_04] 색인 방식의 선택은 검색 단계에서 어떤 종류의 검색이 가능한지를 규정한다.

### 5.4.2 정보 선택 통제: 검색이 결정하는 것들

색인이 "무엇을 찾을 수 있는가"를 규정한다면, 검색 단계는 "지금 이 질문에 대해 무엇을 꺼낼 것인가"를 결정한다. 겉으로 보면 쿼리를 던지고 결과를 받아오는 단순한 과정처럼 보이지만, 그 안에는 여러 설계 결정이 들어 있다.

#### 쿼리 구성 방식

사용자의 원래 질문을 그대로 검색 쿼리로 사용할 것인가, 아니면 재작성할 것인가. 사용자는 종종 불완전하거나 모호한 방식으로 질문한다. "지난번 회의에서 이야기한 내용이 뭐였지?"라는 질문은 벡터 공간에서 관련 문서를 찾기 어려운 형태다. 쿼리 재작성(query rewriting)이나 쿼리 분해(query decomposition)는 이런 경우에 검색 품질을 높이는 설계 선택이다. 그러나 재작성 과정 자체가 오류를 도입할 수도 있다는 점에서, 이 또한 트레이드오프가 있는 결정이다.[^web_02]

#### 유사도 계산과 Top-K

벡터 유사도 기반 검색은 의미적으로 유사한 텍스트를 잘 찾지만, 정확한 용어나 코드, 고유명사의 매칭에는 취약하다. "ISO 27001 3.4.2 조항"처럼 정확한 표현이 중요한 검색에서 의미 기반 유사도만 사용하면 엉뚱한 결과가 나올 수 있다.

Top-K, 즉 검색 결과를 몇 개나 가져올 것인지도 설계 변수다. K가 클수록 관련 정보를 빠뜨릴 가능성(재현율 저하)은 낮아지지만, 관련 없는 정보가 섞일 가능성(노이즈)은 높아진다. K값은 단순히 "더 많이 가져오면 좋다"는 방식으로 결정할 수 없다. 컨텍스트 구성 단계의 처리 용량, 모델의 컨텍스트 창 크기, 노이즈가 생성 품질에 미치는 영향을 함께 고려해야 한다.

#### 조건부 검색의 가능성

모든 질문에 검색이 필요한가? 이 질문은 검색 단계 설계에서 가장 근본적인 것 중 하나다.

::: note Self-RAG — 검색을 조건부로 만드는 설계
Asai et al. (2023)의 Self-RAG는 검색 여부 자체를 모델이 동적으로 판단하도록 설계한 접근이다.[^web_05] 모든 입력에 대해 무조건 검색을 수행하는 대신, 특수 토큰(reflection token)을 통해 "이 질문에 외부 정보가 필요한가"를 먼저 판단한다.

실험 결과, 무차별 검색(unconditional retrieval)은 오히려 성능을 저하시키는 경우가 있었다. 단순한 사실 확인이나 일반 상식에 대한 질문에서는 검색 없이 생성한 결과가 더 정확했다. 검색된 외부 정보가 오히려 노이즈로 작용해 생성 품질을 낮춘 것이다.

이 연구는 "RAG는 항상 써야 한다"는 전제가 설계 결정이 아니라 관성임을 보여준다. 검색 여부, 검색 시점, 검색 결과의 활용 여부를 모두 설계 변수로 다루어야 한다.
:::

### 5.4.3 정보 구성 통제와 출력 통제: 전달과 검증

검색 단계가 어떤 정보를 꺼낼지를 결정한다면, 컨텍스트 구성 단계는 꺼낸 정보를 어떻게 모델에 전달할지를 결정한다. 이 단계는 RAG 파이프라인에서 가장 과소평가되는 설계 지점 중 하나다.

#### 컨텍스트 구성: 순서, 중복, 길이

검색된 청크들을 어떤 순서로 배치할 것인가는 단순한 형식 문제가 아니다.

::: box 컨텍스트 구성에서 순서가 중요한 이유
Liu et al. (2023)의 연구에서 기술된 "Lost in the Middle" 현상은 LLM이 긴 컨텍스트에서 정보를 균등하게 활용하지 않는다는 것을 보여준다. 모델은 컨텍스트의 앞부분과 끝부분에 위치한 정보를 중간 부분보다 더 잘 활용하는 경향이 있다. 즉, 검색된 문서 중 가장 관련성이 높은 청크를 컨텍스트의 중간에 배치하면, 그 정보가 실제 생성에 충분히 반영되지 않을 수 있다.

이 현상은 컨텍스트 배치 순서가 단순한 구현 세부사항이 아니라 생성 품질에 직접 영향을 미치는 설계 결정임을 보여준다. 관련성이 높은 청크를 컨텍스트의 앞 또는 뒤에 배치하는 전략, 재순위화(reranking)를 통해 배치 순서를 조정하는 전략이 이에 대한 설계 응답이다.
:::

중복 제거도 컨텍스트 구성 단계의 설계 과제다. 여러 문서에서 동일한 내용이 다른 표현으로 등장하는 경우, 이를 그대로 모델에 전달하면 컨텍스트 창을 낭비하고 모델이 동일 정보를 중복 처리하게 만든다. 길이 제한은 모델의 컨텍스트 창 크기라는 하드 제약에서 비롯되지만, 어떤 정보를 우선적으로 포함하고 어떤 정보를 잘라낼지를 결정하는 것 자체가 설계 판단이다.

#### 생성 단계의 설계 책임

생성 단계는 파이프라인에서 유일하게 LLM이 직접 개입하는 지점이다. 그러나 이 단계가 "모델에 맡긴다"는 의미는 아니다. 모델이 컨텍스트에 주어진 정보를 따를지, 사전 학습에서 습득한 지식을 우선할지는 프롬프트 설계와 모델 선택에 의해 상당 부분 결정된다.[^web_04]

"제공된 문서에 근거해서만 답변하라"는 지시를 프롬프트에 포함하는 것, 컨텍스트 없이 답변할 수 없는 경우 "모른다"고 응답하도록 유도하는 것, 출력 형식을 JSON이나 마크다운으로 제한하는 것 모두 생성 단계에서의 설계 결정이다. 이 결정들이 명시적으로 이루어지지 않으면, 모델의 기본 동작이 파이프라인의 의도를 결정하게 된다.

#### 출력 후처리: 결과를 검증·정제한다

생성 단계가 출력을 만들었다고 해서 파이프라인이 끝나는 것은 아니다. 출력 후처리 단계에서는 생성된 결과를 시스템이 한 번 더 검토한다. 주요 작업은 세 가지다.

첫째, **출처 확인**이다. 생성된 응답이 실제로 컨텍스트에 있는 정보에 근거하고 있는지를 확인한다. RAGAS의 Faithfulness 메트릭이 이를 자동화하는 방식 중 하나다.[^web_06] 둘째, **신뢰도 임계값 적용**이다. 검색 결과의 유사도 점수나 생성 응답의 확신 수준이 임계값 미만이면 응답을 보류하거나 "확인이 필요합니다"와 같은 안전 응답으로 대체하는 전략이다. 셋째, **포맷 검증**이다. 출력 형식이 다운스트림 시스템에서 기대하는 구조와 일치하는지를 확인한다. JSON 파싱 오류, 필수 필드 누락, 인코딩 문제 등이 여기서 걸러진다.

출력 후처리는 Barnett et al. (2024)이 지적한 것처럼, 파이프라인의 실패가 운영 중에 드러나는 지점이기도 하다.[^web_03] 이 단계를 파이프라인에 명시적으로 설계하지 않으면, 오류는 사용자에게 그대로 전달된다.

---

각 단계가 독립적인 설계 결정 지점이라는 인식은, 실패가 발생했을 때 원인을 어디서 찾아야 하는지를 결정한다. "RAG가 잘못된 답변을 했다"는 진술은 진단 언어가 아니다. "컨텍스트 구성 단계에서 관련성 낮은 청크가 포함되어 생성 단계에서 노이즈로 작용했다"는 진술이 실제 수정 가능한 진단 언어다. 이 언어를 갖추기 위해 다음 절에서는 각 단계에서 발생할 수 있는 실패 유형을 구체적으로 분류한다.

## 5.5 RAG 실패

앞 절에서는 RAG 파이프라인의 각 단계가 독립적인 설계 결정 지점임을 살펴보았다. 입력 처리부터 출력 후처리까지, 각 단계에서 이루어지는 판단이 파이프라인 전체의 정보 흐름을 결정한다. 이제 그 인식을 실패 진단에 적용해보자. RAG 시스템이 기대에 못 미치는 결과를 냈을 때, 그 원인을 어디서 어떻게 찾아야 하는가?

### 5.5.1 "RAG가 이상하다"는 진단 언어가 아니다

RAG를 도입한 팀이 문제를 처음 인식하는 방식은 대체로 비슷하다. "답변이 틀렸다", "엉뚱한 내용을 가져온다", "검색이 이상한 것 같다". 이 표현들은 문제가 있다는 신호를 담고 있지만, 수정 가능한 진단 언어는 아니다. 어디를 고쳐야 하는지를 지목하지 못하기 때문이다.

RAG를 단일 기능으로 이해하면 이런 표현에 머물게 된다. "RAG가 작동하지 않는다"는 진단은 RAG를 하나의 블랙박스로 보는 관점에서 나온다. 그러나 RAG를 아키텍처로 이해하면, 실패는 특정 단계에서 발생한 결함으로 해석된다. "컨텍스트 구성 단계에서 관련성 낮은 청크가 포함되었다", "청킹 경계가 논리 단위를 끊어서 검색 단계에서 불완전한 정보가 회수되었다", "생성 단계에서 컨텍스트 대신 사전 학습 지식이 우선되었다". 이 수준의 진단이 가능해야 실제로 무엇을 수정해야 하는지를 결정할 수 있다.

Barnett et al. (2024)은 RAG 시스템을 운영하면서 반복적으로 나타나는 실패 지점을 일곱 가지로 분류했다.[^web_03] 이 분류에서 주목할 점은 실패가 생성 단계에만 집중되지 않는다는 것이다. 청킹 전략, 임베딩 품질, 인덱스 구조, 검색 쿼리 구성, 컨텍스트 배치 방식, 프롬프트 설계, 출력 검증에 이르기까지, 실패는 파이프라인 전 구간에 분산되어 나타난다. 그리고 이 실패들 중 다수는 시스템이 운영되기 전에는 드러나지 않는다.[^web_03]

### 5.5.2 단계별 실패 유형

파이프라인의 각 단계에서 어떤 유형의 실패가 발생하는지를 정리하면 다음과 같다.

**표 5-1. RAG 파이프라인 단계별 주요 실패 유형**

| 단계 | 실패 유형 | 증상 |
|------|-----------|------|
| 사전 처리 (청킹) | 논리 단위를 무시한 기계적 분할 | 검색된 청크가 문맥 없이 잘려 있어 생성 품질 저하 |
| 임베딩 | 도메인 불일치 임베딩 | 의미적으로 관련 있는 문서가 낮은 유사도로 평가됨 |
| 검색 | 관련성 낮은 청크 회수 | 컨텍스트에 노이즈가 증가하고 생성 정확도 하락 |
| 컨텍스트 구성 | 중요 청크를 컨텍스트 중간에 배치 | Lost in the Middle 현상으로 해당 정보가 생성에 반영되지 않음 |
| 생성 | 컨텍스트 대신 사전 학습 지식 우선 | 제공된 문서와 다른 내용이 출력됨 |
| 출력 후처리 | 검증 단계 부재 | 오류가 걸러지지 않고 사용자에게 전달됨 |

이 표를 보면 실패가 어떤 단계에서도 발생할 수 있다는 점이 분명해진다. 그리고 어느 한 단계의 결함이 다음 단계로 연쇄될 수 있다. 청킹이 잘못되면 임베딩이 정확해도 검색 결과가 나빠지고, 검색이 잘못되면 컨텍스트 구성이 정교해도 생성 품질이 저하된다.

::: box 실패의 연쇄 구조
RAG 파이프라인의 실패는 단계 내부에 머물지 않고 하류로 전파되는 경향이 있다. 청킹 단계에서 문장이 의미 단위 중간에 잘리면, 그 청크는 임베딩 단계에서 불완전한 의미 벡터를 갖게 된다. 불완전한 벡터는 검색 단계에서 낮은 유사도 점수를 받거나 부정확한 문서와 함께 묶인다. 부정확한 청크가 컨텍스트에 포함되면, 생성 단계에서 모델은 그 노이즈를 처리해야 한다. 노이즈가 많아질수록 생성된 답변의 Faithfulness(생성 결과가 제공된 컨텍스트에 얼마나 충실한가를 나타내는 지표)가 낮아진다.[^web_06]

이 연쇄 구조는 RAGEC 프레임워크에서도 확인된다. 청킹과 검색 오류가 환각보다 빈번하게 발생한다는 실증 결과는, 생성 단계가 아니라 파이프라인 앞단의 설계 결함이 실제 운영 실패의 주요 원인임을 시사한다.[^web_12]
:::

### 5.5.3 검증은 운영 중에 드러난다

RAG 실패를 아키텍처 문제로 다루어야 하는 또 다른 이유는, 많은 실패가 개발 단계에서는 드러나지 않는다는 점이다. 개발 환경에서 테스트할 때는 제한된 수의 쿼리와 잘 정비된 문서 세트로 실험하게 된다. 이 조건에서는 청킹 전략의 결함이나 임베딩 불일치가 표면에 나타나지 않을 수 있다.

실제 운영 환경에서는 상황이 달라진다. 사용자 쿼리는 개발 팀이 예상하지 못한 표현 방식과 맥락을 가진다. 문서 저장소는 시간이 지나면서 구조가 불균등해지고, 일부 도메인은 다른 도메인에 비해 청크 품질이 낮아진다. 검색 부하가 높아지면 임계값 설정이나 재순위화 로직에 숨어 있던 설계 결함이 특정 조건에서만 발동되는 방식으로 나타난다.

Barnett et al. (2024)이 "검증은 운영 중에만 가능하다"고 결론 내린 것은 이 맥락에서다.[^web_03] 이 말이 개발 단계의 테스트가 불필요하다는 뜻은 아니다. 오히려 RAG 파이프라인을 설계할 때, 운영 중에 각 단계의 품질을 독립적으로 모니터링하고 진단할 수 있는 구조를 처음부터 갖추어야 한다는 의미다.

::: tip 단계별 독립 모니터링 설계
RAG 파이프라인을 구성할 때 각 단계의 출력을 로깅하고 평가할 수 있도록 설계하는 것이 권장된다. 예를 들어, 검색 단계에서는 회수된 청크의 유사도 점수와 개수를 기록하고, 컨텍스트 구성 단계에서는 최종적으로 포함된 청크의 출처와 순서를 남기며, 생성 단계에서는 Faithfulness 점수를 함께 기록한다. RAGAS와 같은 평가 프레임워크는 이 각 단계를 독립적으로 평가하는 메트릭을 제공한다.[^web_06] 각 단계를 독립적으로 측정할 수 있어야, 실패가 발생했을 때 어느 단계를 수정해야 하는지 빠르게 판단할 수 있다.
:::

---

5장에서 다룬 내용을 정리하면 다음과 같다. LLM은 파라메트릭 메모리의 구조적 한계로 인해 지식 고착, 환각, 폐쇄 도메인 지식 부재라는 세 가지 문제를 내재한다. RAG는 이 한계를 비파라메트릭 메모리를 파이프라인에 통합하는 방식으로 보완한다. 그 과정에서 RAG는 단순히 "검색 기능을 붙이는 것"이 아니라, 사전 처리·임베딩·검색·컨텍스트 구성·생성·출력 후처리에 이르는 각 단계에서 정보 흐름을 통제하는 아키텍처적 선택이다. 실패는 이 중 어느 단계에서도 발생할 수 있으며, 연쇄적으로 전파될 수 있고, 운영 중에만 드러나는 경우가 많다.

이 인식 위에서 중요한 질문이 하나 남는다. RAG를 선택해야 하는 상황과 그렇지 않은 상황은 어떻게 구분하는가? RAG를 도입하는 것은 동시에 새로운 설계 과제를 떠안는 것이기도 하다.[^web_09] 6장에서는 이 판단의 기준, 즉 RAG를 선택한다는 것이 어떤 트레이드오프를 수반하는지를 구체적으로 살펴본다.
