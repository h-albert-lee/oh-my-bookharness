# Chapter Brief: ch05
## RAG를 기능이 아니라 아키텍처로 이해하기

---

## 1. 이 장의 목표

ch04가 LLM 시스템의 기본 파이프라인과 구성 요소를 **무엇이 있는가**의 관점에서 정리했다면, ch05는 그 구성 요소 중 RAG가 **왜 그 자리에 놓이는가**를 구조적으로 설명한다.

RAG(검색 증강 생성, Retrieval-Augmented Generation)는 현재 LLM 기반 시스템에서 가장 널리 쓰이는 패턴이지만, 대부분의 실무 팀이 이를 "외부 지식을 끌어오는 기능"으로만 이해한다. 이 장의 목표는 그 인식을 교정하는 것이다. RAG는 기능이 아니라 **정보 흐름을 통제하는 아키텍처적 선택**이다.

구체적으로, 이 장은 세 가지를 달성한다.

- RAG가 왜 등장했는지를 LLM의 구조적 한계에서 도출한다.
- RAG를 구성하는 각 단계가 정보 흐름 통제의 어느 지점을 담당하는지 설명한다.
- RAG를 잘못 설계했을 때 발생하는 실패를 아키텍처 문제로 재정의한다.

이 장은 RAG 구현 방법을 가르치지 않는다. 왜 이 구조가 필요하며, 어떤 설계 판단이 그 구조 안에 들어 있는지를 설명한다.

## 2. 독자가 얻을 것

| 얻는 것 | 설명 |
|---|---|
| **관점 전환** | "RAG는 검색 기능이다" → "RAG는 정보 흐름 통제 아키텍처다" |
| **실패 진단 언어** | RAG 관련 장애를 "검색이 이상하다"가 아니라 설계 결함으로 기술하는 언어 |
| **구조 독해 능력** | RAG 파이프라인의 각 단계가 무엇을 통제하고 무엇을 위임하는지 읽는 능력 |
| **설계 판단 기준** | RAG를 쓸지 말지, 어떤 방식으로 구성할지를 결정하는 판단 기준의 시작점 |

## 3. 장의 핵심 질문

> RAG는 왜 '검색 기능을 붙이는 것'이 아니라 '정보 흐름을 설계하는 것'인가?

이 질문에 답하기 위해 이 장이 순서대로 다루는 하위 질문들:

1. LLM은 왜 지식을 신뢰하기 어려운가? (구조적 한계)
2. 그 한계를 보완하기 위해 RAG는 무엇을 하는가? (기능이 아닌 역할)
3. RAG 파이프라인의 각 단계는 정보 흐름의 어느 지점을 통제하는가?
4. RAG가 아키텍처라면, 잘못된 RAG는 어떤 아키텍처 결함인가?
5. RAG를 선택하는 것은 어떤 설계 판단인가?

## 4. 반드시 들어갈 개념

### 4-1. LLM의 지식 구조적 한계 (RAG 등장의 근거)

RAG를 설명하기 전에, RAG가 왜 필요해졌는지를 LLM 자체의 구조에서 끌어내야 한다. 세 가지를 짚는다.

- **지식 고착(Knowledge Cutoff)**: 학습 시점 이후의 정보를 모른다.
- **환각(Hallucination)**: 모르는 것을 모른다고 말하지 않고 그럴듯하게 생성한다.
- **폐쇄 도메인 지식 부재**: 기업 내부 문서, 규제 문서, 도메인 특화 지식은 사전 학습에 포함되지 않는다.

이 세 가지가 합쳐지면, LLM 단독으로는 **신뢰 가능한 정보를 안정적으로 생성하기 어렵다**는 결론이 나온다. RAG는 이 결론에서 나온 아키텍처적 대응이다.

### 4-2. RAG의 아키텍처적 정의

RAG를 다음과 같이 재정의한다.

> RAG는 LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조다.

"검색해서 붙여 넣는다"가 아니라, "어떤 정보가 모델에 도달할 수 있는가"를 결정하는 통제 레이어다. 이 재정의가 이 장의 핵심 명제다.

### 4-3. RAG 파이프라인 단계별 역할 분리

RAG 파이프라인의 각 단계를 정보 흐름 통제의 관점에서 설명한다.

| 단계 | 담당 역할 | 통제 대상 |
|---|---|---|
| 문서 전처리·색인 | 어떤 정보를 검색 가능한 상태로 만들 것인가 | 정보 진입 통제 |
| 검색(Retrieval) | 어떤 정보를 꺼낼 것인가 | 정보 선택 통제 |
| 컨텍스트 구성 | 꺼낸 정보를 어떻게 모델에 전달할 것인가 | 정보 구성 통제 |
| 생성(Generation) | 주어진 정보를 바탕으로 출력을 만든다 | 정보 활용 |
| 출력 후처리 | 생성된 결과를 검증·정제한다 | 출력 통제 |

각 단계가 독립적인 설계 결정 지점임을 강조한다.

### 4-4. RAG 실패의 아키텍처적 분류

"RAG가 이상한 답을 냈다"는 진단을 구체화한다. 실패 유형을 단계별로 분류하면:

- 검색이 실패한 경우: 관련 정보가 색인에 없거나, 검색 쿼리가 잘못 구성됐거나, 유사도 계산이 실제 의미를 포착하지 못한 경우
- 컨텍스트 구성이 실패한 경우: 검색된 정보는 맞지만 모델에 전달되는 방식이 잘못된 경우 (순서, 길이, 형식)
- 생성이 컨텍스트를 무시한 경우: 모델이 주어진 정보 대신 사전 학습 지식에 의존한 경우

이 분류가 있어야 디버깅이 가능하다. RAG를 기능으로 보면 "검색이 이상하다"로 뭉뚱그리지만, 아키텍처로 보면 어느 단계의 어떤 결함인지를 특정할 수 있다.

### 4-5. RAG는 설계 선택이다

RAG 도입 여부 자체가 아키텍처 판단임을 명시한다. RAG가 적합한 상황과 그렇지 않은 상황을 대비한다.

- RAG가 적합한 경우: 폐쇄 도메인 지식이 필요하고, 지식이 자주 갱신되며, 정보 출처의 추적 가능성이 중요한 경우
- RAG가 과잉 설계인 경우: 범용 지식으로 충분하고, 정보 최신성이 중요하지 않으며, 시스템 복잡도 증가가 더 큰 리스크인 경우

## 5. 피해야 할 오해

| 오해 | 교정 |
|---|---|
| "RAG = 벡터 DB + LLM 연결" | 벡터 DB는 RAG의 한 구현 선택일 뿐이다. RAG는 정보 흐름을 통제하는 구조 전체를 가리킨다. |
| "RAG를 쓰면 환각이 없어진다" | RAG는 환각을 줄이는 구조적 수단이지, 환각을 제거하는 장치가 아니다. 검색이 잘못되면 환각은 오히려 더 확신에 찬 방식으로 발생한다. |
| "RAG는 항상 써야 한다" | RAG는 복잡도를 수반한다. 필요하지 않은 상황에서 도입하면 새로운 실패 지점이 추가될 뿐이다. |
| "RAG 품질 = 검색 정확도" | 검색 정확도는 RAG 품질의 일부다. 컨텍스트 구성, 생성 단계의 설계가 동등하게 중요하다. |
| "임베딩 모델을 좋은 것으로 바꾸면 해결된다" | 임베딩 품질은 검색 단계의 한 요소다. 문서 전처리, 청킹 전략, 컨텍스트 구성이 함께 설계되어야 한다. |

## 6. 앞 장과의 연결

### ch04에서 ch05로

ch04는 LLM 시스템의 기본 파이프라인을 구성하는 요소들을 정리했다. 프롬프트, 모델 호출, 출력 처리, 외부 도구 연결 등 파이프라인에 **무엇이 있는지**를 독자가 이해한 상태다.

ch05는 그 파이프라인에서 **RAG라는 구성 선택이 왜 등장하는지**를 설명한다. ch04가 지도를 그렸다면, ch05는 그 지도 위에서 특정 경로를 선택하는 이유를 설명하는 장이다.

ch04에서 독자가 가져오는 것:
- LLM 파이프라인의 입력-처리-출력 구조
- 외부 데이터 연결이 파이프라인에서 어떻게 위치하는지에 대한 기초 인식

ch05가 그것을 어떻게 확장하는가:
- "외부 데이터 연결"이 왜 단순한 기능 추가가 아니라 정보 흐름 설계의 문제인지를 설명한다.

### ch01-ch03과의 맥락 연결

ch01에서 도입한 **하네스(Harness)** 개념 — 실행과 평가를 감싸는 외부 구조 — 이 RAG 아키텍처와 연결된다. RAG의 각 단계(검색, 구성, 생성)는 하네스가 통제해야 할 지점들이다. 이 연결을 명시적으로 짚는 문단이 필요하다.

ch03의 "문제는 모델이 아니라 시스템이다"라는 논지를 RAG 맥락에 적용한다. RAG 실패를 "검색 모델이 나쁘다"로 귀인하는 것이 ch03이 비판한 오류와 같은 구조임을 지적한다.

## 7. 뒤 장으로의 연결

### ch05 → ch06

ch05가 정보 흐름 통제를 RAG 아키텍처로 설명했다면, ch06은 **역할과 책임의 분리**를 에이전트 시스템으로 설명한다.

연결 고리: RAG는 "어떤 정보를 모델에 전달할 것인가"를 설계하는 구조다. 에이전트는 "어떤 행동을 어떤 역할이 수행할 것인가"를 설계하는 구조다. 둘 다 **단일 모델 호출 이상의 구조적 선택**이라는 점에서 같은 계층의 문제다.

ch05의 마지막 지점에서 다음 질문을 열어야 한다: "정보 흐름을 통제하는 것만으로는 부족한 경우가 있다. 여러 작업을 여러 역할이 나눠서 처리해야 한다면, 그것은 어떤 구조인가?"

### 이후 파트와의 연결

- **PART 3 (ch07-09)**: ch05에서 정의한 RAG 실패 유형들이 ch08의 "설계 가능한 위험" 목록에 포함된다. RAG 파이프라인의 각 단계가 가드레일(ch09)이 작동해야 하는 지점이기도 하다.
- **PART 4 (ch10-12)**: RAG 파이프라인의 각 단계는 평가 대상이기도 하다. 검색 품질 평가, 컨텍스트 구성 평가, 생성 품질 평가는 ch11의 평가 설계에서 구체화된다.
- **PART 5 (ch13, 15)**: RAG는 ch13의 평가 가능한 시스템 패턴과 ch15의 도메인 특화 아키텍처에서 실제 설계 선택으로 재등장한다.

## 8. 원하는 톤 메모

- **교정의 어조**: "당신이 알고 있는 RAG와 다를 수 있다"는 전제를 독자에게 조용히 깔아야 한다. 기존 이해를 부정하지 않되, 더 정밀한 언어로 교체한다.
- **구조 중심 서술**: 구현 코드나 라이브러리 언급을 피한다. 설명의 단위는 항상 "단계", "흐름", "통제 지점"이다.
- **실패 사례 활용**: 추상 논의보다 "RAG를 붙였는데 왜 이상한 답이 나오는가"라는 실무 경험에서 출발한다. 독자가 이미 겪었을 상황을 진단 언어로 재기술하는 방식이 효과적이다.
- **과잉 약속 금지**: "RAG를 아키텍처로 보면 모든 문제가 해결된다"는 식의 서술 금지. 관점 전환이 가져다주는 것은 더 나은 진단 능력이지, 문제의 소멸이 아니다.
- **독립 가독성 확보**: ch04를 읽지 않은 독자도 이 장의 논지를 따라올 수 있도록 최소한의 파이프라인 맥락을 도입부에서 간략히 복기한다.

## 9. 참고할 이전 장 요약

### ch01 요약 (참고 포인트 발췌)

- **핵심 개념**: 하네스는 "입력 통제, 출력 평가, 실패 처리"를 수행하는 외부 구조다.
- **연결 포인트**: ch05에서 RAG 파이프라인의 각 단계(색인, 검색, 컨텍스트 구성, 생성, 후처리)는 하네스가 감싸야 할 지점들로 해석될 수 있다. "RAG를 아키텍처로 본다"는 것은 곧 각 단계에 하네스가 필요하다는 인식과 연결된다.
- **주의**: ch01의 하네스 개념을 ch05에서 전면에 끌어내지는 않는다. 이 장의 논지는 RAG의 구조적 재해석이 주이며, 하네스와의 연결은 독자가 스스로 포착하거나, 한 문단 내에서 간략히 지적하는 수준으로 처리한다.
