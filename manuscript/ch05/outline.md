# 5장. RAG를 기능이 아니라 아키텍처로 이해하기

---

## 장 전체 Argument Flow

LLM은 학습 시점에 고착된 지식, 환각, 폐쇄 도메인 지식 부재라는 구조적 한계를 가진다. RAG는 이 한계에 대한 아키텍처적 대응으로 등장했다. 그러나 대부분의 실무 팀은 RAG를 "외부 검색을 붙이는 기능"으로 다루기 때문에, 실패가 발생했을 때 원인을 특정하지 못하고 "검색이 이상하다"는 막연한 진단에 머문다. 이 장은 그 인식을 교정한다. RAG를 기능이 아니라 정보 흐름을 통제하는 아키텍처로 재정의하면, 파이프라인의 각 단계—색인, 검색, 컨텍스트 구성, 생성, 후처리—가 독립적인 설계 결정 지점으로 보이고, 실패도 어느 단계의 어떤 결함인지 특정할 수 있게 된다. 관점 전환이 가져다주는 것은 문제의 소멸이 아니라 더 정밀한 진단 언어다. 장의 끝에서 독자는 RAG 도입 여부와 설계 방식을 판단하는 기준의 시작점을 갖게 된다.

### Section 5.1 LLM은 왜 지식을 신뢰하기 어려운가

**목적**
RAG가 왜 필요한지를 LLM 자체의 구조에서 도출한다. "RAG는 검색을 붙이는 기능"이라는 인식이 생기는 이유 중 하나는 RAG의 필요성을 제대로 이해하지 못한 채 도입하기 때문이다. 이 절은 그 근거를 세 가지 구조적 한계로 정리한다.

**사용 source**
- `web_01` (Lewis et al. 2020): 파라메트릭 메모리 vs. 비파라메트릭 메모리 개념 도입
- `web_08`: 환각의 분류 체계, RAG 이후에도 환각이 잔존함을 언급
- `web_11`: 검색 실패 → 생성 환각 연쇄 구조 예비 언급

**포함 예시**
- 법률 AI가 실제로 존재하지 않는 판례를 인용한 사례 (환각의 실무 결과)
- 기업 내규 질의 시스템에서 최신 개정 내용이 반영되지 않은 응답이 나온 사례 (지식 고착의 결과)

**핵심 요점**
1. 지식 고착(Knowledge Cutoff): LLM의 지식은 학습 시점에 동결된다. 이후의 세계를 모른다.
2. 환각(Hallucination): 모르는 것을 "모른다"고 말하지 않고 그럴듯하게 채운다. 오류가 자신감 있게 출력된다.
3. 폐쇄 도메인 지식 부재: 기업 내부 문서, 규제 문서, 도메인 특화 지식은 사전 학습 데이터에 포함되지 않는다.
4. 세 가지가 결합되면, LLM 단독으로는 신뢰 가능한 정보를 안정적으로 생성하기 어렵다. 이것이 RAG 등장의 구조적 근거다.

**앞뒤 연결**
- 앞: ch04에서 정리한 LLM 파이프라인의 입력-처리-출력 구조를 전제로 받는다. 이 절 도입부에서 "ch04에서 우리는 LLM 시스템이 어떤 요소로 구성되는지 봤다. 이제 그 시스템이 왜 외부 정보를 필요로 하는지를 살펴본다"는 한 문단 복기를 넣는다.
- 뒤: 이 한계들이 곧 Section 5.2에서 다룰 "기능으로 보는 RAG"가 왜 불충분한지의 배경이 된다.

**구조물 배치 계획**
- <!-- note: 파라메트릭 메모리와 비파라메트릭 메모리 --> — '파라메트릭 메모리'(모델 가중치에 저장된 지식)와 '비파라메트릭 메모리'(외부 저장소에서 검색되는 지식)의 개념 정의. Lewis et al. 2020의 용어를 기반으로 한다.
- <!-- note: 환각(Hallucination) --> — 환각의 정의와 "자신감 있는 오류"라는 특성 한 문장 정의.
- <!-- figure: 그림 5.1. LLM 단독 시스템의 지식 구조적 한계 세 가지 --> — 지식 고착, 환각, 폐쇄 도메인 지식 부재를 하나의 다이어그램으로 시각화.

#### 5.1.1 고착된 지식과 사라지는 정확도

- 학습 데이터 컷오프의 의미: LLM이 알고 있는 세계는 학습 완료 시점에 멈춘다.
- 컷오프 이후의 정보를 질의하면 오답이 나오거나, 오래된 정보를 최신 정보처럼 제시한다.
- 운영 기간이 길수록 컷오프 문제는 누적된다.

#### 5.1.2 모른다고 말하지 않는 모델

- 환각의 구조: 확률 기반 생성 모델은 "모른다"는 판단보다 "그럴듯한 토큰"을 선택하는 쪽으로 작동한다.
- 환각이 위험한 이유는 잘못된 정보를 틀린 것처럼 보여주지 않기 때문이다.
- 도메인 특화 지식이 부재한 경우, 환각은 더 빈번하고 더 자신감 있게 나타난다.
- 폐쇄 도메인 지식 부재: 기업 내규, 제품 사양서, 규제 문서는 사전 학습에 없다. 이 영역에서 LLM 단독 시스템은 구조적으로 취약하다.

### Section 5.2 RAG를 기능으로 보면 무슨 일이 일어나는가

**목적**
"RAG를 붙였는데 왜 이상한 답이 나오는가"라는 실무 경험에서 출발한다. RAG를 기능으로 이해할 때 발생하는 진단 실패, 설계 결함, 디버깅 불가 상황을 구체적으로 보여준다. 이 절은 독자가 이미 겪었을 상황을 인식하게 만드는 공감 지점이다.

**사용 source**
- `web_02` (Gao et al. 2024 서베이): Naive RAG의 구조적 한계 명시
- `web_03` (Barnett et al. 2024): 실무 실패 사례 기반 논문, "검색이 이상하다"로 뭉뚱그리는 진단의 한계
- `web_12`: RAGEC 프레임워크, 청킹·검색 오류가 환각보다 빈번하다는 실증

**포함 예시**
- "RAG를 붙였더니 오히려 더 이상한 답이 나왔다" — 검색된 맥락이 질문과 맞지 않을 때 모델이 그 맥락을 활용해 더 확신에 찬 잘못된 답을 생성하는 패턴
- "검색은 됐는데 답변이 엉뚱하다" — 검색은 성공했지만 컨텍스트 구성이 실패한 경우를 기능 관점에서는 진단할 수 없다는 상황
- RAG 도입 후 "어디서 고쳐야 하는지 모르겠다"는 실무 팀의 전형적 막힘 상황

**핵심 요점**
1. RAG를 기능으로 보면, 실패는 "검색이 이상하다" 또는 "모델이 나쁘다" 두 가지로만 귀인된다.
2. 실제 실패는 파이프라인의 여러 단계에서 독립적으로 발생한다. 기능 관점에서는 이것을 구분할 언어가 없다.
3. 기능 관점의 RAG 구현(Naive RAG)은 '검색해서 붙여 넣기'로 끝난다. 각 단계가 설계 결정 지점이라는 인식이 없다.
4. 이 진단 실패가 ch03의 논지("문제는 모델이 아니라 시스템이다")와 같은 구조임을 지적한다.

**앞뒤 연결**
- 앞: Section 5.1에서 확립한 LLM의 구조적 한계 → RAG 도입 → "그런데 RAG를 붙였어도 문제가 생긴다"는 흐름으로 자연스럽게 진입.
- 뒤: Section 5.3의 "아키텍처로 보면 무엇이 보이는가"를 위한 대비 구도 설정. "기능으로 보면 이것이 안 보인다"를 먼저 보여주어야 "아키텍처로 보면 이것이 보인다"가 선명해진다.

**구조물 배치 계획**
- <!-- box: Naive RAG의 전형적 구조 — 검색 후 붙여 넣기 --> — Naive RAG 패턴을 한 박스에 정리. 단계별 설계 판단이 없는 구조임을 명시. `web_02` 기반.
- <!-- note: "RAG 품질 = 검색 정확도"라는 오해 --> — 이 절에서 교정하는 핵심 오해를 한 문장 노트로 배치.
- <!-- figure: 그림 5.2. 기능으로 본 RAG — 검색과 생성 사이의 블랙박스 --> — RAG를 블랙박스로 보는 관점을 시각화. 실패 발생 시 어디를 봐야 할지 모르는 상황을 도식화.

#### 5.2.1 "검색이 이상하다"는 진단의 한계

- 실패를 "검색 품질" 하나로 귀인하면, 검색 이외 단계의 결함을 못 본다.
- 검색이 맞아도 컨텍스트 구성이 틀리면 답이 이상하게 나온다.
- 검색과 생성이 모두 맞아도 후처리에서 걸러지지 않으면 잘못된 출력이 나간다.
- 기능 관점의 진단 언어는 이 세 상황을 구별하지 못한다.

#### 5.2.2 Naive RAG가 만들어내는 새로운 실패 지점

- Naive RAG의 정의: 질문을 그대로 쿼리로 전환 → 유사 문서 검색 → 검색 결과를 그대로 컨텍스트에 붙임 → 생성. 각 단계에 설계 판단이 없다.
- 이 구조는 "검색 이후를 설계하지 않는다"는 암묵적 선택을 포함한다.
- 검색된 컨텍스트가 잘못되면, 모델은 그 잘못된 컨텍스트를 바탕으로 더 확신에 찬 오답을 생성한다(환각보다 나쁜 패턴).
- 청킹 전략, 문서 품질, 컨텍스트 순서 등이 결과에 영향을 미치지만, 기능 관점에서는 이 변수들이 보이지 않는다.

### Section 5.3 RAG를 아키텍처로 재정의하면 무엇이 보이는가

**목적**
이 장의 핵심 명제를 선언하고 논증한다. "RAG는 LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조다"라는 재정의를 중심으로, 이 관점 전환이 가져다주는 것이 무엇인지를 보여준다.

**사용 source**
- `web_04` (Gao et al. 2024 Modular RAG): RAG를 재구성 가능한 프레임워크로 정의, 각 모듈이 독립적 설계 결정 지점임을 이론적으로 뒷받침
- `web_07` (IBM 2023): "아키텍처 패턴"으로 RAG를 명시한 공식 산업 문서
- `web_01` (Lewis et al. 2020): 파라메트릭 vs. 비파라메트릭 메모리 — 아키텍처 재정의의 이론적 기반

**포함 예시**
- "검색 기능을 추가한다" vs. "어떤 정보가 모델에 도달할 수 있는가를 결정한다"는 두 관점의 설계 의사결정 차이를 구체적인 질문 목록으로 대비
- 동일한 RAG 시스템을 두 관점에서 기술한 사례: 기능 관점 기술 vs. 아키텍처 관점 기술 비교

**핵심 요점**
1. RAG의 아키텍처적 정의: "LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조."
2. 이 정의가 "검색해서 붙여 넣기"와 다른 점: 통제의 주체가 시스템이며, 통제 지점이 여럿이다.
3. 아키텍처 관점에서 RAG는 정보 진입 통제, 정보 선택 통제, 정보 구성 통제, 출력 통제의 레이어 집합이다.
4. Modular RAG로의 진화 계보: Naive → Advanced → Modular. 이 진화는 "기능에서 아키텍처로"의 이동이기도 하다.
5. ch01의 하네스 개념 연결: RAG의 각 통제 지점은 하네스가 감싸야 할 지점들이다. 이 연결을 한 문단에서 간략히 지적한다.

**앞뒤 연결**
- 앞: Section 5.2에서 "기능으로 보면 이것이 안 보인다"를 확인했다. Section 5.3은 "아키텍처로 보면 이것이 보인다"를 제시한다.
- 뒤: Section 5.4에서 아키텍처의 각 단계를 구체적으로 펼친다. 이 절이 전체 구조를 선언하고, 5.4가 그것을 분해한다.

**구조물 배치 계획**
- <!-- note: RAG의 아키텍처적 정의 --> — "RAG는 LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조다"를 박스 형태가 아닌 인라인 노트로 강조.
- <!-- figure: 그림 5.3. RAG 아키텍처의 정보 흐름 통제 레이어 구조 --> — 정보가 외부 저장소에서 출발해 색인 → 검색 → 컨텍스트 구성 → 생성 → 후처리를 거쳐 출력까지 흐르는 구조. 각 경계에 "통제 지점" 표시.
- <!-- box: Naive → Advanced → Modular RAG 진화 계보 --> — RAG의 구조적 진화를 3단계로 정리. 각 단계에서 "기능"이 "아키텍처 선택"으로 이동하는 양상을 간략히 서술. `web_02`, `web_04` 기반.
- <!-- table: 표 5.1. 기능 관점 vs. 아키텍처 관점에서의 RAG 이해 비교 --> — 두 관점에서 RAG를 바라보는 언어, 실패 진단 방식, 설계 질문의 차이를 4~5행 표로 대비.

#### 5.3.1 정보 흐름을 통제한다는 것의 의미

- "검색을 추가한다"는 행위는 정보 흐름의 한 지점만을 바꾼다.
- "정보 흐름을 통제한다"는 것은 어떤 정보가 어느 단계에서 시스템에 진입하고, 어떻게 선택되며, 어떻게 구성되어 모델에 전달되는지를 전 구간에 걸쳐 설계하는 것이다.
- 통제 지점이 여럿이라는 것은 설계 결정이 여럿이라는 뜻이다. 각 결정은 독립적으로 실패할 수 있다.

#### 5.3.2 하네스와의 연결: 통제 구조는 외부에서 감싼다

- ch01에서 도입한 하네스 개념 간략 복기: 실행과 평가를 감싸는 외부 구조.
- RAG의 각 통제 지점(색인, 검색, 컨텍스트 구성, 생성, 후처리)은 하네스가 감시하고 개입할 수 있는 경계이기도 하다.
- "RAG를 아키텍처로 본다"는 것은 곧 "각 통제 지점에 하네스가 필요하다"는 인식과 연결된다.
- 이 연결은 한 문단으로 처리하며, 전면에 끌어내지 않는다.

### Section 5.4 RAG 파이프라인 단계별 역할과 통제 지점

**목적**
RAG 파이프라인의 다섯 단계 각각이 정보 흐름 통제의 어느 지점을 담당하는지, 그리고 각 단계에서 어떤 설계 결정이 이루어지는지를 구체적으로 펼친다. 이 절이 이 장에서 가장 분량이 많은 핵심 분석 구간이다.

**사용 source**
- `web_02` (Gao et al. 2024 서베이): 각 단계의 구조적 한계와 역할 정의
- `web_04` (Modular RAG): 각 모듈이 독립적 설계 결정 지점임을 이론적으로 뒷받침
- `web_06` (Es et al. 2023 RAGAS): 각 단계를 독립적으로 평가할 수 있다는 근거 — 단계 독립성의 증거로 활용
- `web_05` (Asai et al. 2023 Self-RAG): 검색 자체를 조건부로 만드는 설계 선택의 사례

**포함 예시**
- 청킹 전략의 차이가 검색 품질에 미치는 영향 (문장 단위 vs. 단락 단위 vs. 의미 단위 청킹)
- 동일한 검색 결과를 다른 순서로 컨텍스트에 배치했을 때 생성 품질이 달라지는 사례
- Self-RAG의 조건부 검색: "모든 질문에 검색이 필요한가"라는 설계 질문의 사례로

**핵심 요점**
1. 문서 전처리·색인 단계: "어떤 정보를 검색 가능한 상태로 만들 것인가" — 정보 진입 통제. 청킹, 메타데이터, 색인 전략이 여기서 결정된다.
2. 검색 단계: "어떤 정보를 꺼낼 것인가" — 정보 선택 통제. 쿼리 구성 방식, 유사도 계산 방식, 검색 결과 수가 설계 변수다.
3. 컨텍스트 구성 단계: "꺼낸 정보를 어떻게 모델에 전달할 것인가" — 정보 구성 통제. 순서, 길이, 형식, 중복 제거가 여기서 결정된다.
4. 생성 단계: "주어진 정보를 바탕으로 출력을 만든다" — 정보 활용. 모델이 컨텍스트를 따를지 사전 학습 지식을 따를지는 설계로 통제해야 한다.
5. 출력 후처리 단계: "생성된 결과를 검증·정제한다" — 출력 통제. 출처 표시, 신뢰도 검증, 포맷 정제가 여기서 이루어진다.

**앞뒤 연결**

- 뒤: Section 5.5에서 각 단계의 실패를 진단 언어로 분류할 때, 이 절에서 정의한 단계별 역할이 직접 기준이 된다.

#### 5.4.1 정보 진입 통제: 색인이 결정하는 것들

- 문서 전처리 단계가 RAG의 "상류(upstream)"임을 강조. 여기서 잘못되면 이후 단계가 아무리 잘 설계되어도 회복이 어렵다.
- 청킹 전략: 문서를 어떤 단위로 쪼갤 것인가. 너무 작으면 맥락이 사라지고, 너무 크면 관련 없는 정보가 섞인다.
- 메타데이터 설계: 문서 출처, 날짜, 버전 정보가 검색 단계에서 활용되기 위해서는 색인 시점에 구조화되어 있어야 한다.
- 색인 전략: 어떤 방식으로 색인을 만들 것인가(벡터, 키워드, 하이브리드)는 검색 단계의 선택지를 규정한다.

#### 5.4.2 정보 선택 통제: 검색이 결정하는 것들

- 쿼리 구성 방식: 사용자의 원래 질문을 그대로 쿼리로 쓸 것인가, 재작성할 것인가.
- 유사도 계산: 벡터 유사도는 의미적 유사성을 포착하지만, 정확한 용어 매칭에 약하다. 어느 쪽이 이 시스템에 더 중요한가.
- 검색 결과 수(Top-K): 몇 개를 가져올 것인가. 많을수록 재현율은 높아지지만 노이즈도 증가한다.
- 조건부 검색의 가능성: 모든 질문에 검색이 필요한가, 아니면 검색 여부 자체를 판단해야 하는가.

#### 5.4.3 정보 구성 통제와 출력 통제: 전달과 검증

- **컨텍스트 구성**: 검색 결과를 어떤 순서와 형식으로 모델에 전달할 것인가. 순서, 중복 제거, 길이 제한, 출처 표시 방식이 모두 설계 변수다.
- **생성 단계의 설계 책임**: 모델이 컨텍스트를 따르도록 유도하는 것은 프롬프트 설계와 모델 선택의 문제다. 생성 단계는 "정보 활용" 지점이지만, 모델이 사전 학습 지식을 우선할 경우 통제가 필요하다.
- **출력 후처리**: 생성된 결과를 시스템이 검증·정제하는 단계. 출처 참조 확인, 포맷 검증, 신뢰도 임계값 적용이 포함된다.
