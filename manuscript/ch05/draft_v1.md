# 5장.  RAG를 기능이 아니라 아키텍처로 이해하기

## 5.1 LLM은 왜 지식을 신뢰하기 어려운가

4장에서 우리는 LLM 기반 시스템이 어떤 요소로 구성되는지를 살펴봤다. 입력이 들어오고, 모델이 처리하고, 출력이 나오는 파이프라인 구조였다. 그런데 그 파이프라인의 중심에 놓인 LLM 자체는 정보를 어떻게 보유하고 있을까? 그리고 그 방식이 실무에서 어떤 문제를 일으킬까?

이 절에서는 RAG가 등장하게 된 이유를 LLM 자체의 구조적 특성에서 찾는다. 먼저 LLM이 지식을 저장하는 방식을 정리하고, 거기서 비롯되는 세 가지 한계를 하나씩 살펴본다. 이 한계들을 이해하지 않으면, RAG를 왜 도입하는지, 그리고 왜 단순히 "검색 기능을 붙이는 것"으로는 부족한지를 설명하기 어렵다.

::: note 파라메트릭 메모리와 비파라메트릭 메모리
Lewis et al. (2020)은 LLM이 지식을 보유하는 방식을 두 가지로 구분한다.[^web_01]

- **파라메트릭 메모리(Parametric Memory)**: 모델의 가중치(파라미터) 안에 학습을 통해 인코딩된 지식이다. 모델이 실행될 때 외부 저장소를 참조하지 않고 가중치에서 직접 꺼내 쓰는 지식이다.
- **비파라메트릭 메모리(Non-Parametric Memory)**: 모델 외부의 저장소(데이터베이스, 문서 저장소 등)에 보관되며, 필요할 때 검색을 통해 가져오는 지식이다. 가중치를 변경하지 않아도 내용을 갱신할 수 있다.

이 구분이 ch05 전체 논의의 뼈대가 된다. RAG는 비파라메트릭 메모리를 파이프라인에 통합하는 방식이다.
:::

GPT 계열, Claude, Gemini와 같은 현재의 대형 언어 모델은 파라메트릭 메모리에 의존한다. 수천억 개의 파라미터에 방대한 텍스트 데이터로부터 추출한 패턴과 사실이 녹아 있다. 사용자가 질문을 던지면 모델은 그 파라미터를 활성화해 답변을 생성한다. 별도의 데이터베이스를 조회하거나 인터넷을 탐색하는 것이 아니다. 모델 안에 고정된 지식을 꺼내 쓰는 것이다. 이 구조는 추론 속도와 범용성이라는 강점을 가져다주지만, 동시에 세 가지 구조적 한계를 내재한다.

**그림 5-1. LLM 단독 시스템의 지식 구조적 한계 세 가지**

```
┌─────────────────────────────────────────────────────────┐
│                   LLM (파라메트릭 메모리)                   │
│                                                         │
│   ┌─────────────┐  ┌──────────────┐  ┌──────────────┐  │
│   │  지식 고착   │  │     환각      │  │ 폐쇄 도메인  │  │
│   │             │  │              │  │  지식 부재   │  │
│   │ 학습 완료   │  │ 모르는 것을  │  │ 내부 문서,  │  │
│   │ 시점에서    │  │ 그럴듯하게   │  │ 규제 문서는  │  │
│   │ 지식 동결   │  │ 채워서 출력  │  │ 학습 불포함  │  │
│   └─────────────┘  └──────────────┘  └──────────────┘  │
│                              │                          │
└──────────────────────────────┼──────────────────────────┘
                               ▼
              LLM 단독으로는 신뢰 가능한 정보를
              안정적으로 생성하기 어렵다
```

### 5.1.1 고착된 지식과 사라지는 정확도

LLM의 학습은 특정 시점에 수집된 데이터로 이루어진다. 그 시점 이후에 발생한 사건, 변경된 법령, 새로 발표된 제품 사양, 갱신된 기업 정책은 모델이 알 수 없다. 학습이 완료되는 순간 모델의 세계 인식은 그 시점에서 멈춘다. 이를 **지식 고착(Knowledge Cutoff)** 또는 **컷오프 문제**라고 부른다.

컷오프는 처음에는 그다지 심각해 보이지 않을 수 있다. 모델을 배포한 직후라면 컷오프와 현재 시점의 간극이 몇 달에 불과하기 때문이다. 그러나 시스템이 운영을 지속하면서 이 간극은 조용히 누적된다. 1년 뒤, 2년 뒤에도 같은 모델이 운영되고 있다면, 그 모델은 갈수록 오래된 세계를 기준으로 답변하고 있는 것이다.

기업 내규 질의 시스템을 예로 들어보자. 인사팀이 LLM 기반 사내 Q&A 챗봇을 도입했다. 초기에는 직원 만족도가 높았다. 그런데 6개월 뒤 육아휴직 관련 내규가 개정되었다. 법령 변경에 따라 급여 지급 기준이 바뀌었고, 신청 절차도 달라졌다. 그러나 챗봇은 여전히 개정 전 내규를 기반으로 답변을 제공한다. 직원들은 잘못된 정보를 신뢰하고 그에 따라 행동한다. 오류는 챗봇이 잘못된 정보를 틀렸다는 신호 없이 자신 있게 제공한다는 점에서 더욱 위험하다.

모델을 주기적으로 재학습하면 해결되지 않을까? 이론적으로는 가능하지만, 실제로 수십억~수천억 개의 파라미터를 가진 대형 언어 모델을 재학습하는 데는 막대한 컴퓨팅 자원과 시간이 필요하다. 비용과 주기의 현실적 제약 때문에, 지식 고착은 운영 중인 시스템에서 구조적으로 발생하는 문제로 봐야 한다.

### 5.1.2 모른다고 말하지 않는 모델

지식 고착보다 더 까다로운 문제가 있다. LLM은 자신이 모른다는 사실을 명시적으로 인식하고 표현하는 구조가 아니다. 모델은 확률 기반의 토큰 생성 방식으로 작동한다. 다음에 올 가장 그럴듯한 단어를 예측하는 과정을 반복해 문장을 완성한다. 이 구조에서 "모른다"는 답변은 "관련 있는 토큰 시퀀스를 생성하지 않는다"는 결정을 필요로 하는데, 이것이 학습된 방식으로 작동하지 않을 때 **환각(Hallucination)** 이 발생한다.

::: note 환각(Hallucination)
환각이란 LLM이 사실적으로 틀리거나 근거가 없는 정보를 사실인 것처럼 생성하는 현상이다.[^web_08] 환각의 핵심적 위험은 그 출력이 오류처럼 보이지 않는다는 데 있다. 자신감 있는 문체, 일관된 논리적 흐름, 구체적인 세부사항을 갖추고 있기 때문에 독자는 검토 없이 수용하기 쉽다.
:::

환각이 실무에서 어떤 결과를 낳는지 보여주는 사례 중 하나가 법률 AI의 판례 인용 오류다. 2023년 미국에서 변호사가 법률 AI가 생성한 소장을 그대로 제출했다가, 해당 소장에 인용된 판례 다수가 실제로 존재하지 않는다는 사실이 드러난 사건이 있었다. 모델은 존재하지 않는 판례의 이름, 법원명, 연도를 구체적으로 생성했고, 그 출력은 실제 판례를 인용한 문서와 구분하기 어려울 만큼 자연스러웠다.[^web_08]

환각은 도메인 특화 지식이 없는 영역에서 더 빈번하게, 더 자신감 있게 나타나는 경향이 있다. 모델이 학습 데이터에서 충분한 패턴을 학습하지 못한 영역일수록, 모델은 인접한 패턴으로 빈칸을 채우려 한다.[^web_11] 그리고 그 결과물은 겉보기에 설득력이 있다.

여기에 세 번째 한계가 겹친다. **폐쇄 도메인 지식 부재**다. 기업 내부의 제품 사양서, 계약 템플릿, 내부 감사 기준, 특정 산업의 규제 가이드라인—이런 문서들은 사전 학습 데이터에 포함되어 있지 않다. 공개된 인터넷 텍스트와 서적, 학술 논문 등이 사전 학습의 주요 재료인데, 기업 내부 문서는 그 범위에 속하지 않는다. 이 영역에서 사용자가 질문을 던지면, 모델은 일반적인 지식으로 그럴듯한 답변을 조합하거나, 없는 세부사항을 채워 넣는다.[^web_01]

지식 고착, 환각, 폐쇄 도메인 지식 부재—이 세 가지는 각각 독립적인 문제가 아니다. 동시에 작동한다. 컷오프 이후의 내부 문서를 물어보면 세 가지 한계가 한꺼번에 활성화된다. 모델은 그 문서를 학습한 적이 없고(폐쇄 도메인 지식 부재), 해당 시점 이후의 내용도 모르며(지식 고착), 모른다는 사실을 표현하는 대신 그럴듯한 내용으로 빈칸을 채운다(환각). 이 조합이 LLM 단독 시스템을 신뢰 가능한 정보 제공 채널로 쓰기 어렵게 만드는 구조적 이유다.

::: box 실무에서의 영향: 세 한계가 결합될 때
세 가지 한계가 동시에 작동하는 상황을 하나의 시나리오로 정리하면 다음과 같다.

- **상황**: 특정 제조 기업의 직원이 LLM 챗봇에게 "이번 달 개정된 안전 점검 기준에 대해 알려줘"라고 질문한다.
- **지식 고착**: 모델의 학습 컷오프는 해당 개정 이전이다. 최신 기준을 알 수 없다.
- **폐쇄 도메인 지식 부재**: 해당 기업의 내부 안전 점검 기준 문서는 사전 학습 데이터에 없다.
- **환각**: 모델은 "모른다"고 말하는 대신, 일반적인 제조업 안전 점검 기준을 조합하고, 일부 세부사항을 그럴듯하게 생성해 답변한다.
- **결과**: 직원은 내용이 자신감 있게 서술되어 있어 정확하다고 판단한다. 실제와 다른 기준을 현장에 적용한다.

이 시나리오가 보여주듯, 세 한계는 개별적으로도 문제지만 결합될 때 그 영향이 증폭된다.
:::

이 지점에서 RAG의 등장을 단순히 "유용한 기능"으로 설명하는 것이 왜 부족한지가 드러나기 시작한다. RAG는 LLM의 파라메트릭 메모리가 가진 구조적 한계를 비파라메트릭 메모리를 통해 보완하려는 시도다.[^web_01] 그런데 이를 "검색 기능을 붙이는 것"으로만 이해하면, 어느 지점에서 무엇을 보완하는지에 대한 설계 판단이 빠져버린다. 그 판단이 빠진 RAG가 어떤 문제를 낳는지를 다음 절에서 살펴본다.

[^web_01]: Lewis, P. et al. (2020). Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks. *NeurIPS 2020*.
[^web_08]: 환각의 분류 체계 및 RAG 적용 이후에도 환각이 잔존할 수 있다는 논의를 포함하는 자료.
[^web_11]: 검색 실패가 생성 환각으로 연결되는 연쇄 구조에 대한 분석을 담은 자료.

## 5.2 RAG를 기능으로 보면 무슨 일이 일어나는가

앞 절에서는 LLM이 지식 고착, 환각, 폐쇄 도메인 지식 부재라는 세 가지 구조적 한계를 동시에 갖고 있음을 확인했다. 그 결론은 자연스럽게 RAG를 향한다. RAG는 이 한계를 보완하기 위해 파라메트릭 메모리 외부에서 정보를 끌어오는 구조다. 그런데 현장에서는 RAG를 도입하고 나서도 기대와 다른 일들이 일어난다. "RAG를 붙였는데 왜 여전히 이상한 답이 나오는가"라는 질문이 그것이다. 이 절에서는 그 원인을 살펴본다. 핵심 주장은 하나다. RAG를 기능으로 이해하면, 무엇이 잘못되었는지를 진단하는 언어 자체가 없어진다.

::: note "RAG 품질 = 검색 정확도"라는 오해
RAG를 기능으로 볼 때 가장 흔히 발생하는 오해는 "RAG의 품질은 검색이 얼마나 정확한가에 달려 있다"는 것이다. 그러나 검색은 RAG 파이프라인의 한 단계일 뿐이다. 검색이 성공해도 컨텍스트 구성이 잘못되거나, 청킹 전략이 부적절하거나, 생성 단계에서 후처리가 누락되면 결과는 틀리게 나온다. 검색 정확도는 필요 조건이지 충분 조건이 아니다.
:::

### 5.2.1 "검색이 이상하다"는 진단의 한계

RAG를 도입한 팀이 출력 품질 문제를 마주쳤을 때 가장 먼저 도달하는 진단이 있다. "검색이 이상하다"는 것이다. 관련 문서가 제대로 불러와지지 않았거나, 검색된 문서가 질문과 맞지 않는다는 판단이다. 이 진단이 완전히 틀린 것은 아니다. 검색 품질은 RAG 파이프라인에서 결과에 영향을 미치는 중요한 변수다. 문제는 이 진단이 유일한 언어가 될 때 생긴다.

실제로 RAG 파이프라인에서 출력이 잘못되는 경로는 여러 갈래다. 우선 검색 단계가 실패하는 경우가 있다. 쿼리가 잘못 구성되거나 임베딩 공간에서 관련 문서와의 거리 계산이 빗나가면, 엉뚱한 문서가 검색된다. 이 경우 진단은 비교적 명확하다. 그런데 검색 자체는 성공했지만 컨텍스트 구성이 실패하는 경우가 있다. 관련 문서를 불러왔음에도 해당 문서를 어떻게 자르고(청킹), 어떤 순서로 배열하고, 얼마나 많은 분량을 컨텍스트에 포함할지가 잘못되면, 모델이 받아보는 정보는 이미 왜곡되어 있다.[^web_02] 마지막으로 검색도 성공하고 컨텍스트도 올바르게 구성되었지만, 생성 단계에서 후처리가 제대로 이루어지지 않아 잘못된 출력이 나가는 경우도 있다.

기능 관점의 진단 언어는 이 세 경로를 구별하지 못한다. 결과물이 이상하면 "검색 문제"로, 또는 "모델 문제"로 귀인된다. 두 범주 사이에 있는 컨텍스트 구성, 청킹 전략, 입력 순서, 후처리 같은 변수들은 이름조차 부여받지 못한다. 이름이 없으면 수정할 수 없다. 팀은 벡터 DB를 교체하거나 모델을 다른 것으로 바꾸는 시도를 반복하면서, 정작 문제가 있는 지점을 건드리지 못한 채 시간을 소진한다.

Barnett 등(2024)은 실무 RAG 구현에서 발생하는 실패를 분석하면서, 문제의 원인이 단일 지점이 아니라 파이프라인의 여러 단계에 독립적으로 분산되어 있다는 점을 강조한다.[^web_03] 진단이 "검색이 이상하다" 한 마디로 뭉뚱그려지는 한, 이 분산된 실패 지점들은 발견되지 않는다. 더 나아가 RAGEC 프레임워크가 실증한 바에 따르면, RAG 파이프라인에서 청킹 오류와 검색 오류가 환각보다 오히려 더 빈번하게 발생한다.[^web_12] 즉, 많은 팀이 환각을 억제하려는 목적으로 RAG를 도입하지만, 실제 운영에서는 파이프라인의 앞단(청킹, 인덱싱, 쿼리 구성)에서 더 많은 실패가 만들어지고 있다는 뜻이다.

이 상황은 3장에서 다룬 논지와 같은 구조를 가진다. "문제는 모델이 아니라 시스템이다"라는 명제가 LLM 시스템 전반에 적용된다면, "RAG의 문제는 검색이 아니라 파이프라인의 설계다"라는 명제가 RAG에 특화되어 적용된다. 진단 언어를 갖지 못한 팀은 시스템 어딘가가 잘못되었다는 것을 알면서도 어디를 고쳐야 하는지 알 수 없는 상태에 머문다.

### 5.2.2 Naive RAG가 만들어내는 새로운 실패 지점

기능으로서의 RAG를 가장 직접적으로 구현한 형태가 **Naive RAG**다. Gao 등(2024)은 이를 RAG 구현의 초기 형태로 규정하면서, 세 단계로 구성된 단순한 구조라고 설명한다.[^web_02] 질문을 그대로 쿼리로 변환하고, 벡터 유사도 기반으로 관련 문서를 검색한 다음, 검색된 문서를 그대로 컨텍스트에 붙여 LLM에 전달한다. 각 단계 사이에 설계 판단이 없다. 청킹은 고정된 크기로 기계적으로 이루어지고, 검색된 결과는 점수 순서 그대로 컨텍스트에 나열된다. 이 구조는 "검색해서 붙여 넣기"라는 한 문장으로 요약된다.

::: box Naive RAG의 전형적 구조
Naive RAG는 다음 네 단계로 작동한다.

1. **쿼리 변환 없음**: 사용자의 질문을 그대로 검색 쿼리로 사용한다.
2. **고정 청킹**: 문서를 일정한 토큰 단위로 잘라 인덱싱한다. 문서의 논리 구조나 의미 단위는 고려하지 않는다.
3. **유사도 기반 검색**: 쿼리 임베딩과 문서 청크 임베딩 사이의 코사인 유사도를 기준으로 상위 K개를 검색한다.
4. **컨텍스트 직접 삽입**: 검색된 청크를 순서 그대로 프롬프트에 붙여 넣고 생성을 요청한다.

각 단계는 기본값으로 작동하며, 명시적 설계 결정이 개입하지 않는다.[^web_02]
:::

Naive RAG의 가장 큰 문제는 이 구조 자체가 "검색 이후를 설계하지 않는다"는 암묵적 선택을 포함하고 있다는 점이다. 청킹 전략은 결과에 직접적인 영향을 미친다. 문서의 논리 단위를 무시하고 고정 크기로 자르면, 핵심 정보가 청크의 경계에서 잘려나가거나 관련 없는 내용과 함께 묶인다. 컨텍스트 순서도 마찬가지다. 언어 모델은 컨텍스트 내에서 위치에 따라 정보를 다르게 처리하는 경향이 있다. 가장 관련성이 높은 내용이 컨텍스트의 중간에 묻혀 있으면 모델이 그것을 효과적으로 활용하지 못할 수 있다.[^web_02] 이런 변수들이 결과에 영향을 미치고 있지만, Naive RAG 구조에서는 이 변수들이 존재하는지조차 인식되지 않는다.

더 역설적인 실패 패턴이 있다. 검색된 컨텍스트가 질문과 맞지 않을 때, 모델은 그 잘못된 컨텍스트를 무시하는 것이 아니라 활용한다. LLM은 주어진 컨텍스트를 참조하려는 경향이 있기 때문에, 엉뚱한 문서가 검색되어 들어오면 모델은 그 문서를 바탕으로 더 구체적이고 자신감 있는 오답을 생성한다.[^web_11] RAG가 없을 때는 모델이 불확실한 상황에서 다소 두루뭉술하게 답변하거나 모른다고 말할 여지라도 있지만, 잘못된 컨텍스트가 있으면 그 컨텍스트가 오답 생성의 근거가 된다. 이것은 환각보다 더 나쁜 패턴이다. 출처처럼 보이는 것이 있기 때문에 잘못된 출력에 대한 신뢰가 오히려 높아진다.

이 상황을 하나의 사례로 구체화해보자. 고객 서비스 부서가 RAG 기반 질의응답 시스템을 도입했다. 고객이 "A 제품의 환불 기준이 어떻게 되나요?"라고 묻는다. 검색 과정에서 제품 A와 B의 반품 정책이 혼재된 청크가 상위로 검색된다. 모델은 그 청크를 참조하여 제품 B의 환불 기준을 제품 A의 것처럼 답변한다. 답변에는 실제 정책 문서의 숫자와 조건이 구체적으로 포함되어 있다. 고객은 물론 이를 검토하는 담당자도 이것이 잘못된 정보라는 것을 즉시 알아채기 어렵다. 오류는 조용히 나간다.

기능 관점에서는 이 실패를 어떻게 진단할까. "검색이 잘못되었다"는 결론에 도달할 수 있다. 그런데 그 이후가 막힌다. 무엇을 어떻게 고쳐야 하는지를 결정하려면, 청킹 방식을 바꿔야 하는지, 쿼리 구성 방식을 개선해야 하는지, 검색 후 재순위화(reranking)를 도입해야 하는지, 컨텍스트 필터링 단계를 추가해야 하는지를 논의해야 한다. 그러나 RAG를 기능으로 이해하는 팀에게는 이 항목들이 선택지로 보이지 않는다. 파이프라인 안에 독립적인 설계 결정 지점이 있다는 인식 자체가 없기 때문이다.

결국 RAG를 기능으로 이해할 때 발생하는 핵심 문제는 두 가지로 정리된다. 첫째, 실패를 귀인할 수 있는 언어가 "검색"과 "모델" 두 가지뿐이다. 실제 실패는 파이프라인의 여러 단계에 독립적으로 분산되어 있음에도, 진단 언어가 그것을 구별하지 못한다. 둘째, Naive RAG 구조는 각 단계가 설계 결정 지점이라는 인식 없이 기본값으로 작동한다. 이 기본값들은 단순히 중립적인 것이 아니라, 특정 실패를 구조적으로 내포한다.

이 두 문제를 동시에 해결하는 접근은 RAG를 바라보는 시각 자체를 바꾸는 것이다. "어떤 기능을 붙이는가"가 아니라 "정보 흐름의 어느 지점을 어떻게 통제하는가"라는 질문으로 전환할 때, 파이프라인 안의 각 단계가 비로소 설계 판단의 대상으로 보이기 시작한다. 다음 절에서는 바로 그 전환이 무엇을 가능하게 하는지를 살펴본다.

## 5.3 RAG를 아키텍처로 재정의하면 무엇이 보이는가

앞 절에서는 RAG를 기능으로 이해할 때 어떤 것들이 보이지 않는지를 확인했다. 실패를 귀인할 언어가 부족하고, 파이프라인 안의 설계 결정 지점들이 존재조차 인식되지 않는다는 것이 핵심 문제였다. 이제 시각을 바꿔보자. RAG를 아키텍처로 바라볼 때 비로소 보이기 시작하는 것들을 이 절에서 살펴본다.

::: note RAG의 아키텍처적 정의
RAG는 LLM에 들어가는 정보의 출처와 범위를 시스템이 통제하는 구조다.[^web_07] 이 정의에서 핵심은 '통제'라는 단어다. 어떤 정보가 모델에 도달할 수 있는지를 우연이나 기본값에 맡기지 않고, 시스템이 명시적으로 결정한다는 뜻이다.
:::

IBM(2023)은 공식 기술 문서에서 RAG를 "아키텍처 패턴(architectural pattern)"으로 명시한다.[^web_07] 학계에서도 같은 방향의 논의가 이어진다. Gao 등(2024)은 Modular RAG 논문에서 RAG를 "재구성 가능한 프레임워크(reconfigurable framework)"로 정의하면서, 각 모듈이 독립적인 설계 결정 지점임을 체계적으로 논증한다.[^web_04] 이 두 흐름은 같은 방향을 가리킨다. RAG는 단일한 기능이 아니라, 여러 층위의 설계 선택이 중첩된 구조라는 것이다.

이 재정의가 실무적으로 무엇을 달라지게 하는지를 아래 표로 대비해보자.

**표 5-1. 기능 관점 vs. 아키텍처 관점에서의 RAG 이해 비교**

| 비교 항목 | 기능 관점 | 아키텍처 관점 |
|---|---|---|
| RAG를 기술하는 언어 | "외부 지식을 검색해서 붙인다" | "정보 흐름의 각 단계에서 통제 결정을 내린다" |
| 실패 진단 언어 | "검색이 이상하다", "모델이 잘못 답했다" | "청킹 전략 실패", "컨텍스트 구성 오류", "재순위화 부재" |
| 핵심 설계 질문 | "검색 API를 어떻게 연결하는가?" | "어느 단계에서 어떤 정보를 걸러내는가?" |
| 실패 발생 시 책임 소재 | 검색 또는 모델, 둘 중 하나 | 파이프라인의 특정 단계, 독립적으로 식별 가능 |
| 개선 방식 | 모델 교체 또는 검색 엔진 변경 | 해당 단계의 설계 결정을 수정 |

관점이 바뀌면 묻는 질문이 달라진다. "RAG를 도입할까?"가 아니라 "어떤 단계에서 어떤 방식으로 정보를 통제할 것인가?"를 묻게 된다. 이 질문들은 각각 독립적으로 답할 수 있으며, 독립적으로 실패할 수 있다.

### 5.3.1 정보 흐름을 통제한다는 것의 의미

"검색을 추가한다"는 행위는 정보 흐름의 한 지점만을 건드린다. 외부 데이터를 가져와 컨텍스트에 삽입하는 그 순간에만 개입한다. 그 전후에서 무슨 일이 일어나는지는 기본값이 처리한다.

"정보 흐름을 통제한다"는 것은 다르다. 어떤 정보가 시스템에 진입하는지(색인 단계), 그 정보가 어떻게 선택되는지(검색 단계), 선택된 정보가 어떻게 구성되어 모델에 전달되는지(컨텍스트 구성 단계), 모델이 생성한 결과가 어떻게 처리되는지(후처리 단계)를 전 구간에 걸쳐 설계하는 것이다. Lewis 등(2020)이 제안한 '비파라메트릭 메모리(non-parametric memory)'[^web_01]라는 개념도 이 맥락에서 다시 읽힌다. 모델의 가중치 안에 고착된 지식이 아니라, 외부에서 동적으로 공급되고 통제될 수 있는 지식의 층위를 가리킨다. 이 '통제 가능성'이야말로 RAG가 단순한 검색 기능과 구별되는 지점이다.

통제 지점이 여럿이라는 것은 설계 결정이 여럿이라는 뜻이며, 각 결정은 독립적으로 실패할 수 있다는 뜻이기도 하다. 아래 그림은 이 구조를 시각화한 것이다.

**그림 5-1. RAG 아키텍처의 정보 흐름 통제 레이어 구조**

```
외부 저장소
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  [통제 지점 1] 색인(Indexing)                        │
│  청킹 전략, 임베딩 모델 선택, 메타데이터 구성           │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  [통제 지점 2] 검색(Retrieval)                       │
│  쿼리 변환, 검색 방식 선택, 재순위화(reranking)         │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  [통제 지점 3] 컨텍스트 구성(Context Assembly)        │
│  청크 선택, 순서 배치, 불필요 정보 필터링               │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  [통제 지점 4] 생성(Generation)                      │
│  프롬프트 구조, 지시문 설계, 모델 파라미터              │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  [통제 지점 5] 후처리(Post-processing)               │
│  출력 검증, 출처 확인, 필터링                          │
└─────────────────────────────────────────────────────┘
    │
    ▼
최종 출력
```

이 구조에서 각 통제 지점은 독립적인 설계 결정을 요구한다. 색인 단계에서 청킹 전략을 잘못 선택하면 그 오류는 검색 단계, 컨텍스트 구성 단계, 생성 단계를 거쳐 출력까지 전파된다. 반대로 말하면, 출력이 잘못되었을 때 그 원인을 특정 통제 지점으로 추적할 수 있다. 이것이 아키텍처 관점이 가져다주는 진단 능력이다.

동일한 RAG 시스템을 두 관점에서 기술해보면 이 차이가 더 분명해진다.

**기능 관점의 기술**: "이 시스템은 사용자 질문에 맞는 문서를 검색하여 LLM에 전달하고 답변을 생성한다."

**아키텍처 관점의 기술**: "이 시스템은 문서를 512토큰 단위로 청킹하여 색인한다. 사용자 질문은 HyDE(Hypothetical Document Embeddings) 방식으로 변환된 후 하이브리드 검색(밀집 벡터 + 키워드)을 통해 상위 10개 청크를 가져온다. 검색 결과는 교차 인코더(cross-encoder) 기반 재순위화를 거쳐 상위 3개만 컨텍스트에 포함된다. 생성된 출력은 인용 확인 단계를 통해 컨텍스트와의 일치 여부를 검증한 뒤 반환된다."

두 번째 기술 방식에서는 각 통제 지점의 설계 선택이 명시되어 있다. 이 기술 방식을 쓰는 팀은 실패가 발생했을 때 어느 통제 지점을 먼저 들여다봐야 하는지를 알고 있다.

::: box Naive → Advanced → Modular RAG: 진화는 어느 방향으로 이동했는가

Gao 등(2024)은 RAG의 구조적 진화를 세 단계로 정리한다.[^web_02][^web_04]

**Naive RAG**: 색인-검색-생성의 단순한 선형 구조. 각 단계 사이에 명시적 설계 결정이 없다. 기본값이 전체를 처리한다. 이것은 RAG를 "기능"으로 구현한 형태다.

**Advanced RAG**: 쿼리 변환, 재순위화, 컨텍스트 압축 등의 기법이 특정 단계에 추가된다. 각 기법은 특정 실패를 해결하기 위한 설계 개입이다. Naive RAG의 실패 지점을 인식하기 시작한 단계이며, 일부 통제 지점이 명시적 설계 결정의 대상이 된다.

**Modular RAG**: 각 단계가 독립적으로 교체 가능한 모듈로 재구성된다. 검색 전략, 컨텍스트 구성 방식, 후처리 단계를 각 도메인과 요구사항에 맞게 조합할 수 있다. Self-RAG[^web_05]처럼 검색 여부 자체를 모델이 동적으로 결정하는 방식도 이 범주에 포함된다. 이 단계에 이르면 RAG는 완전히 아키텍처적 선택의 집합이 된다.

이 진화의 방향은 "기능에서 아키텍처로"의 이동이다. 각 단계에서 암묵적으로 처리되던 것들이 명시적 설계 결정의 대상이 되어간다.
:::

### 5.3.2 하네스와의 연결: 통제 구조는 외부에서 감싼다

1장에서 소개한 **평가 하네스(harness)** 개념을 잠깐 되짚자. 하네스는 실행과 평가를 반복 가능하게 수행하는 외부 구조다. 시스템의 내부 로직을 바꾸지 않고, 외부에서 감싸면서 관찰하고 개입한다.

RAG를 아키텍처로 이해하면, 하네스가 어디에 붙어야 하는지가 보인다. 색인, 검색, 컨텍스트 구성, 생성, 후처리 — 이 다섯 통제 지점은 각각 하네스가 감시하고 개입할 수 있는 경계이기도 하다. 색인 단계의 청킹 품질을 측정하고, 검색 단계의 정밀도와 재현율을 추적하고, 컨텍스트 구성이 적절한 정보를 담고 있는지를 확인하는 것 — 이 모든 작업이 하네스의 역할이다. RAGAS[^web_06]처럼 RAG의 각 단계를 독립적으로 평가하는 프레임워크가 등장한 것도 같은 인식에서 비롯된다. 각 단계가 독립적 설계 지점이라면, 각 단계는 독립적으로 평가될 수 있어야 한다. "RAG를 아키텍처로 본다"는 것은 곧 "각 통제 지점에 하네스가 필요하다"는 인식과 맞닿아 있다.

---

지금까지 RAG의 아키텍처적 정의를 확인하고, 그 정의가 기능 관점과 무엇이 다른지를 살펴보았다. 정보 흐름의 통제 레이어가 여럿이고, 각 레이어가 독립적 설계 결정 지점이라는 것이 핵심이다. 그런데 이 구조를 실제로 설계하고 운영하려면, 각 단계가 구체적으로 무엇을 담당하고 어디에서 실패하는지를 더 세밀하게 이해해야 한다. 다음 절에서는 RAG 파이프라인의 각 단계를 하나씩 펼쳐보면서, 각 단계가 통제하는 것과 위임하는 것, 그리고 그 경계에서 발생하는 실패 유형을 구체적으로 살펴본다.

[^web_01]: Lewis, P. et al. (2020). Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks. 파라메트릭 메모리(parametric memory)와 비파라메트릭 메모리(non-parametric memory)의 구분을 통해 RAG의 이론적 기반을 제시한다.
[^web_02]: Gao, Y. et al. (2024). Retrieval-Augmented Generation for Large Language Models: A Survey. Naive → Advanced → Modular RAG의 진화 계보를 체계적으로 정리한다.
[^web_04]: Gao, Y. et al. (2024). Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks. RAG를 재구성 가능한 프레임워크로 정의하며, 각 모듈이 독립적 설계 결정 지점임을 논증한다.
[^web_05]: Asai, A. et al. (2023). Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection. ICLR 2024 Oral. 검색 여부를 모델이 동적으로 결정하는 방식으로, Modular RAG의 대표적 사례다.
[^web_06]: Es, S. et al. (2023). RAGAS: Automated Evaluation of Retrieval Augmented Generation. RAG의 각 단계(검색, 컨텍스트 구성, 생성)를 독립적으로 평가하는 메트릭을 제시한다.
[^web_07]: IBM (2023). What is retrieval-augmented generation? RAG를 아키텍처 패턴으로 명시적으로 정의한 공식 산업 문서.

## 5.4 RAG 파이프라인 단계별 역할과 통제 지점

앞에서 RAG를 "여러 통제 지점의 집합"으로 정의하고, 그 정의가 기능 관점과 어떻게 다른지를 살펴보았다. 이제 그 통제 지점들을 하나씩 구체적으로 펼쳐볼 차례다. 파이프라인의 각 단계가 무엇을 통제하고, 어떤 설계 변수가 거기서 결정되며, 그 결정이 이후 단계에 어떤 영향을 미치는지를 순서대로 짚어보자.

먼저 전체 구조를 한눈에 파악할 수 있도록 정리하면 다음과 같다.

**표 5-2. RAG 파이프라인 단계별 역할과 통제 대상**

| 단계 | 역할 | 통제 대상 | 주요 설계 변수 |
|------|------|-----------|---------------|
| 문서 전처리·색인 | 정보 진입 통제 | 어떤 정보를 검색 가능한 상태로 만들 것인가 | 청킹 전략, 메타데이터 설계, 색인 방식 |
| 검색 | 정보 선택 통제 | 어떤 정보를 꺼낼 것인가 | 쿼리 구성 방식, 유사도 계산 방식, Top-K, 조건부 검색 여부 |
| 컨텍스트 구성 | 정보 구성 통제 | 꺼낸 정보를 어떻게 모델에 전달할 것인가 | 배치 순서, 중복 제거, 길이 제한, 출처 표시 방식 |
| 생성 | 정보 활용 | 주어진 정보를 바탕으로 출력을 만든다 | 프롬프트 설계, 모델 선택, 컨텍스트 우선도 유도 |
| 출력 후처리 | 출력 통제 | 생성된 결과를 검증·정제한다 | 출처 참조 확인, 포맷 검증, 신뢰도 임계값 |

**그림 5-4. RAG 파이프라인의 단계별 정보 흐름과 통제 지점**

```
외부 문서
    │
    ▼
┌─────────────────────┐
│   문서 전처리·색인   │ ◀── [통제 지점 1] 정보 진입
│  (청킹, 메타데이터,  │
│     벡터 색인)       │
└─────────┬───────────┘
          │ 벡터 저장소
          ▼
┌─────────────────────┐
│       검색           │ ◀── [통제 지점 2] 정보 선택
│ (쿼리 구성, 유사도   │
│  계산, Top-K 선택)   │
└─────────┬───────────┘
          │ 검색된 청크
          ▼
┌─────────────────────┐
│   컨텍스트 구성       │ ◀── [통제 지점 3] 정보 구성
│ (순서, 중복 제거,    │
│   길이 제한)         │
└─────────┬───────────┘
          │ 구성된 컨텍스트
          ▼
┌─────────────────────┐
│        생성          │ ◀── [통제 지점 4] 정보 활용
│  (LLM 추론, 프롬프트 │
│     + 컨텍스트)      │
└─────────┬───────────┘
          │ 생성된 출력
          ▼
┌─────────────────────┐
│     출력 후처리       │ ◀── [통제 지점 5] 출력 통제
│ (출처 확인, 포맷,    │
│   신뢰도 검증)       │
└─────────┬───────────┘
          │
          ▼
       최종 출력
```

각 통제 지점이 무엇을 결정하는지를 이제 단계별로 들여다보자.[^web_02][^web_04]

### 5.4.1 정보 진입 통제: 색인이 결정하는 것들

RAG 파이프라인에서 문서 전처리·색인 단계는 가장 상류(upstream)에 위치한다. 여기서의 오류는 이후 어떤 단계도 완전히 만회하기 어렵다. 검색이 아무리 정밀하게 설계되어 있어도 색인에 들어 있지 않은 정보는 꺼낼 수 없고, 잘못 분할된 청크는 잘못된 검색 결과를 지속적으로 생산한다. 이 단계를 설계한다는 것은 "어떤 정보를, 어떤 형태로, 검색 가능한 상태로 만들 것인가"를 결정하는 일이다.

이 단계의 핵심 설계 변수는 세 가지다.

**청킹 전략**은 문서를 어떤 단위로 분할할 것인가의 문제다. 이 결정은 이후 검색 품질 전체에 영향을 미친다.

::: box 청킹 전략은 왜 설계 결정인가

청킹은 단순한 기술 구현이 아니다. "어떤 단위로 정보를 검색 가능하게 만들 것인가"라는 설계 선택이다.

**문장 단위 청킹**: 각 문장을 독립적인 검색 단위로 만든다. 정밀한 매칭이 가능하지만, 문장 하나만으로는 의미를 파악하기 어려운 경우 컨텍스트가 사라진다. "전년 대비 12% 감소했다"는 문장은 무엇이 감소했는지를 알 수 없다.

**단락 단위 청킹**: 하나의 단락을 검색 단위로 묶는다. 맥락이 보존되지만 단락 내에 관련 없는 정보가 섞일 수 있고, 청크 크기가 일정하지 않다.

**의미 단위 청킹**: 문서의 의미적 경계(소제목, 논리 단락 구분 등)를 기준으로 분할한다. 가장 많은 전처리 비용이 들지만, 검색된 청크가 하나의 완결된 개념 단위를 담을 가능성이 높다.

트레이드오프는 명확하다. 청크가 작을수록 검색 정밀도(precision)는 높아지지만 맥락 보존이 어렵고, 청크가 클수록 맥락은 보존되지만 노이즈가 섞일 가능성이 높아진다. 어느 쪽이 이 시스템에 더 중요한가는 도메인에 따라 다르다. 법률 문서라면 조항 단위의 정확성이 우선이고, 매뉴얼 문서라면 절차 흐름을 보존하는 단락 단위가 더 적합할 수 있다.[^web_02]
:::

**메타데이터 설계**는 청크와 함께 색인되는 구조화된 정보를 어떻게 설계할 것인가의 문제다. 문서 출처, 작성일, 버전 번호, 섹션 계층 — 이 정보들은 검색 단계에서 필터링 조건으로 활용된다. 예컨대 "2024년 이후에 작성된 문서만 검색"하거나 "특정 제품 버전에 해당하는 문서만 포함"하려면, 그 기준이 되는 메타데이터가 색인 시점에 구조화되어 있어야 한다. 이것이 나중에 추가될 수 없는 이유다.

**색인 전략**은 어떤 방식으로 색인을 구성할 것인가의 문제다. 밀집 벡터 색인(dense vector index)은 의미적 유사성에 강하지만 정확한 용어 매칭에 약하고, 키워드 기반 역색인(inverted index)은 정확한 용어 매칭에 강하지만 표현이 다르면 놓친다. 두 방식을 결합한 하이브리드 검색은 두 장점을 취하지만 구성이 복잡해진다. 어떤 색인 전략을 선택하느냐는 검색 단계에서 사용할 수 있는 방식의 범위를 규정한다.[^web_04]

---

### 5.4.2 정보 선택 통제: 검색이 결정하는 것들

색인이 정보의 진입을 통제한다면, 검색 단계는 색인된 정보 중 무엇을 꺼낼 것인가를 결정한다. 여기서의 설계 선택은 컨텍스트 구성 단계에 전달되는 정보의 질과 양을 직접 결정한다.

**쿼리 구성 방식**은 사용자의 원래 질문을 어떻게 검색 쿼리로 변환할 것인가의 문제다. 사용자 입력을 그대로 쿼리로 쓰는 것이 가장 단순한 방법이지만, 사용자의 질문이 모호하거나 검색 시스템이 기대하는 형태와 다를 때 검색 품질이 낮아진다. 쿼리 재작성(query rewriting), 쿼리 분해(query decomposition), 또는 가상 문서를 생성해 그것을 쿼리로 쓰는 HyDE(Hypothetical Document Embeddings) 방식 등은 모두 이 단계에서의 설계 선택이다.[^web_02]

**유사도 계산 방식**은 색인 전략과 밀접하게 연결된다. 벡터 유사도(코사인 유사도, 내적 등)는 의미적으로 가까운 청크를 찾는 데 효과적이지만, "RFC 2119"나 "BERT-base-uncased"처럼 정확한 용어 매칭이 필요한 경우에는 성능이 떨어질 수 있다. 유사도 계산 방식의 선택은 어떤 종류의 질문에 이 시스템이 잘 작동하는가를 결정한다.

**검색 결과 수(Top-K)**는 몇 개의 청크를 검색할 것인가의 문제다. K를 크게 설정하면 관련 정보를 놓칠 가능성(재현율 손실)이 줄어들지만, 무관한 청크도 포함될 가능성(노이즈)이 높아진다. K를 작게 설정하면 컨텍스트가 깔끔해지지만 중요한 정보가 빠질 수 있다. 이 파라미터는 종종 기본값으로 방치되지만, 도메인과 질문 유형에 따라 최적값이 달라지는 명백한 설계 변수다.

::: note Self-RAG — 검색을 조건부로 만드는 설계

Asai 등(2023)이 제안한 Self-RAG는 검색 자체를 조건부로 만드는 설계다.[^web_05] 기존 RAG가 모든 질문에 대해 검색을 수행한다면, Self-RAG는 모델이 스스로 "이 질문에 검색이 필요한가"를 먼저 판단한다. 검색이 필요하다고 판단될 때만 검색이 일어나고, 검색된 청크가 실제로 유용한지도 모델이 평가한다.

이 설계가 시사하는 바는 명확하다. 무차별적 검색은 오히려 성능을 낮출 수 있다. "오늘 날씨는 어때?"처럼 외부 문서가 필요 없는 질문에 검색이 일어나면, 관련 없는 청크가 컨텍스트에 섞이고 모델의 응답 품질이 저하된다. "모든 질문에 검색이 필요한가"는 RAG 파이프라인 설계에서 반드시 물어야 하는 질문이다.
:::

### 5.4.3 정보 구성 통제와 출력 통제: 전달과 검증

검색이 어떤 정보를 꺼낼지를 결정한다면, 컨텍스트 구성 단계는 꺼낸 정보를 어떻게 모델에 전달할지를 결정한다. 이 단계는 종종 "검색된 문서를 그냥 붙여넣는 것"으로 처리되지만, 실제로는 생성 품질에 상당한 영향을 미치는 설계 지점이다.

#### 컨텍스트 구성: 순서와 형식이 만드는 차이

검색된 청크를 어떤 순서로 컨텍스트에 배치하느냐는 모델이 해당 정보를 얼마나 잘 활용하는가에 영향을 미친다.

::: box 컨텍스트 구성에서 순서가 중요한 이유

Liu 등의 연구에서 확인된 "Lost in the middle" 현상은 LLM이 긴 컨텍스트를 처리할 때 중간 부분의 정보를 상대적으로 잘 활용하지 못하는 경향을 보인다는 것이다.[^web_02] 즉, 가장 관련성이 높은 청크를 컨텍스트의 중간에 배치하면, 컨텍스트 시작이나 끝에 배치했을 때보다 생성 품질이 낮아질 수 있다.

이 발견이 갖는 설계 함의는 분명하다. 검색된 결과를 재순위화(reranking)하고, 가장 관련성 높은 청크를 컨텍스트의 앞이나 뒤에 배치하는 것이 단순한 최적화가 아니라 통제 지점에서의 명시적 설계 결정이라는 점이다. 교차 인코더(cross-encoder) 기반 재순위화는 이 목적을 위해 흔히 사용되는 방법이다.

같은 정보를 담고 있더라도, 그것을 어떤 순서로 모델에 전달하느냐에 따라 생성 결과가 달라진다. 이것은 컨텍스트 구성이 단순한 취합 작업이 아님을 보여준다.
:::

컨텍스트 구성 단계의 다른 설계 변수들도 주목할 필요가 있다. **중복 제거**는 여러 청크가 비슷한 내용을 담고 있을 때 컨텍스트 길이를 낭비하지 않도록 한다. **길이 제한**은 모델의 컨텍스트 윈도우(context window) 한계와 비용 사이에서 균형을 잡는다. **출처 표시 방식**은 모델이 생성 시 출처를 참조하도록 유도하는 방법으로, 출력 후처리 단계에서 인용 검증을 가능하게 하는 전제 조건이기도 하다.

#### 생성 단계: 통제와 위임의 경계

생성 단계는 RAG 파이프라인에서 LLM이 실제로 추론하는 지점이다. 모델은 컨텍스트로 전달된 정보와 사전 학습을 통해 내재화된 지식 — 이른바 파라메트릭 메모리(parametric memory) — 을 함께 활용해 출력을 생성한다.[^web_01]

이 단계에서 설계자가 직접 통제할 수 있는 범위는 다른 단계에 비해 제한적이다. 모델이 컨텍스트를 따를지, 아니면 사전 학습 지식을 따를지는 모델 내부의 작동 방식에 달려 있기 때문이다. 그러나 프롬프트 설계와 모델 선택이라는 두 가지 수단이 남아 있다. "다음 문서만을 근거로 답하라"는 지시는 모델이 컨텍스트를 우선하도록 유도하는 프롬프트 설계의 전형적인 사례다. 이 유도가 효과적인지는 모델마다 다르며, 이것이 모델 선택이 설계 변수인 이유 중 하나다.

#### 출력 후처리: 생성된 결과를 시스템이 다시 잡는다

출력 후처리 단계는 생성된 결과를 시스템이 검증하고 정제하는 마지막 통제 지점이다. Barnett 등(2024)이 정리한 실무 실패 분석에서도 생성 단계 이후의 검증 부재가 출력 오류의 주요 원인으로 지목된다.[^web_03]

이 단계에서 이루어지는 주요 작업은 세 가지다. 첫째, **출처 참조 확인**은 생성된 답변이 컨텍스트에 실제로 포함된 정보를 인용하고 있는지를 검증한다. 모델이 컨텍스트에 없는 내용을 그럴듯하게 생성했을 때 이를 걸러내는 역할을 한다. 둘째, **포맷 검증**은 시스템이 요구하는 출력 형식(JSON, 특정 구조의 텍스트 등)이 충족되었는지를 확인한다. 셋째, **신뢰도 임계값 적용**은 RAGAS[^web_06] 같은 평가 프레임워크에서 제시하는 것처럼, 검색된 컨텍스트와 생성된 답변 사이의 일관성 점수가 일정 기준 미만일 때 답변 대신 "정보를 찾을 수 없습니다" 같은 안전한 응답을 반환하도록 한다.

이 다섯 단계를 합쳐 보면, RAG 파이프라인은 단순한 정보 검색 + 생성 구조가 아니라 각각이 독립적 설계 결정 지점인 통제 레이어의 연쇄임을 확인할 수 있다. 각 단계에서 어떤 선택이 이루어지느냐에 따라 동일한 질문에 대해 완전히 다른 결과가 나올 수 있다.[^web_04]

---

각 단계가 무엇을 결정하는지를 이제 파악했다. 그런데 이 구조를 실제로 운영하다 보면 한 가지 불편한 사실과 마주하게 된다. 파이프라인의 각 단계는 서로 연결되어 있기 때문에, 특정 단계에서 발생한 결함이 출력에서는 전혀 다른 형태로 나타날 수 있다는 것이다. 색인 단계의 청킹 오류가 검색 단계의 저품질 결과로, 다시 컨텍스트 구성의 노이즈로, 최종적으로는 생성 단계의 환각으로 이어지는 연쇄가 가능하다. 출력이 잘못되었을 때 그 원인을 어느 단계에서 찾아야 하는가. 다음 절에서는 RAG의 실패 유형을 단계별로 분류하는 진단 언어를 구체적으로 살펴본다.

[^web_01]: Lewis, P. et al. (2020). Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks. 파라메트릭 메모리(parametric memory)와 비파라메트릭 메모리(non-parametric memory)의 구분을 통해 RAG의 이론적 기반을 제시한다.
[^web_02]: Gao, Y. et al. (2024). Retrieval-Augmented Generation for Large Language Models: A Survey. Naive → Advanced → Modular RAG의 진화 계보를 체계적으로 정리한다.
[^web_03]: Barnett, S. et al. (2024). Seven Failure Points When Engineering a Retrieval Augmented Generation System. 실무 실패 사례를 기반으로 RAG의 7가지 실패 지점을 분류한다.
[^web_04]: Gao, Y. et al. (2024). Modular RAG: Transforming RAG Systems into LEGO-like Reconfigurable Frameworks. RAG를 재구성 가능한 프레임워크로 정의하며, 각 모듈이 독립적 설계 결정 지점임을 논증한다.
[^web_05]: Asai, A. et al. (2023). Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection. ICLR 2024 Oral. 검색 여부를 모델이 동적으로 결정하는 방식으로, Modular RAG의 대표적 사례다.
[^web_06]: Es, S. et al. (2023). RAGAS: Automated Evaluation of Retrieval Augmented Generation. RAG의 각 단계를 독립적으로 평가하는 메트릭을 제시한다.

## 5.5 RAG 실패

앞에서 RAG 파이프라인의 각 단계가 정보 흐름의 어느 지점을 통제하는지를 살펴보았다. 색인, 쿼리 처리, 검색, 컨텍스트 구성, 생성, 출력 후처리로 이어지는 각 단계는 서로 독립적인 설계 결정 지점이지만, 동시에 긴밀하게 연결되어 있다. 이 연결성이 RAG 실패를 진단하기 어렵게 만드는 핵심 이유다. 출력이 잘못되었을 때, 그 원인은 출력 단계에 있지 않은 경우가 많다.

이번 절에서는 RAG의 실패를 "검색이 이상하다"거나 "모델이 환각을 일으켰다"는 현상 기술이 아니라, 파이프라인의 어느 단계에서 어떤 설계 결함이 발생했는지를 짚어낼 수 있는 언어로 정리한다.

### 5.5.1 실패는 단계별로 발생한다

RAG 파이프라인에서 발생하는 실패는 출력에서 단일한 형태로 나타나더라도, 그 원인은 서로 다른 단계에 분산되어 있다. Barnett 등(2024)은 실무 시스템에서 반복적으로 관찰된 실패를 7가지 지점으로 분류했으며, 그 대부분이 생성 단계가 아니라 그 이전 단계에서 기원한다는 점을 지적한다.[^web_03]

**표 5-1. RAG 파이프라인 단계별 주요 실패 유형**

| 단계 | 실패 유형 | 출력에서의 증상 |
|---|---|---|
| 색인 | 부적절한 청킹, 메타데이터 누락 | 검색 결과의 품질 저하, 문맥 단절 |
| 쿼리 처리 | 사용자 의도 오해석, 쿼리 미변환 | 관련 없는 청크 검색 |
| 검색 | 유사도 임계값 오설정, 벡터 공간 미스매치 | 무관한 정보 또는 공백 컨텍스트 |
| 컨텍스트 구성 | 중복 과다, 배치 순서 미조정 | 모델의 핵심 정보 누락 또는 혼동 |
| 생성 | 컨텍스트 무시, 파라메트릭 메모리 과신 | 환각, 출처 불일치 |
| 출력 후처리 | 검증 부재, 포맷 미확인 | 잘못된 형식의 출력, 미검증 인용 |

이 표가 보여주는 중요한 점은 각 단계의 실패가 고유한 원인과 증상을 가진다는 것이다. 색인 단계에서 청킹이 잘못 설정되면 그 오류는 검색 단계의 저품질 결과로, 다시 컨텍스트 구성 단계의 노이즈로, 최종적으로는 생성 단계의 환각으로 연쇄될 수 있다. 출력만을 보고 "환각이 발생했다"고 진단하면 실제 원인인 청킹 오류는 수정되지 않는다.

::: box 실패 연쇄의 실제 사례
금융 규제 문서를 대상으로 구축된 RAG 시스템을 가정해 보자. 청킹 경계가 조항의 중간에서 잘려 "이 경우에 해당하는 기업은"이라는 문장이 앞 조항의 맥락 없이 독립 청크로 생성되었다고 하자.

검색 단계에서는 이 청크가 질문과 표면적 유사도가 높아 상위에 배치된다. 컨텍스트 구성 단계에서는 이 단절된 청크가 그대로 모델에 전달된다. 생성 단계에서 모델은 "이 경우에 해당하는 기업"의 범주를 자신이 알고 있는 정보 — 즉 파라메트릭 메모리 — 로 채워 넣는다. 출력은 자신감 있는 환각이 된다.

이 실패의 원인은 생성 단계가 아니라 색인 단계의 청킹 설계다. 출력 후처리에서 출처 참조 확인을 수행했더라면 단절 징후를 잡을 수도 있었겠지만, 근본 원인을 고치지 않는 한 같은 실패는 반복된다.
:::

### 5.5.2 실패 유형의 아키텍처적 재정의

RAG 실패를 단계별로 분류할 수 있다면, 다음 단계는 그 실패를 아키텍처 결함의 언어로 재정의하는 것이다. 실무에서 자주 쓰이는 "검색이 잘 안 된다"거나 "모델이 이상하게 답한다"는 표현은 진단 언어가 아니다. 어느 통제 지점에서 무엇이 빠져 있는지를 명시해야 설계 수정이 가능해진다.

Gao 등(2024)의 서베이는 Naive RAG에서 Advanced RAG로의 전환이 검색 전(pre-retrieval)과 검색 후(post-retrieval) 처리를 체계화하는 방향으로 이루어졌다는 점을 보여준다.[^web_02] 이 전환은 "검색만 잘 하면 된다"는 관점에서 "파이프라인 전체를 설계해야 한다"는 관점으로의 이동이다. 다시 말해, RAG의 성숙은 실패를 더 정밀한 언어로 기술할 수 있게 되는 과정과 일치한다.

아래에 자주 관찰되는 실패 유형을 아키텍처 결함으로 재정의한 표현을 정리한다.

- **"검색 결과가 무관하다"** → 쿼리와 색인 사이의 의미 공간 불일치. 쿼리 변환 또는 임베딩 모델 재선택이 필요한 설계 문제.
- **"답이 너무 짧거나 피상적이다"** → 검색된 청크의 밀도 부족 또는 컨텍스트 길이 제한의 과도한 적용. 청킹 전략 재검토가 필요한 색인 단계의 결함.
- **"모델이 컨텍스트를 무시하고 자기 마음대로 답한다"** → 생성 단계에서 파라메트릭 메모리와 비파라메트릭 메모리 간의 우선순위 설정 실패.[^web_01] 프롬프트 설계 또는 모델 교체가 요구되는 생성 단계의 통제 부재.
- **"같은 질문에 매번 다른 답이 나온다"** → 검색 결과의 비결정성(non-determinism) 또는 청크 배치 순서의 불안정. 재순위화(reranking) 부재 또는 컨텍스트 구성 로직의 불일치.
- **"출처를 잘못 인용한다"** → 출력 후처리 단계의 출처 참조 확인 부재. 또는 색인 단계에서 메타데이터가 청크와 함께 저장되지 않은 구조적 결함.

::: note 비파라메트릭 메모리와 파라메트릭 메모리
Lewis 등(2020)이 제시한 구분으로, RAG의 이론적 기반을 이루는 개념이다.[^web_01] **파라메트릭 메모리(parametric memory)**는 모델의 가중치에 내재화된 지식, 즉 사전 학습을 통해 모델 내부에 저장된 정보를 가리킨다. **비파라메트릭 메모리(non-parametric memory)**는 외부 저장소에서 실시간으로 검색되어 컨텍스트로 제공되는 정보다. RAG의 핵심은 이 두 가지를 어떻게 조합하느냐에 있으며, 생성 단계에서의 많은 실패는 이 두 메모리 사이의 우선순위가 명확히 설계되지 않은 데서 비롯된다.
:::

### 5.5.3 실패 진단이 가능하려면

RAG의 실패를 단계별로 진단하려면 전제가 하나 있다. 각 단계의 출력을 관찰할 수 있어야 한다는 것이다. 최종 출력만을 보고 원인 단계를 역추적하는 것은 구조적으로 매우 어렵다. Barnett 등(2024)이 강조하는 점도 이것이다. 실패의 상당수는 운영 중에야 드러나며, 그것을 포착하려면 각 단계의 중간 결과를 로깅하고 추적할 수 있는 관측 구조가 필요하다.[^web_03]

Es 등(2023)의 RAGAS 프레임워크는 이 방향에서 실용적인 접근을 제시한다.[^web_06] RAGAS는 검색 단계, 컨텍스트 구성 단계, 생성 단계를 각각 독립적으로 평가하는 메트릭을 제공한다. 예를 들어 **충실도(faithfulness)**는 생성된 답변이 컨텍스트에 실제로 포함된 정보를 기반으로 하는지를 측정하고, **컨텍스트 정밀도(context precision)**는 검색된 청크 중 실제로 답변에 기여한 비율을 측정한다. 이 메트릭들이 단계별로 분리되어 있다는 것 자체가, 각 단계가 독립적인 진단 대상임을 전제한다.

::: tip 단계별 로깅을 구조로 만들어라
RAG 파이프라인을 처음 구축할 때부터 각 단계의 입출력을 로깅할 수 있는 구조를 포함하는 것이 좋다. 검색 단계에서 어떤 청크가 반환되었는지, 컨텍스트 구성 단계에서 어떤 순서로 배치되었는지, 생성 단계에서 어떤 프롬프트가 전달되었는지를 기록해 두면, 출력 오류가 발생했을 때 원인 단계를 훨씬 빠르게 특정할 수 있다. 이 로깅 구조는 나중에 추가하기가 어렵다. 처음 설계할 때 통제 지점으로 간주하고 포함시켜야 한다.
:::

RAG 실패를 아키텍처 결함으로 기술하는 능력은 단순한 진단 기술이 아니다. 이것은 RAG를 기능이 아니라 아키텍처로 이해한다는 것이 실무에서 무엇을 의미하는지를 보여주는 가장 구체적인 지점이다. "검색이 이상하다"고 말하는 팀과 "쿼리 변환 단계에서 의미 공간 불일치가 발생하고 있으며, 이는 색인 단계의 임베딩 모델 선택과 연관될 수 있다"고 말하는 팀은 같은 시스템을 보더라도 전혀 다른 수정 경로를 밟게 된다.

---

5장에서는 RAG가 왜 등장했는지, 각 단계가 정보 흐름의 어느 지점을 통제하는지, 그리고 실패가 어떻게 단계별 결함으로 재정의될 수 있는지를 살펴보았다. 이 논의를 관통하는 주제는 하나다. RAG는 검색 기능을 붙이는 것이 아니라, LLM 단독으로는 신뢰할 수 없는 정보 흐름을 외부 구조로 통제하는 아키텍처적 선택이다.

6장에서는 이 아키텍처가 실제로 어떻게 구성되는지를 더 구체적인 설계 판단의 수준에서 다룬다. RAG를 도입할지 말지, 도입한다면 어떤 방식으로 구성할지, 그리고 그 판단을 어떤 기준으로 내릴 것인지가 다음 장의 주제다.
