### MF-02 · temperature=0 서술 기술적 오류 수정
**출처:** technical 리뷰  
**문제:** 현재 서술은 temperature=0을 '샘플링 억제' 수단으로 단순 제시하며, 실제 구현에서 완전한 결정론이 보장되지 않는다는 사실을 누락하고 있다. OpenAI API는 seed 파라미터 없이 temperature=0이어도 재현 가능성(reproducibility)을 공식적으로 보장하지 않는다.  
**요구 조치:** 해당 문장을 다음 방향으로 재작성하라.
- temperature=0은 argmax decoding에 근접하여 가장 높은 확률의 토큰을 선택하는 경향을 보이지만, floating point 연산 순서나 타이 브레이킹 로직 등 구현 세부 사항에 따라 동일 입력에도 다른 출력이 나올 수 있다.
- API 기반 서비스에서 재현 가능성을 높이려면 temperature 외에 seed 파라미터를 함께 사용해야 하며, 이조차도 보장이 아닌 근사에 가깝다는 점을 명시하라.
- 박스 설명의 'temperature를 낮추면 비결정성을 줄일 수 있다'는 서술은 유지하되, '억제한다' · '제거한다'는 표현은 삭제하라.

### MF-03 · top-k / nucleus 샘플링 비결정성 원인 서술 재구성
**출처:** technical 리뷰  
**문제:** 현재 서술이 top-k와 nucleus(top-p)를 비결정성의 독립적 원인처럼 병렬 나열하고 있다. 두 샘플링 전략은 temperature > 0일 때 확률적 선택을 수행하는 메커니즘이며, 서로 다른 전략이다.  
**요구 조치:** 다음 구조로 재서술하라.
> "LLM은 다음 토큰을 생성할 때 확률 분포에서 샘플링하는 방식을 사용한다. temperature가 0보다 클 때, top-k 샘플링(확률 상위 k개 토큰 중 선택)이나 nucleus 샘플링(top-p, 누적 확률 p 이내 토큰 중 선택)과 같은 전략이 적용되며, 이로 인해 동일 입력에도 매 실행마다 다른 토큰이 선택될 수 있다."

top-k와 top-p 각각에 한 줄 한글 설명을 병기하라 (→ MF-05와 연동).

## 4. Optional Improvements (선택적 개선)

### OI-01 · 로컬 실행 시나리오 각주 처리

### OI-02 · 소프트웨어 엔지니어링 '테스트 하네스' 설명 보강
'소프트웨어 엔지니어링에는 테스트 하네스라는 오래된 개념이 있다'는 언급에 한 줄 설명을 추가하라. 예: "테스트 하네스는 실제 시스템 대신 테스트 환경에서 컴포넌트를 구동·검증하는 도구 집합을 가리킨다." 엔지니어링 배경이 없는 독자(PM, 규제 실무자)를 고려한 조치다.

### OI-03 · temperature 첫 등장 시 한글 선행 병기 검토
`temperature`를 '온도 파라미터(temperature)' 형태로 첫 등장 시 병기하는 것을 검토하라. tone_guide의 영문 단독 기술 용어 처리 규정과 충돌 여부를 확인한 뒤 결정하라.

### OI-04 · 계약서 예시 재참조 방식 간결화
2-3절에서 동일한 계약서 요약 시나리오를 재소환할 때, 앞 절의 사례를 알고 있다는 전제 하에 더 간결하게 참조하라. 예: "앞서 살펴본 계약서 요약 서비스를 다시 떠올려보자."

### [RI-02] 비결정성 절 — 샘플링 서술 재작성 (MF-02, MF-03 실행)

**위치:** '비결정성: AI가 소프트웨어와 다른 이유' 절 내 해당 문단  
**작업:**

**[삭제]** 현재 top-k / nucleus 병렬 나열 문장  
**[삭제]** temperature=0을 '샘플링 억제' 수단으로 단순 기술하는 문장  

**[삽입]** 아래 두 블록으로 대체하라.

**블록 1 — 샘플링 메커니즘 설명:**
> "LLM은 다음 토큰을 생성할 때 확률 분포에서 샘플링하는 방식을 사용한다. temperature가 0보다 클 때, 모델은 top-k 샘플링(상위 k개 후보 토큰 중 확률적으로 선택) 또는 nucleus 샘플링(누적 확률 p 이내의 토큰 집합에서 선택, top-p라고도 한다) 같은 전략을 적용한다. 이 때문에 동일한 입력을 넣어도 실행마다 다른 토큰이 선택될 수 있다. top-k와 nucleus는 서로 다른 전략이지만, 두 방식 모두 temperature와 함께 작동하며 비결정성의 독립적 원인이 아니다."

**블록 2 — temperature=0 한계 명시 (박스 설명 또는 본문 내 보충):**
> "temperature=0으로 설정하면 매 단계에서 가장 높은 확률의 토큰을 선택하는 argmax decoding에 근접한다. 그러나 이것이 완전한 결정론을 보장하지는 않는다. floating point 연산 순서나 타이 브레이킹 로직 같은 구현 세부 사항에 따라 동일 입력에도 다른 출력이 나올 수 있으며, API 기반 서비스에서 재현 가능성을 높이려면 temperature 외에 seed 파라미터(재현 가능성을 높이기 위한 난수 초기값)를 함께 사용해야 한다. 이 역시 보장이 아닌 근사임을 주의해야 한다."
