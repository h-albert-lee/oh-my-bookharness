# 1장. AI는 왜 항상 데모까지만 잘 되는가: 하네스의 부재

이 장에서는 AI 프로젝트에서 반복적으로 관찰되는 하나의 패턴을 들여다본다. 데모 환경에서 완벽하게 작동하던 AI가 실제 서비스에 올라간 순간 흔들리기 시작하는 현상이다. 이 장의 목적은 그 현상에 구조적 언어를 부여하는 것이다. 실패의 원인을 모델 성능이나 프롬프트 품질에서 찾는 통상적 반응이 왜 불충분한지를 살펴보고, "있어야 했는데 없었던 것"이 무엇인지를 가리키는 언어 — 하네스(Harness) — 를 처음으로 꺼내 놓는다.

이 장을 다 읽은 독자는 명확한 답 대신 명확한 질문 하나를 가져간다. "있어야 했는데 없었던 것은 무엇인가?" 1–2절에서 두 환경의 구조적 차이를 진단하고, 3절에서 AI 고유의 특성을 확인한 뒤, 4절에서 그 빈자리를 가리키는 언어 — 하네스 — 를 처음 꺼내 놓는다. 하네스의 구체적 설계는 2장에서 시작된다.

## 그날의 데모는 완벽했다

발표 전날 밤, 팀은 늦게까지 남아 질문 목록을 골랐다. 너무 모호하지 않고, 너무 구체적이지도 않은 것들. 모델이 잘 대답하는 유형으로만 추렸다. 다섯 개의 질문이 최종 목록에 올랐고, 각각에 대한 예상 출력도 미리 확인했다. 준비는 완벽했다.

다음 날 고객사 회의실에서 시연이 시작됐다. 계약서 요약 기능을 보여주는 자리였다. 첫 번째 질문을 입력하자 모델이 깔끔하게 핵심 조항을 정리해 냈다. 두 번째, 세 번째도 마찬가지였다. 화면 앞에서 담당자가 "이 정도면 실무에서 바로 쓸 수 있겠는데요"라고 했다. 박수가 나왔다. PoC(개념 검증, Proof of Concept)는 성공으로 마무리됐다.

두 달 후, 서비스가 오픈됐다. 첫 주가 지나기 전에 사내 메신저 채널에 리포트가 쌓이기 시작했다. "요약이 엉뚱한 조항을 골라서 보여준다", "같은 계약서인데 어제랑 오늘 결과가 다르다", "영문 혼합 문서를 넣었더니 아무 응답도 없다". 팀은 황당했다. 데모에서는 분명히 됐었다.

이 이야기는 특정 팀의 실수가 아니다. 비슷한 구조의 장면이 AI 프로젝트 곳곳에서 반복된다. 내부 시연용 챗봇을 만든 팀이 있다고 해보자. 개발 기간 동안 팀원들이 직접 테스트 질문을 입력했을 때는 응답 품질이 훌륭했다. 그런데 전사 배포 이후 일반 직원들이 입력하기 시작하자 품질이 눈에 띄게 떨어졌다. 입력 방식, 맥락 제공 방식, 질문의 형태가 완전히 달랐기 때문이다. 팀원들은 "좋은 질문"을 하도록 훈련된 사람들이었고, 실제 사용자는 그렇지 않았다.

이런 일이 반복되는 이유는 무엇일까. 모델이 부족해서일까, 프롬프트가 잘못 작성되어서일까. 이 장은 그 막연한 불편함에 언어를 부여하는 데서 시작한다. "왜 데모에서는 됐는데 실제로는 안 되는가"라는 경험의 구조적 원인을 차례로 따라가 보자.

## 데모 환경과 운영 환경은 다른 세계다

데모와 운영 사이의 괴리를 "운이 나빴다"거나 "버그가 있었다"고 설명하는 것은 현상의 기술이지 원인의 설명이 아니다. 두 환경은 구조적으로 다르게 설계되어 있으며, 그 차이는 같은 AI 모델을 사용하더라도 전혀 다른 행동을 유발할 수 있다.

가장 뚜렷한 차이는 입력의 다양성이다. 데모 환경에서는 팀이 미리 선별한 입력만 사용된다. 모델이 잘 처리하는 유형의 쿼리, 적절한 길이의 문서, 예상 가능한 형태의 요청이 입력된다. 운영 환경에서는 이 선별이 사라진다. 사용자는 빈 문자열을 입력하기도 하고, 수백 페이지짜리 문서를 통째로 붙여 넣기도 하며, 한국어와 영어가 뒤섞인 표에 대한 처리를 요청하기도 한다. 팀이 예측하지 못한 형태의 입력이 제한 없이 유입된다.

반복 실행도 다르다. 데모는 한 번 혹은 몇 번의 실행으로 끝난다. 운영 환경에서는 같은 종류의 요청이 하루에 수천 번 반복된다. 같은 입력이라도 실행마다 출력이 달라질 수 있는 AI 시스템에서, 반복 실행 횟수가 늘어날수록 예외적인 출력을 마주칠 확률도 높아진다. 데모에서 한 번도 보지 못했던 응답 패턴이 운영에서 반복적으로 나타나는 것은 이 때문이다.

실패의 비용도 전혀 다르다. 데모에서 예상과 다른 응답이 나왔을 때, 발표자는 "네트워크가 좀 느린 것 같네요"라고 넘어가거나 조용히 새로고침을 눌렀다. 운영 환경에서의 실패는 사용자 이탈로 이어지고, 서비스 신뢰도에 영향을 준다. 금융이나 의료 같은 규제 환경에서는 한 번의 잘못된 출력이 법적 책임 문제로 번질 수도 있다.

아래 표는 두 환경의 주요 특성을 정리한 것이다.

**표 1-1. 데모 환경과 운영 환경 비교**

| 항목 | 데모 환경 | 운영 환경 |
|---|---|---|
| 입력 제어 수준 | 팀이 선별한 입력 | 사용자가 자유롭게 입력 |
| 실행 횟수 | 소수의 시연용 실행 | 일 수천 건 이상의 반복 실행 |
| 판단 주체 | 사람이 실시간 판단 | 시스템이 자동으로 응답 |
| 실패 처리 방식 | 비공식적으로 덮거나 재시도 | 사용자에게 즉시 노출 |
| 성공 기준 | "보기에 좋은" 출력 | 측정 가능한 품질 기준(아직 정의되지 않은 경우가 많다) |

이 표에서 눈여겨볼 것은 모델 관련 항목이 하나도 없다는 점이다. 두 환경의 차이는 모델의 품질이나 성능과 무관하다. 동일한 모델을 사용하더라도 환경의 구조가 달라지면 관찰되는 동작이 달라진다. "더 좋은 모델로 바꾸면 해결된다"는 직관이 왜 충분하지 않은지, 그 첫 번째 근거가 여기에 있다.

환경의 구조적 차이를 확인했다면, 이제 그 차이를 더 위험하게 만드는 AI 고유의 특성을 살펴볼 차례다.

## 비결정성: AI가 소프트웨어와 다른 이유

전통적인 소프트웨어 함수는 결정론적(deterministic)이다. `sort([3, 1, 2])`를 실행하면 언제나 `[1, 2, 3]`이 반환된다. 어제도 그랬고, 내일도 그럴 것이며, 서버를 교체하거나 부하가 높아도 결과는 바뀌지 않는다. 이 예측 가능성이 전통적 소프트웨어 품질 관리의 토대다.

LLM(대규모 언어 모델, Large Language Model) 기반 AI 시스템은 다르다. "이 계약서의 위험 조항을 요약해줘"라는 동일한 요청을 동일한 모델에 열 번 보내면, 열 번의 출력이 미묘하게 혹은 뚜렷하게 다를 수 있다. 이것이 비결정성(Nondeterminism) — 동일한 입력에도 다른 출력이 나오는 AI 시스템의 본질적 특성 — 이다.

비결정성의 원인은 층위별로 다르다. 가장 근본적인 층위는 모델 자체의 확률적 샘플링이다. top-k 샘플링이나 nucleus 샘플링 방식은 동일한 입력에도 확률 분포에서 서로 다른 토큰을 선택한다. 그 위 층위에서는 API 서비스 운영 방식이 개입한다. OpenAI처럼 API로 제공되는 서비스에서는 모델 버전 업데이트나 서버사이드 배치 처리 — 여러 요청을 묶어 한꺼번에 처리하는 방식 — 자체가 동일 입력에 대한 출력 변동을 유발한다. 로컬 환경에서 직접 모델을 실행할 경우에는 GPU 병렬 연산의 비결정적 순서, 부동소수점 연산 차이도 추가로 작용한다.

> **[참고] temperature는 비결정성을 줄이는 도구이지 제거하는 도구가 아니다**
>
> temperature는 LLM이 다음 토큰을 선택할 때 적용하는 확률 분포의 온도를 조절하는 파라미터다. 0에 가까울수록 가장 확률이 높은 토큰만 선택하므로 출력이 더 일정해진다. 그러나 temperature=0 설정은 확률적 샘플링을 억제하지만, API 버전 변경이나 배치 처리 방식에서 비롯되는 변동까지 제거하지는 않는다. temperature를 낮추면 비결정성을 줄일 수 있지만, 구조적으로 제거할 수는 없다.

**그림 1-1. 결정론적 시스템과 비결정론적 시스템의 입출력 구조**

```
결정론적 시스템:
  입력 A  ──→  [ 함수 ]  ──→  출력 X
  입력 A  ──→  [ 함수 ]  ──→  출력 X  (언제나 동일)

비결정론적 시스템:
  입력 A  ──→  [ LLM ]  ──→  출력 X1
  입력 A  ──→  [ LLM ]  ──→  출력 X2
  입력 A  ──→  [ LLM ]  ──→  출력 X3  (분포 내에서 변동)
```

비결정성은 버그가 아니다. LLM의 설계 원리에서 비롯된 본질적 특성이며, 이를 완전히 제거하는 것은 현재 기술 수준에서 가능하지 않다. 한편 비결정성은 모델 수준에서 시작되지만, 시스템 수준에서 증폭된다. API 버전 업데이트나 배치 처리 방식은 모델 자체를 바꾸지 않아도 출력을 달라지게 만든다. 이 점이 중요하다. 비결정성의 뿌리가 모델에 있더라도, 그것이 문제로 발현되는 자리는 시스템 전체에 걸쳐 있다.

그렇다면 이 구조적 차이가 왜 운영에서만 치명적으로 드러나는가. 데모에서는 실행 횟수가 적고 입력이 통제된다. 출력이 조금 달라지더라도 사람이 실시간으로 보고 있으므로 이상한 응답은 조용히 걸러진다. 그러나 운영 환경에서는 수천 번의 실행이 자동으로 일어나고, 각각의 출력을 사람이 검토하지 않는다. 더 좋은 모델로 교체하거나 프롬프트를 정교하게 다듬어도, 확률적 샘플링과 API 버전 업데이트에서 비롯되는 비결정성은 구조적으로 남는다. 표 1-1이 보여주듯 두 환경의 차이는 모델 품질과 무관한 차원에서 발생하기 때문이다. 문제의 진원지가 모델이 아니라면, 해결의 출발점도 모델이 아니다.

비결정성은 데모에서도 동일하게 존재했다. 그것이 문제가 되지 않았던 이유는 무엇일까. 그 답이 다음 절에 있다.

## 데모를 성공시킨 비공식 구조

데모가 잘 됐던 이유를 다시 생각해보자. 발표 전날 밤, 팀은 "잘 나오는 질문"을 선별했다. 그 행위가 무엇인지 다시 이름 붙여보면 — 입력 통제다. 운영에서는 누가 이것을 하는가. 아무도 하지 않는다.

발표 당일, 발표자는 출력을 실시간으로 보면서 "이게 맞네요"라고 판단했다. 그 행위는 — 출력 평가다. 운영에서는 누가 이것을 하는가. 아무도 하지 않는다.

예상과 다른 응답이 나왔을 때, 발표자는 "네트워크 문제인 것 같아요"라고 넘어갔다. 그 행위는 — 실패 은폐이자 실패 처리다. 운영에서는 이 실패가 그대로 사용자에게 노출된다.

이 세 가지 — 입력 통제, 출력 평가, 실패 처리 — 를 데모 자리에서 수행한 것은 팀이었다. 의도적으로 설계한 것이 아니었다. 발표를 잘 마치려는 자연스러운 행동이 결과적으로 AI 시스템을 안정적으로 보이게 만드는 구조를 임시로 형성했다. 데모의 성공은 시스템의 성능이 아니라, 사람이 그 자리에서 제공하던 비공식 구조 덕분이었다.

이 비공식 구조는 서비스가 오픈된 순간 사라졌다. 팀원은 자리를 떠났고, 입력을 선별해주는 사람도, 출력을 판단해주는 사람도, 실패를 조용히 처리해주는 사람도 없어졌다. AI 시스템은 혼자서 세상과 마주하게 됐다. 그 순간부터 비결정성의 파장이 통제 없이 사용자에게 전달되기 시작했다.

흥미로운 것은, 이런 비공식 구조의 존재 자체를 팀이 인식하지 못하는 경우가 많다는 점이다. 발표 준비를 열심히 한 것이지, "하네스를 설계했다"는 인식은 없다. 그렇기 때문에 이 구조가 운영으로 이전되어야 한다는 생각도 자연스럽게 떠오르지 않는다. 데모가 성공한 것은 시스템이 안정적이어서가 아니라 사람이 안정성을 임시 제공했기 때문인데, 그 사실이 가려진 채 "우리 모델은 성능이 검증됐다"는 결론으로 마무리된다.

이 비공식 구조가 없어진 자리, 그 빈자리에 이름이 필요하다.

## 하네스라는 이름을 붙이기 전에

자동차 제조사가 신차를 도로에 내보내기 전에 하는 일을 생각해보자. 테스트 트랙에서 반복 주행한다. 노면 상태, 속도, 기후 조건을 통제하고, 주행 결과를 기록하며, 이상이 생기면 즉시 감지해서 수정한다. 트랙이 없으면 어떻게 될까. 실제 도로가 최초의 테스트 환경이 된다. 첫 번째 사용자가 품질 검증의 대상이 된다.

AI 시스템도 마찬가지다. 데모 자리에서 팀이 임시로 제공했던 구조 — 입력 통제, 출력 평가, 실패 처리 — 가 없다면, 운영 환경이 최초의 테스트 환경이 된다. 사용자가 시스템의 한계를 발견해주는 역할을 맡게 된다.

소프트웨어 엔지니어링에는 테스트 하네스(test harness)라는 오래된 개념이 있다. 원래는 코드 단위 테스트를 실행하고 결과를 수집하는 도구 묶음을 가리키는 말이다. 이 책에서 말하는 하네스(Harness)는 그보다 넓은 의미다. AI 시스템이 실행되는 과정 전체를 감싸고, 반복 가능한 방식으로 실행과 판단을 수행하는 외부 구조를 가리킨다. 테스트 도구와의 혼동을 피하기 위해, 이 책에서 하네스는 항상 'AI 시스템 외부에 위치하는 실행·평가 구조'를 의미한다.

하네스가 없다는 것은 구체적으로 어떤 상태인가. 세 가지로 정리할 수 있다. 첫째, AI가 실행되는 환경을 통제하는 외부 구조가 없다. 어떤 입력이 들어오든 그대로 모델에 전달된다. 둘째, 출력이 적절한지 판단하는 구조가 없다. 모델이 생성한 것이 바로 사용자에게 노출된다. 셋째, 실패가 발생했을 때 이를 가시화하고 처리하는 구조가 없다. 문제가 생겨도 기록되지 않고, 반복되어도 발견되지 않는다.

**그림 1-2. AI 모델 단독 실행(데모 구조)과 하네스가 감싸는 실행(운영 구조) 비교**

```
[데모 구조] — 사람이 비공식 하네스 역할을 수행
  입력(선별됨) ──→ [ AI 모델 ] ──→ 출력(사람이 판단)

┌──────────────────── 하네스(Harness) ────────────────────┐
│                                                        │
│  입력 ──→ [ 입력 통제 ] ──→ [ AI 모델 ] ──→ [ 출력 판단 ] ──→ 출력  │
│                │                          │            │
│                └────── [ 실패 처리 ] ◀────┘            │
│                        (오류 포착 및 재시도/폴백)        │
└────────────────────────────────────────────────────────┘
[운영 구조] — 하네스가 구조적으로 수행
```

하네스는 AI 모델 바깥에 존재하는 구조다. 모델 안에서 해결하려는 시도 — 더 나은 프롬프트, 더 큰 모델, 더 낮은 temperature — 와는 다른 층위에 있다. 이 차이가 핵심이다. 데모 자리에서 사람이 제공하던 구조를 시스템 수준에서 설계하는 것, 그것이 하네스의 역할이다.

이 장에서 하네스의 세부 아키텍처를 다루지는 않는다. 지금 독자에게 필요한 것은 이름과 윤곽이다. "데모는 됐는데 왜 운영에서 안 되지?"라는 경험에 이제 언어가 생겼다. 하네스가 없었기 때문이다.

## "모델이 문제다"는 왜 틀린 진단인가

서비스 장애가 발생하면 회의가 열린다. 그 자리에서 가장 자주 나오는 제안은 대체로 비슷하다. "GPT-4로 업그레이드하면 해결될 것 같다." "프롬프트를 다시 짜보자." "파인튜닝을 해봐야 할 것 같다." 모두가 고개를 끄덕인다. 가장 빠르고 직관적인 선택지처럼 보이기 때문이다.

이 진단이 왜 불충분한지 살펴보자. 동일한 모델을 사용하는 두 팀이 있다고 해보자. 두 팀 모두 GPT-4 기반의 계약서 분석 서비스를 운영한다. 한 팀의 서비스는 안정적으로 운영되고, 다른 팀의 서비스는 반복적인 품질 민원에 시달린다. 두 팀이 같은 모델을 쓴다면, 차이는 어디에서 오는가. 답은 입력 전처리 구조, 출력 후처리 구조, 실패 감지 및 폴백 설계에 있다. 즉, 하네스의 차이다.

비유를 하나 들어보자. 자동차 사고가 났다. 조사관이 원인을 분석한다. "엔진 마력이 부족했던 것 같습니다. 더 강력한 엔진으로 교체하면 해결될 겁니다." 그런데 실제 원인이 빙판길을 과속으로 달린 것이라면, 엔진 교체는 아무것도 해결하지 못한다. 문제를 잘못 짚은 것이다. AI 서비스 실패를 모델 성능 부족으로 귀인하는 것은 구조적으로 같은 오류다.[^TODO_sculley2015]

아래 표는 AI 프로젝트에서 흔히 등장하는 실패 귀인 오류의 유형과, 그에 앞서 실제로 확인해야 할 질문을 정리한 것이다.

**표 1-2. 실패 귀인 오류 유형**

| 흔한 진단 | 실제로 확인해야 할 질문 |
|---|---|
| "모델이 틀렸다" | 동일 모델이 다른 환경(다른 입력 통제 구조)에서도 같은 실패를 하는가? |
| "프롬프트가 부족했다" | 프롬프트 변경 없이도 실패 패턴이 달라지는 조건이 있는가? |
| "데이터가 나빴다" | 입력 통제 구조가 없었던 것은 아닌가? |
| "이번에만 운이 나빴다" | 같은 실패가 재현되는지, 동일한 입력과 조건으로 반복 실행하여 확인했는가? |

이 표의 오른쪽 열이 공통으로 가리키는 것은 시스템 설계다. 모델이 아니다. 모델 교체나 프롬프트 수정은 가장 빠른 선택지이지만, 하네스 부재라는 구조적 원인을 건드리지 않는다. 실패를 모델이나 개인 역량에 귀인하는 것은 같은 실패의 반복을 허용한다.

이 지점에서 두 가지 극단적 태도를 경계할 필요가 있다. 하나는 "AI는 원래 불안정하다"는 숙명론이다. 비결정성이 존재한다는 사실이 안정적인 운영을 포기해야 한다는 의미는 아니다. 다른 하나는 "모델만 완벽하면 모든 것이 해결된다"는 낙관론이다. 앞서 살펴봤듯, 두 환경의 구조적 차이는 모델 품질과 무관한 차원에서 발생한다. 이 장이 가리키는 방향은 세 번째다. 실패를 설계 가능한 문제로 바라보는 관점이다.

## 빠진 것에 이름을 붙이는 것부터

처음 장면으로 잠시 돌아가보자. 고객사 회의실에서 팀이 다섯 개의 질문으로 완벽한 데모를 마쳤던 그 순간으로. 지금의 언어로 그 자리를 다시 바라보면, 무엇이 보이는가. 팀은 입력을 통제했고, 출력을 실시간으로 판단했으며, 실패를 조용히 처리했다. 의도하지 않았지만, 사람이 하네스 역할을 하고 있었다.

서비스가 오픈된 순간, 그 사람들이 사라졌다. 하네스가 없는 상태로 AI 시스템이 혼자 세상과 마주하게 됐다. 슬랙 채널에 쌓이기 시작한 리포트는 버그 보고가 아니었다. 없어야 할 것이 있거나, 있어야 할 것이 없다는 신호였다.

엔지니어링 세계에서 문제에 이름을 붙이는 것은 단순한 명명(命名) 이상의 의미를 갖는다. 이름이 있어야 설계를 시작할 수 있다. 이름 없이 "왠지 불안정한 것 같다"는 느낌으로는 무엇을 고쳐야 하는지 알 수 없다. "하네스가 없다"는 진단은 다음 질문을 자연스럽게 열어준다. 그렇다면 하네스를 어떻게 만드는가. 어떻게 설계하는가. 어떻게 운영에 포함시키는가.

이 책의 나머지 부분이 그 질문에 답한다. 단, 그 전에 해결해야 할 선행 질문이 하나 있다. AI 시스템의 품질을 보증하는 데 기존의 방법론 — 소프트웨어 개발에서 검증된 테스트 주도 개발(TDD) 방식 — 은 왜 충분하지 않은가. 비결정성을 가진 시스템에서 "Assert Equal"이 무너지는 이유가 무엇인가. 2장에서는 이 질문을 들고 기존 소프트웨어 테스트 방법론이 AI 시스템에 왜 그대로 적용되지 않는지를 따라간다. 하네스를 설계하는 방법은 그 이후에 비로소 제대로 논의할 수 있다.
