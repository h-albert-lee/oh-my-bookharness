# 1장. AI는 왜 항상 데모까지만 잘 되는가: 하네스의 부재

데모 환경에서 완벽하게 작동하던 AI가 실제 서비스에 올라간 순간 흔들리기 시작하는 현상은 특정 팀의 실수가 아니라 구조의 문제다. 이 장은 그 구조적 원인을 가리키는 언어 — 하네스(Harness) — 를 처음 꺼내 놓는다. 이 장을 다 읽은 독자는 "있어야 했는데 없었던 것은 무엇인가"라는 질문을 명확하게 가져갈 수 있다. 1절과 2절에서 두 환경의 구조적 차이를 진단하고, 3절에서 AI 고유의 특성을 확인한 뒤, 4절에서 데모를 실제로 작동시켰던 비공식 구조의 정체를 드러내고, 5절에서 하네스를 처음 꺼내 놓는다. 하네스를 설계하기 전에 먼저 AI 시스템에서 '테스트'가 의미하는 바를 재정의해야 한다. 그 질문은 다음 장에서 시작된다.

---

## 1.1 그날의 데모는 완벽했다

발표 전날 밤, 팀은 늦게까지 남아 질문 목록을 골랐다. 너무 모호하지 않고, 너무 구체적이지도 않은 것들. 모델이 잘 대답하는 유형으로만 추렸다. 다섯 개의 질문이 최종 목록에 올랐고, 각각에 대한 예상 출력도 미리 확인했다. 준비는 완벽했다.

다음 날 고객사 회의실에서 시연이 시작됐다. 계약서 요약 기능을 보여주는 자리였다. 첫 번째 질문을 입력하자 모델이 핵심 조항을 깔끔하게 정리해 냈다. 두 번째, 세 번째도 마찬가지였다. 화면 앞에서 담당자가 "이 정도면 실무에서 바로 쓸 수 있겠는데요"라고 했다. 박수가 나왔다. PoC(개념 검증, Proof of Concept)는 성공으로 마무리됐다.

두 달 후, 서비스가 오픈됐다. 첫 주가 지나기 전에 사내 메신저 채널에 리포트가 쌓이기 시작했다. "요약이 엉뚱한 조항을 골라서 보여준다", "같은 계약서인데 어제랑 오늘 결과가 다르다", "영문 혼합 문서를 넣었더니 아무 응답도 없다". 팀은 황당했다. 데모에서는 분명히 됐었다.

이 이야기는 특정 팀의 실수가 아니다. 비슷한 구조의 장면이 AI 프로젝트 곳곳에서 반복된다. 내부 시연용 챗봇을 만든 팀이 있다고 해보자. 개발 기간 동안 팀원들이 직접 테스트 질문을 입력했을 때는 응답 품질이 훌륭했다. 그런데 전사 배포 이후 일반 직원들이 입력하기 시작하자 품질이 눈에 띄게 떨어졌다. 입력 방식, 맥락 제공 방식, 질문의 형태가 완전히 달랐기 때문이다. 팀원들은 "좋은 질문"을 하도록 훈련된 사람들이었고, 실제 사용자는 그렇지 않았다.

이런 일이 반복되는 이유는 무엇일까. 모델이 부족해서일까, 프롬프트가 잘못 작성되어서일까. 이 장은 그 막연한 불편함에 언어를 부여하는 데서 시작한다. "왜 데모에서는 됐는데 실제로는 안 되는가"라는 경험의 구조적 원인을 차례로 따라가 보자.

## 1.2 데모 환경과 운영 환경은 다른 세계다

데모와 운영 사이의 괴리를 "운이 나빴다"거나 "버그가 있었다"고 설명하는 것은 현상의 기술이지 원인의 설명이 아니다. 두 환경은 구조적으로 다르게 설계되어 있으며, 그 차이는 같은 AI 모델을 사용하더라도 전혀 다른 행동을 유발할 수 있다.

가장 뚜렷한 차이는 입력의 다양성이다. 데모 환경에서는 팀이 미리 선별한 입력만 사용된다. 모델이 잘 처리하는 유형의 쿼리, 적절한 길이의 문서, 예상 가능한 형태의 요청이 입력된다. 운영 환경에서는 이 선별이 사라진다. 사용자는 빈 문자열을 입력하기도 하고, 수백 페이지짜리 문서를 통째로 붙여 넣기도 하며, 한국어와 영어가 뒤섞인 표에 대한 처리를 요청하기도 한다. 팀이 예측하지 못한 형태의 입력이 제한 없이 유입된다.

반복 실행도 다르다. 데모는 한 번 혹은 몇 번의 실행으로 끝난다. 운영 환경에서는 같은 종류의 요청이 하루에 수천 번 반복된다. 같은 입력이라도 실행마다 출력이 달라질 수 있는 AI 시스템에서, 반복 실행 횟수가 늘어날수록 예외적인 출력을 마주칠 확률도 높아진다. 데모에서 한 번도 보지 못했던 응답 패턴이 운영에서 반복적으로 나타나는 것은 이 때문이다.

실패의 비용도 전혀 다르다. 데모에서 예상과 다른 응답이 나왔을 때, 발표자는 "네트워크가 좀 느린 것 같네요"라고 넘어가거나 조용히 새로고침을 눌렀다. 운영 환경에서의 실패는 사용자 이탈로 이어지고, 서비스 신뢰도에 영향을 준다. 금융이나 의료 같은 규제 환경에서는 한 번의 잘못된 출력이 법적 책임 문제로 번질 수도 있다.

아래 표는 두 환경의 주요 특성을 정리한 것이다.

**표 1-1. 데모 환경과 운영 환경 비교**

| 항목 | 데모 환경 | 운영 환경 |
|---|---|---|
| 입력 제어 수준 | 팀이 선별한 입력 | 사용자가 자유롭게 입력 |
| 실행 횟수 | 소수의 시연용 실행 | 일 수천 건 이상의 반복 실행 |
| 판단 주체 | 사람이 실시간 판단 | 시스템이 자동으로 응답 |
| 실패 처리 방식 | 비공식적으로 덮거나 재시도 | 사용자에게 즉시 노출 |
| 성공 기준 | "보기에 좋은" 출력 | 측정 가능한 품질 기준 |

> 운영 초기에는 측정 가능한 품질 기준 자체가 부재한 경우가 적지 않다. 기준 없이 운영을 시작하면 실패가 발생해도 얼마나 실패했는지 알기 어렵다.

이 표에서 눈여겨볼 것은 모델 관련 항목이 하나도 없다는 점이다. 두 환경의 차이는 모델의 품질이나 성능과 무관하다. 동일한 모델을 사용하더라도 환경의 구조가 달라지면 관찰되는 동작이 달라진다. "더 좋은 모델로 바꾸면 해결된다"는 직관이 왜 충분하지 않은지, 그 첫 번째 근거가 여기에 있다.

환경의 구조적 차이를 확인했다면, 이제 그 차이를 더 위험하게 만드는 AI 고유의 특성을 살펴볼 차례다.

## 1.3 비결정성: AI가 소프트웨어와 다른 이유

전통적인 소프트웨어 함수는 결정론적(deterministic)이다. `sort([3, 1, 2])`를 실행하면 언제나 `[1, 2, 3]`이 반환된다. 어제도 그랬고, 내일도 그럴 것이며, 서버를 교체하거나 부하가 높아도 결과는 바뀌지 않는다. 이 예측 가능성이 전통적 소프트웨어 품질 관리의 토대다.

LLM(대규모 언어 모델, Large Language Model) 기반 AI 시스템은 다르다. "이 계약서의 위험 조항을 요약해줘"라는 동일한 요청을 동일한 모델에 열 번 보내면, 열 번의 출력이 미묘하게 혹은 뚜렷하게 다를 수 있다. 이것이 비결정성(Nondeterminism) — 동일한 입력에도 다른 출력이 나오는 AI 시스템의 본질적 특성 — 이다.

비결정성의 가장 근본적인 원인은 모델 자체의 확률적 토큰 생성 방식이다. LLM은 다음 토큰을 생성할 때 확률 분포에서 샘플링하는 방식을 사용한다. temperature(출력 확률 분포의 날카로움을 조절하는 파라미터)가 0보다 클 때, 모델은 top-k 샘플링(상위 k개 후보 토큰 중 확률적으로 선택) 또는 nucleus 샘플링(누적 확률 p 이내의 토큰 집합에서 선택, top-p라고도 한다) 같은 전략을 적용한다. 이 때문에 동일한 입력을 넣어도 실행마다 다른 토큰이 선택될 수 있다. top-k와 nucleus는 서로 다른 전략이지만, 두 방식 모두 temperature와 함께 작동하며 비결정성의 독립적 원인이 아니라 확률적 샘플링이라는 하나의 메커니즘 안에서 작동한다.

그 위 층위에서는 서비스 운영 방식이 개입한다. API로 제공되는 LLM 서비스에서는 모델 버전 업데이트나 서버사이드 배치 처리 — 여러 요청을 묶어 한꺼번에 처리하는 방식 — 자체가 동일 입력에 대한 출력 변동을 유발할 수 있다. 로컬 환경에서 직접 모델을 실행하는 경우에는 GPU 병렬 연산에서 부동소수점(floating point) 연산이 결합 법칙을 따르지 않는 특성, 그리고 처리 순서 차이가 추가로 작용한다.

> **[참고] temperature는 비결정성을 줄이는 도구이지 제거하는 도구가 아니다**
>
> temperature=0으로 설정하면 매 단계에서 가장 높은 확률의 토큰을 선택하는 argmax decoding에 근접한다. 출력이 더 일정해지는 경향이 있다는 것은 사실이다. 그러나 이것이 완전한 결정론을 보장하지는 않는다. 부동소수점 연산 순서나 동일 확률 토큰 간의 타이 브레이킹 로직 같은 구현 세부 사항에 따라 동일 입력에도 다른 출력이 나올 수 있다. API 기반 서비스에서 재현 가능성(reproducibility)을 높이려면 temperature 외에 seed(재현 가능성을 높이기 위한 난수 초기값 파라미터)를 함께 사용하는 것이 권장되지만, 이 역시 보장이 아닌 근사에 가깝다는 점을 주의해야 한다. temperature를 낮추면 비결정성을 줄일 수 있지만, 구조적으로 제거할 수는 없다.

**그림 1-1. 결정론적 시스템과 비결정론적 시스템의 입출력 구조**

```
결정론적 시스템:
  입력 A  ──→  [ 함수 ]  ──→  출력 X
  입력 A  ──→  [ 함수 ]  ──→  출력 X  (언제나 동일)

비결정론적 시스템:
  입력 A  ──→  [ LLM ]  ──→  출력 X₁
  입력 A  ──→  [ LLM ]  ──→  출력 X₂
  입력 A  ──→  [ LLM ]  ──→  출력 X₃  (분포 내에서 변동)
```

비결정성은 버그가 아니다. LLM의 설계 원리에서 비롯된 본질적 특성이다. 그리고 바로 이 특성이 앞 절에서 살펴본 환경의 구조적 차이와 결합할 때 데모-운영 괴리가 증폭된다. 데모 환경에서는 실행 횟수가 적고 입력이 통제되어 있어서, 비결정성이 만들어내는 출력의 분산이 표면에 드러나지 않는다. 운영 환경에서는 수천 번의 반복 실행과 통제되지 않은 다양한 입력이 결합되면서, 비결정성이 만들어내는 변동이 직접적으로 사용자에게 노출된다.

그렇다면 역방향으로 질문해볼 필요가 있다. 비결정성이 데모 환경에서도 동일하게 존재했다면, 데모는 왜 잘 됐는가?

## 1.4 데모를 성공시킨 비공식 구조

데모의 성공에는 모델의 성능 외에 다른 무언가가 있었다. 그것은 팀이 의식하지 못한 채 수행하고 있던 세 가지 역할이다.

첫째는 입력 통제였다. 앞서 언급했듯이, 팀은 발표 전날 밤 질문 목록을 골랐다. 이 행위는 단순한 준비가 아니었다. 모델이 잘 처리하는 입력의 범위 안으로 실행 환경을 제한하는 입력 통제였다. 무한한 입력의 가능성을 다섯 개의 선별된 쿼리로 좁혀 놓은 것이다. 사람이 이 역할을 수행하고 있었기 때문에 비결정성이 만들어내는 출력 변동이 문제가 될 만한 입력 자체가 제거됐다.

둘째는 출력 평가였다. 발표자가 모델의 응답을 화면에서 보며 "맞네요, 잘 됐습니다"라고 판단하는 행위가 있었다. 이것은 출력의 적절성을 실시간으로 판단하는 평가였다. 자동화된 기준이 없었지만, 발표 맥락에서 어떤 출력이 "좋은" 출력인지를 판단할 수 있는 사람이 그 자리에 있었다.

셋째는 실패 처리였다. 시연 중 예상과 다른 응답이 나왔을 때, 팀은 조용히 새로고침하거나 "이 예시는 조금 특수한 케이스라서요"라고 넘어갔다. 이것은 실패를 사용자에게 노출하지 않고 내부에서 흡수하는 실패 처리였다. 운영 환경에서는 이 역할을 할 사람이 없다.

데모를 성공시킨 비공식 구조는 결국 세 가지로 요약된다. 팀이 유리한 입력만 골라 사용한 입력 통제, 모델 출력을 사람이 실시간으로 검토한 출력 평가, 그리고 예상과 다른 결과를 조용히 덮거나 재시도한 실패 처리다.

중요한 것은 이 구조가 의도적으로 설계된 것이 아니었다는 점이다. 팀은 "우리가 지금 비공식 하네스를 운영하고 있다"고 생각하지 않았다. 발표를 잘 하기 위해 자연스럽게 행동했을 뿐이다. 그래서 서비스가 오픈된 순간, 이 구조는 흔적도 없이 사라졌다. 입력을 골라주던 사람이 사라지고, 출력을 판단하던 사람이 사라지고, 실패를 처리하던 사람이 사라졌다. AI 시스템은 그 어떤 외부 구조도 없이 혼자 세상과 마주하게 됐다.

이 관찰이 이 장의 핵심 질문으로 이어진다. 데모를 성공시켰던 그 구조 — 의도하지 않았지만 실제로 작동하고 있었던 그것 — 을 무엇이라고 불러야 할까.

## 1.5 하네스가 있어야 할 자리

소프트웨어 엔지니어링에는 테스트 하네스(test harness)라는 오래된 개념이 있다. 테스트 하네스는 실제 시스템 대신 통제된 환경에서 컴포넌트를 구동하고 검증하는 도구 집합을 가리킨다. 이 개념을 AI 시스템으로 확장하면 새로운 의미가 생긴다.

하나의 비유로 접근해보자. 자동차 제조사는 신차를 도로에 내보내기 전에 테스트 트랙에서 수만 킬로미터를 주행한다. 테스트 트랙은 실제 도로와 다르다. 노면 상태를 통제할 수 있고, 코너의 각도를 설정할 수 있으며, 극한 기상 조건을 재현할 수 있다. 주행 중 이상이 발생하면 즉시 포착되고 기록된다. 이 트랙이 없다면, 실제 도로가 최초의 테스트 환경이 된다. 사용자가 처음으로 차를 몰고 나가는 순간이 결함을 발견하는 순간이 되는 것이다.

AI 시스템에서 하네스(Harness)란 바로 이 역할을 하는 외부 구조다. 하네스(평가 하네스)는 반복 가능하게 실행과 평가를 수행하는 외부 구조다. 모델 안에 있는 것이 아니라, 모델을 감싸는 층위에 존재한다. 모델이 어떤 입력을 받는지, 그 출력이 적절한지, 실패가 발생했을 때 이를 어떻게 처리할지를 관리하는 구조다.

데모 환경에서는 이 구조가 사람에 의해 비공식적으로 채워졌다. 입력을 선별하는 사람, 출력을 평가하는 사람, 실패를 처리하는 사람이 하네스의 역할을 수행하고 있었다. 문제는 그것이 비공식적이었기 때문에, 시스템의 일부로 설계되지 않았기 때문에, 운영으로 이전될 수 없었다는 점이다.

**그림 1-2. AI 모델 단독 실행(데모 구조)과 하네스가 감싸는 실행(운영 구조) 비교**

```
[데모 구조 — 비공식]
  사람이 선별한 입력
        │
        ▼
    [ AI 모델 ]
        │
        ▼
  사람이 판단한 출력
  (실패는 조용히 처리)

[하네스가 있는 구조]
  ┌─────────────────────────────┐
  │           하네스              │
  │  ┌──────────────────────┐   │
  │  │  입력 관리           │   │
  │  └──────────┬───────────┘   │
  │             ▼               │
  │         [ AI 모델 ]         │
  │             │               │
  │  ┌──────────▼───────────┐   │
  │  │  출력 평가           │   │
  │  └──────────┬───────────┘   │
  │             ▼               │
  │  ┌──────────────────────┐   │
  │  │  오류 포착 및 재처리  │   │
  │  └──────────────────────┘   │
  └─────────────────────────────┘
```

이 그림에서 하네스는 모델 외부를 감싸는 구조다. 모델 안을 바꾸는 것 — 더 좋은 가중치, 더 정교한 프롬프트 — 과는 다른 층위에서 작동한다. 하네스가 없다는 것은 구체적으로 다음과 같은 상태를 가리킨다. AI가 실행되는 환경을 통제하는 외부 구조가 없고, 출력이 적절한지 판단하는 구조가 없으며, 실패가 발생했을 때 이를 가시화하고 처리하는 구조가 없는 상태다.

하네스의 세부 아키텍처가 어떻게 구성되는지, 어떻게 설계하는지는 이 장에서 다루지 않는다. 지금 독자에게 필요한 것은 구현의 상세가 아니라 이름과 윤곽이다. "있어야 했는데 없었던 것"에 이름이 생겼다.

## 1.6 "모델이 문제다"는 왜 틀린 진단인가

서비스 장애 이후의 회의를 상상해보자. 누군가가 먼저 말을 꺼낸다. "GPT-4로 업그레이드하면 나아지지 않을까요?" 테이블 주변에서 고개를 끄덕이는 사람들이 보인다. 이것이 AI 프로젝트 실패 이후 가장 자주 등장하는 반응이다. 모델이 문제라는 진단, 그리고 더 좋은 모델로 교체하면 해결된다는 처방.

이 반응이 왜 불충분한지를 살펴보자. 앞서 다룬 계약서 요약 서비스를 다시 떠올려보자. 그 서비스가 실패한 이유는 모델의 성능이 부족해서가 아니었다. 동일한 모델이 데모 환경에서는 잘 작동했다. 환경이 달라졌고 — 입력이 통제되지 않고, 반복 실행이 늘어났으며, 실패를 처리하는 구조가 없었다 — 그때 문제가 드러났다. 모델을 GPT-4로 교체한다면 일부 케이스에서는 응답 품질이 나아질 수 있다. 그러나 입력 통제 구조가 없고, 출력 평가 구조가 없으며, 실패 처리 구조가 없다는 사실은 변하지 않는다. 같은 실패가 조금 나중에, 조금 다른 형태로 다시 나타날 가능성이 높다.

동일한 모델을 사용하는 두 팀의 사례를 비교해보면 이 점이 더 선명해진다. A팀과 B팀이 같은 LLM API를 사용해 유사한 문서 분석 서비스를 운영하고 있다고 해보자. A팀의 서비스는 출시 두 달 만에 사용자 불만이 쌓이기 시작했다. B팀의 서비스는 같은 기간 동안 안정적으로 운영됐다. 두 팀이 다른 모델을 쓴 것이 아니었다. A팀과 B팀의 차이는 입력 전처리 구조, 출력 후처리 구조, 그리고 비정상 응답을 감지하고 처리하는 구조에 있었다. B팀에는 그 구조가 있었고, A팀에는 없었다.

실패를 모델이나 개인 역량에 귀인하는 것은 같은 실패의 반복을 허용한다. 왜냐하면 진단이 잘못되었기 때문이다. 아래 표는 AI 프로젝트에서 자주 등장하는 귀인 오류 유형과 그에 대응하는 올바른 질문을 정리한 것이다.

**표 1-2. 실패 귀인 오류 유형**

| 흔한 진단 | 실제로 확인해야 할 질문 |
|---|---|
| "모델이 틀렸다" | 동일한 모델이 구조화된 환경에서도 같은 실패를 하는가? 입력 통제와 출력 평가 구조가 있었는가? |
| "프롬프트가 부족했다" | 프롬프트 변경 없이도 다른 실행 환경에서는 실패 패턴이 달라지는가? |
| "데이터가 나빴다" | 입력 통제 구조가 없어서 비정상 입력이 그대로 모델에 전달된 것은 아닌가? |
| "모델을 업그레이드해야 한다" | 더 나은 모델을 넣어도 하네스 없이 운영하면 다른 형태의 실패가 재현될 가능성은 없는가? |
| "사용자가 잘못 입력했다" | 사용자의 다양한 입력을 감당할 수 있는 입력 관리 구조가 설계되어 있었는가? |

이 표의 오른쪽 열에 있는 질문들은 모두 모델 바깥을 향한다. 하네스의 부재를 가리키는 질문들이다.

"AI는 원래 불안정하다"는 체념도, "모델만 완벽하면 다 해결된다"는 낙관도 이 장이 도달하려는 지점이 아니다. 실패의 원인을 설계 가능한 문제로 바라보는 관점이 필요하다. 그 관점의 출발점이 하네스다.

## 1.7 빠진 것에 이름을 붙이는 것부터

이 장의 첫 장면으로 잠깐 돌아가보자. 발표 전날 밤, 팀은 질문 목록을 골랐다. 발표 당일, 발표자는 응답을 보며 실시간으로 판단했다. 중간에 예상과 다른 응답이 나왔을 때, 팀은 조용히 넘어갔다. 그리고 두 달 후, 그 모든 역할을 수행하던 사람들이 사라진 자리에 AI 시스템 혼자 남겨졌다.

무엇이 없었는가. 반복 가능하게 실행과 평가를 수행하는 외부 구조가 없었다. 비공식적으로, 임시로, 사람에 의해 채워지던 그 구조가 없었다. 이름이 없었기 때문에 설계되지 않았고, 설계되지 않았기 때문에 운영으로 이전되지 못했다.

엔지니어링 세계에서 문제에 이름을 붙이는 것은 단순한 명명 행위가 아니다. 이름이 있어야 설계할 수 있고, 설계할 수 있어야 개선할 수 있다. "우리 서비스가 왜 이렇게 불안정하죠?"라는 질문과 "하네스가 없어서 입력 통제와 출력 평가 구조가 빠져 있습니다"라는 진단 사이에는 실천 가능성의 차이가 있다. 전자는 막연하고, 후자는 다음 행동으로 이어진다.

이 장에서 하네스의 세부 아키텍처를 설명하지 않은 것은 의도적이다. 지금 중요한 것은 구현의 상세가 아니라, "있어야 했는데 없었던 것"에 이름을 붙이는 것이다. AI 시스템의 실패는 설계 가능한 문제다. 숙명이 아니다.

하네스가 무엇이고 어떻게 만드는지는 이 책 나머지의 출발점이다. 그 전에 먼저 물어야 할 질문이 하나 있다. AI 시스템의 품질을 어떻게 정의하고 검증할 것인가. 전통적인 소프트웨어 개발에서 이 질문에 답해온 방식 — 테스트 주도 개발(TDD, Test-Driven Development) — 이 AI 환경에서 왜 그대로 적용되기 어려운지, 그리고 어떤 새로운 접근이 필요한지가 다음 장의 주제다.
