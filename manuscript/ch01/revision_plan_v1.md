## 2. Must Fix (반드시 수정) — 이번 사이클 8건

> 우선순위 순 정렬. 모두 출판 전 필수 해결 사항이다.

### MF-01. 표 1-2 마지막 행 미완성 문장 복구
**문제:** `'이번에만 운이 나빴다' | 같은 실패가 재현되는` — 문장이 중간에 잘림. 원고 탈락 오류.  
**조치:** '실제로 확인해야 할 질문' 셀을 완성된 문장으로 복구. 예: "같은 실패가 재현되는지 동일 조건에서 반복 실행으로 확인했는가?"

### MF-02. `[^concept_dictionary]` 편집 마커 즉시 제거
**문제:** `'비결정성(Nondeterminism)이다.[^concept_dictionary]'` — 내부 작업 마커가 본문에 노출된 상태.  
**조치:** 각주 내용이 준비된 경우 실제 각주로 교체. 미준비 상태면 마커를 즉시 삭제하고, 비결정성 개념 보충 설명은 본문 내 괄호 또는 사이드바로 흡수.

### MF-03. 비결정성 원인 설명의 기술적 정확성 보정 (API 환경 중심으로)
**문제:** 초안은 비결정성 원인으로 '부동소수점 연산 차이, 서버 환경, 배치 처리 방식, 인프라 설정'을 나열하나, 주요 독자 컨텍스트(API 기반 서비스)에서 훨씬 지배적인 원인인 **모델 버전 업데이트, 서버사이드 배치 처리(server-side batching), top-k 샘플링의 확률적 특성**이 누락되었다.  
**조치:** 원인 목록을 다음과 같이 재구성. "API 서비스에서 비결정성은 주로 세 가지 층위에서 발생한다: ①모델 자체의 확률적 샘플링(top-k·nucleus sampling), ②API 레이어의 모델 버전 업데이트, ③서버사이드 배치 처리 방식의 차이. 로컬 추론 환경에서는 부동소수점 연산 순서와 GPU 병렬 처리 비결정성이 추가로 작용한다." 이 순서로 서술하면 API 독자와 로컬 독자 모두를 커버할 수 있다.

### MF-04. 도입부 2문단 편집 메모형 문장 교체
**문제:** `"해법의 상세한 서술은 이 장의 범위 밖이다"` — 편집 메모 어투. 기술서 도입부 톤에 부적합.  
**조치:** 독자 지향형 문장으로 교체. 예: "이 장을 다 읽은 독자는 명확한 답 대신 명확한 질문 하나를 가져간다. 그 질문이 2장에서 다룰 방법론의 출발점이 된다."

### MF-05. 선언문 스타일 당위 표현 제거
**문제:** `"'더 확실한 모델'이나 '더 정교한 프롬프트'로 비결정성을 제거하려는 시도는 방향 자체를 다시 검토할 필요가 있다"` — tone_guide 금지 항목(선언문 스타일 당위 표현).  
**조치:** 구체적 근거 서술로 교체. 예: "더 좋은 모델로 교체하거나 프롬프트를 정교하게 다듬어도, 확률적 샘플링과 API 버전 업데이트에서 비롯되는 비결정성은 구조적으로 남는다. 표 1-1이 보여주듯 두 환경의 차이는 모델 품질과 무관한 차원에서 발생하기 때문이다."

### MF-06. 장 제목 부제 표기 방식 통일
**문제:** 장 제목 `하네스(Harness)의 부재`에서 영문 병기 → 본문 첫 등장 시 재병기 불필요해짐. 편집 방침과 충돌.  
**조치:** 둘 중 하나를 선택하고 전체에 일관 적용. **권장안:** 장 제목은 `하네스의 부재`로 영문 제거, 본문 첫 등장 시 `하네스(Harness)`로 병기. 이후 본문에서는 '하네스'만 사용. (편집 방침 확정 후 전수 교열 필요)

### MF-07. 그림 1-2 ASCII 다이어그램 구조 재설계 또는 교체
**문제:** 다이어그램 내부 파이프라인 라인이 하네스 박스 너비를 초과, 인쇄·전자책 조판 시 깨짐 가능성. `[실패 처리]` 피드백 화살표(↑) 참조 대상 불명확.  
**조치(단기):** ASCII 박스를 다음 원칙으로 재설계. ①파이프라인을 수평이 아닌 **수직 또는 2행 분할** 구조로 변경해 박스 너비를 80자 이내로 제한. ②`[실패 처리]` 화살표의 연결 대상을 명시적으로 레이블. **조치(중기):** 편집 단계에서 벡터 도식으로 교체 예약.

### MF-08. 입력 공간 '무한' 표현 기술적 완화
**문제:** `"입력 공간(input space)이 사실상 무한하게 열린다"` — 컨텍스트 윈도우 한계, 토큰 한도, API 입력 제한 등 실제 물리적 경계를 무시한 과장 표현.  
**조치:** `"통제되지 않은 광범위한 입력 공간으로 열린다"` 또는 `"팀이 예측하지 못한 형태의 입력이 제한 없이 유입된다"`로 완화.

## 3. Should Fix (권장 수정) — 다음 사이클 우선

### SF-01. 비결정성의 층위 구분 (모델 레벨 vs. 시스템 레벨)
비결정성이 모델 자체의 특성인지 시스템 레벨 특성인지 구분하는 한 문장을 삽입하면 "모델 교체로 해결 안 된다"는 핵심 논지가 강화된다. 예: "비결정성은 모델 수준에서 시작되지만, 시스템 수준에서 증폭된다. API 버전 업데이트나 배치 처리 방식은 모델 자체를 바꾸지 않아도 출력을 달라지게 만든다."

### SF-02. '모델'과 'AI 시스템' 혼용 정비
본문 내 '모델'과 'AI 시스템'을 혼용 중. SF-01 대응과 함께, 등장할 때마다 지시 대상을 명확히 하거나 문서 상단에 용어 사용 기준을 각주로 명시.

### SF-03. EDD 개념 1장 말미 예고 삽입
argument_map의 핵심 논제인 EDD가 1장 본문에 전혀 등장하지 않음. 마지막 절 이후 브리지 문단 1개 추가. 예: "이 구조적 문제가 개발 방법론까지 어떻게 바꾸는지 — 그 이야기는 2장에서 시작된다."

### SF-04. 2장 연결 브리지 문단 추가
장 말미에 독자가 다음 장에서 무엇을 기대해야 하는지 한두 문장 안내. 현재 `"이 질문이 이 책 나머지 부분의 출발점이 된다"`는 너무 추상적. 예: "2장에서는 이 질문을 들고 기존 소프트웨어 테스트 방법론이 AI 시스템에 왜 적용되지 않는지를 따라간다."

### SF-05. '데모가 성공한 진짜 이유' 소제목 중립화
`"진짜 이유"` → `"데모를 성공시킨 비공식 구조"` 또는 `"데모 환경이 AI를 안정적으로 보이게 만드는 이유"` 등 서술형 중립 표현으로 조정.

### SF-06. temperature 설명 문단 분리
temperature 설명 문단이 주 논지(비결정성의 운영 영향)에서 기술 심화 방향으로 이탈. '참고' 박스 또는 각주로 분리 권장. 박스 제목 예: `[참고] temperature는 비결정성을 줄이는 도구이지 제거하는 도구가 아니다`

### SF-07. 비유 밀도 조정 ('하네스라는 이름을 붙이기 전에' 섹션)
자동차 테스트 트랙 + 외과 수술 체크리스트 비유가 연달아 등장. 하나를 압축하거나, 대상 독자(AI/ML 엔지니어)에게 더 직접적인 CI/CD 파이프라인 또는 테스트 하네스 원어 비유로 교체 검토.

### SF-08. 외부 근거 확보 후 최소 1건 각주 삽입
source_grounding 2/5. 핵심 주장('PoC-운영 괴리는 구조적 문제다', '비결정성이 핵심 원인이다')에 대해 외부 자료(예: Sculley et al. 2015, Chip Huyen 2023) 확보 후 각주 1–2건 삽입. 외부 자료 확보 전까지는 플레이스홀더 각주로 표시.

### SF-09. PoC 표기 방침 확정
tone_guide 규정(첫 등장 이후 한글 사용)에 따르면 이후 'PoC' 대신 '개념 검증'을 써야 하나 초안은 'PoC' 약어를 계속 사용. 책 전체 용어 방침 확정 후 일관 적용. 방침 확정 전까지 당장은 현행 유지.

### SF-10. 도입부 로드맵 한 줄 추가
하네스 정의가 나오기까지 4개 절을 거침. 도입부에 구조 예고 한 줄 추가. 예: "1–2절에서 문제를 진단하고, 3절에서 AI 고유의 특성을 확인한 뒤, 4절에서 하네스 개념을 처음 꺼내 놓는다."

### [지시 W-03] 비결정성 원인 문단 재작성 `(MF-03 + SF-01)`

**위치:** `"temperature를 0으로 설정해도..."` 이후 원인 열거 문단

**현재 텍스트 (요약):**
> 부동소수점 연산의 미세한 차이, 서버 환경과 배치 처리 방식, 인프라 설정에 따라 출력이 달라질 수 있다.

**수정 후 텍스트:**
```
비결정성의 원인은 층위별로 다르다. 가장 근본적인 층위는 모델 자체의 
확률적 샘플링이다. top-k 샘플링이나 nucleus 샘플링 방식은 동일한 
입력에도 확률 분포에서 서로 다른 토큰을 선택한다. 
그 위 층위에서는 API 서비스 운영 방식이 개입한다. OpenAI처럼 
API로 제공되는 서비스에서는 모델 버전 업데이트나 서버사이드 배치 처리
(여러 요청을 묶어 한꺼번에 처리하는 방식) 자체가 동일 입력에 대한 
출력 변동을 유발한다. 로컬 환경에서 직접 모델을 실행할 경우에는 
GPU 병렬 연산의 비결정적 순서, 부동소수점 연산 차이도 추가로 작용한다.

따라서 temperature=0 설정은 확률적 샘플링을 억제하지만, 
API 버전 변경이나 배치 처리 방식에서 비롯되는 변동까지 제거하지는 않는다.
비결정성을 줄이는 도구이지, 제거하는 도구가 아닌 것이다.
```

> **NOTE:** temperature 메커니즘 세부 설명은 별도 '참고' 박스로 분리 (지시 W-08 참조).
