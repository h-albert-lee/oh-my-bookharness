## 2. Must Fix (반드시 수정)

> 이번 사이클 내 반드시 해결해야 할 항목. 우선순위 순으로 정렬.

### MF-01. 코드 블록 미완성 — `PromptBuilder.build()` 메서드 절단
- **위치**: `프롬프트 구성기` 절, `PromptBuilder` 클래스 의사코드
- **문제**: `'context': co`에서 코드가 잘려 있어 독자가 설계 패턴을 이해할 수 없음. 동일한 절단이 두 리뷰어에게 별도로 지적됨(중복 확인).
- **조치**: 완성된 `build()` 메서드 전체를 작성하거나, 코드가 당장 완성 불가능하면 잘린 위치에 `# ... (이하 MF-02와 함께 완성 예정)` 표시 후 별도 완성본을 제출. **제출 전 전체 코드 블록 완결 필수.**

### MF-02. 의사코드 내 미정의 변수 `session_id`
- **위치**: `log_input` 로깅 단계, `metadata={"timestamp": now(), "session_id": session_id}`
- **문제**: `session_id`가 함수 파라미터로 전달되지 않은 미정의 변수. 기술서 의사코드에서 독자 혼란을 유발.
- **조치**: 함수 시그니처에 `session_id` 파라미터를 추가하거나, 로깅 메타데이터를 `{"timestamp": now()}` 로 단순화. 변수 출처를 코드 내에서 명확히 해야 함.

### MF-03. 프롬프트 인젝션 방어 과도한 단순화
- **위치**: `입력 처리기` 절, 프롬프트 인젝션 관련 서술
- **문제**: 단순 패턴 차단(예: '이전의 모든 지시를 무시하고')이 입력 처리기 수준에서 완전히 방어 가능한 것처럼 서술되어 있어 기술적으로 부정확.
- **조치**: '첫 번째 방어선'이라는 표현은 유지하되, 다음 문장을 추가: **"단, 입력 필터링은 인젝션 공격의 첫 번째 방어선일 뿐이며, 정교한 인젝션은 패턴 매칭만으로 차단되지 않는다. 출력 검증, 권한 분리, 프롬프트 구조 설계 등 다층적 방어가 병행되어야 한다."**

### MF-04. 프롬프트 드리프트 원인 유형 불명확
- **위치**: 드리프트 개념 도입 문단
- **문제**: '배포된 프롬프트와 실제 동작하는 프롬프트가 달라지는 현상'이라는 정의는 있으나, 원인이 코드 분기 문제인지 런타임 변조 문제인지 불분명.
- **조치**: 정의 뒤에 원인 유형을 다음과 같이 구체화: **"드리프트는 주로 두 경로로 발생한다. 첫째, 코드베이스 내 여러 경로에서 서로 다른 프롬프트 버전이 분기 조건에 따라 선택적으로 실행되는 경우. 둘째, 런타임 환경에서 외부 컨텍스트(검색 결과, 사용자 이력 등)가 예측과 다르게 프롬프트에 삽입되는 경우."**

### MF-05. `진단 불가능성(diagnosability의 부재)` 표현 오류 및 과장 선언
- **위치**: 도입부 세 번째 문단
- **문제 A**: `diagnosability의 부재`는 tone_guide '불필요한 영어 용어 남발 금지' 위반. 어색한 혼용 표현.
- **문제 B**: '가장 위험한 상태로 만든다'는 과장·선언적 표현.
- **조치 (두 문제를 한 번에 수정)**:
  - 표현: `진단 가능성의 부재` 또는 `추적 불가능성`으로 교체
  - 문장 전체: **"이 추적 불가능성 때문에 운영 환경의 LLM 시스템은 장애가 발생해도 원인을 특정하기 어렵다."**

### MF-06. 도입부 '~할 것이다' 반복 및 메타 선언문 과잉
- **위치**: 도입부 네 번째·다섯 번째 문단 (파이프라인 개요 예고 + 5장·6장 예고)
- **문제**: '나열하는 방식이 아니라…정의할 것이다', '예측할 수 있게 될 것이다' 등 선언문 나열이 tone_guide 금지 패턴.
- **조치**: 두 문단을 다음 1~2문장으로 압축, '~할 것이다' 종결 최대 1회 허용:
  > **"이 장에서는 파이프라인을 구성하는 여섯 가지 요소를 분해하고, 각 요소가 없을 때 어떤 문제가 생기는지를 먼저 보여준다. 이 구조를 이해하면 5장의 RAG와 6장의 에이전트가 파이프라인의 어느 지점을 확장하는지 파악할 수 있다."**

### MF-07. 챕터 참조 표기 통일 — `ch03` → `3장`
- **위치**: 도입부 두 번째 문단 `ch03에서 살펴봤듯이`
- **문제**: 편집 스타일상 한빛미디어 기술서는 챕터를 `3장에서`로 표기. `ch03` 약어 형식은 부적절.
- **조치**: 원고 전체에서 `ch0N` 형식을 `N장에서` / `N장의` 로 일괄 치환. 이번 사이클에서 도입부 해당 문장 수정 필수.

### MF-08. 2장 EDD 개념 연결 누락 (장간 논리 단절)
- **위치**: 파이프라인 개념 도입 절 또는 구성 요소 분리 이유 절
- **문제**: 4장이 파이프라인의 '테스트·진단 가능성'을 강조하면서도 2장의 EDD(평가 주도 개발) 개념과 전혀 연결되지 않아 Part 1→Part 2 논리 흐름이 단절.
- **조치**: Chip Huyen 인용 또는 '각 단계 독립 테스트' 원칙 언급 직후 다음 1문장 삽입:
  > **"2장에서 다룬 평가 주도 개발(EDD)의 관점에서 보면, 파이프라인의 각 단계가 독립된 단위일 때 비로소 각 단계를 평가의 단위로도 다룰 수 있다."**

### MF-09. 1장 하네스 재언급 시 핵심 기능 누락
- **위치**: 하네스(harness)를 재소개하는 단락
- **문제**: 1장에서 하네스의 핵심 기능으로 명시된 '입력 통제·출력 평가·실패 처리' 중 '출력 평가'와 '실패 처리'가 재언급 단락에서 누락. 독자가 하네스의 전체 역할을 4장에서 재확인할 수 없음.
- **조치**: 하네스 첫 언급 문장 또는 그림 4-1 캡션 직후에 다음 1문장 추가:
  > **"하네스는 입력 통제, 출력 평가, 실패 처리 세 기능으로 파이프라인 전체를 감싸는 외부 구조다(1장 참조)."**

### MF-10. 3장 참조 인용 정확성 검증 및 조건부 처리
- **위치**: 도입부 `LLM 기반 시스템이 운영 환경에서 실패할 때 그 원인은 대부분 시스템 설계에 있다` 문장
- **문제**: 3장 요약 미확보 상태에서 인용 문구의 정확성 미검증. blueprint 논지와는 부합하나 실제 3장 문장과 불일치 가능성 있음.
- **조치**: 3장 완성본 확보 후 인용 문구 대조 필수. 당장 수정이 어렵다면 해당 문장을 직접 인용 형식에서 **간접 서술형**으로 전환:
  > **"3장에서 살펴봤듯이, 운영 환경의 LLM 시스템 실패 원인은 모델 자체보다 시스템 설계에서 비롯하는 경우가 많다."** (정확한 원문 확보 후 최종 조정)

## 3. Should Fix (권장 수정)

### SF-01. 그림 4-1 캡션과 다이어그램 구조 불일치
- **문제**: 캡션에서 오류 처리기·상태 관리자가 '모든 단계와 연결'된다고 하나 다이어그램은 주 흐름 아래 병렬 나열.
- **조치 옵션 A**: 다이어그램에 양방향 점선 화살표로 연결 관계 표시. **조치 옵션 B**: 캡션을 "오류 처리기와 상태 관리자는 주 흐름 외부에서 각 단계를 지원한다"로 수정하여 다이어그램 구조에 맞춤. 둘 중 하나 선택.

### SF-02. LLM 파이프라인과 웹 미들웨어 비유의 차이점 미명시
- **문제**: 미들웨어 체인 비유가 효과적이나 LLM 파이프라인의 비결정성(non-determinism) 차이가 언급되지 않아 독자 오개념 유발 가능.
- **조치**: 비유 단락 마지막에 1문장 추가: **"다만 웹 미들웨어와 결정적으로 다른 점이 있다. LLM 호출 단계의 출력은 동일한 입력에도 매번 달라질 수 있다. 이 비결정성이 파이프라인 각 단계를 더욱 명확히 분리해야 하는 이유다."**

### SF-03. '나쁜 설계' 의사코드 `user_tier` 비대칭 지시 의도 불명확
- **문제**: premium 사용자에게만 JSON 응답 지시가 포함된 것이 의도적 설계 결함 예시인지 작성 실수인지 모호.
- **조치**: 해당 코드 직전 또는 직후 문장에서 명시: **"아래 코드에서 premium 계층에만 JSON 응답 지시가 포함된 것은 의도적인 결함 예시다. 이처럼 계층별 지시가 비대칭으로 구성되면, 같은 시스템이 사용자 등급에 따라 다른 형식의 응답을 반환하여 출력 처리기가 파싱 오류를 낸다."**

### SF-04. 멀티모달 언급 처리
- **문제**: 입력 처리기 절 말미의 멀티모달 언급이 이 장의 scope를 벗어나 집중력 분산.
- **조치**: 삭제하거나 다음 박스로 분리:
  ```
  > **참고**: 이미지·오디오·JSON 등 멀티모달 입력의 처리 방식은 이 장의 범위를 
  > 벗어난다. 입력 처리기의 확장 설계는 9장에서 다룬다.
  ```

### SF-05. 프롬프트 드리프트 용어 정식 병기 형식
- **문제**: 'drift'가 첫 등장 시 정식 용어 병기 처리 미흡.
- **조치**: 첫 등장 문장을 다음과 같이 교체: **"프롬프트 드리프트(prompt drift, 배포된 프롬프트와 실제 동작하는 프롬프트 간 불일치 현상)는 단일 함수에 프롬프트 로직이 산재할 때 가장 빠르게 누적된다."**

### SF-06. 동어반복 문장 제거 — '단계라는 단위가 존재해야 한다'
- **문제**: Chip Huyen 인용 직후 '이 원칙이 성립하려면 먼저 단계라는 단위 자체가 존재해야 한다'는 자명한 결론으로 동어반복.
- **조치**: 삭제하거나 **"즉, 파이프라인 설계가 테스트 가능성의 전제 조건이다."** 로 압축.

### SF-07. 블로그형 전환 문장 교체
- **문제**: '더 큰 문제는 다른 곳에 있다' — 전환 근거 없이 분위기로 연결하는 블로그형 표현.
- **조치**: **"그러나 더 실질적인 문제는 프롬프트 구성 로직의 여러 곳에 산재한다는 점이다."** 로 교체.

### SF-08. 과장 단언 표현 완화
- **문제**: '유지보수가 불가능한 상태로 귀결된다' — 절대적 단언.
- **조치**: **"어느 시점부터 코드 수정 비용이 급격히 증가하고, 변경 영향 범위를 예측하기 어려워진다."** 로 교체.

### SF-09. 1장 비결정성 개념 연결
- **문제**: 파이프라인 필요성을 설명할 때 1장의 핵심 문제 원인인 비결정성이 언급되지 않아 1장과의 논리적 연속성 약화.
- **조치**: SF-02 수정과 병행하여, '단일 API 호출로 충분하지 않은 이유' 서술에 다음 연결 추가: **"1장에서 다룬 비결정성 문제가 이를 더욱 복잡하게 만든다. 동일한 입력이 매번 다른 출력을 낸다는 사실은, 각 처리 단계를 분리하지 않으면 오류 원인 추적이 구조적으로 불가능하다는 것을 의미한다."**

### SF-10. 1장 실패 귀인 오류 표 명시적 참조
- **문제**: 도입부 시나리오('모델이 문제다', '프롬프트가 문제다', '사용자 입력이 문제다')가 1장 표 1-2 '실패 귀인 오류 유형'을 묘사하나 참조 없음.
- **조치**: 시나리오 단락 마지막 문장 뒤에 추가: **"이는 1장의 표 1-2에서 정의한 실패 귀인 오류의 전형적인 패턴이다."**

### OP-04. SoC 원칙 명시적 언급
- 소프트웨어 경험 독자를 위해 구성 요소 분리 이유 절에 `"이것이 LLM 시스템에 적용된 관심사 분리(SoC, Separation of Concerns) 원칙이다."` 한 문장 추가 가능.

### OP-05. 그림 4-1 ASCII 다이어그램 편집팀 재제작 요청 명시
- 인쇄 환경 가독성 저하 대비, 원고 해당 위치에 편집 지시 삽입: `[편집팀: 현재 ASCII는 초안 전달용. 벡터 도식으로 재제작 요청]`

### OP-06. 5·6장 예고 톤 완화
- '예측할 수 있게 될 것이다' → '5장과 6장의 구조적 맥락을 이해하는 데 도움이 된다' (MF-06 수정 시 자동 해소)

### OP-07. 10~12장 전방 참조 추가
- EDD 연결(MF-08) 문장 뒤에 `"이 파이프라인의 각 단계가 실제로 어떻게 평가 단위가 되는지는 10~12장에서 다룬다."` 추가 가능.

### OP-08. 도입 시나리오 원인 범위 명시
- '며칠 전까지만 해도 정상이었다'는 드리프트·모델 업데이트·데이터 변화 등 여러 원인을 동시에 암시. 이 모호성이 의도적임을 한 문장으로 명시하거나, 이 장에서 다루는 파이프라인 구조 문제로 범위를 좁히는 문장 추가.

### OP-09. `나쁜 설계/좋은 설계` 코드 내 레이블 처리
- 코드 블록 내 가치판단 주석 대신, 코드 앞 본문에서 `"다음은 구성 로직이 혼재된 나쁜 설계 예시다."` 형식으로 처리하고 코드 내 주석은 제거하는 것이 한빛미디어 실무서 스타일에 부합.

### OP-10. 각주 설명 레이블 보강
- `[^web_01]`, `[^web_02]` 등 각주에 최소한 출처 설명 레이블 병기 (예: `[^web_01]: a16z 레퍼런스 아키텍처 문서`). 집필 단계 리뷰 가능성 향상.

## 5. Contradictory Comments (상충 의견)

### CC-01. 도입 시나리오 원인 범위: 의도적 모호성 vs. 범위 축소
| 의견 A | 의견 B |
|--------|--------|
| 시나리오의 모호성이 의도적이라면 명시하라 (Optional) | 이 장 주제인 파이프라인 구조 문제로 범위를 좁히는 것이 낫다 (Optional) |

**권장**: 도입 시나리오는 독자 공감용으로 모호성 자체가 기능한다. 단, 세 번째 문단에서 '이 장에서 다루는 문제는 파이프라인 설계 문제에 한정한다'는 범위 명시 문장을 추가하면 두 의견을 동시에 만족할 수 있다. MF-06 수정과 병행 처리 권장.

### CC-02. 각주 참조 형식: 출처 레이블 상세화 vs. 현행 유지
| 의견 A | 의견 B |
|--------|--------|
| 독자용 기술서에는 제목·저자·URL 필요, 최소한 설명 레이블 병기 필요 (Should Fix) | 집필 단계 문서이므로 현행 source_id 참조로 충분 (암묵적 현행 유지 입장) |

**권장**: Should Fix 등급을 유지하되, 이번 사이클에서는 Optional(OP-10)로 처리. 편집 단계에서 정식 각주로 전환하는 것이 현실적. 단, 동일 출처 중복 인용(`[^web_10]` 2회, `[^web_01]` 2회)은 편집 단계에서 통합 필수.

### CC-03. 그림 4-1 오류 처리기·상태 관리자 연결 표현: 다이어그램 수정 vs. 캡션 수정
| 의견 A | 의견 B |
|--------|--------|
| 다이어그램에 연결 관계를 시각적으로 추가 | 캡션 설명을 다이어그램 구조에 맞게 조정 |

**권장**: ASCII 다이어그램 수정보다 캡션 수정이 이번 사이클에서 현실적. **"오류 처리기와 상태 관리자는 주 흐름 외부에서 각 단계를 지원하는 횡단 관심사(cross-cutting concerns)다."** 로 캡션 수정 후, 벡터 재제작 요청 시(OP-05) 연결 관계를 다이어그램에 반영하도록 편집팀에 지시.

## 6. Rewrite Instructions for Writer (구체적 수정 지시)

> 아래 지시를 순서대로 따라 수정하라. 각 항목 완료 후 체크.

### [STEP 1] 코드 완성 — 최우선 (MF-01, MF-02)

**1-A. `PromptBuilder.build()` 완성**

현재 `'context': co` 에서 잘린 코드를 다음과 같이 완성하라:

```python
class PromptBuilder:
    def __init__(self, template: str, config: dict):
        self.template = template
        self.config = config

    def build(self, user_input: str, context: str, session_id: str) -> str:
        """
        시스템 지시, 컨텍스트, 사용자 입력을 조합하여 최종 프롬프트를 반환한다.
        """
        components = {
            'system': self.config.get('system_instruction', ''),
            'context': context,
            'user_input': user_input,
        }
        return self.template.format(**components)
```

**1-B. `log_input`의 `session_id` 미정의 수정**

함수 시그니처에 `session_id: str` 파라미터를 추가하거나, 로깅 메타데이터를 단순화:

```python
# 수정 전
def process_input(user_input: str) -> ProcessedInput:
    log_input(user_input, metadata={"timestamp": now(), "session_id": session_id})

# 수정 후 (옵션 A: 파라미터 추가)
def process_input(user_input: str, session_id: str) -> ProcessedInput:
    log_input(user_input, metadata={"timestamp": now(), "session_id": session_id})

# 수정 후 (옵션 B: 단순화)
def process_input(user_input: str) -> ProcessedInput:
    log_input(user_input, metadata={"timestamp": now()})
```
