### MF-01. 프롬프트 인젝션 방어의 단일 책임 서술 오류

**원인**: 프롬프트 인젝션 방어가 입력 처리기 단독 책임처럼 서술되어 있어 기술적으로 부정확하다.

**수정 방향**:
- `'파이프라인 진입 시점에서 차단하는 첫 번째 방어 지점이다'`
  → `'파이프라인 진입 시점에서 차단하는 첫 번째 방어 지점 중 하나다'`
- 이어서 한 문장 추가: 프롬프트 인젝션에 대한 완전한 방어는 시스템 프롬프트를 구조화하는 프롬프트 구성기, 응답을 검증하는 출력 처리기까지 포함하는 다층 방어(Defense in Depth)를 통해 이루어진다는 사실을 명시한다.

### MF-02. 컨텍스트 윈도우 초과를 입력 처리기 단독 문제로 귀결하는 서술 오류

**원인**: 컨텍스트 윈도우 초과는 입력 길이뿐 아니라 대화 이력(State Manager), 시스템 프롬프트(Prompt Builder), RAG 검색 결과 등 복수 단계의 누적으로 발생한다. 입력 처리기의 길이 검증만으로 해결된다는 인상을 주는 현재 서술은 오류다.

**수정 방향**:
- 해당 문장에서 '컨텍스트 윈도우 초과'를 입력 처리기 부재의 대표 증상으로만 제시하는 표현을 삭제하거나 완화한다.
- 컨텍스트 윈도우 초과가 입력 처리기·상태 관리자·프롬프트 구성기 등 복수 단계의 누적에서 비롯될 수 있음을 명시한다. 입력 처리기의 역할은 그 누적을 줄이는 '첫 번째 기여 단계'로 한정하여 서술한다.

### MF-08. 볼드 강조 남용 교정

**원인**: '**실패는 하위 단계에서 발생하지만, 원인은 상위 단계에 있다.**' 등 문장 전체를 굵게 처리한 사례가 다수이며, 이는 기술서의 레이아웃 품질과 독자 집중도를 저하시킨다.

**수정 방향**:
- 문장 전체 볼드를 전면 해제한다.
- 볼드는 **핵심 용어** 및 **개념 단위**에만 제한 적용한다. 예: `실패는 하위 단계에서 발생하지만, 원인은 **상위 단계**에 있다.`
- 초안 전체에서 볼드 처리된 문장을 순차 검토하고, 문장 단위 볼드는 모두 해제한다.

### MF-09. 도입부 중복 논지 압축

**원인**: '덩어리 구조 → 진단 불가 → 파이프라인 필요'라는 동일 논지가 도입부 6개 단락에서 3회 이상 반복 순환된다. redundancy 점수 2/5의 직접적 원인이다.

**수정 방향**:
- 도입부를 3단락 이내로 압축한다.
  - 1단락: 실패 시나리오(챗봇 오작동 → 프롬프트 수정 → 모델 교체 → 벽)
  - 2단락: 문제의 본질 (덩어리 구조와 진단 불가능성)
  - 3단락: 이 장의 접근 방식 (파이프라인 분해, 진단 가능성 기준)
- 나머지 논거('프롬프트 엔지니어링의 한계', '하네스 재확인')는 '파이프라인이라는 관점' 섹션 또는 해당 구성 요소 섹션으로 이동한다.

## 3. Should Fix (권장 수정)

### SF-01. 의사코드 길이 단위 명확화 (문자 수 vs 토큰 수)

`len(normalized)`가 문자 수 기반 측정처럼 보이나, LLM 파이프라인에서 실질적으로 유효한 단위는 토큰 수다. 해당 의사코드 직후에 다음 내용의 주석 또는 본문 설명을 추가한다: "문자 수 기반 검증은 근사값이며, 언어와 모델에 따라 실제 토큰 수와 차이가 발생한다. 정확한 제한을 위해서는 모델별 토크나이저를 사용한 토큰 수 기반 검증이 적합하다."

### SF-02. 의사코드 처리 순서의 설계적 근거 설명 추가

인코딩 정규화(`normalize_encoding`) → 길이 검증 순서가 현재 근거 없이 나열된 것처럼 보인다. 인코딩 정규화 후 길이가 확정되므로 이 순서가 의도적임을 한 문장으로 설명한다: "정규화 이후의 길이를 기준으로 검증해야 정확한 제한이 적용된다."

### SF-03. 미들웨어 체인 비유의 LLM 특수성 보완

현재 미들웨어 체인 비유가 LLM과 기존 소프트웨어의 구조적 유사성만 다루고, 차이—비결정성(non-determinism), 레이턴시 변동, 토큰 비용—에 대한 언급이 전혀 없다. 비유 직후에 다음 내용의 한 단락을 추가한다: "다만 기존 미들웨어 체인과 달리, LLM 파이프라인의 각 단계는 동일한 입력에 대해 동일한 출력을 보장하지 않는다. 비결정성, 레이턴시 변동, 호출 비용이라는 LLM 고유의 특성은 파이프라인 설계 시 추가적인 고려를 요구한다."

### SF-04. web_07 인용 범위 확인 및 조정

`[^web_07]`의 Source Pack Notes 정의는 '실패 분류 논문'이다. '진단 가능성을 시스템 설계의 첫 번째 목표로 삼아야 한다'는 규범적 주장이 해당 논문에서 직접 도출되는지 검토한다. 직접 도출되지 않는다면 인용을 제거하거나, '실패 원인을 시스템 구조로 귀인하는 연구 결과[^web_07]'처럼 논문의 실제 기여 범위로 좁혀 인용한다.

### SF-05. 그림 4-1 내용 명세 구체화 (편집 지시 형식)

편집 지시에 '그림(파이프라인 흐름도) 1개 필수 삽입'이 명시되어 있으나, 현재 초안에 그림의 구체적 내용 명세가 없다. 본문 또는 편집 주석으로 다음을 명시한다: ① 6개 구성 요소의 선형 흐름(입력 처리기→프롬프트 구성기→LLM 호출기→출력 처리기→상태 관리자 / 오류 처리기), ② 하네스가 전체를 감싸는 외곽 구조, ③ 각 단계 간 화살표와 데이터 흐름 방향.

### SF-06. ch01 '입력 통제·출력 평가·실패 처리' 3축과 6개 구성 요소 매핑 명시

ch01의 핵심 개념 3축(입력 통제, 출력 평가, 실패 처리)이 4장의 6개 컴포넌트와 어떻게 대응되는지를 한 문장으로 연결한다. 예: "1장에서 정의한 입력 통제는 입력 처리기와 프롬프트 구성기가, 출력 평가는 출력 처리기가, 실패 처리는 오류 처리기가 각각 구체화하는 역할을 담당한다."

### SF-07. 표 1-2(실패 귀인 오류 유형) 명시적 참조 추가

'실패가 LLM 호출 단계에서 발생하지만 원인은 입력 단계에 있다'는 논의 지점에서 ch01 표 1-2를 직접 참조한다: "1장의 실패 귀인 오류 유형(표 1-2)에서 살펴본 것처럼, 증상이 나타나는 단계와 원인이 있는 단계는 다르다."

### SF-08. ch01 비결정성(nondeterminism) 개념 연결

파이프라인 구성 요소 분리의 필요성을 논증하는 지점에서 ch01의 '비결정성' 개념을 연결한다: "비결정성으로 인해 동일한 입력이라도 매번 다른 출력이 생성될 수 있다. 각 단계의 경계가 명확할수록, 어느 단계의 비결정성이 최종 출력에 영향을 미쳤는지를 좁혀갈 수 있다."

### SF-09. '파이프라인이라는 관점' 섹션 재편

현재 이 섹션이 도입부와 동일한 논지(덩어리 구조 → 진단 불가)를 반복한다. 이 섹션은 다음 두 가지에 집중하도록 재편한다: ① 파이프라인의 구조적 정의 (순서와 역할이 명시된 처리 흐름), ② 기존 소프트웨어 아키텍처 비유(미들웨어 체인 주 비유, MVC 보조 비유)와 LLM 파이프라인의 유사점 및 차이점(SF-03 통합). 반복 논증은 삭제한다.

### SF-10. 의사코드 성격 명시

blueprint에 '코드 비중 최소화, 이론 중심'이 명시되어 있다. 입력 처리기 섹션의 의사코드 직전 또는 직후에 다음을 명시한다: "아래는 구현을 위한 코드가 아니라, 입력 처리기가 수행하는 단계를 개념 수준에서 나타낸 의사코드다." 또는 의사코드 대신 단계 목록(텍스트 기반 플로우) 형식으로 대체하는 방안을 검토한다.

## 4. Optional Improvements (선택적 개선)

### OP-01. 정제(Sanitization)의 규칙 기반 vs 모델 기반 구분 추가
'악성 지시 제거' 설명에 각주 또는 짧은 설명으로 규칙 기반 필터링과 모델 기반 분류의 트레이드오프를 언급한다. 구현 복잡도, 오탐율, 처리 비용이 접근 방식에 따라 크게 달라진다는 점이 실무 독자에게 유용하다.

### OP-02. 원본 입력 로깅의 규제 환경 주의사항
로깅 포인트에서 원본 입력 기록을 강조하는 문장 근처에 한 줄을 추가한다: "규제 환경(GDPR, 개인정보보호법 등)에서는 원본 입력의 무조건적 로깅이 제약될 수 있으므로, 로깅 정책은 별도로 수립해야 한다."

### OP-03. 도입부 시나리오 좌절감 구체화
'프롬프트 수정 → 모델 교체 → 막다른 벽'의 각 시도에서 독자가 느끼는 좌절감을 한 문장씩 구체화하면 공감도가 높아진다. 예: "지시문을 더 명확하게 고쳐도, 다음 날이면 같은 오류가 다른 입력에서 다시 나타난다."

### OP-04. ch01 데모-운영 격차 서사 연결
도입부 챗봇 실패 시나리오 직후에 한 문장을 추가한다: "이것은 1장에서 다룬 데모-운영 격차가 실제 운영 환경에서 어떤 형태로 나타나는지를 보여주는 전형적인 사례다."

### OP-05. 관심사의 분리 원칙의 LLM 특수성 보강
관심사의 분리(Separation of Concerns)를 도입할 때, LLM 고유 특성(비결정성, 레이턴시 변동, 버전 드리프트)이 이 원칙을 더욱 강하게 요구한다는 내용을 한 단락으로 추가하면 기존 소프트웨어 개발자 독자에게 강한 동기부여가 된다.

### OP-06. 섹션 전환 문장 다양화
각 섹션 말미의 '다음 섹션의 주제다', '시작하자', '살펴볼 차례다' 형태 반복을 피하고, 섹션별로 서로 다른 전환 방식을 사용한다.

### OP-07. 소제목 형식 통일
현재 명령형('통제하라'), 명사형('관점'), 서술형('조립하는 단계')이 혼재한다. 장 전체 소제목을 명사형 또는 서술형 중 하나로 통일한다.

### OP-08. 미들웨어 체인 vs MVC 비유 위계 정리
미들웨어 체인을 주 비유, MVC를 보조 비유로 명확히 위계를 두어 제시한다. 독자층이 백엔드 개발자 중심임을 감안할 때, 미들웨어 체인이 더 직접적인 비유다.

### OP-09. 프롬프트 인젝션 용어 첫 등장 확인
프롬프트 인젝션이 책 전체에서 처음 등장하는 위치인지 확인한다. 첫 등장이면 현재 병기 표기는 적절하나, 동일 장 내 이후 사용에서는 한글 단독 표기를 유지한다.

### OP-10. 그림 참조 독립성 보완
'그림 4-1은 이 장에서 다룰 LLM 파이프라인의 전체 흐름을 보여준다' 직후에 그림 없이도 이해할 수 있는 1~2문장을 추가한다: "이 흐름은 입력 처리기에서 시작해 프롬프트 구성기, LLM 호출기, 출력 처리기를 거치며, 상태 관리자와 오류 처리기가 각 단계를 지원하는 구조다."
