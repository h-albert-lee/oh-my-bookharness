# 4장. LLM 파이프라인 기본 구성 요소

## 4.0 도입: 어디서부터 생각해야 하는가

### 4.0.1 진단할 수 없는 실패

챗봇이 어느 날부터 이상한 답변을 내놓기 시작했다고 상상해 보자. 어제까지는 분명히 잘 동작했는데, 오늘부터는 엉뚱한 내용을 반환하거나, 응답 형식이 무너지거나, 가끔은 아예 답변 자체를 거부한다. 첫 번째 시도는 프롬프트 수정이다. 지시문을 더 명확하게 고쳐 보고, 예시를 추가해 본다. 나아지는 것 같다가도 다시 무너진다. 두 번째 시도는 모델 교체다. 더 최신 버전으로 바꾸면 해결될 것이라는 기대로 API 엔드포인트를 변경해 본다. 증상이 조금 달라지긴 하지만, 근본적으로는 같은 문제가 다른 형태로 재현된다. 결국 어느 지점에서 막다른 벽에 부딪힌다. 문제가 어디에 있는지 특정할 수 없기 때문이다.

이 상황이 낯설지 않다면, 그것은 시스템이 분해되어 있지 않기 때문일 가능성이 높다. LLM을 호출하는 코드 한 줄과 사용자 입력을 받아 응답을 돌려주는 몇 개의 함수가 뒤섞인 채 하나의 덩어리처럼 동작할 때, 실패는 어디서든 발생할 수 있지만 어디서 발생했는지는 아무도 알 수 없다. 3장에서 살펴봤듯, 실패의 원인은 모델 자체가 아닌 시스템에 있는 경우가 많다.[^web_07] 그런데 그 시스템이 내부적으로 구분된 단계 없이 하나로 뭉쳐 있다면, 진단은 원리적으로 불가능해진다.

3장에서 분류한 실패 유형 중 하나를 다시 꺼내 보자. 같은 입력에서도 맥락이나 이력이 달라질 때 품질이 저하되는 경우가 있다. 이 실패는 파이프라인의 어느 단계에서 발생한 것인가—입력이 잘못 처리된 것인가, 프롬프트 구성이 불완전한 것인가, 아니면 이전 대화 이력이 잘못 관리된 것인가. 분리된 구조 없이는 이 질문 자체를 물을 수 없다. 프롬프트를 고치는 것은 그 덩어리의 어느 한 부분을 건드리는 일이고, 모델을 바꾸는 것은 또 다른 부분에 손을 대는 일이지만, 어느 쪽이 실제로 문제였는지는 끝내 확인할 수 없다.

### 4.0.2 "프롬프트만 잘하면 된다"는 가정의 한계

"프롬프트 엔지니어링(Prompt Engineering, 모델에 전달하는 지시문을 체계적으로 작성하는 기법)만 잘하면 된다"는 인식은 이 덩어리 구조를 전제로 한다. 프롬프트를 시스템의 전부인 것처럼 다루는 접근은, 규모가 작고 요구사항이 단순한 초기 단계에서는 어느 정도 통할 수 있다. 하지만 입력의 다양성이 높아지고, 실패에 대한 허용 범위가 좁아지며, 유지보수가 필요해지는 시점부터 이 접근은 한계를 드러낸다.[^web_10] 프롬프트는 시스템의 구성 요소 중 하나일 뿐이다. 그것이 전부인 구조에서는, 프롬프트가 원인이 아닌 실패도 프롬프트를 탓하게 된다.

::: note 진단 가능성(Diagnosability)
어느 단계에서 무엇이 잘못되었는지를 특정할 수 있는 시스템의 속성. 구성 요소가 역할 단위로 분리되어 있을 때 비로소 확보된다. 진단이 불가능한 시스템은 수정 방향을 알 수 없기 때문에, 유지보수 비용이 기하급수적으로 증가한다.
:::

1장에서 소개한 하네스(Harness) 개념을 다시 떠올려 보자. 그림 1-2는 모델 단독 실행과, 하네스가 모델을 감싸는 실행 구조를 대비해서 보여준다. 하네스는 입력을 통제하고, 출력을 평가하며, 실패를 처리하는 구조다. 그런데 하네스가 실제로 작동하려면, 그 내부가 역할 단위로 분리되어 있어야 한다. 입력을 처리하는 부분, 프롬프트를 구성하는 부분, 모델을 호출하는 부분, 응답을 처리하는 부분이 구분되어 있어야만 각 단계에서 무엇이 들어오고 무엇이 나가는지를 추적할 수 있다. 그 추적이 가능해야만 실패가 어느 단계에서 발생했는지를 특정할 수 있다.

LLM을 "호출"하는 것은 한 줄의 코드로 가능하지만, LLM 시스템을 "설계"한다는 것은 실패를 진단할 수 있는 구조를 갖추는 일이다. 진단 가능성이 시스템 설계의 첫 번째 목표가 되어야 하는 이유가 여기에 있다.[^web_07] 이 장에서는 그 구조를 구성하는 부품들을 하나씩 분해해서 살펴본다. 각 구성 요소가 어떤 역할을 하는지, 그것이 독립된 단위로 존재해야 하는 이유가 무엇인지, 그리고 그것이 실패할 때 어떤 증상이 나타나는지를 순서대로 정리할 것이다.

그 전에 먼저 한 가지 관점을 정리할 필요가 있다. LLM 시스템을 파이프라인(Pipeline, 순서와 역할이 명시된 처리 흐름)이라는 틀로 바라보는 것이 왜 유용한지, 그리고 그 틀이 기존 소프트웨어 설계 경험과 어떻게 연결되는지부터 살펴보자.

[^web_07]: 실패 분류 논문. LLM 기반 시스템의 실패 원인을 분류하고, 이를 시스템 엔지니어링 문제로 프레이밍하는 학술적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 중심 접근의 산업적 한계를 다루며, 시스템 수준의 설계 전환 필요성을 논증한다.

## 4.1 파이프라인이라는 관점

앞에서 LLM 시스템의 실패가 "분해되지 않은 구조"에서 비롯된다는 점을 살펴보았다. 진단 가능성이 시스템 설계의 첫 번째 목표가 되어야 한다는 결론까지 이어졌는데, 그렇다면 그 분해를 어떤 방식으로 접근할 것인지 틀부터 정리해야 한다. 이 섹션에서는 LLM 시스템을 바라보는 관점으로서의 파이프라인 개념을 소프트웨어 설계 경험과 연결하여 살펴본다.

### 4.1.1 파이프라인은 프레임워크가 아니다

"파이프라인(Pipeline, 순서와 역할이 명시된 처리 흐름)"이라는 단어를 들으면 LangChain이나 LlamaIndex 같은 특정 프레임워크를 먼저 떠올리기 쉽다. 이 연상은 자연스럽지만, 오히려 개념을 좁히는 함정이 된다. 파이프라인은 특정 도구나 라이브러리가 아니라 순서(Sequence)와 역할(Role)이 명시된 처리 흐름이라는 개념적 구조다.[^web_01] LangChain은 파이프라인을 구현하는 한 가지 방법일 뿐이다. LangChain 없이도 파이프라인은 존재할 수 있고, LangChain을 사용한다고 해서 자동으로 파이프라인이 갖춰지지도 않는다. 파이프라인이라는 관점은 특정 프레임워크를 도입하기 이전에, 시스템을 어떻게 생각할 것인가의 문제다.

소프트웨어 개발 경험이 있다면 이미 유사한 구조를 다뤄본 적이 있을 것이다. 웹 서버의 미들웨어 체인(Middleware Chain, HTTP 요청이 처리되는 동안 순서대로 실행되는 처리 레이어의 연결)을 떠올려 보자. HTTP 요청이 들어오면 먼저 인증 미들웨어가 요청의 유효성을 확인하고, 그다음 로깅 미들웨어가 요청 내용을 기록하며, 마지막으로 비즈니스 로직이 실제 처리를 수행한다. 각 미들웨어는 자신의 역할만 담당하고, 처리 결과를 다음 단계로 넘긴다. 어느 미들웨어에서 오류가 발생했는지는 즉시 특정할 수 있다.

LLM 파이프라인의 단계 분리는 이와 동일한 원리로 작동한다.[^web_05] 입력 처리, 프롬프트 구성, 모델 호출, 출력 처리 각각이 독립된 책임을 가진 레이어로 존재한다. MVC 아키텍처(Model-View-Controller, 소프트웨어를 모델·뷰·컨트롤러 세 개의 관심사로 분리하는 설계 패턴)에서 각 레이어를 분리하는 이유—관심사의 분리(Separation of Concerns, SoC)—와 본질적으로 같은 동기에서 출발한다.[^web_09] 다른 점은 레이어 사이를 흐르는 데이터가 구조화된 객체가 아니라 자연어 텍스트라는 것, 그리고 각 단계의 출력이 결정론적으로 보장되지 않는다는 것이다.

::: note 관심사의 분리(Separation of Concerns, SoC)
시스템의 각 부분이 하나의 명확한 책임만을 가지도록 구조를 설계하는 원칙. 소프트웨어 공학에서 오래전부터 사용되어 온 개념으로, LLM 파이프라인에서는 각 구성 요소가 단 하나의 처리 역할만 담당하도록 경계를 긋는 것으로 적용된다.
:::

### 4.1.2 파이프라인 관점이 주는 설계적 이점

파이프라인이라는 틀이 유용한 이유는 시스템을 보기 좋게 정리하는 데 있지 않다. 각 단계를 독립적으로 교체하고, 테스트하고, 진단할 수 있는 경계가 생긴다는 데 있다.[^web_02] 하나로 뭉쳐 있는 구조에서는 출력이 잘못되었을 때 원인이 어디에 있는지 추적할 수 없다. 반면 단계가 분리된 구조에서는 각 단계의 입력과 출력을 개별적으로 검사할 수 있다. 프롬프트 구성 단계의 출력이 기대와 다르다면, 그것은 입력 처리 단계의 문제일 수 있다. 출력 처리 단계가 실패한다면, 모델이 반환한 형식이 기대와 달랐기 때문일 수 있다.

**그림 4-1. LLM 파이프라인 전체 흐름도**

```
┌─────────────────────────────────────────────────────────────┐
│                        하네스(Harness)                        │
│                                                             │
│  사용자   ┌──────────┐  ┌──────────┐  ┌──────────┐         │
│  입력  → │ 입력 처리기 │→│프롬프트  │→│ LLM 호출기 │         │
│         └──────────┘  │  구성기   │  └────┬─────┘         │
│                       └──────────┘       │                 │
│         ┌──────────┐  ┌──────────┐       │ 모델 응답        │
│  최종 ← │ 출력 처리기 │←│ 오류 처리기│←──────┘                │
│  응답    └──────────┘  └──────────┘                         │
│                ↕                                            │
│         ┌──────────┐                                        │
│         │ 상태 관리자 │  (대화 이력·세션 유지)                    │
│         └──────────┘                                        │
└─────────────────────────────────────────────────────────────┘
```

그림 4-1은 이 장에서 다룰 여섯 개 구성 요소가 어떤 순서로 연결되는지, 그리고 1장의 하네스가 이 파이프라인 전체를 어떤 위치에서 감싸는지를 보여준다. 하네스는 파이프라인의 한 구성 요소가 아니라, 파이프라인 전체를 외부에서 둘러싸며 입력 통제·출력 평가·실패 처리를 가능하게 하는 구조다. 파이프라인이 내부의 부품 배열이라면, 하네스는 그것이 실제 운영 환경에서 안전하게 동작할 수 있도록 보호하는 외피다.

파이프라인 관점에서 보면 "프롬프트 엔지니어링(Prompt Engineering)만 잘하면 된다"는 접근이 왜 불충분한지도 명확해진다. 프롬프트 엔지니어링은 파이프라인 전체가 아니라 그 중 하나의 단계—프롬프트 구성기—에 해당하는 작업이다. 그것이 잘 동작한다고 해도, 입력이 잘못 정제되거나, 대화 이력이 올바르게 관리되지 않거나, 출력이 적절히 파싱되지 않는다면 시스템 전체는 여전히 실패할 수 있다.[^web_10]

::: tip 파이프라인 관점을 팀에 적용할 때
설계 초기에 "우리 시스템에서 각 구성 요소 간의 경계가 어디인가"를 팀 내에서 명시적으로 합의하는 것이 중요하다. 이 합의 없이 구현을 시작하면, 각자가 다른 경계를 암묵적으로 가정한 채 코드를 작성하게 되고, 이후 진단과 테스트가 어려운 구조로 굳어진다.
:::

파이프라인이라는 틀을 이해했다면, 이제 각 구성 요소를 순서대로 들여다볼 준비가 된 셈이다. 파이프라인의 첫 번째 관문—사용자로부터 들어오는 입력이 시스템 안으로 진입하는 지점—인 입력 처리기(Input Processor)부터 살펴보자. 여기서의 처리가 이후 모든 단계의 품질을 결정한다는 점에서, 입력 처리기는 파이프라인에서 가장 먼저, 가장 신중하게 설계해야 할 단계 중 하나다.

[^web_01]: a16z 레퍼런스 아키텍처. LLM 시스템을 단계화된 흐름으로 분류하는 산업계 기준점으로 활용된다.
[^web_02]: Chip Huyen. "각 단계를 독립적으로 테스트해야 한다"는 설계 원칙을 명확히 서술한다.
[^web_05]: Craig Risi. 기존 소프트웨어 아키텍처(MVC, 미들웨어)와 LLM 파이프라인의 구조적 유사성 및 차이를 분석한다.
[^web_09]: 모듈형 파이프라인 가이드. SoC 원칙을 LLM 시스템에 적용하는 실용적 접근을 다룬다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 중심 접근의 산업적 한계와 시스템 수준 설계 전환 필요성을 논증한다.

## 4.2 입력 처리기

파이프라인이라는 틀을 갖추었다면, 이제 그 흐름의 첫 번째 관문을 구체적으로 들여다볼 차례다. 사용자로부터 들어오는 입력이 시스템 안으로 진입하는 지점—입력 처리기(Input Processor)—에서 어떤 일이 일어나야 하는지, 그리고 이 단계가 부재할 때 시스템이 어떻게 반응하는지를 살펴보자.

### 4.2.1 입력 처리기의 역할과 실패 시 증상

입력 처리기의 역할을 한 문장으로 정리하면 이렇다. 사용자 입력을 정제·검증하여 파이프라인이 처리할 수 있는 형태로 변환하는 단계다.[^web_01] 여기서 '변환'이 핵심이다. 입력 처리기는 단순히 입력을 받아 그대로 다음 단계로 넘기는 통로가 아니라, 입력의 형태와 내용이 시스템이 기대하는 범위 안에 있는지를 판단하고 조정하는 능동적인 레이어다.

이 레이어가 없으면 어떤 일이 벌어지는지를 구체적인 상황으로 떠올려 보자. 사용자가 1만 자 분량의 텍스트를 그대로 입력했다고 가정한다. 입력 처리기가 존재하지 않는 구조에서는 이 텍스트가 아무런 가공 없이 프롬프트 구성기로 전달된다. 프롬프트 구성기는 시스템 지시문과 대화 이력, 그리고 이 1만 자짜리 입력을 조합하여 최종 프롬프트를 만든다. 이때 프롬프트의 전체 길이가 모델이 처리할 수 있는 컨텍스트 윈도우(Context Window, 모델이 한 번에 처리할 수 있는 토큰의 최대 범위)를 초과한다. 오류는 이 시점—LLM 호출 단계—에서 발생한다. 로그에는 "컨텍스트 초과" 오류만 남는다. 그런데 실제 문제의 원인은 입력 단계에 있었다. 입력이 지나치게 길었다는 사실을 가장 일찍 감지했어야 할 지점은 입력 처리기였지만, 그 지점이 존재하지 않았기 때문에 오류는 몇 단계 뒤에서 발생하고, 원인 추적에 불필요한 시간이 소요된다.[^web_03]

이것이 입력 처리기 부재가 만드는 전형적인 증상이다. **실패는 입력 단계가 아닌 하위 단계에서 발생하고, 원인은 상위 단계에 있으며, 그 사이의 거리만큼 진단이 어려워진다.** 악성 입력이나 예상치 못한 형식의 데이터가 파이프라인 깊숙이 침투한 뒤에야 문제가 드러나는 것도 같은 이유다.

::: note 컨텍스트 윈도우(Context Window)
LLM이 하나의 요청에서 처리할 수 있는 토큰(텍스트를 분할한 최소 단위)의 최대 개수. 모델마다 이 한계가 다르며, 이를 초과하는 입력은 오류를 발생시키거나 초과 부분이 잘려나갈 수 있다. 입력 처리기는 이 한계를 사전에 검사하는 첫 번째 지점이 되어야 한다.
:::

### 4.2.2 설계 시 고려사항: 방어선이자 기록 지점

입력 처리기를 설계할 때 고려해야 할 핵심 관점은 두 가지다. 첫째는 **시스템의 첫 번째 방어선**이라는 역할이고, 둘째는 **첫 번째 기록 지점(로깅 포인트)**이라는 역할이다.[^web_03]

방어선으로서의 입력 처리기는 최소한 다음 세 가지를 검사한다.

- **입력 길이 제한**: 컨텍스트 윈도우 초과를 사전에 차단한다. 허용 범위를 초과하는 입력은 잘라내거나 거부 메시지를 반환한다.
- **인코딩 정규화**: 동일한 문자가 다른 인코딩으로 들어올 때 일관된 형식으로 변환한다. 이를 처리하지 않으면 하위 단계에서 예측 불가능한 문자 처리 오류가 발생한다.
- **허용 입력 유형 명세**: 텍스트만 허용하는 시스템에 이미지 URL이나 바이너리 데이터가 들어올 경우, 이 단계에서 명확한 오류를 반환한다.

다음은 이 세 가지 검사를 의사코드(Pseudocode) 수준으로 표현한 것이다.

```
function process_input(raw_input):
    # 1. 입력 유형 검사
    if type(raw_input) is not text:
        raise InputTypeError("텍스트 입력만 허용됩니다.")

    # 2. 인코딩 정규화
    normalized = normalize_encoding(raw_input, target="UTF-8")

    # 3. 입력 길이 제한
    if token_count(normalized) > MAX_INPUT_TOKENS:
        raise InputLengthError("입력이 허용 길이를 초과했습니다.")

    # 4. 로깅
    log(input=normalized, timestamp=now())

    return normalized
```

기록 지점으로서의 역할도 방어선만큼 중요하다. 입력 처리기는 파이프라인에서 원본 사용자 입력을 처음으로 수신하는 단계이므로, 이 지점에서 입력을 기록해 두면 이후 어느 단계에서 오류가 발생하더라도 "어떤 입력이 들어왔는가"를 재현할 수 있다. 1장에서 하네스의 역할로 정의한 '입력 통제'가 실제 컴포넌트로 구현되는 지점이 바로 여기다. 입력 처리기는 하네스의 입력 통제 기능을 파이프라인 안에서 구체화한 첫 번째 구성 요소다.[^web_01]

::: tip 입력 처리기를 먼저 독립 단위로 구현하라
시스템 초기 설계 단계에서 입력 처리 로직을 다른 단계와 분리된 함수 또는 클래스로 구현해 두면, 이후 허용 입력 유형이나 길이 제한 정책이 변경될 때 해당 단계만 수정하면 된다. 입력 처리 로직이 프롬프트 구성기나 LLM 호출 코드와 뒤섞여 있으면, 정책 변경이 시스템 전체에 예측 불가능한 영향을 미칠 수 있다.
:::

입력 처리기가 제 역할을 다했을 때, 그 산출물은 "정제되고 검증된 입력"이다. 이것이 다음 단계로 넘어간다. 그런데 정제된 입력이 있다고 해서 LLM이 곧바로 이해할 수 있는 형태가 되는 것은 아니다. 시스템 지시문을 어디에 배치할 것인지, 대화 이력을 어떻게 포함할 것인지, 사용자의 요청을 어떤 구조로 감쌀 것인지—이 조합의 문제가 그다음 단계인 프롬프트 구성기(Prompt Builder)의 몫이다.

[^web_01]: a16z 레퍼런스 아키텍처. 입력 처리 단계의 역할 범주화 및 파이프라인 단계 분류의 산업계 기준점으로 활용된다.
[^web_03]: Anthropic. 입력 통제를 빌딩 블록 개념으로 설명하며, 각 처리 단계를 독립된 구성 요소로 정의하는 맥락을 제공한다.

## 4.3 프롬프트 구성기

입력 처리기가 정제한 입력이 파이프라인의 다음 단계로 전달된다. 그런데 정제된 입력이 있다고 해서 LLM이 곧바로 그것을 이해할 수 있는 것은 아니다. "사용자가 무엇을 요청했는가"만으로는 부족하다. "이 시스템이 어떤 역할을 해야 하는가", "이전 대화에서 어떤 맥락이 쌓였는가", "어떤 형식으로 답변해야 하는가"—이 모든 정보가 하나의 입력으로 조합되어야 비로소 LLM에 전달할 수 있는 형태가 갖춰진다. 이 조합의 역할을 맡는 것이 프롬프트 구성기(Prompt Builder)다.

### 4.3.1 역할과 실패 시 증상

프롬프트 구성기의 역할을 한 문장으로 정리하면 이렇다. 시스템 지시(System Instruction), 맥락(Context), 사용자 입력(User Turn), 예시(Few-shot Example)를 조합하여 LLM에 전달할 최종 입력을 구성하는 단계다.[^web_04] 이 조합은 단순한 문자열 연결이 아니다. 어떤 정보를 어떤 순서로, 어떤 구조로 배치하느냐에 따라 모델의 응답 품질이 달라진다는 것이 실증 연구를 통해 확인되어 있다.[^web_04]

그런데 많은 초기 시스템에서 이 단계는 독립된 컴포넌트로 존재하지 않는다. 비즈니스 로직 안에 프롬프트 조립 코드가 인라인으로 끼워지거나, 사용자 입력 처리와 시스템 지시문 삽입이 같은 함수 안에서 뒤섞인 형태로 구현된다. 다음은 이 패턴을 의사코드 수준으로 재현한 것이다.

```
# 나쁜 설계 사례: 비즈니스 로직, 포맷팅, 시스템 지시가 한 함수에 혼재
function handle_user_request(user_id, raw_input):
    user_profile = db.get_user(user_id)
    # 시스템 지시 + 사용자 데이터 + 비즈니스 규칙이 한 문자열 안에
    prompt = "당신은 고객 상담원입니다. "
             + "고객 등급: " + user_profile.tier + ". "
             + "반드시 정중하게 응답하고, 환불은 7일 이내만 허용됩니다. "
             + "고객 질문: " + raw_input
    response = llm.call(prompt)
    return response
```

이 구조에서 프롬프트를 수정해야 하는 상황을 생각해 보자. 예를 들어 "환불 정책이 30일로 변경되었다"는 요구사항이 들어왔다고 가정한다. 이 변경은 비즈니스 정책의 변경이지만, 위 코드에서는 프롬프트 문자열과 비즈니스 로직이 같은 함수 안에 얽혀 있기 때문에 함수 전체를 건드려야 한다. 프롬프트를 수정하면서 실수로 사용자 프로필 조회 로직에 영향을 줄 수 있고, 반대로 비즈니스 로직을 변경하다가 프롬프트 구조를 깨뜨릴 수도 있다. 회귀 테스트(Regression Test, 변경 이전에 정상이었던 기능이 변경 이후에도 정상인지 확인하는 테스트)를 수행하려 해도, 테스트 대상이 "비즈니스 로직"인지 "프롬프트 텍스트"인지 구분이 되지 않는다.[^web_22]

이것이 프롬프트 구성기가 독립된 단위여야 하는 가장 직접적인 이유다. 프롬프트 변경이 비즈니스 로직 변경과 결합된 구조에서는 두 가지가 동시에 무너진다. 하나는 테스트 가능성이고, 다른 하나는 프롬프트 드리프트(Prompt Drift)에 대한 대응 능력이다.

::: note 프롬프트 드리프트(Prompt Drift)
운영 중인 시스템에서 프롬프트가 점진적으로 의도와 어긋나게 변형되는 현상. 코드 수정 과정에서 프롬프트가 조금씩 변경되거나, 서로 다른 팀원이 각기 다른 의도로 프롬프트를 수정하면서 원래의 설계 의도가 희석된다. 프롬프트 구성기가 독립된 단위로 존재하고 버전이 관리될 때만 이 현상을 추적하고 되돌릴 수 있다.[^web_19]
:::

### 4.3.2 설계 고려사항: 분리와 버전 관리

프롬프트 구성기를 독립된 단위로 설계한다는 것은 구체적으로 무엇을 의미하는가. 핵심은 두 가지다. 시스템 지시와 사용자 입력의 명확한 분리, 그리고 프롬프트 템플릿의 버전 관리다.[^web_10]

다음은 앞의 나쁜 설계 사례를 프롬프트 구성기로 분리한 구조다.

```
# 좋은 설계 사례: 시스템 지시, 맥락, 사용자 입력이 분리된 조립 구조
function build_prompt(system_instruction, context, user_turn, few_shot_examples):
    return {
        "system": system_instruction,          # 시스템 지시 (고정 또는 버전 관리)
        "context": context,                    # 외부에서 주입된 맥락 정보
        "examples": few_shot_examples,         # 예시 (선택적)
        "user": user_turn                      # 사용자 입력 (입력 처리기에서 전달)
    }

# 호출 시
prompt = build_prompt(
    system_instruction = template_store.get("customer_support_v2"),
    context            = user_profile.to_context_string(),
    user_turn          = processed_input,
    few_shot_examples  = example_store.get("refund_cases")
)
response = llm_caller.call(prompt)
```

이 구조에서 환불 정책이 변경되면 `system_instruction`에 해당하는 템플릿만 수정한다. 비즈니스 로직(`user_profile.to_context_string()`)은 변경되지 않으며, 사용자 입력도 영향을 받지 않는다. 각 요소가 독립적으로 수정, 테스트, 버전 관리될 수 있다.[^web_04]

::: tip 프롬프트 템플릿을 코드가 아닌 설정으로 관리하라
프롬프트 텍스트를 소스 코드 안에 하드코딩하는 대신, 별도의 파일이나 데이터베이스에서 로드하는 구조를 고려하라. 이렇게 하면 코드 배포 없이도 프롬프트를 수정하고 롤백할 수 있으며, 버전별 품질 차이를 추적하는 기반이 생긴다.
:::

프롬프트 구성기를 이 방식으로 설계해 두면, 5장에서 다룰 RAG(Retrieval-Augmented Generation, 외부 정보를 검색하여 프롬프트에 주입하는 구조)가 자연스럽게 연결된다. RAG의 핵심 동작은 결국 외부에서 검색된 문서를 `context` 슬롯에 주입하는 것이다. 프롬프트 구성기가 조립 구조로 분리되어 있을 때만, 이 주입이 다른 구성 요소에 영향을 주지 않고 깔끔하게 이루어질 수 있다. 프롬프트 구성기에 외부 정보를 주입하는 구조—이것이 RAG 아키텍처의 출발점이다.

프롬프트 구성기가 완성한 산출물은 LLM이 처리할 준비가 된 최종 입력이다. 이제 그 입력이 실제로 모델에 전달되어야 한다. 그런데 이 호출 역시 단순히 API를 호출하는 한 줄로 처리하기에는 고려해야 할 것들이 있다. 재시도 정책, 타임아웃, 모델 선택—이 모든 것을 담당하는 것이 다음 단계인 LLM 호출기다.

[^web_04]: 프롬프트 템플릿 논문. 프롬프트 구성의 구조화가 응답 품질에 미치는 실증적 영향을 분석하며, 조립 구조의 설계적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 구성이 시스템 설계의 일부임을 보여주는 산업적 관점. 프롬프트 중심 접근의 한계와 구성 요소 분리의 필요성을 논증한다.
[^web_19]: 프롬프트 드리프트 분석. 운영 환경에서 프롬프트가 점진적으로 변형되는 현상을 추적하고, 독립된 프롬프트 구성기의 운영적 필요성을 제시한다.
[^web_22]: 운영 실패 사례. 프롬프트와 비즈니스 로직이 혼재된 코드에서 발생하는 실제 유지보수 실패 맥락을 다룬다.

## 4.4 LLM 호출기

프롬프트 구성기가 완성한 입력은 이제 모델에 전달될 준비가 되었다. 그런데 이 전달 과정—즉 LLM 호출—을 단순히 API 호출 한 줄로 처리하는 구조가 가진 문제를 이번 절에서 집중적으로 살펴본다. 모델을 호출하는 행위 자체는 간단하다. 하지만 그 호출이 파이프라인의 나머지 로직과 뒤섞일 때 어떤 일이 벌어지는지를 이해하면, 왜 이 단계를 독립된 단위로 격리해야 하는지가 분명해진다.

### 4.4.1 역할: 호출 자체를 파이프라인의 한 단계로 두기

LLM 호출기(LLM Caller)의 역할은 단순하게 정의할 수 있다. 프롬프트 구성기에서 완성된 입력을 받아, 지정된 모델 API를 호출하고, 원시 응답(Raw Response, 파싱이나 변환이 적용되지 않은 모델의 직접 출력)을 그대로 다음 단계로 전달하는 것이다.[^web_01] 이 역할 정의에서 중요한 것은 "원시 응답을 그대로 전달한다"는 부분이다. 응답을 파싱하거나 변환하거나 검증하는 것은 이 단계의 책임이 아니다. 그것은 출력 처리기(Output Processor)의 몫이다.

그런데 많은 초기 구현에서 이 경계가 무너진다. LLM을 호출하는 코드가 비즈니스 로직, 응답 파싱, 사용자에게 반환하는 코드와 한 함수 안에 뒤섞인다. 그 결과 어떤 문제가 생기는지를 직접 비교해 보자.

```
# 나쁜 구조: LLM 호출이 비즈니스 로직 전반에 산재
function handle_user_request(user_input, user_profile):
    response = openai.chat(
        model    = "gpt-4",
        messages = [{"role": "user", "content": build_prompt(user_input)}]
    )
    # 응답 파싱, 후처리, 로깅이 모두 같은 함수 안에 존재
    text = response.choices[0].message.content
    log_usage(response.usage.total_tokens)
    return format_for_ui(text)
```

이 구조에서 모델을 GPT-4에서 Claude로 교체해야 한다고 가정하자. 변경해야 하는 것은 `openai.chat` 호출 방식뿐만 아니라, 응답 객체의 구조(`response.choices[0].message.content`와 Claude의 응답 구조는 다르다), 토큰 사용량을 읽는 방법, 오류 코드 체계까지 모두 포함된다. 이 함수만 고치면 끝나는 게 아니라, 동일한 패턴으로 작성된 다른 모든 함수를 찾아 수정해야 한다.[^web_02]

```
# 좋은 구조: LLM 호출기 레이어로 격리
function llm_caller.call(prompt, model_config):
    # 모델 공급자 추상화 레이어
    raw_response = provider_adapter.call(prompt, model_config)
    return raw_response          # 원시 응답만 반환, 파싱·가공 없음

function handle_user_request(user_input, user_profile):
    prompt      = prompt_builder.build(user_input, user_profile)
    raw         = llm_caller.call(prompt, model_config)
    processed   = output_processor.parse(raw)
    return format_for_ui(processed)
```

이제 모델 교체는 `provider_adapter` 내부 변경으로 격리된다. 비즈니스 로직, 프롬프트 구성기, 출력 처리기는 변경되지 않는다.[^web_02]

### 4.4.2 설계 고려사항: 격리가 가져오는 세 가지 이점

LLM 호출기를 독립된 레이어로 설계했을 때 얻는 이점은 크게 세 가지로 정리된다.

**첫째, 모델 교체 가능성(Model Swappability)이다.** 앞의 비교 예시에서 확인했듯, 호출 레이어가 분리되어 있으면 모델 공급자 변경의 영향 범위가 이 레이어 하나로 제한된다.[^web_02] LangChain과 LlamaIndex가 LLM 호출을 추상화하는 방식은 서로 다르지만, 두 프레임워크 모두 이 격리를 핵심 설계 목표로 삼는다는 점은 동일하다.[^web_15][^web_16] 구성 요소가 특정 모델 공급자에 종속되지 않아야 한다는 원칙—이것이 두 프레임워크가 서로 다른 API로 같은 문제를 해결하는 방식이다.

**표 4-1. LLM 호출 단계 추상화 방식 비교**

| 접근 방식 | 호출 추상화 단위 | 모델 교체 시 변경 범위 | 스트리밍 지원 |
|---|---|---|---|
| LangChain | `ChatModel` / `LLM` 인터페이스 | 모델 클래스 교체만으로 격리 | 내장 지원 |
| LlamaIndex | `LLM` 추상 클래스 | 동일 인터페이스 구현체 교체 | 내장 지원 |
| 직접 구현 | 자체 어댑터 레이어 | 어댑터 내부 변경으로 격리 | 직접 구현 필요 |

**둘째, 호출 방식 변경의 격리다.** 응답을 한 번에 받는 방식(Batch)에서 토큰이 생성되는 즉시 전달하는 스트리밍(Streaming) 방식으로 전환할 때, 이 변경이 LLM 호출기 내부에서만 일어나야 한다. 호출 방식이 비즈니스 로직 안에 직접 박혀 있다면, 스트리밍 전환은 시스템 전반의 수정을 요구한다.

**셋째, 지연(Latency)과 비용 측정 지점이 확보된다.** LLM 호출기가 독립된 레이어로 존재할 때, 이 레이어를 통과하는 모든 호출의 응답 시간과 토큰 사용량을 한 곳에서 계측할 수 있다.[^web_01] 호출 코드가 전체 코드베이스에 산재되어 있다면 이 계측은 불가능하거나 매우 번거롭다. 비용을 통제하고 성능 병목을 진단하기 위해서도 이 측정 지점은 필수적이다.

::: note 모델 공급자 추상화(Provider Abstraction)
OpenAI, Anthropic, Google 등 서로 다른 모델 공급자의 API를 동일한 인터페이스로 감싸는 설계 패턴. 호출 코드가 특정 공급자 API에 직접 의존하지 않도록 중간 레이어를 두는 것이 핵심이다. 이를 통해 모델 교체 시 변경 범위를 어댑터 레이어 안으로 제한할 수 있다.
:::

::: tip 타임아웃과 재시도는 LLM 호출기 안에서 처리하라
네트워크 불안정이나 공급자 측 일시적 장애로 인한 호출 실패는 LLM 호출기 레이어에서 타임아웃과 재시도 정책을 통해 처리하는 것이 적절하다. 이 로직을 비즈니스 로직 안에 두면 중복이 발생하고, 정책 변경 시 여러 지점을 동시에 수정해야 한다.
:::

LLM 호출 자체가 LLM 시스템의 완성이라는 오해는 여기서 명확히 반박된다. 모델 API를 호출하는 것은 파이프라인의 한 단계이지 전부가 아니다.[^web_01] LLM 호출기의 출력—원시 응답—은 아직 시스템이 사용할 수 있는 형태가 아니다. JSON으로 파싱해야 하거나, 특정 필드만 추출해야 하거나, 응답 형식이 기대와 다를 때 처리해야 하는 로직이 남아 있다. 이 모든 처리를 담당하는 다음 단계가 출력 처리기다.

[^web_01]: a16z 레퍼런스 아키텍처. LLM 파이프라인의 단계 분류와 각 단계의 역할을 산업계 기준점으로 제시한다.
[^web_02]: Chip Huyen. 모델 교체 가능성을 위한 호출 레이어 분리 원칙을 명확하게 서술한다.
[^web_15]: LangChain 프레임워크 비교. LLM 호출 추상화 방식과 모델 공급자 격리 구조를 다룬다.
[^web_16]: LlamaIndex 프레임워크 비교. LLM 호출 단계의 추상화 접근이 LangChain과 어떻게 다르며, 동일한 개념 문제를 어떻게 해결하는지를 보여준다.

## 4.5 출력 처리기

LLM 호출기가 모델의 원시 응답(Raw Response)을 받아 파이프라인으로 넘긴다는 것을 앞에서 확인했다. 그런데 그 원시 응답은 아직 시스템이 바로 사용할 수 있는 상태가 아닌 경우가 많다. 출력 처리기(Output Processor)는 그 응답을 실제로 활용 가능한 구조화된 형태로 변환하는 단계다. 이 변환 과정이 생략되거나 단순하게 취급될 때 어떤 문제가 발생하는지, 그리고 출력 처리기를 독립된 설계 단위로 두어야 하는 이유는 무엇인지를 이번 절에서 살펴보자.

### 4.5.1 원시 응답과 파싱 실패: 보이지 않는 장애

모델에게 JSON 형식으로 응답해 달라고 요청했는데, 돌아온 것이 아래와 같다고 가정하자.

```
```json
{
  "category": "환불",
  "priority": "high",
  "summary": "고객이 3일 이내 환불을 요청하고 있습니다."
}
```
```

실제로 이런 일은 드물지 않다. 모델은 지시를 따라 JSON을 생성했지만, 마크다운 코드블록으로 감싸서 반환했다. 이 응답을 그대로 `JSON.parse()`에 넘기면 파싱은 실패한다. 더 까다로운 점은, 이 오류가 어디서 발생한 것처럼 보이느냐다. 오류 메시지는 "JSON 파싱 오류"라고 표시되고, 로그를 처음 보는 사람은 LLM 호출 단계에 문제가 있다고 오해하기 쉽다.[^web_17] 실제로는 모델이 잘못된 것이 아니라, 응답을 처리하는 레이어가 없거나 불충분한 것이 원인이지만, 출력 처리기가 독립된 단위로 존재하지 않으면 이 둘을 구분할 근거가 없다.

비정형 출력이 다운스트림 컴포넌트에 그대로 전파되는 경우도 있다. 출력 처리가 파이프라인에서 별도 단계로 분리되어 있지 않으면, 모델의 응답 형식 변화—예컨대 버전 업그레이트 이후 응답 구조가 미묘하게 달라지는 경우—가 최종 사용자 화면까지 그대로 흘러들어 장애로 이어진다.[^web_17] 이때도 장애 원인을 특정하기 어렵다. "모델이 갑자기 이상해졌다"는 진단이 내려지기 쉽지만, 실제로는 출력 처리 레이어의 부재가 원인이다.

::: note 파싱 실패(Parsing Failure)
모델이 반환한 텍스트를 기대하는 데이터 구조(JSON, XML 등)로 변환하는 과정에서 발생하는 오류. 모델의 언어 생성 실패가 아니라, 응답 형식과 처리 코드 사이의 불일치에서 비롯된다. 출력 처리기가 없으면 이 오류가 상위 레이어의 문제로 잘못 귀인되기 쉽다.
:::

### 4.5.2 출력 처리기의 역할: 모델과 시스템 사이의 계약

출력 처리기는 단순히 JSON을 파싱하는 코드 한 줄이 아니다. 이 단계는 모델과 나머지 시스템 사이에서 출력 형식에 관한 계약(Contract)을 구현한다.[^web_18] 모델은 자연어를 생성하는 확률론적 시스템이고, 다운스트림 컴포넌트는 정해진 구조의 데이터를 기대한다. 출력 처리기는 이 두 세계 사이의 인터페이스 역할을 한다.

구체적으로, 출력 처리기가 수행하는 작업은 크게 세 가지로 구분된다. 첫째, **파싱(Parsing)**이다. 마크다운 코드블록을 벗겨내거나, 자연어 응답에서 구조화된 데이터를 추출하는 작업이다. 둘째, **정제(Cleaning)**다. 모델이 종종 추가하는 불필요한 접두어—"물론입니다!", "다음은 요청하신 내용입니다." 같은 문구—를 제거하거나, 일관된 표기 형식으로 정규화하는 과정이다. 셋째, **검증(Validation)**이다. 파싱된 결과가 출력 스키마(Output Schema)에 부합하는지 확인한다. 필수 필드가 누락되었거나, 값의 타입이 맞지 않거나, 허용 범위를 벗어나는 경우를 탐지한다.[^web_18]

1장에서 소개한 "출력 평가" 개념은 이 단계에서 구체적인 컴포넌트로 구현된다. 모델의 응답이 기대한 구조에 부합하는지를 판단하는 로직이 출력 처리기 안에 자리잡는다. 이 평가 없이는 파이프라인이 잘못된 출력을 정상으로 오인한 채 계속 진행되고, 오류는 훨씬 나중에, 훨씬 파악하기 어려운 형태로 표면화된다.

::: box 출력 스키마 버전 관리
모델이 교체되거나 프롬프트가 변경될 때, 출력 형식이 달라질 수 있다. 출력 스키마(Output Schema)를 명시적으로 정의하고 버전 관리하면, 스키마 변경이 다운스트림 컴포넌트에 미치는 영향을 사전에 파악할 수 있다. 이는 소프트웨어 API의 버전 관리 원칙과 동일한 논리다. 스키마가 암묵적으로 존재하면, 변경 시 영향 범위를 파악하는 데 상당한 비용이 든다.[^web_18]
:::

### 4.5.3 파싱 실패와 오류 처리기의 연계

출력 처리기는 파싱이 실패했을 때 어떻게 대응할 것인지도 설계 범위에 포함한다. 파싱 실패를 단순히 예외(Exception)로 던지고 끝내는 것은 가장 단순하지만, 가장 취약한 선택이기도 하다.[^web_21]

파싱 실패 시 대응 경로는 크게 두 가지다. 하나는 **재시도 트리거**다. 파싱에 실패했을 때 오류 처리기(Error Handler)로 신호를 보내고, 오류 처리기가 동일한 입력으로 모델을 다시 호출하도록 요청한다. 이때 재시도 프롬프트에 "응답을 JSON 형식으로만 반환하라, 마크다운 블록 없이"와 같은 보정 지시를 추가하는 경우도 있다. 다른 하나는 **폴백 경로(Fallback Path)**다. 재시도 후에도 파싱에 실패한다면, 사전에 정의된 기본값이나 오류 메시지를 반환하여 다운스트림 장애를 차단한다.[^web_21]

```
# 출력 처리기 — 파싱 실패 처리 예시 (의사코드)
function output_processor.parse(raw_response):
    cleaned = strip_markdown_fences(raw_response)
    result  = try_parse_json(cleaned)

    if result is None:
        # 오류 처리기에 재시도 요청
        error_handler.trigger_retry(reason="parse_failure")
        return None

    if not schema_validator.validate(result):
        # 스키마 불일치 → 폴백 반환
        return fallback_response()

    return result
```

이 의사코드에서 주목할 점은 출력 처리기가 파싱 결과를 직접 최종 반환하지 않는다는 것이다. 실패 시 오류 처리기와 협력하고, 스키마 검증까지 완료된 결과만 다음 단계로 전달한다. 이 협력 관계가 명시적으로 설계되어 있지 않으면, 파싱 실패는 파이프라인 전체를 예기치 않게 중단시키거나 오류 원인을 추적하기 어렵게 만든다.[^web_21]

::: tip 파싱 실패 로그는 별도 채널로 분리하라
파싱 실패 이벤트는 일반 애플리케이션 오류와 다른 채널로 기록하는 것이 좋다. 파싱 실패 빈도가 높아진다면, 그것은 모델 출력 형식이 변하고 있거나 프롬프트와 실제 출력 사이의 기대 불일치가 커지고 있다는 조기 신호일 수 있다. 이 신호를 일반 오류 로그에 묻어두면 포착하기 어렵다.
:::

출력 처리기가 정상적으로 동작하면, 그 결과—검증된 구조화 데이터—는 두 가지 방향으로 흘러간다. 최종 반환이 목적이라면 사용자나 호출 시스템에 전달된다. 대화 흐름이 계속된다면 현재 교환의 결과를 어딘가에 기록해야 한다. 모델이 이 응답을 기억하지 못한다는 점—LLM의 비상태성(Statelessness)—이 여기서 문제가 된다. 이 기록을 담당하는 것이 다음에 살펴볼 상태 관리자(State Manager)다.

[^web_17]: 출력 처리 심화. 출력 파싱 오류 유형과 파싱 실패가 상위 레이어 문제로 오인되는 메커니즘을 다룬다.
[^web_18]: data contract. 출력 처리기를 시스템 간 계약으로 보는 관점과 출력 스키마 설계 원칙을 설명한다.
[^web_21]: 파싱 오류 처리. 출력 처리기와 오류 처리기의 협력 관계, 재시도 트리거 설계를 다룬다.

## 4.6 상태 관리자

출력 처리기가 모델의 응답을 검증된 구조화 데이터로 변환하는 과정을 살펴봤다. 그런데 그 결과가 다음 대화 턴(Turn)에서도 활용되어야 한다면 어떻게 해야 할까. 모델이 직전 응답을 스스로 기억하고 있을 것이라고 기대할 수 있다면 문제가 없다. 그러나 LLM은 그런 방식으로 작동하지 않는다. 이 절에서는 그 본질적 한계에서 출발하여, 상태 관리자(State Manager)라는 구성 요소가 왜 독립된 설계 단위가 되어야 하는지를 살펴본다.

### 4.6.1 무상태 LLM과 상태 관리의 필요성

LLM은 근본적으로 무상태(Stateless) 시스템이다. 각 API 호출은 독립적으로 처리되며, 모델은 이전 호출에서 무엇이 오갔는지를 내부적으로 보존하지 않는다.[^web_08] Andrej Karpathy는 이를 간결하게 표현한 바 있다. LLM은 매 호출마다 "빈 상태로 깨어난다"는 것이다. 직전 대화를 기억하는 것처럼 보인다면, 그것은 대화 이력 전체가 매번 컨텍스트로 함께 전달되기 때문이지, 모델이 이전 상태를 저장하고 있기 때문이 아니다.

이 특성이 단순한 단발성 질의응답에서는 큰 문제가 되지 않는다. 그러나 멀티턴(Multi-turn) 대화—사용자와 시스템이 여러 번 교환을 주고받으며 맥락이 누적되는 방식—에서는 즉시 문제로 드러난다. 사용자가 세 번째 메시지에서 "아까 말한 그 조건으로 다시 계산해줘"라고 요청했다고 하자. 상태 관리자가 없는 파이프라인에서는 "아까 말한 그 조건"이 무엇인지를 복원할 방법이 없다. 이전 교환의 결과가 어디에도 기록되어 있지 않기 때문이다. 모델은 세 번째 메시지만을 입력으로 받아 처리하며, 첫 번째와 두 번째 교환에서 무슨 일이 있었는지 알지 못한다.

이 문제를 파이프라인 외부—예를 들어 애플리케이션 레이어의 전역 변수나 세션 쿠키—에서 임시로 해결하는 경우가 많다. 초기에는 작동하는 것처럼 보인다. 하지만 이 방식은 상태 관리 로직이 여러 레이어에 분산되는 결과를 낳는다. 어느 시점에 어떤 이력이 모델에 전달됐는지를 추적하기 어려워지고, 세션이 복잡해질수록 상태 불일치 버그가 발생하기 시작한다.[^web_20] 상태 관리를 명시적인 컴포넌트로 분리하는 이유가 여기에 있다.

::: note 무상태(Stateless)와 컨텍스트 전달
LLM의 무상태성은 모델이 대화를 "기억할 수 없다"는 의미가 아니라, 대화 이력이 매 호출 시 컨텍스트로 명시적으로 전달되어야 한다는 의미다. 상태 관리자는 이 전달에 필요한 정보를 수집·저장·조합하는 역할을 담당한다. 기억처럼 보이는 동작은 상태 관리자의 작업 결과다.
:::

### 4.6.2 상태의 유형과 설계 고려사항

상태 관리자가 관리하는 상태는 단일한 종류가 아니다. 실무에서는 크게 세 가지 유형으로 구분하는 것이 유용하다.[^web_03]

**표 4-1. 상태 유형 분류**

| 유형 | 내용 | 생명주기 |
|---|---|---|
| 단기 메모리(Short-term Memory) | 현재 대화 세션의 교환 이력 | 세션 종료 시 만료 |
| 장기 메모리(Long-term Memory) | 사용자 프로필, 선호도, 누적 맥락 | 세션 간 유지 |
| 실행 상태(Execution State) | 파이프라인 단계별 중간 결과 | 단일 파이프라인 실행 범위 |

단기 메모리는 현재 세션에서 오간 대화 이력이다. 모델 호출 시 이 이력을 컨텍스트로 조합하여 전달함으로써, 모델이 이전 교환을 "알고 있는" 것처럼 동작하게 만든다. 장기 메모리는 세션이 끝나도 유지되어야 하는 사용자 수준의 맥락이다. 사용자가 어떤 언어를 선호하는지, 이전 세션에서 어떤 주제를 다뤘는지 같은 정보가 여기에 해당한다. 실행 상태는 파이프라인 내부의 중간 결과로, 도구 호출 결과나 검색 결과처럼 단일 실행 내에서만 의미 있는 데이터다.

이 세 가지를 구분하지 않고 하나의 저장소에 섞어 두면, 컨텍스트 윈도우 예산(Context Window Budget) 관리가 어려워진다.[^web_08] 컨텍스트 윈도우에는 한계가 있고, 모든 이력을 매번 전부 전달할 수는 없다. 단기 메모리에서 얼마를 포함할지, 장기 메모리에서 어떤 항목이 현재 쿼리와 관련 있는지를 선별하는 로직이 상태 관리자 안에 위치해야 한다. 이 선별 로직이 없으면 컨텍스트 윈도우는 빠르게 오래된 정보로 가득 차고, 정작 현재 교환에 필요한 맥락이 누락되는 역설이 발생한다.

Google의 Agent Development Kit(ADK)는 상태 관리와 컨텍스트 구성을 별도의 레이어로 명시적으로 분리한 사례다.[^web_20] 어떤 상태를 보존할지와 그 상태를 어떻게 컨텍스트로 조합할지를 독립적으로 설계함으로써, 두 관심사가 하나의 코드에 뒤엉키는 상황을 방지한다. 이는 큰 규모에서의 설계 결정이지만, 그 원칙은 규모에 무관하게 적용된다.

::: tip 상태 저장소는 파이프라인과 독립적으로 유지하라
상태 저장소(메모리 내 딕셔너리든 외부 데이터베이스든)가 특정 파이프라인 구현에 강하게 결합되면, 파이프라인을 교체하거나 테스트할 때 상태도 함께 갱신해야 하는 문제가 생긴다. 상태 저장소를 독립적인 인터페이스 뒤에 두면, 파이프라인은 상태의 구체적인 저장 방식을 알 필요 없이 읽기·쓰기만 요청할 수 있다.
:::

상태 관리자의 중요성은 단일 파이프라인 실행에서 그치지 않는다. 6장에서 다룰 에이전트는 하나의 목표를 달성하기 위해 파이프라인을 여러 번 반복 실행한다. 이때 각 실행 간에 정보가 이어지지 않으면, 에이전트는 매 실행마다 처음부터 다시 시작하는 것과 다름없어진다. 상태 관리자가 실행 간 정보를 연결하는 구조가 되어 있어야, 에이전트가 진전을 누적하며 목표에 접근할 수 있다. 이 연결이 에이전트 아키텍처에서 상태 관리자가 핵심적인 위치를 차지하는 이유다.

상태가 올바르게 관리되더라도, 파이프라인은 언제나 실패할 가능성을 안고 있다. 모델 호출이 타임아웃될 수도 있고, 파싱이 예기치 않게 실패할 수도 있다. 이러한 실패를 예외적인 사건이 아니라 설계 대상으로 다루는 것이 다음에 살펴볼 오류 처리기(Error Handler)의 역할이다.

[^web_08]: 컨텍스트 관리. stateless LLM의 본질에서 상태 관리 필요성을 도출하는 논거, Karpathy 인용 포함.
[^web_20]: Google ADK 사례. 상태 관리와 컨텍스트 구성 분리를 실제 Big Tech 결정으로 뒷받침.
[^web_03]: Anthropic. 메모리와 상태를 빌딩 블록으로 다루는 관점.

## 4.7 오류 처리기(Error Handler)

상태 관리자가 LLM의 무상태성을 보완하여 대화의 연속성을 확보한다면, 오류 처리기는 그 연속성이 실패로 인해 끊기지 않도록 방어하는 역할을 맡는다. 상태가 올바르게 유지되더라도, 파이프라인은 언제든 실패할 수 있다. 모델 호출이 네트워크 지연으로 타임아웃될 수도 있고, 응답 파싱이 예상치 못한 형식에서 실패할 수도 있다. 이 절에서는 그러한 실패를 예외적인 사건이 아닌 설계 대상으로 다루는 방법을 살펴본다.

### 4.7.1 실패는 예외가 아니라 정상 운영의 일부다

LLM 파이프라인을 처음 구현할 때 오류 처리는 흔히 나중으로 미뤄진다. 기본 흐름이 잘 동작하는 것을 먼저 확인한 다음, "오류가 발생하면 그때 처리하자"는 식이다. 이 접근이 초기에는 합리적으로 느껴지지만, 운영 환경에서는 비용이 급격히 커진다. LLM 기반 시스템에서 실패는 간헐적으로 발생하는 예외적 사건이 아니라, 정상적인 운영 과정에서 예측 가능한 빈도로 반복되는 현상이기 때문이다.[^web_06]

외부 API에 의존하는 시스템에서 네트워크 불안정이나 서비스 일시 중단은 언제든 발생할 수 있다. 모델이 비정형적인 출력을 반환하는 경우도 마찬가지다. 이러한 실패를 처리하는 로직이 파이프라인 곳곳에 흩어져 있거나, 아예 존재하지 않는다면, 한 단계의 실패가 파이프라인 전체를 중단시키는 결과로 이어진다. 사용자는 오류 메시지를 받거나 아무런 응답도 받지 못한다. 운영 팀은 로그에서 실패 원인을 추적하려 하지만, 어느 단계에서 무엇이 잘못됐는지를 특정하기 어렵다.

오류 처리기를 파이프라인의 독립적 구성 요소로 분리해야 하는 이유는 여기에 있다. 각 단계의 실패 유형이 다르고, 그에 적합한 대응 방식도 다르다. 단일한 예외 핸들러가 모든 실패를 동일하게 처리하는 구조는, 입력 형식 오류와 API 타임아웃과 파싱 실패를 구분하지 못한다.[^web_07] 이 세 가지는 원인도, 적절한 대응도 전혀 다르다.

### 4.7.2 단계별 실패 유형과 대응 전략

파이프라인의 각 단계는 고유한 실패 유형을 가진다. 이를 구분하지 않으면 대응 전략도 획일화될 수밖에 없다.

**표 4-2. 파이프라인 단계별 실패 유형과 대응**

| 단계 | 주요 실패 유형 | 적합한 대응 |
|---|---|---|
| 입력 처리기 | 형식 오류, 인코딩 실패, 길이 초과 | 입력 거부 및 사용자 피드백 반환 |
| 프롬프트 구성기 | 템플릿 렌더링 실패, 컨텍스트 누락 | 기본값 적용 또는 단순화된 프롬프트로 폴백 |
| LLM 호출기 | 타임아웃, API 오류, 속도 제한(Rate Limit) | 재시도(Retry), 대체 모델 폴백 |
| 출력 처리기 | 파싱 실패, 스키마 불일치, 형식 오류 | 재시도 요청 또는 원시 텍스트(Raw Text) 반환 |

출력 처리기의 파싱 실패는 오류 처리기와 협력하는 계층 구조가 필요한 대표적 사례다.[^web_21] 모델이 유효한 JSON을 반환하지 않았을 때, 출력 처리기가 이를 즉시 오류로 상위에 전파하는 것이 항상 최선은 아니다. 파싱 실패를 오류 처리기에 전달하면, 오류 처리기는 재시도 조건을 확인하고 LLM 호출기에 다시 요청할지, 아니면 더 단순한 형식의 응답으로 폴백할지를 결정할 수 있다.

세 가지 주요 대응 패턴을 정리하면 다음과 같다.

**재시도(Retry)**는 일시적인 실패—네트워크 순단, API 속도 제한 등—에 유효하다. 단, 재시도 횟수와 간격에 제한을 두지 않으면 시스템이 실패 상태에서 무한 루프에 빠질 수 있다. 지수 백오프(Exponential Backoff, 재시도 간격을 점진적으로 늘리는 방식)를 적용하는 것이 일반적이다.[^web_06]

**폴백(Fallback)**은 주요 경로가 실패했을 때 대안 경로로 전환하는 방식이다. 주 모델 호출이 실패하면 더 작고 응답 속도가 빠른 모델로 전환하거나, 파싱이 실패하면 구조화된 응답 대신 원시 텍스트를 반환하는 것이 그 예다.

**서킷 브레이커(Circuit Breaker)**는 전기 회로의 차단기에서 유래한 패턴이다. 차단기가 과부하를 감지하면 전체 회로를 일시 차단하여 더 큰 손상을 방지하듯, LLM 호출기에서 연속적인 실패가 감지되면 일정 시간 동안 호출 자체를 중단한다.[^web_06] 이미 응답 불가 상태인 서비스에 계속 요청을 보내는 것은 시스템 부하를 가중시킬 뿐이다. 서킷 브레이커는 그 낭비를 막고, 서비스가 복구될 시간을 확보한다.

::: box 서킷 브레이커 패턴의 세 가지 상태
서킷 브레이커는 **닫힘(Closed)**, **열림(Open)**, **반열림(Half-Open)** 세 가지 상태를 순환한다. 닫힘 상태에서는 정상적으로 호출이 통과된다. 실패가 임계치를 초과하면 열림 상태로 전환되어 호출이 차단된다. 일정 시간이 지난 후 반열림 상태에서 제한적으로 호출을 시도하고, 성공하면 닫힘으로 복귀한다. 이 전이 로직 자체가 오류 처리기 안에 명시적으로 위치해야 한다.
:::

::: tip 재시도 조건과 횟수 제한을 명시적으로 정의하라
재시도 로직은 반드시 최대 횟수와 대기 간격 정책을 명시해야 한다. "실패하면 재시도"만 정의하고 상한을 두지 않으면, 연속 실패 시 시스템이 응답 불가 상태로 빠질 수 있다. 어떤 오류 코드에 재시도를 적용하고 어떤 오류는 즉시 실패 처리할지도 구분해 두는 것이 좋다. 예를 들어 네트워크 타임아웃은 재시도 대상이지만, 잘못된 API 키로 인한 인증 실패는 재시도해도 결과가 바뀌지 않는다.
:::

1장에서 소개한 하네스의 "실패 처리" 기능이 실제 컴포넌트로 구체화되는 지점이 바로 여기다. 하네스가 실패를 다룰 수 있으려면, 실패를 감지하고 분류하며 대응 방식을 결정하는 논리가 명확히 위치해야 한다. 그 위치가 오류 처리기다.

지금까지 여섯 가지 구성 요소—입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기—를 각각의 역할 중심으로 살펴봤다. 각 요소를 따로 정의하는 것과, 왜 이것들이 분리된 설계 단위여야 하는지를 논증하는 것은 다른 작업이다. 다음 절에서는 이 분리가 시스템 설계에서 어떤 구조적 이점을 만들어 내는지를 구체적으로 따져본다.

[^web_06]: Maxim AI. 재시도·폴백·서킷 브레이커의 독립 레이어화 논거를 제공한다.
[^web_07]: 실패 분류 논문. 파이프라인 각 단계별 실패 유형 분류의 학술적 근거를 제공한다.
[^web_21]: 파싱 오류 처리. 출력 처리기 실패가 오류 처리기와 협력하는 계층 구조를 설명한다.

## 4.8 구성 요소 분리의 설계적 이유

지금까지 입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기—여섯 가지 구성 요소를 각각의 역할 중심으로 살펴봤다. 각 요소가 무엇을 하는지는 이제 어느 정도 분명하다. 그런데 "왜 이것들을 분리해야 하는가"라는 질문은 별도로 답할 필요가 있다. 분리는 코드를 깔끔하게 정리하는 취향의 문제가 아니라, 특정 설계 목표를 달성하기 위한 의도적인 선택이기 때문이다.

### 4.8.1 분리하지 않았을 때 발생하는 구체적 결과

분리의 가치를 가장 선명하게 드러내는 방법은 분리하지 않은 구조가 실제로 어떤 결과를 만드는지를 살펴보는 것이다.

입력 검증, 프롬프트 구성, 모델 호출, 출력 파싱이 하나의 함수 또는 하나의 클래스 안에 뒤섞여 있다고 가정해 보자. 이 구조에서 출력 파싱이 실패했을 때, 원인이 모델의 응답 형식인지, 프롬프트 구성의 문제인지, 아니면 입력 단계에서 이미 잘못된 데이터가 들어온 것인지를 알 수 없다. 로그를 아무리 들여다봐도, 어느 시점에 어느 단계가 어떤 값을 다뤘는지를 추적할 수 없다. 실패는 발생하지만 어디서 발생했는지는 특정할 수 없는 상태—이것이 분리되지 않은 구조의 첫 번째 문제다.

두 번째 문제는 교체 비용이다. 사용하던 모델을 다른 모델로 바꾸는 상황을 생각해 보자. LLM 호출기가 독립된 단위로 존재한다면, 모델 교체는 그 단위의 내부만 수정하는 일이다. 프롬프트 구성기나 출력 처리기는 영향을 받지 않는다. 반면 모든 로직이 한 덩어리로 얽혀 있다면, 모델 교체는 전체 코드를 손봐야 하는 작업이 된다. 어디까지가 모델 호출 관련 코드이고, 어디서부터가 출력 처리인지가 불분명하기 때문이다.[^web_09]

::: box 관심사 분리(Separation of Concerns)의 LLM 적용
소프트웨어 설계에서 관심사 분리(SoC, Separation of Concerns)는 각 모듈이 하나의 명확한 책임만 가져야 한다는 원칙이다. LLM 파이프라인에 이 원칙을 적용하면, 각 구성 요소는 입력을 받아 처리하고, 결과를 다음 단계로 넘기는 단일한 역할을 가져야 한다. 이 원칙이 지켜질 때, 각 구성 요소는 독립적으로 수정하거나 교체하거나 테스트할 수 있는 단위가 된다.[^web_09]
:::

### 4.8.2 분리가 만들어 내는 세 가지 설계적 이점

구성 요소를 분리했을 때 얻을 수 있는 이점은 크게 세 가지로 정리된다.

**첫째, 교체 가능성(Replaceability)이다.** 각 구성 요소가 명확한 경계를 가지면, 그 내부 구현을 다른 것으로 바꿔도 파이프라인 전체가 영향을 받지 않는다. 주 모델을 대체 모델로 전환하거나, 프롬프트 템플릿 방식을 바꾸거나, 출력 파싱 로직을 교체할 때—어느 경우든 해당 구성 요소의 경계 안에서만 변경이 일어난다. 이것은 시스템이 특정 모델이나 특정 방식에 종속되지 않는다는 의미이기도 하다.

**둘째, 독립 평가 가능성(Independent Evaluability)이다.** 파이프라인의 각 단계를 독립적으로 테스트하고 평가할 수 있어야 한다.[^web_02] 프롬프트 구성기가 올바른 컨텍스트를 조립하는지, 출력 처리기가 다양한 응답 형식을 정확히 파싱하는지—이 각각을 별도로 검증할 수 있어야, 전체 시스템 품질이 어느 단계에서 저하되는지를 특정할 수 있다. 모든 단계가 한 덩어리라면, 테스트는 전체 파이프라인의 최종 출력만 확인할 수 있을 뿐이며, 어느 단계가 실패에 기여했는지를 분리해 낼 수 없다.

**셋째, 실패 격리(Failure Isolation)이다.** 한 구성 요소의 실패가 파이프라인 전체로 전파되지 않도록 경계를 두는 것이 실패 격리의 핵심이다. 오류 처리기가 독립된 단위로 존재하고, 각 구성 요소가 실패 시 명확한 인터페이스로 오류를 상위에 전달한다면, 실패의 영향 범위는 해당 단계에 국한된다. 4.7절에서 살펴본 재시도, 폴백, 서킷 브레이커 패턴이 실제로 작동하려면, 그 전제로 각 구성 요소가 실패를 격리할 수 있는 경계를 갖고 있어야 한다.

::: tip 구성 요소의 경계를 인터페이스로 명시하라
분리의 효과는 구성 요소가 서로 어떻게 데이터를 주고받는지를 명시적으로 정의할 때 온전히 발휘된다. 각 구성 요소의 입력과 출력 형식—어떤 데이터를 받아 어떤 형태로 내보내는지—을 명확히 정의해 두면, 내부 구현이 바뀌어도 인접한 구성 요소는 영향을 받지 않는다. 이 인터페이스 정의는 코드보다 먼저, 설계 단계에서 이루어지는 것이 좋다.
:::

---

이 장에서는 LLM 파이프라인을 구성하는 여섯 가지 핵심 요소를 역할 중심으로 분해하고, 각 요소가 독립된 설계 단위여야 하는 이유를 살펴봤다. 분리는 코드 구조의 정리가 아니라, 교체 가능성·독립 평가 가능성·실패 격리라는 설계 목표를 실현하는 수단이다. 그리고 이 세 가지 이점은 시스템이 복잡해질수록, 운영 기간이 길어질수록 그 가치가 더 분명하게 드러난다.

다음 5장에서는 이 파이프라인 구조 위에 검색 기능이 결합되는 방식을 살펴본다. RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 흔히 단순한 검색 기능 추가로 이해되지만, 실제로는 파이프라인의 구성 방식 자체를 바꾸는 아키텍처적 선택이다. 4장에서 정립한 구성 요소의 언어를 갖고 있다면, RAG가 기존 파이프라인의 어느 지점을 어떻게 확장하는지를 훨씬 명확하게 읽을 수 있을 것이다.
