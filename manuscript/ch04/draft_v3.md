## API 호출 한 줄로는 충분하지 않은 이유

어느 날 서비스가 이상해지기 시작했다. 고객 응대 챗봇이 며칠 전까지만 해도 정상적으로 동작했는데, 오늘은 엉뚱한 답변을 반복하고 있다. 팀원들이 모여 원인을 찾기 시작한다. 누군가는 모델의 응답 품질이 갑자기 나빠진 것 아니냐고 말한다. 다른 사람은 어제 수정한 프롬프트가 문제인 것 같다고 한다. 모두가 의심을 갖고 있지만 아무도 확신하지 못한다. 어디서부터 손을 대야 할지 모르겠다. 이는 1장의 표 1-2에서 정의한 실패 귀인 오류의 전형적인 패턴이다.

3장에서 살펴봤듯이, LLM 기반 시스템이 운영 환경에서 실패할 때 그 원인은 모델 자체보다 시스템 설계에서 비롯하는 경우가 많다.[^web_07] 그렇다면 자연스럽게 다음 질문이 뒤따른다. "시스템은 무엇으로 구성되어 있는가?" 그리고 더 실용적인 질문은 이것이다. "어느 부분이 문제인지 어떻게 특정할 수 있는가?" 이 장은 그 두 질문에 답하는 것을 목표로 한다.

진단이 불가능한 근본 이유는 시스템이 분해되어 있지 않기 때문이다. LLM을 호출하는 코드, 프롬프트를 구성하는 코드, 응답을 파싱하는 코드, 대화 이력을 관리하는 코드가 하나의 함수에 뒤섞여 있다면, 그 함수가 잘못된 결과를 낸다는 사실만 알 뿐, 왜 그런 결과가 나오는지 추적하기 어렵다. 품질 저하가 입력 처리 문제인지, 프롬프트 구성 문제인지, 모델 자체의 문제인지, 아니면 이전 대화 이력의 오염 때문인지 알 수 없다. 이 추적 불가능성 때문에 운영 환경의 LLM 시스템은 장애가 발생해도 원인을 특정하기 어렵다.[^web_10] 이 장에서 다루는 문제는 이 파이프라인 설계 문제에 한정한다.

1장에서 다룬 비결정성(nondeterminism) 문제가 이를 더욱 복잡하게 만든다. 동일한 입력이 매번 다른 출력을 낸다는 사실은, 각 처리 단계를 분리하지 않으면 오류 원인 추적이 구조적으로 불가능하다는 것을 의미한다. 단순히 코드가 지저분해서 어려운 것이 아니라, 비결정적 출력과 분리되지 않은 구조의 조합이 진단을 원천적으로 막는다.[^web_01]

이 장에서는 그 해답을 파이프라인이라는 관점에서 찾는다. 먼저 파이프라인이라는 개념 자체를 정의하고, 이어서 여섯 개의 핵심 구성 요소를 하나씩 분해해 살펴본다. 각 구성 요소는 단순히 나열하는 방식이 아니라, "이것이 없으면 어떤 일이 벌어지는가"라는 역순 서술 방식으로 정의한다. 장의 마지막에는 이 구성 요소들이 왜 분리된 단위로 설계되어야 하는지의 논리를 정리하고, 1장에서 도입한 평가 하네스(harness)가 이 파이프라인 전체를 어떻게 감싸는지 확인한다. 이 구조를 이해하면 5장의 RAG(검색 증강 생성)와 6장의 에이전트가 파이프라인의 어느 지점을 확장하는지 파악할 수 있다.

## 파이프라인: 순서와 역할이 있는 처리 흐름

"파이프라인"이라는 단어를 들으면 LangChain이나 LlamaIndex 같은 특정 프레임워크를 먼저 떠올리는 독자가 많다. 그러나 파이프라인은 특정 도구나 라이브러리가 아니다. 파이프라인은 개념적 구조다. 순서(sequence)와 역할(role)이 명시된 처리 흐름을 파이프라인이라고 부른다. 프레임워크는 그 개념을 특정 방식으로 구현한 것일 뿐이며, 프레임워크 없이도 파이프라인 구조로 시스템을 설계할 수 있다.[^web_01] "파이프라인을 도입하겠다"는 말은 특정 라이브러리를 채택하겠다는 뜻이 아니라, 처리 흐름을 역할 단위로 분해하여 관리하겠다는 설계적 결정이다.

이 개념이 소프트웨어 개발자에게 전혀 낯선 것은 아니다. 웹 서버를 개발해본 경험이 있다면 미들웨어 체인(middleware chain)을 이미 알고 있다. HTTP 요청이 들어오면 인증 미들웨어가 먼저 처리하고, 로깅 미들웨어가 기록하고, 비즈니스 로직이 실행되고, 응답이 반환되기 전에 직렬화 레이어를 거친다. 각 미들웨어는 자신의 역할만 수행하고 다음 단계로 넘긴다. MVC(Model-View-Controller) 패턴도 마찬가지다. 요청 처리, 데이터 가공, 뷰 렌더링이 각각 독립된 레이어로 분리되어 있다. LLM 파이프라인은 이와 동일한 추상화 수준에 있다.[^web_05] 사용자 입력이 들어오면 정제 단계를 거치고, 프롬프트가 조립되고, 모델이 호출되고, 응답이 처리되고, 결과가 반환된다. 각 단계는 자신의 역할이 있고, 그 경계가 분명하다. 이것이 LLM 시스템에 적용된 관심사 분리(SoC, Separation of Concerns) 원칙이다.[^web_09]

다만 웹 미들웨어와 결정적으로 다른 점이 있다. LLM 호출 단계의 출력은 동일한 입력에도 매번 달라질 수 있다. 인증 미들웨어는 동일한 토큰에 항상 동일한 결과를 반환하지만, LLM은 그렇지 않다. 이 비결정성이 파이프라인 각 단계를 더욱 명확히 분리해야 하는 이유다. 결정론적 시스템에서는 전체 흐름을 하나로 묶어도 디버깅이 가능하지만, 비결정론적 구성 요소가 포함된 시스템에서는 각 단계의 경계가 없으면 어떤 단계가 가변적인지조차 파악할 수 없다.

파이프라인 관점이 가져다주는 가장 큰 이점은 교체·테스트·진단의 단위가 생긴다는 점이다. 각 단계가 독립적으로 분리되어 있다면, 입력 처리 단계만 따로 테스트할 수 있고, 프롬프트 구성 단계만 교체할 수 있으며, 오류가 발생했을 때 어느 단계에서 생긴 것인지 특정할 수 있다. Chip Huyen은 LLM 시스템 설계에서 "각 단계를 독립적으로 테스트해야 한다"는 원칙을 강조한다.[^web_02] 즉, 파이프라인 설계가 테스트 가능성의 전제 조건이다. 2장에서 다룬 평가 주도 개발(EDD)의 관점에서 보면, 파이프라인의 각 단계가 독립된 단위일 때 비로소 각 단계를 평가의 단위로도 다룰 수 있다. 이 파이프라인의 각 단계가 실제로 어떻게 평가 단위가 되는지는 10~12장에서 다룬다.

또 하나 명확히 해야 할 오해가 있다. 프롬프트 엔지니어링은 파이프라인 전체가 아니라, 파이프라인 안의 한 단계—프롬프트 구성기(Prompt Builder)—에 해당한다. 프롬프트를 아무리 정교하게 다듬어도, 입력 처리 단계가 잘못된 데이터를 넘기거나 상태 관리 단계가 잘못된 이력을 제공한다면 좋은 결과를 기대하기 어렵다.[^web_10] "프롬프트만 잘 쓰면 된다"는 생각은 파이프라인의 한 단계를 전체로 착각한 데서 비롯한다.

**그림 4-1. LLM 파이프라인 전체 흐름도**

```
┌──────────────────────────────────────────────────────────────────────┐
│                    평가 하네스 (Harness)                               │
│             [입력 통제 · 출력 평가 · 실패 처리]                          │
│                                                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐            │
│  │  입력    │─▶│ 프롬프트  │─▶│   LLM   │─▶│  출력    │            │
│  │  처리기  │  │  구성기   │  │  호출기  │  │  처리기  │            │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘            │
│       ↕               ↕             ↕             ↕                  │
│  ┌──────────────────────────────────────────────────────────┐       │
│  │         오류 처리기  (횡단 관심사: 각 단계 지원)            │       │
│  └──────────────────────────────────────────────────────────┘       │
│  ┌──────────────────────────────────────────────────────────┐       │
│  │         상태 관리자  (횡단 관심사: 각 단계 지원)            │       │
│  └──────────────────────────────────────────────────────────┘       │
└──────────────────────────────────────────────────────────────────────┘
```

*그림 4-1 설명: 사용자 입력이 파이프라인의 각 단계를 순서대로 통과하며 처리된다. 오류 처리기와 상태 관리자는 주 흐름 외부에서 각 단계를 지원하는 횡단 관심사(cross-cutting concerns)다. 평가 하네스는 입력 통제, 출력 평가, 실패 처리의 세 기능으로 이 전체 구조를 외부에서 감싼다(1장 참조).*

1장에서 도입한 평가 하네스(harness) 개념을 여기서 다시 확인하자. 하네스는 "반복 가능하게 실행과 평가를 수행하는 외부 구조"로 정의된다. 하네스는 입력 통제, 출력 평가, 실패 처리 세 기능으로 파이프라인 전체를 감싸는 외부 구조다(1장 참조). 그림 4-1에서 볼 수 있듯이, 하네스는 파이프라인의 특정 단계가 아니라 파이프라인 전체를 외부에서 감싸는 구조다. 파이프라인이 "무엇을 처리하는가"를 정의한다면, 하네스는 "그 처리가 잘 이루어지고 있는지 어떻게 확인하는가"를 담당한다. 두 개념은 독립적이지 않다. 하네스가 효과적으로 작동하려면 감싸는 대상—파이프라인—이 명확한 경계를 가진 단계들로 구성되어 있어야 한다. 경계가 없는 파이프라인은 하네스가 개입할 지점이 없다.

이제 파이프라인을 구성하는 여섯 개의 핵심 요소를 하나씩 살펴보자. 각 요소는 "이것이 없으면 어떤 일이 벌어지는가"라는 질문에서 출발한다. 역할(Role)을 정의하기 전에 문제를 먼저 보여주는 방식이다. 첫 번째는 파이프라인의 입구에 해당하는 입력 처리기다.

## 입력 처리기: 시스템이 받아들이는 것을 통제하라

입력 처리기(Input Processor)가 없는 시스템이 어떻게 동작하는지부터 살펴보자. 사용자가 2만 자 분량의 문서를 붙여넣으면 그 텍스트 전체가 아무런 처리 없이 다음 단계로 전달된다. 프롬프트를 구성하는 단계에서 이 텍스트를 받아 조립하고, 완성된 프롬프트가 LLM 호출 단계에 도달했을 때 비로소 컨텍스트 윈도우 초과 오류가 발생한다. 오류 메시지는 LLM 호출 단계에서 나왔지만, 실제 원인은 입력 단계에 있다. 팀은 LLM 호출 코드를 한참 들여다보다가 결국 입력 크기 제한이 없었다는 사실을 뒤늦게 발견한다. 이것이 입력 처리기 부재가 만들어내는 전형적인 오진(misdiagnosis) 상황이다.[^web_01]

**역할:** 입력 처리기의 역할은 사용자 입력을 정제·검증하여 파이프라인이 처리할 수 있는 형태로 변환하는 것이다. 구체적으로는 다음 세 가지 기능을 수행한다. 첫째, 입력의 형식을 검증한다. 기대한 인코딩 방식인지, 허용된 문자 집합인지, 필수 필드가 존재하는지를 확인한다. 둘째, 입력의 크기와 내용을 정제한다. 길이 초과 입력을 잘라내거나 분할하고, 정규화가 필요한 문자열을 처리한다. 셋째, 입력을 기록한다. 입력 처리기는 파이프라인에서 사용자 입력이 원본 형태로 존재하는 유일한 지점이므로, 여기에서의 로깅은 이후 모든 디버깅의 기반이 된다. 이 세 가지 기능은 1장에서 정의한 하네스의 "입력 통제" 기능이 구체적인 컴포넌트로 구현되는 지점이기도 하다.[^web_03]

```python
# 입력 처리기 의사코드

def process_input(user_input: str, session_id: str) -> ProcessedInput:
    # 1. 로깅: 원본 입력을 가장 먼저 기록한다
    log_input(user_input, metadata={"timestamp": now(), "session_id": session_id})

    # 2. 형식 검증
    if not is_valid_encoding(user_input):
        raise InputValidationError("지원하지 않는 인코딩 형식입니다.")

    # 3. 길이 제한 적용
    if len(user_input) > MAX_INPUT_LENGTH:
        user_input = truncate(user_input, MAX_INPUT_LENGTH)
        log_warning("입력이 최대 길이를 초과하여 잘렸습니다.")

    # 4. 정규화
    normalized = normalize_whitespace(user_input)

    return ProcessedInput(content=normalized, original_length=len(user_input))
```

**실패 시 증상:** 입력 처리기가 없거나 불완전할 때 나타나는 증상은 대부분 하위 단계에서 나타난다. LLM 호출 단계에서 컨텍스트 윈도우 초과 오류가 발생하거나, 프롬프트 구성 단계에서 예상치 못한 특수문자로 인한 형식 오류가 생기거나, 출력 처리 단계에서 인코딩 문제가 불거진다. 오류의 출처가 입력 단계임에도 증상은 다른 단계에서 발생하기 때문에, 파이프라인이 분리되어 있지 않다면 원인을 찾기까지 상당한 시간이 소요된다. 더 심각한 경우는 오류가 발생하지 않고 잘못된 데이터가 조용히 통과하는 경우다. 입력 처리기가 악성 패턴을 걸러내지 못하면, 그 패턴은 프롬프트 구성 단계를 거쳐 모델에게 전달된다.

**설계 시 고려사항:** 입력 처리기를 설계할 때 핵심 질문은 "어떤 입력이 들어올 수 있는가"가 아니라 "어떤 입력을 허용할 것인가"이다. 방어적 프로그래밍의 관점에서, 시스템이 처리할 수 있는 입력의 범위를 명시적으로 정의하고, 그 범위를 벗어나는 입력은 파이프라인에 진입하기 전에 거부하거나 변환해야 한다. 프롬프트 인젝션(prompt injection) 방어도 이 단계에서 시작된다. 단, 입력 필터링은 인젝션 공격의 첫 번째 방어선일 뿐이며, 정교한 인젝션은 패턴 매칭만으로 차단되지 않는다. 출력 검증, 권한 분리, 프롬프트 구조 설계 등 다층적 방어가 병행되어야 한다.[^web_09] 또한 입력 처리기는 시스템의 첫 번째 로깅 지점(logging point)이어야 한다. 사용자가 실제로 무엇을 입력했는지를 기록하는 것은 이후 모든 문제 추적의 출발점이 된다.

> **참고**: 이미지, 오디오, 구조화된 JSON 등 멀티모달 입력의 처리 방식은 이 장의 범위를 벗어난다. 입력 처리기의 확장 설계는 9장에서 다룬다.

정제된 입력은 이제 파이프라인의 다음 단계, 프롬프트 구성기로 넘어간다. 입력 처리기가 "무엇을 받아들일 것인가"를 결정한다면, 프롬프트 구성기는 "그것을 어떻게 모델에게 전달할 것인가"를 결정한다.

## 프롬프트 구성기: 맥락을 조립하는 단계

프롬프트 구성기(Prompt Builder)가 독립된 단계로 분리되지 않은 코드베이스를 한 번이라도 유지보수해본 사람이라면, 그 어려움을 직관적으로 이해할 것이다. 비즈니스 로직을 담은 조건 분기 사이에 프롬프트 템플릿 문자열이 산재해 있고, 사용자 등급에 따라 다른 지시문이 각기 다른 함수에 하드코딩되어 있으며, 어디선가 컨텍스트 정보가 문자열 연산으로 프롬프트에 끼워 넣어진다. 이 구조에서 프롬프트 하나를 수정하려면 코드 전체에 어떤 영향을 미치는지 파악해야 한다. 더 큰 문제는 프롬프트 구성 로직이 여러 곳에 산재한다는 점이다.[^web_22]

다음은 이 상황을 보여주는 나쁜 설계 예시다. 아래 코드에서 premium 계층에만 JSON 응답 지시가 포함된 것은 의도적인 결함 예시다. 이처럼 계층별 지시가 비대칭으로 구성되면, 같은 시스템이 사용자 등급에 따라 다른 형식의 응답을 반환하여 출력 처리기가 파싱 오류를 낸다.

```python
# 나쁜 설계: 비즈니스 로직과 프롬프트 구성이 혼재

def handle_user_request(user_id: str, message: str) -> str:
    user = get_user(user_id)
    # 비즈니스 로직과 프롬프트가 뒤섞여 있다
    if user.tier == "premium":
        prompt = f"당신은 도움이 되는 어시스턴트입니다. JSON으로 응답하세요.\n\n사용자: {message}"
    else:
        prompt = f"당신은 어시스턴트입니다.\n\n사용자: {message}"

    if user.language == "en":
        prompt += "\nRespond in English."

    # 대화 이력도 여기서 직접 조립
    history = db.get_history(user_id)
    for turn in history[-3:]:
        prompt = f"{turn['role']}: {turn['content']}\n" + prompt

    return call_llm(prompt)
```

이 코드의 문제는 프롬프트 변경이 비즈니스 로직 변경과 결합되어 있다는 것이다. 프롬프트의 어시스턴트 페르소나를 바꾸려면 이 함수를 열어야 하고, 이 함수를 열면 비즈니스 로직도 함께 노출된다. 대화 이력 조립 방식을 변경하면 프롬프트 구조도 함께 바뀐다. 어느 시점부터 코드 수정 비용이 급격히 증가하고, 변경 영향 범위를 예측하기 어려워진다.[^web_04]

**역할:** 프롬프트 구성기의 역할은 시스템 지시(system instruction), 컨텍스트(context), 사용자 입력(user turn), 예시(few-shot examples)를 조합하여 LLM에 전달할 최종 입력을 구성하는 것이다. 이 단계의 핵심은 "조합"이다. 각 구성 요소—시스템 지시, 외부 컨텍스트, 사용자 입력—는 서로 다른 출처와 생애 주기(lifecycle)를 갖는다. 시스템 지시는 배포 시점에 정의되고, 외부 컨텍스트는 실행 시점에 주입되며, 사용자 입력은 요청마다 달라진다. 이 세 요소를 명확한 구조 없이 한 곳에서 조립하면, 각 요소의 경계가 흐려지고 변경의 영향을 통제할 수 없게 된다.[^web_19]

```python
# 좋은 설계: 프롬프트 구성기를 독립된 클래스로 분리

class PromptBuilder:
    def __init__(self, template: str, config: dict):
        self.template = template
        self.config = config

    def build(self, user_input: str, context: str, session_id: str) -> str:
        """
        시스템 지시, 컨텍스트, 사용자 입력을 조합하여 최종 프롬프트를 반환한다.
        각 구성 요소의 출처와 역할이 분리되어 있어, 개별 수정이 다른 요소에 영향을 미치지 않는다.
        """
        components = {
            'system': self.config.get('system_instruction', ''),
            'context': context,
            'user_input': user_input,
        }
        return self.template.format(**components)
```

이 구조에서는 시스템 지시를 변경하려면 `config`를 수정하면 되고, 프롬프트 템플릿을 교체하려면 `template`만 바꾸면 된다. 비즈니스 로직은 이 클래스의 존재 자체를 알 필요가 없다.

**실패 시 증상:** 프롬프트 구성기가 독립된 단위로 분리되지 않았을 때 나타나는 대표적인 증상은 프롬프트 드리프트(prompt drift, 배포된 프롬프트와 실제 동작하는 프롬프트 간 불일치 현상)다.[^web_19] 드리프트는 주로 두 경로로 발생한다. 첫째, 코드베이스 내 여러 경로에서 서로 다른 프롬프트 버전이 분기 조건에 따라 선택적으로 실행되는 경우다. 둘째, 런타임 환경에서 외부 컨텍스트(검색 결과, 사용자 이력 등)가 예측과 다르게 프롬프트에 삽입되는 경우다. 두 경우 모두, 프롬프트 구성 로직이 단일 지점에 집중되어 있지 않으면 어떤 프롬프트가 실제로 모델에 전달되었는지 사후 추적이 어렵다. 또 다른 증상은 프롬프트 변경이 비즈니스 로직 변경과 결합되어 회귀 테스트(regression test)가 불가능해지는 것이다. 프롬프트만 바꾸고 싶어도 함수 전체를 다시 테스트해야 한다.

**설계 시 고려사항:** 프롬프트 구성기를 설계할 때 중요한 것은 프롬프트 템플릿의 버전 관리다. 프롬프트는 코드와 마찬가지로 변경 이력을 추적해야 하는 산출물이다. 특정 날짜에 어떤 프롬프트가 사용되었는지 확인할 수 없다면, 그 시점의 모델 동작을 재현하거나 원인을 소급해서 파악하기 어렵다.[^web_04] 또한 시스템 지시(system prompt)와 사용자 입력(user turn)은 명확하게 분리되어야 한다. 두 요소를 구분 없이 한 문자열로 연결하면, 시스템 지시의 의도가 사용자 입력에 의해 덮어써질 위험이 생긴다.

5장(RAG)을 위한 포석을 여기서 남겨두자. 프롬프트 구성기가 독립된 단계로 분리되어 있다면, 이 단계에 외부 정보를 주입하는 구조를 추가하는 것이 자연스럽다. 검색 결과를 컨텍스트로 전달하거나, 데이터베이스에서 관련 문서를 조회하여 프롬프트에 포함시키는 것이 RAG(검색 증강 생성)의 핵심 구조다. 프롬프트 구성기가 비즈니스 로직 안에 묻혀 있다면 이 확장은 훨씬 복잡해진다.

조립된 프롬프트는 이제 파이프라인의 핵심 단계로 넘어간다. LLM 호출기가 이것을 받아 모델에게 전달한다.

## LLM 호출기: 모델 호출을 격리된 단계로 두어야 하는 이유

많은 개발자가 LLM 시스템을 처음 만들 때 저지르는 실수 중 하나는 API 호출 코드를 비즈니스 로직 곳곳에 흩어두는 것이다. 데이터 처리 함수 안에서 직접 `openai.chat.completions.create()`를 호출하고, 조건 분기 안에서 다른 모델을 호출하고, 유틸리티 함수에서 또 다른 호출이 일어난다. 이 구조에서 GPT-4를 Claude로 교체하기로 결정하면, 코드베이스 전체에서 호출 코드를 찾아 하나씩 수정해야 한다. 또한 어느 호출이 얼마나 걸렸는지, 비용은 얼마나 발생했는지 측정할 방법도 없다. LLM 호출기(LLM Caller)를 격리된 단계로 두어야 하는 이유가 바로 여기에 있다.[^web_02]

**역할:** LLM 호출기의 역할은 프롬프트 구성기의 출력을 받아 모델 API를 호출하고, 원시 응답(raw response)을 다음 단계로 반환하는 것이다. 이 단계의 핵심은 "격리(isolation)"다. LLM 호출기는 모델 공급자(provider)와 나머지 시스템 사이의 경계를 담당한다. 이 경계 덕분에 모델을 교체할 때 영향 범위가 이 단계 안에 한정되고, 호출 방식(동기/비동기, 스트리밍/배치)을 변경할 때도 다른 단계가 영향을 받지 않는다. 또한 모든 모델 호출이 단일 지점을 통과하기 때문에, 지연 시간(latency)과 비용 측정, 호출 로깅이 한 곳에서 처리될 수 있다.[^web_01]

```python
# LLM 호출기 의사코드

class LLMCaller:
    def __init__(self, provider: str, model: str, timeout: int):
        self.client = create_client(provider)
        self.model = model
        self.timeout = timeout

    def call(self, prompt: str) -> RawResponse:
        start_time = now()
        try:
            response = self.client.complete(
                model=self.model,
                prompt=prompt,
                timeout=self.timeout
            )
            log_call_metrics(
                model=self.model,
                latency=now() - start_time,
                tokens=response.token_count
            )
            return RawResponse(content=response.text, metadata=response.metadata)
        except TimeoutError:
            raise LLMTimeoutError(f"{self.timeout}초 내에 응답이 없습니다.")
        except APIError as e:
            raise LLMAPIError(f"API 호출 실패: {e}")
```

이 구조에서 모델을 교체하려면 `LLMCaller`를 초기화하는 설정 값만 바꾸면 된다. 나머지 파이프라인은 변경 없이 동작한다.

**실패 시 증상:** LLM 호출기가 나머지 로직과 분리되어 있지 않으면 모델 교체 시 전체 코드베이스 수정이 필요하다. 이것이 가장 직접적인 증상이다. 덜 가시적이지만 더 심각한 증상은 측정 지점의 부재다. 호출 비용이 어디서 발생하는지, 어떤 요청이 가장 오래 걸리는지, API 오류가 얼마나 자주 발생하는지 알 수 없다. 이 정보가 없으면 시스템 최적화나 운영 결정을 데이터에 근거하여 내리기 어렵다. 또한 LLM 호출이 여러 코드 경로에 흩어져 있으면 타임아웃 설정, 재시도 정책, 오류 처리가 일관성 없이 구현되어 장애 상황에서 예측할 수 없는 동작을 보인다.[^web_02]

다음은 세 가지 주요 LLM 통합 방식이 호출 레이어를 어떻게 다루는지를 간략히 비교한 것이다.

| 접근 방식 | 호출 추상화 방식 | 모델 교체 용이성 |
|---|---|---|
| LangChain | LLM/ChatModel 추상 클래스 | 높음 (공급자만 교체) |
| LlamaIndex | LLM 인터페이스 | 높음 (서비스 컨텍스트 교체) |
| 직접 구현 | 직접 설계한 호출기 클래스 | 설계에 따라 다름 |

세 방식이 구체적인 구현은 다르지만, 모두 동일한 개념 문제—모델 호출을 나머지 로직으로부터 격리하는 것—를 해결하고 있다는 점이 중요하다.[^web_15] 특정 프레임워크를 사용하든 직접 구현하든, LLM 호출기라는 개념적 단위는 파이프라인 어딘가에 반드시 존재해야 한다.

**설계 시 고려사항:** LLM 호출기를 설계할 때 중요한 세 가지 결정이 있다. 첫째, 모델 공급자 추상화다. 한 공급자에만 의존하는 클라이언트를 직접 사용하는 대신, 공급자를 교체할 수 있는 인터페이스를 두는 것이 좋다. 둘째, 타임아웃 설정이다. LLM 응답은 수 초에서 수십 초까지 걸릴 수 있으며, 타임아웃 없이 호출하면 하나의 느린 응답이 전체 시스템 응답성에 영향을 준다. 셋째, 스트리밍 응답 처리다. 대부분의 LLM API는 스트리밍 방식을 지원하며, 사용자 경험을 위해 전체 응답이 완성되기 전에 부분 응답을 전달하는 구조가 필요할 수 있다. 이 세 결정이 LLM 호출기 안에 캡슐화되어 있다면, 나머지 파이프라인은 이 세 가지 사항을 신경 쓰지 않아도 된다.

"LLM 호출 = LLM 시스템 완성"이라는 오해를 여기서 명시적으로 반박하자. LLM 호출기는 파이프라인의 핵심 단계이지만, 그것만으로 시스템이 완성되지 않는다. 모델이 반환한 원시 응답은 아직 시스템이 사용할 수 있는 형태가 아닐 수 있다. 그것을 처리하는 것이 다음 단계인 출력 처리기의 역할이다.

## 출력 처리기: 모델 응답을 시스템이 사용할 수 있는 형태로

JSON 형식으로 응답해달라고 LLM에게 요청한 경험이 있다면, 모델이 항상 깔끔한 JSON을 반환하지 않는다는 것을 알고 있을 것이다. 때로는 JSON을 마크다운 코드 블록(` ```json ... ``` `)으로 감싸서 반환하고, 때로는 JSON 앞뒤로 설명 문장을 붙이고, 때로는 JSON 구조 자체가 요청한 스키마와 다르다. 출력 처리기(Output Processor)가 없다면, 이 원시 응답이 다음 단계로 그대로 전달된다. 다음 단계에서 파싱 오류가 발생하면, 그것이 모델의 응답 형식 문제인지, 파싱 코드의 버그인지, 스키마 정의 문제인지 구분하기 어렵다. 오류의 원인이 LLM 호출 단계에 있는 것으로 오인되어, 모델 교체나 프롬프트 수정을 먼저 시도하게 된다.[^web_17]

**역할:** 출력 처리기의 역할은 모델의 원시 응답을 파싱·정제·변환하여 다운스트림 컴포넌트(downstream component, 파이프라인의 후속 단계 또는 최종 소비자)가 사용할 수 있는 구조화된 형태로 변환하는 것이다. 이 단계는 모델과 나머지 시스템 사이의 계약(contract)을 구현한다.[^web_18] 모델이 어떤 형식으로 응답하든, 출력 처리기를 통과한 데이터는 항상 동일한 구조를 가져야 한다. 이 보장이 있을 때 다운스트림 컴포넌트는 모델의 응답 형식 변동성에 무관하게 동작할 수 있다. 1장에서 정의한 하네스의 "출력 평가" 기능은 이 단계에서 처리된 결과를 대상으로 작동한다. 정제되지 않은 원시 응답을 평가하는 것은 의미가 없다.

```python
# 출력 처리기 의사코드

class OutputProcessor:
    def __init__(self, output_schema: dict):
        self.schema = output_schema

    def process(self, raw_response: RawResponse) -> ProcessedOutput:
        text = raw_response.content

        # 마크다운 코드 블록 제거
        text = strip_markdown_code_blocks(text)

        # JSON 파싱 시도
        try:
            parsed = parse_json(text)
        except JSONParseError:
            # 파싱 실패 시 오류 처리기에 신호
            raise OutputParseError(
                message="JSON 파싱 실패",
                raw_content=text,
                retry_possible=True
            )

        # 스키마 검증
        validation_result = validate_against_schema(parsed, self.schema)
        if not validation_result.is_valid:
            raise OutputSchemaError(
                message=f"스키마 불일치: {validation_result.errors}",
                parsed_content=parsed
            )

        return ProcessedOutput(data=parsed, validation_status="ok")
```

**실패 시 증상:** 출력 처리기가 없거나 불완전할 때 가장 흔한 증상은 파싱 오류가 상위 단계 문제로 오인되는 것이다. 앞서 언급한 마크다운 감싸기 문제 외에도, 모델이 스키마 구조를 미묘하게 변경하거나 필드 이름의 대소문자를 다르게 쓰는 경우가 있다. 이때 비정형 출력이 출력 처리 없이 다운스트림으로 전파되면, 그 오류는 훨씬 나중에, 훨씬 이해하기 어려운 형태로 나타난다. 최악의 경우 오류 없이 잘못된 데이터가 조용히 처리되어 데이터베이스에 저장되기도 한다.[^web_21]

**설계 시 고려사항:** 출력 처리기를 설계할 때 핵심은 출력 스키마(output schema)를 명시적으로 정의하고 버전으로 관리하는 것이다. 스키마가 암묵적으로 코드에 내재되어 있으면, 모델의 응답 형식이 바뀌었을 때 어디가 영향을 받는지 파악하기 어렵다. 또한 파싱 실패 탐지 조건을 명확히 정의하고, 실패 시 오류 처리기와 어떻게 연계할지를 설계 단계에서 결정해야 한다. 재시도를 시도할 것인지, 폴백 응답을 사용할 것인지, 사용자에게 오류를 전달할 것인지의 판단은 출력 처리기가 단독으로 결정하지 않고 오류 처리기에 신호를 보내는 방식이 더 바람직하다. 처리 성공 여부에 따라 결과는 상태 관리자에 기록되거나 오류 처리기로 전달된다.

정상적으로 처리된 출력은 이제 두 방향으로 흐를 수 있다. 최종 응답으로 반환되거나, 상태 관리자에 대화 이력으로 기록된다. 여기서 파이프라인의 "운영 지지대" 역할을 하는 두 구성 요소—상태 관리자와 오류 처리기—를 살펴볼 차례다.

## 상태 관리자: LLM은 기억하지 못한다

LLM에 대한 가장 흔한 오해 중 하나는 모델이 이전 대화를 "기억한다"는 것이다. ChatGPT나 Claude와 대화할 때 이전 발언이 반영되는 것처럼 보이지만, 이것은 모델이 기억하는 것이 아니다. 매번 호출할 때마다 이전 대화 이력을 함께 전달하기 때문에 연속성이 유지되는 것처럼 보일 뿐이다. LLM은 본질적으로 무상태(stateless)다. 각 호출은 독립적이며, 호출과 호출 사이에 무언가를 기억하는 능력이 모델 자체에는 없다.[^web_08]

이 사실이 파이프라인에 어떤 함의를 갖는지 구체적인 상황으로 살펴보자. 사용자가 세 번의 메시지를 주고받은 뒤 "아까 말한 그 조건으로 계산해줘"라고 요청한다. 상태 관리자(State Manager)가 없다면 이 요청은 독립된 단일 호출로 처리된다. "아까 말한 그 조건"이 무엇인지 파이프라인 어디에도 존재하지 않기 때문에 모델은 그것을 알 수 없다. 시스템은 명백히 잘못된 응답을 내놓거나, 조건이 무엇인지 다시 물어봐야 하는 상황이 된다.[^web_08]

**역할:** 상태 관리자의 역할은 대화 이력, 세션 정보, 사용자 맥락 등 파이프라인 실행 간에 유지되어야 하는 상태를 관리하는 것이다. 중요한 것은 상태를 "어디에 저장하는가"가 아니라 "파이프라인의 어느 단계가 상태를 책임지는가"를 명확히 하는 것이다. 상태 관리를 파이프라인 외부—예를 들어 웹 프레임워크의 세션 객체나 전역 변수—에 위임하면, 파이프라인 내부에서는 상태가 어디서 오는지, 어떻게 변경되는지 추적하기 어렵다. 상태 관리자를 명시적인 컴포넌트로 분리할 때 이 책임이 한 곳에 모인다.[^web_20]

상태는 크게 세 가지 유형으로 구분할 수 있다.

- **단기 메모리(short-term memory):** 현재 대화 세션의 이력. 사용자와 어시스턴트의 발언이 시간 순서대로 유지된다. 컨텍스트 윈도우에 포함하여 모델에게 전달된다.
- **장기 메모리(long-term memory):** 세션을 넘어 유지되는 사용자 프로필, 선호도, 이전 상호작용 요약 등. 별도 저장소에 유지되며 필요할 때 컨텍스트로 주입된다.
- **작업 상태(task state):** 진행 중인 작업의 중간 결과나 단계 정보. 멀티턴으로 진행되는 복잡한 작업에서 이전 단계의 결과를 다음 단계에 전달하기 위해 사용된다.

```python
# 상태 관리자 의사코드

class StateManager:
    def __init__(self, storage: StateStorage, max_history_turns: int):
        self.storage = storage
        self.max_history_turns = max_history_turns

    def get_context(self, session_id: str) -> ConversationContext:
        history = self.storage.get_history(session_id)
        # 컨텍스트 윈도우 예산 내에서 이력을 관리
        trimmed_history = self._trim_to_budget(history, self.max_history_turns)
        user_profile = self.storage.get_profile(session_id)
        return ConversationContext(history=trimmed_history, profile=user_profile)

    def update(self, session_id: str, user_turn: str, assistant_turn: str):
        self.storage.append_history(session_id, user_turn, assistant_turn)

    def _trim_to_budget(self, history: list, max_turns: int) -> list:
        return history[-max_turns:] if len(history) > max_turns else history
```

**실패 시 증상:** 상태 관리자가 없거나 부적절하게 구현되었을 때 나타나는 증상은 멀티턴 대화에서의 맥락 단절이다. 사용자가 이전 발언을 참조하면 모델이 엉뚱한 응답을 하고, 세션별 개인화가 작동하지 않으며, 같은 질문을 반복해야 한다. 더 미묘한 증상은 컨텍스트 윈도우의 비효율적 사용이다. 상태 관리 없이 모든 이력을 무한정 축적하다 보면 컨텍스트 윈도우가 빠르게 소진되고, 오래된 관련 없는 대화가 최근의 중요한 맥락을 밀어낸다.[^web_03]

**설계 시 고려사항:** 상태 관리자를 설계할 때 핵심 질문은 "어떤 상태가 얼마나 오래 유지되어야 하는가"이다. 단기 이력은 컨텍스트 윈도우 예산(context window budget)을 초과하지 않도록 길이를 제한하고, 오래된 이력은 요약하거나 제거하는 정책이 필요하다. 장기 메모리는 언제 업데이트되고 언제 만료되는지를 명시적으로 정의해야 한다. 또한 상태 저장소는 파이프라인의 다른 부분과 독립적이어야 한다. 파이프라인을 재시작하거나 교체하더라도 상태가 유지될 수 있어야 하기 때문이다.[^web_20]

6장(에이전트)을 위한 포석을 여기서 남겨두자. 4장에서 다루는 파이프라인이 단일 실행 흐름이라면, 에이전트는 이 파이프라인을 여러 번, 동적으로 실행하는 구조다. 에이전트가 여러 번의 실행을 통해 목표를 달성하려면, 각 실행 간에 무엇을 했는지, 어떤 도구를 사용했는지, 중간 결과가 무엇인지를 기억해야 한다. 상태 관리자가 그 정보를 연결하는 핵심 구조가 된다.

상태 관리자와 함께 파이프라인의 운영 지지대를 구성하는 마지막 요소가 남아 있다. 오류 처리기다.

## 오류 처리기: 실패를 예외가 아닌 설계 대상으로

LLM 시스템에서 오류는 예외적 상황이 아니다. LLM 기반 시스템이 운영 환경에서 마주치는 실패 중 상당 부분은 예측 가능하고 반복적인 유형을 갖는다.[^web_07] API 타임아웃, 컨텍스트 윈도우 초과, JSON 파싱 실패, 네트워크 오류, 모델 공급자의 일시적 장애—이것들은 예외가 아니라 시스템 설계 시 반드시 대비해야 하는 정상적인 운영 조건이다. 오류 처리기(Error Handler)를 독립된 구성 요소로 분리하지 않으면, 각 단계에서 오류를 만날 때마다 즉흥적으로 처리하게 되고, 그 결과는 일관성 없는 사용자 경험과 예측할 수 없는 시스템 동작으로 나타난다.[^web_06]

오류 처리가 파이프라인 각 단계에 산재된 구조와 독립된 오류 처리기가 있는 구조의 차이를 구체적 상황으로 살펴보자. 전자의 구조에서 LLM 호출 타임아웃이 발생했다고 가정하자. 그 예외를 처리하는 코드가 호출 함수 안에 있을 수도 있고, 호출 함수를 부른 비즈니스 로직에 있을 수도 있고, 최상위 API 핸들러에 있을 수도 있다. 어디에 있는지는 개발자가 그때그때 결정했을 것이다. 결과적으로 동일한 유형의 오류가 시스템 내 위치에 따라 다르게 처리된다. 재시도를 하는 경우도 있고, 오류를 그대로 던지는 경우도 있고, 조용히 무시하는 경우도 있다. 이 불일관성이 장애 상황에서 예측 불가능한 동작을 만들어낸다.[^web_06]

**역할:** 오류 처리기의 역할은 파이프라인 각 단계에서 발생하는 실패를 탐지하고, 실패 유형에 따른 적절한 대응(재시도·폴백·알림)을 실행하는 것이다. 이 단계가 독립적으로 분리되어야 하는 이유는 적절한 대응이 실패 원인에 따라 달라지기 때문이다. 모든 오류를 동일하게 처리하면 회복 가능한 오류를 불필요하게 사용자에게 노출하거나, 반대로 심각한 오류를 조용히 무시할 수 있다.[^web_21]

파이프라인 단계별 실패 유형과 그에 적합한 대응을 정리하면 다음과 같다.

| 단계 | 대표적 실패 유형 | 적합한 대응 |
|---|---|---|
| 입력 처리기 | 형식 오류, 길이 초과 | 즉시 거부, 사용자 피드백 반환 |
| 프롬프트 구성기 | 템플릿 변수 누락, 컨텍스트 조립 실패 | 기본값 적용 또는 오류 반환 |
| LLM 호출기 | 타임아웃, API 오류, 요청 한도 초과 | 재시도 (지수 백오프), 폴백 모델 |
| 출력 처리기 | 파싱 실패, 스키마 불일치 | 재시도 요청 또는 폴백 응답 |

서킷 브레이커(circuit breaker) 패턴은 오류 처리기의 중요한 설계 요소다. 전기 회로의 차단기가 과부하를 감지해 전체 시스템을 보호하듯, LLM 호출기에서 연속 실패가 일정 횟수 이상 발생하면 추가 호출을 일시 중단하고 즉시 폴백 응답을 반환하는 패턴이다. 이 패턴 없이 연속 실패를 계속 재시도하면, 이미 과부하 상태인 API 서버에 요청이 집중되어 문제가 악화될 수 있다.[^web_06]

```python
# 오류 처리기 의사코드

class ErrorHandler:
    def __init__(self, retry_policy: RetryPolicy, fallback: FallbackHandler):
        self.retry_policy = retry_policy
        self.fallback = fallback
        self.failure_counter = FailureCounter()

    def handle(self, error: PipelineError, stage: str) -> HandlingResult:
        log_error(error, stage=stage)
        self.failure_counter.increment(stage)

        # 서킷 브레이커: 연속 실패가 임계값 초과 시 폴백 즉시 반환
        if self.failure_counter.exceeds_threshold(stage):
            return self.fallback.get_response(error)

        # 재시도 가능한 오류인지 판단
        if error.retry_possible and self.retry_policy.should_retry(error):
            return HandlingResult(action="retry", delay=self.retry_policy.next_delay())

        # 재시도 불가 또는 재시도 한도 초과
        return self.fallback.get_response(error)
```

**실패 시 증상:** 오류 처리기가 없거나 각 단계에 산재되어 있을 때 나타나는 증상은 한 단계의 오류가 파이프라인 전체를 중단시키는 것이다. 재시도로 회복 가능한 일시적 API 오류가 사용자에게 노출되거나, 폴백 응답이 없어 서비스 불가 상태가 발생한다. 또한 실패 원인 유형에 무관한 획일적 처리로 인해, 입력 형식 오류(사용자 측 문제)와 API 타임아웃(시스템 측 문제)이 동일하게 처리되어 사용자에게 적절한 피드백을 줄 수 없다.

**설계 시 고려사항:** 오류 처리기를 설계할 때 중요한 세 가지 결정이 있다. 첫째, 재시도 조건과 횟수 제한이다. 모든 오류에 재시도를 적용하는 것은 위험하다. 입력 형식 오류처럼 재시도해도 결과가 달라지지 않는 오류에는 재시도가 의미 없다. 둘째, 폴백 정책(fallback policy)이다. LLM 호출이 실패했을 때 더 저렴하거나 빠른 대체 모델로 폴백할 것인지, 캐시된 응답을 반환할 것인지, 사용자에게 오류를 알릴 것인지를 미리 정의해야 한다. 셋째, 실패 로깅과 알림이다. 어떤 단계에서 어떤 유형의 오류가 얼마나 자주 발생하는지를 기록하는 것이 1장에서 정의한 하네스의 "실패 처리" 기능과 연결되는 지점이다.

여섯 개의 구성 요소를 각각 살펴봤다. 이제 이 구성 요소들을 왜 분리된 단위로 설계해야 하는지의 논리를 정리할 차례다.

## 왜 분리하는가: 교체 가능성과 평가 가능성

앞의 여섯 섹션이 "각 구성 요소가 무엇인가"를 다뤘다면, 이 섹션은 "왜 그것들을 분리해야 하는가"를 다룬다. 분리는 코드 정리 취향이나 개발자의 강박이 아니다. 그것은 교체 가능성(replaceability), 독립 평가 가능성(independent evaluability), 실패 격리(failure isolation)라는 세 가지 설계 목표를 달성하기 위한 의도적 선택이다.[^web_09]

분리하지 않았을 때 어떤 일이 생기는지를 먼저 보자. 입력 처리, 프롬프트 구성, LLM 호출, 출력 처리가 하나의 함수 안에 있다고 가정하자. 이 시스템에서 다음 질문에 답할 수 있는가?

- 이번 달 LLM 호출 평균 응답 시간은 얼마인가?
- 프롬프트 템플릿을 수정했을 때 출력 품질이 어떻게 변했는가?
- JSON 파싱 실패는 전체 요청 중 몇 퍼센트에서 발생하는가?
- 입력 길이와 모델 응답 품질 사이에 상관관계가 있는가?

단일 함수 구조에서는 이 질문들에 답하기 위해 로깅, 측정, 분석 코드를 함수 전체에 섞어 넣어야 한다. 그 결과 함수는 더욱 복잡해지고, 측정 코드와 비즈니스 로직이 뒤섞인다. 각 질문에 독립적으로 답하려면, 각 질문에 해당하는 단계가 독립된 단위로 존재해야 한다.[^web_02]

**교체 가능성**의 의미는 단순히 모델을 바꾸기 쉽다는 것에 그치지 않는다. LLM 호출기가 격리되어 있으면 모델을 교체해도 나머지 파이프라인이 영향을 받지 않는다. 프롬프트 구성기가 독립되어 있으면 프롬프트 템플릿을 교체해도 호출 코드를 수정할 필요가 없다. 출력 처리기가 분리되어 있으면 출력 스키마가 바뀌어도 파싱 코드만 수정하면 된다. 이 각각의 교체가 다른 단계에 영향을 주지 않는다는 보장이 있을 때, 시스템은 변화에 유연하게 대응할 수 있다.[^web_09] 구성 요소가 많다고 해서 복잡해지는 것이 아니다. 오히려 단계 분리가 충분하지 않을 때 변경의 영향 범위가 통제 불가능하게 확산된다.

**독립 평가 가능성**은 각 단계를 개별적으로 측정하고 개선할 수 있다는 것을 의미한다. 입력 처리기에 들어오는 입력 분포를 분석하면 어떤 유형의 입력이 많은지 파악할 수 있다. 프롬프트 구성기의 출력을 저장하면 어떤 프롬프트가 실제로 모델에게 전달되었는지 추적할 수 있다. 출력 처리기의 파싱 성공률을 측정하면 모델 응답 형식의 일관성을 정량적으로 평가할 수 있다. 이 평가들은 파이프라인 전체에 대한 것이 아니라 각 단계에 대한 것이다. 단계가 분리되어 있을 때 비로소 각 단계를 평가의 단위로 다룰 수 있다.[^web_02] 2장에서 다룬 평가 주도 개발(EDD)의 관점에서 보면, 파이프라인의 각 단계가 독립된 단위일 때 비로소 각 단계를 평가의 단위로도 다룰 수 있다.

**실패 격리**는 한 단계의 실패가 다른 단계에 전파되지 않도록 경계를 유지하는 것이다. 출력 처리기에서 파싱 실패가 발생했을 때, 그 실패가 LLM 호출기의 문제로 오인되지 않는 것이 실패 격리다.[^web_07] 각 단계가 명확한 입력과 출력 계약을 가지고 있으면, 오류가 발생했을 때 어느 단계의 계약이 위반되었는지 즉시 특정할 수 있다. 이것이 이 장 도입부에서 제기한 "어느 부분이 문제인지 어떻게 특정할 수 있는가"에 대한 답이다.

소프트웨어 공학에서 이 원칙은 관심사 분리(SoC, Separation of Concerns)로 오래전부터 알려져 있다.[^web_09] LLM 파이프라인이 전혀 새로운 아키텍처 도전을 제기하는 것이 아니다. 다만 비결정론적 구성 요소가 포함됨으로써 이 원칙의 중요성이 더욱 커진다는 점에서 차이가 있다. 결정론적 시스템에서는 단계 구분이 없어도 입출력 추적이 가능하지만, 비결정론적 구성 요소가 포함된 시스템에서는 단계 분리 없이는 어떤 단계가 결과 변동의 원인인지 구분할 수 없다.

## 정보 흐름: 데이터가 파이프라인을 어떻게 통과하는가

파이프라인의 구성 요소를 정의하는 것만으로는 충분하지 않다. 구성 요소 간에 데이터가 어떻게 흐르는지—무엇이 입력되고 무엇이 출력되는지, 그 형태가 어떻게 변형되는지—를 명시적으로 설계하는 것이 정보 흐름(information flow) 설계다. 정보 흐름이 명시적으로 정의되지 않으면, 각 단계는 독립적으로 존재하지만 그 경계에서 어떤 변환이 일어나는지 불분명한 상태가 된다.[^web_13]

각 단계의 입력과 출력을 정의하면 다음과 같다.

| 단계 | 입력 | 출력 |
|---|---|---|
| 입력 처리기 | 원시 사용자 입력(raw string) | 정제된 입력 객체(ProcessedInput) |
| 프롬프트 구성기 | 정제된 입력 + 컨텍스트 | 최종 프롬프트 문자열(prompt string) |
| LLM 호출기 | 최종 프롬프트 문자열 | 원시 응답(RawResponse) |
| 출력 처리기 | 원시 응답 | 구조화된 출력 객체(ProcessedOutput) |
| 상태 관리자 | 대화 이력 업데이트 요청 | 업데이트된 세션 상태 |

이 표에서 중요한 것은 각 단계의 출력이 다음 단계의 입력으로 그대로 전달되지 않는다는 점이다. 형태가 변형된다. 원시 사용자 입력은 `ProcessedInput` 객체로 변환되고, 그것은 다시 프롬프트 문자열로 조립되고, 모델의 원시 응답은 구조화된 출력 객체로 변환된다. 이 변환들이 명시적으로 정의되어 있을 때, 각 단계의 입출력 계약이 명확해지고 단계 간 의존성이 통제된다.[^web_18]

정보 흐름 설계에서 자주 간과되는 것은 메타데이터의 흐름이다. 사용자 입력의 원래 길이, 프롬프트 구성에 사용된 컨텍스트의 출처, 모델 호출의 소요 시간, 파싱에 걸린 처리 시간—이런 메타데이터는 최종 응답에 포함되지 않지만, 시스템 진단과 성능 최적화에 필수적이다. 이 메타데이터가 각 단계를 통과하며 누적되는 방식을 처음부터 설계하지 않으면, 나중에 필요한 정보를 추가하기 위해 파이프라인 전체를 수정해야 하는 상황이 생긴다.[^web_17]

정보 흐름 개념은 5장의 RAG를 이해하는 데 직접적으로 연결된다. RAG는 새로운 종류의 파이프라인이 아니다. 프롬프트 구성기 단계에 외부 정보를 주입하는 경로가 추가되는 것이다. 검색 시스템에서 반환된 문서들이 프롬프트 구성기의 컨텍스트 입력으로 전달된다. 이때 정보 흐름의 관점에서 RAG의 핵심 설계 질문은 "어떤 정보를 어느 단계에 주입할 것인가"와 "그 정보의 형태를 어떻게 변환할 것인가"다. 파이프라인의 정보 흐름 개념을 이해하고 있으면, RAG가 어떤 지점을 확장하는지 자연스럽게 파악할 수 있다.

## 이 장을 마치며: 부품을 아는 것이 설계의 시작이다

이 장에서 살펴본 여섯 개의 구성 요소를 간략히 정리하면 다음과 같다.

**표 4-1. LLM 파이프라인 핵심 구성 요소 요약**

| 구성 요소 | 핵심 역할 | 분리의 핵심 이유 |
|---|---|---|
| 입력 처리기 | 사용자 입력 검증·정제·로깅 | 첫 번째 방어선이자 로깅 지점 |
| 프롬프트 구성기 | 시스템 지시·컨텍스트·사용자 입력 조합 | 프롬프트 버전 관리, 드리프트 방지 |
| LLM 호출기 | 모델 API 호출 및 원시 응답 반환 | 모델 교체 격리, 비용·지연 측정 |
| 출력 처리기 | 원시 응답 파싱·검증·구조화 | 모델-시스템 간 계약 구현 |
| 상태 관리자 | 대화 이력·세션·사용자 맥락 관리 | 파이프라인 실행 간 연속성 확보 |
| 오류 처리기 | 단계별 실패 탐지 및 대응 | 실패 격리, 일관된 회복 정책 |

이 여섯 구성 요소는 LLM 파이프라인의 최소 단위다. 실제 시스템에서는 이보다 더 세분화될 수 있고, 일부 단계가 합쳐지거나 확장될 수도 있다. 중요한 것은 특정 구성 방식이 아니라, 각 역할이 명확하게 구분되고 그 경계에서의 입출력이 정의되어 있다는 점이다.[^web_09]

1장에서 제기한 화두로 돌아가자. "AI는 왜 항상 데모까지만 잘 되는가." 그 질문의 답 중 하나가 여기에 있다. 데모 환경에서는 파이프라인이 없어도 된다. 단일 호출로 결과가 나오면 충분하다. 그러나 운영 환경에서는 입력이 예측 불가능하고, 모델이 교체될 수 있고, 오류가 반드시 발생하고, 성능을 지속적으로 개선해야 한다. 이 요구사항들은 모두 단일 API 호출로는 처리할 수 없다. 역할이 분리된 구성 요소들의 연결로만 처리할 수 있다.[^web_10]

하네스(harness)와 파이프라인의 관계도 다시 확인하자. 파이프라인은 "무엇을 어떻게 처리하는가"를 정의한다. 하네스는 "그 처리가 잘 이루어지고 있는지 어떻게 확인하는가"를 담당한다. 파이프라인의 각 단계가 명확히 분리되어 있을 때, 하네스는 각 단계에 개입하여 입력을 통제하고, 출력을 평가하고, 실패를 처리할 수 있다. 파이프라인 없이 하네스는 작동할 수 없다.

5장에서는 이 파이프라인 위에 RAG(검색 증강 생성) 구조를 올린다. RAG는 프롬프트 구성기에 외부 정보를 주입하는 구조라고 이 장에서 예고했다. 5장은 그 구조가 구체적으로 어떻게 설계되는지, 그리고 단순한 검색 기능 추가가 아니라 정보 흐름을 통제하는 아키텍처 결정임을 설명할 것이다. 이 장에서 정의한 프롬프트 구성기와 정보 흐름 개념이 5장 전체의 기반이 된다.
