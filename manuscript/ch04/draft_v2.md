# LLM 파이프라인 기본 구성 요소

## 도입: 어디서부터 생각해야 하는가

챗봇이 갑자기 이상한 답변을 내놓기 시작했다고 가정해 보자. 어제까지는 분명히 잘 동작했는데, 오늘부터는 엉뚱한 내용을 반환하거나, 형식이 뭉개지거나, 가끔은 아예 응답 자체를 거부한다. 첫 번째 시도는 프롬프트 수정이다. 지시문을 더 명확하게 고쳐 보고, 예시를 추가해 본다. 나아지는 것 같다가도 다시 무너진다. 두 번째 시도는 모델 교체다. 더 최신 버전으로 바꾸면 해결될 것이라는 기대감으로 API 엔드포인트를 바꿔 본다. 증상이 조금 달라지긴 하지만, 근본적으로는 같은 문제가 다른 형태로 재현된다. 결국 어느 지점에서 막다른 벽에 부딪힌다. 문제가 어디에 있는지 특정할 수 없기 때문이다.

이 상황이 낯설지 않다면, 그것은 시스템이 분해되어 있지 않기 때문일 가능성이 높다. LLM을 호출하는 코드 한 줄과, 사용자 입력을 받아 응답을 돌려주는 몇 개의 함수가 뒤섞인 채 하나의 덩어리처럼 동작할 때, 실패는 어디서든 발생할 수 있지만 어디서 발생했는지는 아무도 알 수 없다. 3장에서 살펴봤듯, 실패의 원인은 모델 자체가 아닌 시스템에 있는 경우가 많다[^web_07]. 그런데 그 시스템이 내부적으로 구분된 단계 없이 하나로 뭉쳐 있다면, 진단은 원리적으로 불가능하다. 프롬프트를 고치는 것은 그 덩어리의 어느 한 부분을 건드리는 일이고, 모델을 바꾸는 것은 또 다른 부분에 손을 대는 일이지만, 어느 쪽이 실제로 문제였는지는 끝내 확인할 수 없다.

"프롬프트 엔지니어링만 잘하면 된다"는 인식은 이 덩어리 구조를 전제로 한다. 프롬프트를 시스템의 전부인 것처럼 다루는 접근은, 규모가 작고 요구사항이 단순한 초기 단계에서는 어느 정도 통할 수 있다. 하지만 입력의 다양성이 높아지고, 실패에 대한 허용 범위가 좁아지고, 유지보수가 필요해지는 시점부터 이 접근은 한계를 드러낸다[^web_10]. 프롬프트는 시스템의 구성 요소 중 하나일 뿐이다. 그것이 전부인 구조에서는, 프롬프트가 원인이 아닌 실패도 프롬프트를 탓하게 된다.

1장에서 소개한 하네스(Harness) 개념을 다시 떠올려 보자. 그림 1-2는 모델 단독 실행과, 하네스가 모델을 감싸는 실행 구조를 대비해서 보여준다. 하네스는 입력을 통제하고, 출력을 평가하며, 실패를 처리하는 구조다. 그런데 하네스가 실제로 작동하려면, 그 내부가 역할 단위로 분리되어 있어야 한다. 입력을 처리하는 부분, 프롬프트를 구성하는 부분, 모델을 호출하는 부분, 응답을 처리하는 부분이 구분되어 있어야만, 각 단계에서 무엇이 들어오고 무엇이 나가는지를 추적할 수 있다. 그리고 그 추적이 가능해야, 실패가 어느 단계에서 발생했는지 특정할 수 있다.

이것이 이 장의 출발점이다. LLM을 "호출"하는 것은 한 줄의 코드로 가능하지만, LLM 시스템을 "설계"한다는 것은 실패를 진단할 수 있는 구조를 갖추는 일이다. 진단 가능성(Diagnosability)—어느 단계에서 무엇이 잘못되었는지를 특정할 수 있는 능력—이 시스템 설계의 첫 번째 목표가 되어야 한다[^web_07]. 이 장에서는 그 구조를 구성하는 부품들을 하나씩 분해해서 살펴본다. 각 구성 요소가 어떤 역할을 하는지, 그것이 독립된 단위로 존재해야 하는 이유가 무엇인지, 그리고 그것이 실패할 때 어떤 증상이 나타나는지를 정리한다.

그 전에 먼저 한 가지 관점을 정리할 필요가 있다. LLM 시스템을 "파이프라인(Pipeline)"이라는 틀로 바라보는 것, 그리고 그 틀이 왜 유용한지에 대한 이야기부터 시작하자.

[^web_07]: 실패 분류 논문. LLM 기반 시스템의 실패 원인을 분류하고, 이를 시스템 엔지니어링 문제로 프레이밍하는 학술적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 중심 접근의 산업적 한계를 다루며, 시스템 수준의 설계 전환 필요성을 논증한다.

## 파이프라인이라는 관점

앞에서 LLM 시스템의 실패가 "분해되지 않은 구조"에서 비롯된다는 점을 살펴봤다. 이제 그 분해를 어떤 방식으로 접근할 것인지, 그 틀부터 정리하자.

"파이프라인(Pipeline)"이라는 단어를 들으면 LangChain이나 LlamaIndex 같은 특정 프레임워크를 떠올리기 쉽다. 이 두 가지를 동일시하는 것은 자연스러운 연상이지만, 그 연상이 오히려 개념을 좁히는 함정이 된다. 파이프라인은 특정 도구나 라이브러리가 아니라, 순서(Sequence)와 역할(Role)이 명시된 처리 흐름이라는 개념적 구조다[^web_01]. LangChain이 파이프라인을 구현하는 한 가지 방법일 뿐이지, LangChain 없이는 파이프라인이 존재하지 않는다는 뜻이 아니다. 파이프라인이라는 관점은 특정 프레임워크를 도입하기 전에, 시스템을 어떻게 생각할 것인가의 문제다.

소프트웨어 개발자라면 이미 이 구조와 유사한 것을 경험해 본 적이 있다. 웹 서버의 미들웨어 체인(Middleware Chain)을 떠올려 보자. HTTP 요청이 들어오면, 먼저 인증 미들웨어가 요청의 유효성을 확인하고, 그다음 로깅 미들웨어가 요청 내용을 기록하며, 마지막으로 비즈니스 로직이 실제 처리를 수행한다. 각 미들웨어는 자신의 역할만 담당하고, 다음 단계로 데이터를 넘긴다. 어느 미들웨어에서 오류가 발생했는지는 즉시 특정할 수 있다. LLM 파이프라인의 단계 분리는 이와 동일한 원리로 작동한다[^web_05]. 입력 처리, 프롬프트 구성, 모델 호출, 출력 처리 각각이 독립된 책임을 가진 레이어로 존재한다. MVC 아키텍처에서 모델, 뷰, 컨트롤러를 분리하는 이유—각 관심사(Concern)를 격리하여 변경의 파급 범위를 제한하는 것—가 LLM 파이프라인의 단계 분리에도 동일하게 적용된다.

이처럼 파이프라인 관점의 핵심 이점은, 각 단계를 독립적으로 교체·테스트·진단할 수 있는 경계가 생긴다는 데 있다[^web_02]. 단계가 분리되어 있을 때, "프롬프트 구성 로직을 변경했는데 출력 형식이 달라졌다"는 사실을 추적할 수 있다. 반대로, 단계가 뒤섞인 덩어리 구조에서는 어느 변경이 어떤 결과를 낳았는지 인과관계를 파악하기 어렵다. 이는 소프트웨어 설계 원칙인 관심사의 분리(Separation of Concerns, SoC)를 LLM 시스템에 적용한 결과이기도 하다[^web_09]. 각 단계가 단일 책임을 가질 때, 그 단계를 단독으로 평가하거나 교체하는 것이 가능해진다.

그림 4-1은 이 장에서 다룰 LLM 파이프라인의 전체 흐름을 보여준다. 사용자 입력은 입력 처리기(Input Processor)를 거쳐 정제되고, 프롬프트 구성기(Prompt Builder)가 컨텍스트와 시스템 지시를 조합한다. LLM 호출기(LLM Caller)가 모델 API를 실제로 호출하고, 출력 처리기(Output Processor)가 응답을 파싱·변환한다. 상태 관리자(State Manager)는 대화 이력과 세션을 유지하고, 오류 처리기(Error Handler)는 실패와 재시도를 담당한다. 1장의 그림 1-2에서 소개한 하네스는 이 파이프라인 전체를 외부에서 감싸는 구조로, 각 단계가 정의되어 있어야 하네스가 유의미하게 작동할 수 있다. 파이프라인 없는 하네스는 감쌀 대상이 없는 껍데기에 불과하다.

여기서 한 가지 오해를 짚고 넘어갈 필요가 있다. "프롬프트 엔지니어링"은 이 파이프라인 전체가 아니라, 프롬프트 구성기 단계에 해당한다. 프롬프트를 정교하게 다듬는 작업이 중요하지 않다는 뜻이 아니다. 다만 그것이 파이프라인의 다른 단계—입력 검증 실패, 출력 파싱 오류, 상태 누락—로 인한 문제까지 해결해 주지는 않는다. 프롬프트 중심 접근이 한계를 드러내는 시점은, 대부분 파이프라인의 다른 단계가 원인인 실패를 프롬프트 탓으로 돌리기 시작할 때다[^web_10].

파이프라인이라는 관점은 진단 가능성의 기초를 제공한다. 어느 단계에서 무엇이 들어오고 무엇이 나가는지를 정의할 수 있을 때, 실패는 비로소 특정 가능한 사건이 된다. 이제 이 각 단계가 구체적으로 어떤 역할을 수행하는지 살펴볼 차례다. 파이프라인의 첫 번째 관문, 입력 처리기부터 시작하자. 사용자가 무엇을 입력하든 시스템이 안정적으로 처리할 수 있으려면, 그 입력이 내부로 들어오기 전에 어떤 작업이 필요한지를 이해하는 것이 출발점이 된다.

[^web_02]: Chip Huyen. "각 단계를 독립적으로 테스트해야 한다"는 설계 원칙을 명확하게 표현한다.
[^web_05]: Craig Risi. 기존 소프트웨어 아키텍처(MVC, 미들웨어)와 LLM 파이프라인의 구조적 유사성 및 차이를 설명한다.

## 입력 처리기: 시스템이 받아들이는 것을 통제하라

파이프라인이라는 관점이 무엇인지, 그리고 단계 분리가 왜 진단 가능성의 기초가 되는지를 정리했다. 이제 파이프라인의 첫 번째 단계인 입력 처리기(Input Processor)를 구체적으로 들여다보자.

입력 처리기의 역할을 가장 빠르게 이해하는 방법은, 그것이 없을 때 무슨 일이 벌어지는지를 먼저 살펴보는 것이다. 사용자가 1만 자 분량의 텍스트를 입력했다고 가정하자. 입력 처리기가 없는 구조에서는 이 텍스트가 아무런 검증 없이 프롬프트 구성기로 넘어간다. 프롬프트 구성기는 이 텍스트에 시스템 지시와 대화 이력을 덧붙여 LLM 호출기에 전달한다. 그리고 모델 API 호출 시점에 가서야 컨텍스트 윈도우(Context Window) 초과 오류가 발생한다. 이 오류 메시지가 가리키는 위치는 LLM 호출 단계다. 그러나 실제 원인은 입력 단계에서 길이가 제한되지 않은 데 있다. 디버깅하는 엔지니어는 LLM 호출기를 먼저 들여다볼 것이고, 원인 파악에 불필요한 시간을 소모하게 된다[^web_01]. 이것이 입력 처리기 부재의 전형적인 증상이다. **실패는 하위 단계에서 발생하지만, 원인은 상위 단계에 있다.**

1장에서 "입력 통제"를 하네스 설계의 첫 번째 축으로 소개한 바 있다. 입력 처리기는 그 개념이 실제 컴포넌트로 구현되는 지점이다. a16z의 LLM 애플리케이션 아키텍처 분류에 따르면, 입력 처리 단계는 사용자 입력을 파이프라인이 처리할 수 있는 형태로 변환하는 범주로 정의된다[^web_01]. Anthropic의 빌딩 블록 관점에서도 입력 통제는 독립된 처리 단위로 다루어진다[^web_03]. 두 관점 모두 입력 처리기를 파이프라인의 장식이 아닌, 시스템 안정성의 전제 조건으로 본다.

**역할**을 한 문장으로 정리하면, 입력 처리기는 사용자 입력을 정제하고 검증하여 파이프라인 내부가 처리할 수 있는 형태로 변환하는 단계다. 이 정의에는 세 가지 동작이 포함된다. 첫째, 정제(Sanitization)—입력에 포함된 불필요한 문자, 악성 지시, 인코딩 불일치를 제거하거나 정규화한다. 둘째, 검증(Validation)—입력이 시스템이 허용하는 형식과 범위 안에 있는지 확인한다. 셋째, 변환(Transformation)—검증된 입력을 파이프라인 내부 인터페이스에 맞는 구조로 바꾼다. 이 세 동작이 입력 처리기 안에서 완결되어야 이후 단계들이 "입력은 이미 정제된 상태"라는 가정 위에서 자신의 역할에만 집중할 수 있다.

**설계 시 고려사항** 측면에서 입력 처리기는 두 가지 성격을 동시에 가진다. 첫 번째는 방어선(Defense Layer)이다. 허용 입력 유형을 명시하고, 길이 상한을 설정하고, 인코딩을 정규화하는 작업이 여기에 해당한다. 두 번째는 로깅 포인트(Logging Point)다. 파이프라인에서 외부와 내부의 경계가 되는 이 지점은, 원본 입력을 기록해 두기에 가장 적합한 위치다. 나중에 어떤 입력이 어떤 결과로 이어졌는지를 추적하려면, 변환이 일어나기 전 원본 상태가 보존되어 있어야 한다. 입력 처리기가 이 두 역할을 모두 수행할 때, 시스템은 "무엇이 들어왔는가"를 항상 설명할 수 있는 상태가 된다.

다음은 입력 처리기의 핵심 검증 로직을 의사코드 수준으로 표현한 것이다.

```
function process_input(raw_input):
    # 1. 입력 유형 검증
    if type(raw_input) not in ALLOWED_INPUT_TYPES:
        raise InputTypeError("허용되지 않는 입력 유형")

    # 2. 인코딩 정규화
    normalized = normalize_encoding(raw_input, target="UTF-8")

    # 3. 길이 제한 검증
    if len(normalized) > MAX_INPUT_LENGTH:
        raise InputLengthError(f"입력 길이 초과: {len(normalized)} > {MAX_INPUT_LENGTH}")

    # 4. 원본 입력 로깅 (변환 이전 상태 기록)
    log_input(raw_input, stage="input_processor")

    # 5. 정제 및 변환
    sanitized = sanitize(normalized)
    return sanitized
```

이 의사코드에서 주목할 점은 순서다. 로깅은 정제보다 먼저 일어난다. 원본 입력을 기록한 뒤에 변환을 수행해야, 나중에 "어떤 입력이 들어왔을 때 이 결과가 나왔는가"를 정확히 재현할 수 있기 때문이다.

**실패 시 증상**은 대개 두 가지 형태로 나타난다. 하나는 앞서 본 것처럼, 오류가 발생한 위치와 원인이 발생한 위치가 다른 경우다. LLM 호출 단계에서 발생한 오류가 실제로는 입력 길이 미검증 때문이었다면, 이 오류는 원인을 숨긴 채 잘못된 단계를 지목한다. 다른 하나는 악성 입력이 파이프라인을 그대로 통과하는 경우다. 사용자가 프롬프트 인젝션(Prompt Injection)을 시도하는 문자열을 입력했을 때, 입력 처리기가 없다면 이 문자열이 프롬프트 구성기로 그대로 전달되어 시스템 지시를 덮어쓸 위험이 생긴다[^web_03]. 입력 처리기는 이 가능성을 파이프라인 진입 시점에서 차단하는 첫 번째 방어 지점이다.

입력 처리기가 제 역할을 수행한다고 해서 모든 문제가 해결되는 것은 아니다. 정제되고 검증된 입력이 파이프라인 내부로 들어왔더라도, 그것이 실제로 유용한 프롬프트로 조합되기 위해서는 다음 단계가 필요하다. 입력 처리기가 "무엇이 들어오는가"를 통제한다면, 프롬프트 구성기(Prompt Builder)는 "그것을 어떻게 모델에게 전달할 것인가"를 결정한다. 정제된 입력을 받아 컨텍스트와 시스템 지시를 어떻게 조합할 것인지—그 설계 문제가 다음 섹션의 주제다.

[^web_01]: a16z 레퍼런스 아키텍처. LLM 시스템을 단계화된 흐름으로 분류하는 산업계 기준점으로, 입력 처리 단계의 역할 범주화 근거를 제공한다.
[^web_03]: Anthropic. 입력 통제를 독립된 빌딩 블록 개념으로 설명하며, 프롬프트 인젝션 등 입력 단계 위험에 대한 설계적 맥락을 제공한다.

## 프롬프트 구성기(Prompt Builder): 맥락을 조립하는 단계

입력 처리기가 사용자 입력을 정제하고 검증하는 역할을 담당한다면, 프롬프트 구성기는 그 결과물을 받아 모델이 실제로 소화할 수 있는 형태로 조립하는 단계다. 입력 처리기가 "무엇이 들어오는가"를 통제한다면, 프롬프트 구성기는 "그것을 어떻게 모델에게 전달할 것인가"를 결정한다. 두 단계는 역할이 명확히 다르고, 그렇기 때문에 분리되어야 한다.

프롬프트 구성기를 별도 컴포넌트로 두지 않을 때 어떤 일이 벌어지는지를 먼저 살펴보자. 실무에서 흔히 목격되는 구조는 다음과 같다.

```
function handle_user_request(user_id, raw_query):
    user = db.get_user(user_id)
    history = db.get_recent_history(user_id, limit=5)

    # 비즈니스 로직, 포맷팅, 시스템 지시가 한 함수에 뒤섞임
    prompt = f"""
    당신은 {user.subscription_tier}에 맞는 응답을 제공하는 어시스턴트입니다.
    사용자 이름: {user.name}, 가입일: {user.joined_at}
    최근 대화: {history}
    현재 질문: {raw_query}
    반드시 {user.language}로 응답하세요.
    """
    return llm_api.call(prompt)
```

이 코드에서 프롬프트를 수정하려면 비즈니스 로직 함수 전체를 열어야 한다. 사용자 구독 등급에 따른 응답 정책이 바뀌면 프롬프트 문자열도 함께 바뀌고, 프롬프트 문자열을 조정하면 데이터 포맷팅 로직과 충돌이 생길 수 있다. 이 구조에서는 "프롬프트만 바꾸는" 작업이 존재하지 않는다. 프롬프트 변경은 항상 비즈니스 로직 변경과 결합된다. 그 결과, 회귀 테스트(Regression Test)의 범위가 예측 불가능해지고, 무엇이 바뀌었을 때 모델 응답이 달라졌는지를 추적하기 어려워진다[^web_22]. 이것이 프롬프트와 로직이 뒤섞인 코드가 만들어내는 유지보수 실패의 전형적인 패턴이다.

잘 설계된 프롬프트 구성기는 조립의 재료를 명확히 분리한다. 구성 재료는 크게 세 가지다. 첫 번째는 시스템 지시(System Instruction)로, 모델이 어떤 역할을 수행하고 어떤 제약을 따를지를 명시한다. 두 번째는 맥락(Context)으로, 현재 요청과 관련된 배경 정보, 대화 이력, 외부에서 검색된 정보 등이 해당된다. 세 번째는 사용자 입력(User Turn)으로, 입력 처리기를 통과한 정제된 요청이다. 경우에 따라 퓨샷 예시(Few-shot Example)—모델에게 원하는 응답 형식을 보여주는 입출력 쌍—가 여기에 추가된다. 프롬프트 구성기의 역할은 이 재료들을 정해진 순서와 구조로 조합하여 LLM 호출기에 넘길 최종 입력을 만드는 것이다[^web_04].

같은 기능을 구성 요소를 분리하여 구현하면 다음과 같은 구조가 된다.

```
# 각 재료가 독립적으로 관리됨
system_instruction = template_store.get("assistant_v2")
context = context_builder.build(user_id, session_id)
user_turn = processed_input  # 입력 처리기 통과 후

# 프롬프트 구성기가 조립만 담당
prompt = PromptBuilder()
    .set_system(system_instruction)
    .add_context(context)
    .add_user_turn(user_turn)
    .build()
```

이 구조에서 시스템 지시를 수정하려면 `template_store`만 건드리면 된다. 맥락 구성 방식을 바꾸려면 `context_builder`만 수정하면 된다. 비즈니스 로직은 어느 쪽에도 관여하지 않는다. 각 재료가 독립된 소스를 가지기 때문에, 어떤 버전의 시스템 지시가 어떤 결과를 만들었는지를 추적하는 것도 가능해진다[^web_10].

이 추적 가능성은 단순한 편의가 아니라 운영적 필수 조건에 가깝다. 프롬프트 드리프트(Prompt Drift)—시간이 지나면서 프롬프트가 조금씩 변경되어 모델 응답의 품질이 의도치 않게 저하되는 현상—는 프롬프트가 코드 안에 직접 기술되어 있을 때 특히 감지하기 어렵다[^web_19]. 프롬프트 구성기가 독립된 단위로 분리되고, 템플릿에 버전이 부여되어야 비로소 "어느 시점에, 어떤 변경이, 어떤 응답 변화를 일으켰는가"를 비교할 수 있다. 이것이 프롬프트 구성기를 독립 컴포넌트로 두어야 하는 운영적 근거다.

한 가지 더 짚고 넘어갈 것이 있다. 이 섹션에서 정의한 프롬프트 구성기의 구조—시스템 지시, 맥락, 사용자 입력을 분리하여 조립—는 5장에서 다룰 RAG(검색 증강 생성, Retrieval-Augmented Generation)의 기반이 되는 구조이기도 하다. RAG는 외부 데이터베이스에서 검색된 정보를 맥락 슬롯에 주입하는 방식으로 동작한다. 프롬프트 구성기가 맥락을 독립된 재료로 취급하는 설계가 갖춰져 있어야, 그 슬롯에 무엇을 채울지를 런타임에 결정하는 RAG 구조가 자연스럽게 얹힐 수 있다. 지금 설계하는 프롬프트 구성기가 그 확장을 가능하게 하는 구조적 전제가 된다.

프롬프트 구성기가 최종 입력을 조립하고 나면, 그것은 파이프라인의 다음 단계로 넘어간다. 조립된 프롬프트를 실제 모델 API에 전달하고 응답을 받아오는 단계—LLM 호출기(LLM Caller)—가 그것이다. 프롬프트를 잘 만드는 것과 그것을 안정적으로 호출하는 것은 서로 다른 설계 문제이며, 다음 섹션에서는 바로 그 차이를 들여다본다.

[^web_04]: 프롬프트 템플릿 구조화에 관한 실증 연구. 프롬프트를 구조화된 형태로 관리할 때 모델 응답 품질이 유의미하게 향상됨을 보여주며, 프롬프트 구성기를 독립 단위로 설계해야 하는 실증적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 작성이 시스템 설계의 전부가 아니라 파이프라인의 한 단계임을 산업적 관점에서 설명하며, 구성 요소 분리의 필요성을 뒷받침한다.
[^web_19]: 프롬프트 드리프트 분석. 프롬프트가 독립된 버전 관리 단위로 분리되지 않을 때 발생하는 품질 저하 패턴을 분석하며, 프롬프트 구성기를 독립 컴포넌트로 운영해야 하는 근거를 제시한다.
[^web_22]: 운영 실패 사례. 프롬프트와 비즈니스 로직이 혼재된 코드베이스에서 발생한 실제 유지보수 실패 사례를 서술하며, 설계 분리의 필요성을 구체적 맥락으로 보강한다.

## LLM 호출기(LLM Caller): 모델 호출을 격리된 단계로 두어야 하는 이유

프롬프트 구성기가 최종 입력을 조립해 넘기고 나면, 파이프라인은 그 입력을 실제 모델 API에 전달해야 한다. 이 역할을 담당하는 것이 LLM 호출기(LLM Caller)다. 겉으로 보면 단순한 API 호출처럼 보이지만, 이 단계를 나머지 로직과 얼마나 깔끔하게 분리했느냐가 이후 시스템의 유연성을 결정짓는다. "LLM을 호출하는 코드 한 줄이면 충분하다"는 인식은 초기 프로토타입에서는 통하지만, 운영 환경에서는 빠르게 한계를 드러낸다.

LLM 호출기가 별도의 설계 단위가 되어야 하는 가장 직접적인 이유는 **모델 교체 가능성(Model Swappability)** 이다[^web_02]. 개발 초기에는 GPT-4를 사용하다가 비용 절감을 위해 Claude로 전환하거나, 특정 기능에 한해 오픈소스 모델을 도입하는 결정은 실무에서 드물지 않게 일어난다. LLM 호출 코드가 비즈니스 로직 전반에 산재되어 있을 때, 이런 전환이 어떤 결과를 낳는지를 먼저 살펴보자.

```python
# 호출 코드가 비즈니스 로직에 직접 내장된 구조
def generate_summary(document):
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": f"요약해줘: {document}"}]
    )
    return response.choices[0].message.content

def classify_intent(user_query):
    response = openai.ChatCompletion.create(
        model="gpt-4",
        messages=[{"role": "user", "content": f"의도 분류: {user_query}"}]
    )
    return response.choices[0].message.content
```

이 구조에서 GPT-4를 Claude로 교체하려면, `openai.ChatCompletion.create`가 호출되는 모든 지점을 찾아 수정해야 한다. 함수가 10개라면 10곳을 바꾸어야 하고, 각 호출부마다 인증 방식과 응답 형식이 달라지므로 변경 범위는 단순 치환을 넘어선다[^web_01]. 반면 LLM 호출기를 독립 레이어로 분리하면 다음과 같은 구조가 된다.

```python
# LLM 호출기를 격리된 레이어로 분리
class LLMCaller:
    def __init__(self, provider="openai", model="gpt-4"):
        self.provider = provider
        self.model = model

    def call(self, prompt) -> str:
        if self.provider == "openai":
            response = openai.ChatCompletion.create(...)
        elif self.provider == "anthropic":
            response = anthropic_client.messages.create(...)
        return self._extract_text(response)

# 비즈니스 로직은 LLMCaller만 알면 된다
def generate_summary(document, caller: LLMCaller):
    prompt = build_summary_prompt(document)
    return caller.call(prompt)
```

모델 공급자를 바꾸려면 `LLMCaller` 초기화 인자 하나만 수정하면 된다. 비즈니스 로직 함수들은 변경되지 않는다. 이것이 모델 공급자 추상화(Provider Abstraction)가 가져오는 실질적인 이점이다[^web_15][^web_16]. LangChain이나 LlamaIndex 같은 프레임워크가 LLM 호출을 별도의 추상화 레이어로 다루는 것도 같은 이유에서다. 두 프레임워크는 구현 방식이 다르지만, 모델 공급자를 교체할 때 상위 로직에 영향이 가지 않도록 호출 레이어를 격리한다는 설계 목표는 공유한다.

| 구현 방식 | LLM 호출 추상화 방식 | 공급자 교체 범위 |
|---|---|---|
| LangChain | `ChatOpenAI`, `ChatAnthropic` 등 공급자별 클래스, 공통 인터페이스 공유 | 클래스 교체만으로 비즈니스 로직 무변경 |
| LlamaIndex | `LLM` 추상 클래스 상속, 공급자별 구현체 분리 | 설정 변경으로 호출 대상 전환 |
| 직접 구현 | 공급자별 클라이언트를 래핑하는 단일 Caller 클래스 | Caller 내부만 수정, 외부 영향 없음 |

이 표에서 확인할 수 있듯, 구성 요소는 특정 도구에 종속되지 않는다. 프레임워크를 사용하든 직접 구현하든, LLM 호출을 격리된 단계로 다루어야 한다는 원칙은 동일하게 적용된다[^web_01].

LLM 호출기를 분리해야 하는 두 번째 이유는 **측정 지점 확보**다. 호출 비용과 지연 시간(Latency)을 측정하려면 호출이 일어나는 지점이 단일하고 명확해야 한다. 호출 코드가 코드베이스 전반에 흩어져 있으면, "이 기능에서 LLM 호출이 평균 몇 ms 걸리고 몇 토큰을 소비하는가"를 집계하기 위해 별도의 집계 코드를 여러 곳에 심어야 한다[^web_02]. LLM 호출기가 단일 레이어로 존재하면, 그 안에서 타임스탬프와 토큰 사용량을 한 곳에서 기록할 수 있다. 비용 추적과 성능 모니터링의 기반이 여기서 마련된다.

세 번째로, 호출 방식의 변경을 격리할 수 있다. 응답 전체를 기다리는 일반 호출 방식과, 생성되는 토큰을 실시간으로 화면에 표시하는 스트리밍(Streaming) 방식은 사용자 경험 측면에서 차이가 크다. 이 두 방식 간의 전환이 비즈니스 로직 수정을 요구하지 않으려면, 호출 방식의 차이가 LLM 호출기 내부에 캡슐화되어 있어야 한다. 타임아웃 설정, 재시도 전략, 스트리밍 응답의 청크 조립 방식—이런 세부 사항들은 모두 LLM 호출기가 책임지는 영역이다.

한 가지 오해를 짚고 넘어갈 필요가 있다. "LLM을 호출하면 시스템이 완성된다"는 인식이다. 모델 API를 호출하고 텍스트 응답을 받아오는 것은 파이프라인의 한 단계이지 전부가 아니다[^web_10]. LLM 호출기의 역할은 프롬프트 구성기의 출력을 받아 모델을 호출하고, 원시 응답(Raw Response)—아직 파싱되거나 검증되지 않은 모델의 직접 출력—을 다음 단계로 넘기는 것까지다. 그 원시 응답이 실제로 사용 가능한 형태인지, 예상된 형식을 따르고 있는지, 오류 코드가 섞여 있지는 않은지를 판단하는 것은 LLM 호출기의 관심사가 아니다.

바로 이 지점에서 다음 구성 요소의 필요성이 생긴다. 모델이 반환한 원시 응답은 항상 그대로 쓸 수 있는 형태가 아니다. JSON을 요청했는데 자연어 설명이 앞에 붙어 있기도 하고, 필드 이름이 요청한 것과 미묘하게 다르기도 하다. 이 원시 응답을 애플리케이션이 소비할 수 있는 구조로 변환하는 역할—그것이 출력 처리기(Output Processor)의 몫이다. 다음 섹션에서는 이 단계가 왜 별도의 컴포넌트로 존재해야 하는지를 살펴본다.

[^web_01]: a16z LLM 파이프라인 레퍼런스 아키텍처. LLM 호출을 격리된 단계로 다루는 구조의 산업계 기준점을 제시하며, 모델 추상화 레이어의 필요성을 뒷받침한다.

## 출력 처리기(Output Processor): 모델 응답을 시스템이 사용할 수 있는 형태로

앞 섹션에서 LLM 호출기가 프롬프트를 받아 모델을 호출하고, 그 결과로 원시 응답(Raw Response)을 반환하는 역할에 집중한다는 것을 살펴보았다. 그리고 마지막 대목에서 이미 한 가지 사실을 언급했다. 모델이 반환한 원시 응답이 항상 그대로 사용할 수 있는 형태라는 보장은 없다. 이 간극을 메우는 것이 출력 처리기의 역할이다.

출력 처리기(Output Processor)는 모델의 원시 응답을 파싱·정제·변환하여 다운스트림 컴포넌트(Downstream Component), 즉 파이프라인의 다음 단계에 위치한 구성 요소들이 소비할 수 있는 구조화된 형태로 바꾸는 단계다[^web_17]. 여기서 "구조화된 형태"란 단순히 텍스트를 보기 좋게 정리하는 수준이 아니다. 애플리케이션이 기대하는 데이터 타입, 필드 이름, 값의 범위를 만족하는 형태로 변환하는 것을 의미한다. 이 관점에서 출력 처리기는 모델과 나머지 시스템 사이의 계약(Contract)을 구현하는 단계로 볼 수 있다[^web_18].

1장에서 "출력 평가"를 하네스의 세 가지 기능 중 하나로 소개한 바 있다. 출력 처리기는 그 출력 평가 개념이 파이프라인 안에서 구체적인 컴포넌트로 구현된 결과다. 모델 응답이 형식 요건을 충족하는지, 필수 필드가 빠지지 않았는지, 스키마에 부합하는지를 판정하는 로직이 바로 여기에 속한다.

### 실패 시 증상: 파싱 오류가 엉뚱한 곳에서 나타난다

출력 처리기가 없을 때 어떤 일이 벌어지는지를 구체적인 상황으로 살펴보자. 시스템이 모델에게 다음과 같은 요청을 했다고 가정한다.

```
사용자의 감정 분석 결과를 JSON으로 반환하라.
형식: {"sentiment": "positive"|"negative"|"neutral", "score": 0.0~1.0}
```

모델은 다음과 같이 응답할 수 있다.

```
다음은 감정 분석 결과입니다:

```json
{"sentiment": "positive", "score": 0.87}
```

이 결과는 ...
```

요청한 JSON은 맞다. 그러나 마크다운 코드블록(` ```json ` ... ` ``` `)으로 감싸여 있고, 앞뒤에 자연어 설명이 붙어 있다. 애플리케이션이 응답 문자열에 곧바로 `json.parse()`를 호출하면 파싱 오류가 발생한다[^web_17]. 이 오류를 처음 마주친 개발자는 LLM 호출 자체가 실패했거나 네트워크 문제가 있다고 오해하는 경우가 적지 않다. 파싱 오류의 원인이 출력 형식 불일치인데도, 증상이 LLM 호출 단계의 실패처럼 보이기 때문이다. 이런 상황에서 디버깅 시간이 낭비되고, 근본 원인을 찾지 못한 채 "모델이 불안정하다"는 잘못된 진단으로 이어지기도 한다.

비정형 출력이 출력 처리기를 거치지 않고 다운스트림 컴포넌트로 그대로 전달되면 문제는 더 넓게 퍼진다. 데이터베이스에 잘못된 형식의 값이 기록되거나, 후속 처리 로직이 예외를 발생시키거나, 최악의 경우 오류 없이 진행되다가 더 뒤 단계에서 데이터 오염이 발견된다[^web_17]. 장애 지점이 원인과 멀리 떨어져 있을수록 디버깅 비용은 기하급수적으로 늘어난다.

### 설계 고려사항: 스키마 명세와 오류 처리기 연동

출력 처리기를 설계할 때 가장 먼저 정의해야 하는 것은 출력 스키마(Output Schema)다. 출력 스키마는 모델 응답이 어떤 구조를 가져야 하는지를 명세한 계약서다[^web_18]. 필드 이름, 데이터 타입, 허용 값의 범위, 필수 여부가 모두 이 명세에 포함된다. 출력 스키마가 명시적으로 정의되어 있어야만 파싱 로직을 테스트할 수 있고, 모델 응답이 요건을 충족하는지 자동으로 검증할 수 있다.

주의할 점은 출력 스키마의 버전 관리다. 모델이 바뀌거나 프롬프트가 수정되면 출력 형식이 미묘하게 달라질 수 있다. 스키마를 버전으로 관리하지 않으면, 프롬프트 수정이 다운스트림 컴포넌트를 조용히 깨뜨리는 상황이 생긴다. 운영 환경에서 이런 변화는 배포 직후가 아니라 특정 입력 패턴이 들어왔을 때 뒤늦게 드러나는 경향이 있어서 추적이 어렵다.

파싱 실패를 탐지했을 때의 처리 경로도 설계 단계에서 명확히 해두어야 한다. 출력 처리기가 응답을 스키마에 맞게 파싱하지 못했을 때, 두 가지 선택지가 있다. 첫째는 재시도 트리거를 발동하는 것이다. LLM 호출기에 다른 프롬프트 전략이나 강화된 형식 지시를 담아 재호출을 요청한다. 둘째는 오류 처리기(Error Handler)에 제어를 넘기는 것이다[^web_21]. 어느 조건에서 재시도를 시도하고, 어느 조건에서 바로 오류 처리기로 넘길지—이 분기 기준이 명확하지 않으면 파싱 실패 상황에서 파이프라인이 불확정 상태에 빠질 수 있다.

출력 처리기와 오류 처리기의 협력 관계는 이 파이프라인 설계에서 중요한 연결 지점 중 하나다[^web_21]. 출력 처리기는 파싱 실패의 유형과 맥락을 오류 처리기에 전달해야 하고, 오류 처리기는 그 정보를 바탕으로 재시도 전략이나 폴백 응답을 결정한다. 이 두 컴포넌트 사이의 인터페이스가 명확하게 정의되어 있어야 파이프라인이 부분 실패에도 예측 가능하게 동작한다.

출력 처리기가 제 역할을 하면, 파싱에 성공한 구조화된 결과는 두 방향으로 흘러간다. 최종 응답으로 반환되거나, 대화 이력에 기록되어 다음 호출의 컨텍스트를 구성하는 데 활용된다. 그런데 이 "기록"이라는 행위—어디에, 어떤 형태로, 얼마나 오래 보존할 것인가—는 출력 처리기의 관심사가 아니다. 이것은 상태 관리자(State Manager)가 담당하는 영역이다. 모델 자체는 이전 대화를 기억하지 못한다. 대화의 연속성이 필요하다면, 그 연속성을 만들어내는 별도의 구성 요소가 필요하다. 다음 섹션에서는 이 문제를 다룬다.

## 상태 관리자(State Manager): LLM은 기억하지 못한다

출력 처리기가 모델 응답을 구조화된 형태로 변환하면, 그 결과는 두 가지 방향으로 흘러간다고 앞에서 언급했다. 하나는 사용자에게 최종 응답으로 반환되는 경로이고, 다른 하나는 대화 이력에 기록되어 다음 호출의 컨텍스트로 활용되는 경로다. 이 두 번째 경로에서 핵심 역할을 담당하는 것이 상태 관리자다. 그런데 왜 이것이 별도의 구성 요소여야 할까? 이 질문에 답하려면 먼저 LLM의 근본적인 특성부터 확인해야 한다.

LLM은 본질적으로 무상태(stateless)다. 모델은 각 호출마다 입력된 컨텍스트 윈도우만을 바탕으로 응답을 생성한다. 이전 대화에서 무슨 말이 오갔는지, 사용자가 누구인지, 세션이 얼마나 진행되었는지를 모델 자체는 전혀 알지 못한다[^web_08]. Andrej Karpathy는 이를 두고 "모델은 매번 빈 기억 상태에서 깨어난다"고 표현했다[^web_08]. 이것은 결함이 아니라 설계 방식이다. 문제는 이 특성을 무시하고 연속적인 대화를 기대할 때 발생한다.

### 무상태 LLM이 만드는 실제 문제

구체적인 상황을 생각해보자. 사용자가 세 번째 메시지에서 "아까 말한 그 조건으로 다시 계산해줘"라고 입력했다. 이 요청을 처리하려면 시스템은 "아까 말한 그 조건"이 무엇인지를 알아야 한다. 그런데 상태 관리자 없이 모델에게 이 메시지만 단독으로 전달하면, 모델은 맥락 없이 "어떤 조건을 말씀하시는 건가요?"라고 되물을 수밖에 없다. 이전 대화가 모델의 컨텍스트 윈도우에 없기 때문이다.

더 미묘한 경우도 있다. 사용자가 첫 번째 메시지에서 "저는 채식주의자예요"라고 말했고, 다섯 번째 메시지에서 식당 추천을 요청했다고 하자. 상태 관리자가 이 정보를 보존하지 않았다면, 모델은 채식 조건을 고려하지 않은 추천을 할 수 있다. 이때 사용자가 경험하는 것은 단순한 기능 결함이 아니라 "이 시스템은 내 말을 듣지 않는다"는 신뢰 손상이다.

이런 맥락 단절 현상은 멀티턴 대화(multi-turn conversation)—여러 차례 주고받는 대화—에서 가장 빈번하게 나타난다. 단일 질의응답에서는 드러나지 않던 문제가, 대화가 누적될수록 점점 심각해진다.

### 상태의 세 가지 유형

상태 관리자가 다루어야 하는 상태는 성격에 따라 크게 세 가지로 구분된다[^web_03].

첫째는 **단기 메모리(short-term memory)**다. 현재 진행 중인 대화 이력이 여기에 해당한다. 사용자와 시스템이 주고받은 메시지 목록이 순서대로 보존되며, 프롬프트 구성기가 다음 호출의 컨텍스트를 구성할 때 이 이력에서 적절한 양을 선택해 가져간다. 단기 메모리는 세션이 종료되면 함께 소멸하는 경우가 일반적이다.

둘째는 **장기 메모리(long-term memory)**다. 사용자 프로필, 선호 설정, 반복적으로 확인된 사실처럼 세션을 넘어서도 유지되어야 하는 정보가 여기에 속한다. 장기 메모리는 별도의 저장소(데이터베이스, 벡터 스토어 등)에 외부화되어야 하며, 단기 메모리와 생명 주기가 다르다[^web_03].

셋째는 **세션 경계(session boundary) 정보**다. 언제 새로운 세션이 시작되었는지, 현재 세션의 목적이 무엇인지, 이전 세션에서 완료된 작업이 무엇인지를 추적한다. 에이전트가 하나의 목표를 달성하기 위해 파이프라인을 여러 번 실행할 때, 각 실행이 어느 세션에 속하는지를 관리하는 것이 여기에 해당한다.

### 설계 고려사항: 독립성과 컨텍스트 윈도우 예산

상태 관리자를 설계할 때 가장 중요한 원칙은 **저장소의 독립성**이다[^web_20]. 상태는 파이프라인 코드와 분리된 독립 저장소에 보관되어야 한다. 상태를 파이프라인 내부 변수로 관리하면 프로세스 재시작, 서버 교체, 스케일 아웃 시 상태가 소실된다. 실제로 Google ADK(Agent Development Kit)와 같은 프로덕션 수준의 에이전트 프레임워크에서는 상태 관리와 컨텍스트 구성 로직을 명시적으로 분리하고, 상태 저장소를 파이프라인 외부에 두는 방식을 채택하고 있다[^web_20]. 이것은 단순한 설계 선호가 아니라 운영 안정성을 위한 구조적 결정이다.

또 하나 신경 써야 할 것은 **컨텍스트 윈도우 예산(context window budget) 관리**다. 대화 이력이 누적될수록 프롬프트에 포함할 수 있는 이력의 양은 컨텍스트 윈도우 크기에 의해 제한된다. 모든 이력을 무조건 포함하면 컨텍스트 윈도우가 이력으로 가득 차서 현재 작업에 필요한 지시 공간이 줄어들고, 모델 호출 비용도 증가한다[^web_08]. 상태 관리자는 어떤 이력을 포함하고 어떤 것을 요약하거나 제외할지를 결정하는 선택 로직을 갖추어야 한다. 단기 메모리와 장기 메모리를 구분하는 설계는 이 예산 관리 문제와 직결된다.

상태 관리자는 6장에서 에이전트를 다룰 때 훨씬 더 중심적인 위치를 차지하게 된다. 에이전트가 하나의 목표를 위해 파이프라인을 반복 실행할 때, 각 실행의 결과와 중간 상태를 연결하는 것이 상태 관리자의 역할이기 때문이다. 지금 단계에서는 "LLM이 기억하지 못하는 것을 파이프라인이 기억하게 만드는 구성 요소"로 이해해 두면 충분하다.

그런데 상태 관리자가 대화 이력을 잘 보존하고 있더라도, 파이프라인 전체는 언제든 예상치 못한 방식으로 실패할 수 있다. 모델 API가 응답하지 않을 수도 있고, 출력 처리기가 파싱에 실패할 수도 있으며, 외부 도구 호출이 타임아웃될 수도 있다. 상태를 잘 관리하는 것이 파이프라인의 견고함을 보장하지는 않는다. 이러한 실패 상황을 체계적으로 다루는 것이 다음에 살펴볼 오류 처리기의 역할이다.

[^web_08]: 컨텍스트 관리 자료. stateless LLM의 본질에서 상태 관리 필요성을 도출하는 논거를 다루며, Karpathy의 "빈 기억 상태" 비유를 포함한다.
[^web_03]: Anthropic 설계 원칙 자료. 메모리와 상태를 단기/장기로 구분하여 빌딩 블록으로 다루는 관점을 제시한다.
[^web_20]: Google ADK 사례. 상태 관리와 컨텍스트 구성 분리를 실제 프로덕션 프레임워크 결정으로 뒷받침하며, 저장소 독립성의 운영적 근거를 설명한다.

## 오류 처리기(Error Handler): 실패를 예외가 아닌 설계 대상으로

상태 관리자가 대화 이력과 세션 정보를 안정적으로 보존하더라도, 파이프라인이 항상 성공적으로 완주한다는 보장은 없다. 모델 API 서버가 일시적으로 다운될 수 있고, 출력 처리기가 예상하지 못한 형식의 응답을 만나 파싱에 실패할 수도 있다. 네트워크 타임아웃, 토큰 한도 초과, 외부 서비스 장애—이 모든 것은 '언젠가 발생할 수도 있는 일'이 아니라 '운영 중에 반드시 마주치게 되는 일'이다. 오류 처리기는 바로 이 현실을 설계 안으로 끌어들이는 구성 요소다.

1장에서 "실패 처리"를 하네스의 세 가지 역할 중 하나로 언급했다. 오류 처리기는 그 개념이 파이프라인 안에서 구체적인 컴포넌트로 구현되는 지점이다. 오류를 파이프라인 전체가 무너지는 예외 상황으로 다루는 대신, 어느 단계에서 어떤 이유로 실패했는지를 탐지하고 그에 맞는 대응을 실행하는 독립된 처리 단위로 설계해야 한다[^web_06].

### 단계별 실패 유형은 다르다

오류 처리기를 독립 구성 요소로 분리해야 하는 핵심 이유는, **파이프라인의 각 단계에서 발생하는 실패가 서로 다른 성격을 갖기 때문**이다[^web_07]. 다음 세 가지 경우를 비교해보자.

| 실패 발생 단계 | 실패 유형 예시 | 적절한 대응 |
|---|---|---|
| 입력 처리기 | 허용되지 않는 형식, 악의적 입력 | 즉시 거부, 사용자에게 안내 반환 |
| LLM 호출기 | API 타임아웃, 서비스 일시 중단 | 지연 후 재시도, 폴백 모델 전환 |
| 출력 처리기 | 파싱 실패, 스키마 불일치 | 재생성 요청, 폴백 응답 반환 |

입력 처리기에서 발생한 형식 오류를 재시도로 처리하는 것은 의미가 없다. 동일한 잘못된 입력을 다시 시도해도 결과는 달라지지 않기 때문이다. 반면 LLM 호출기의 타임아웃은 일시적인 네트워크 문제일 가능성이 높으므로, 짧은 대기 후 재시도가 효과적인 대응이 된다. 출력 처리기의 파싱 실패는 모델이 예상 형식을 따르지 않았을 때 발생하는데, 이 경우는 보다 명확한 형식 지시를 포함한 재생성 요청이나 간이 폴백 응답이 적합하다[^web_21].

이처럼 실패의 원인에 따라 대응이 달라져야 한다면, 오류 처리 로직을 각 단계 코드 안에 뭉개어 넣는 방식으로는 한계가 생긴다. 입력 처리기 담당자가 LLM 호출 재시도 정책까지 알아야 하고, 출력 처리기 담당자가 서킷 브레이커 임계값까지 관리해야 하는 구조는, 각 구성 요소의 역할 경계를 흐린다. 오류 처리기를 별도 레이어로 두면, 각 구성 요소는 자신의 실패를 오류 처리기에 위임하고 본래 역할에만 집중할 수 있다[^web_06].

### 서킷 브레이커: 실패의 전파를 막는 설계

오류 처리기가 지원해야 하는 핵심 패턴 중 하나가 **서킷 브레이커(circuit breaker)**다. 전기 회로에서 차단기가 과전류를 감지하면 회로를 열어 전체 시스템을 보호하듯, 소프트웨어의 서킷 브레이커는 특정 구성 요소의 연속 실패를 감지하면 그 구성 요소에 대한 호출을 일시적으로 중단한다[^web_06].

LLM 호출기를 예로 들면, API가 연속으로 다섯 번 타임아웃을 반환했다면 여섯 번째 호출도 실패할 가능성이 높다. 그 상황에서 계속 재시도하면 사용자는 느린 응답을 반복해서 경험하고, 시스템은 실패한 API에 불필요한 요청을 계속 쏟아낸다. 서킷 브레이커가 작동하면 일정 시간 동안 호출을 차단하고, 그 사이 폴백(fallback) 응답—예컨대 "현재 서비스가 지연되고 있습니다. 잠시 후 다시 시도해주세요"—을 즉시 반환한다. 일정 시간이 지난 뒤 소수의 탐침 요청으로 복구 여부를 확인하고, 성공하면 다시 정상 흐름으로 돌아온다.

### 설계 고려사항

오류 처리기를 구체적으로 설계할 때는 다음 세 가지를 명확히 결정해두어야 한다.

**재시도 조건과 횟수 제한**이 첫 번째다. 모든 오류를 재시도 대상으로 삼으면 안 된다. 일시적 오류(일정 시간 후 회복 가능한 오류)와 영구적 오류(재시도해도 결과가 달라지지 않는 오류)를 구분하고, 재시도 횟수와 대기 간격(지수 백오프 방식이 일반적이다)을 명시적으로 설정해야 한다[^web_06].

**폴백 정책(fallback policy)**이 두 번째다. 재시도가 모두 소진되었을 때 시스템이 어떻게 행동할지를 미리 정의해야 한다. 대안 모델로 전환할 것인가, 캐시된 이전 응답을 반환할 것인가, 사용자에게 명확한 실패 메시지를 보낼 것인가—이 선택은 시스템의 가용성 목표에 따라 달라진다.

**실패 로깅과 알림**이 세 번째다. 오류가 어느 단계에서, 어떤 유형으로, 얼마나 자주 발생하는지를 기록해두지 않으면 시스템의 취약 지점을 파악하기 어렵다[^web_07]. 특정 패턴의 실패가 임계값을 넘으면 운영팀에 알림을 보내는 구조도 오류 처리기의 설계 범위에 포함된다.

---

여기까지 여섯 개의 구성 요소—입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기—를 각각의 역할 중심으로 살펴보았다. 각 구성 요소가 무엇을 담당하는지는 이제 어느 정도 명확해졌을 것이다. 그런데 이것들을 왜 하나의 덩어리로 묶지 않고 굳이 분리해야 하는가라는 질문은 아직 정면으로 다루지 않았다. 작동만 하면 되지 않느냐는 반문도 충분히 가능하다. 다음 섹션에서는 바로 이 질문에 답한다. 구성 요소를 분리하지 않았을 때 어떤 설계적 부채가 쌓이는지, 그리고 분리를 통해 얻는 것이 코드 정리 이상의 것임을 구체적으로 살펴볼 것이다.

[^web_07]: 실패 분류 논문. 파이프라인 각 단계별 실패 유형을 분류하는 학술적 근거를 제공하며, 실패가 시스템 엔지니어링 문제임을 뒷받침한다.
[^web_21]: 파싱 오류 처리 자료. 출력 처리기의 실패가 오류 처리기와 어떻게 협력하는 계층 구조를 형성하는지를 설명한다.

## 왜 분리하는가: 교체 가능성과 평가 가능성

앞에서 여섯 개의 구성 요소—입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기—를 각각의 역할 중심으로 살펴보았다. 각 구성 요소가 무엇을 담당하는지는 충분히 정리되었을 것이다. 그런데 이것들을 왜 굳이 분리해야 하는가라는 질문은 여전히 열려 있다. "일단 작동하면 되지 않느냐"는 반문도 충분히 타당하게 들린다. 이 섹션에서는 바로 그 질문에 정면으로 답한다.

구성 요소를 분리하는 것은 코드를 깔끔하게 정리하거나 설계 취향을 반영하는 문제가 아니다. 교체 가능성(replaceability), 독립 평가 가능성(independent evaluability), 실패 격리(failure isolation)라는 세 가지 설계 목표를 달성하기 위한 의도적인 선택이다[^web_09]. 이 목표들이 왜 중요한지는, 분리하지 않았을 때 어떤 일이 벌어지는지를 보면 더 명확하게 드러난다.

### 분리하지 않으면 생기는 일

프롬프트 구성 로직, LLM 호출, 응답 파싱이 하나의 함수 안에 섞여 있는 시스템을 상상해보자. 처음에는 단순하고 빠르게 동작한다. 문제는 시스템이 변화를 요구받는 순간부터 시작된다.

모델을 GPT-4에서 Claude로 교체하려 한다고 가정하자. 호출 방식과 응답 형식이 미묘하게 다르기 때문에, 모델 교체는 파싱 로직 수정을 수반한다. 그런데 파싱 로직이 프롬프트 구성 코드와 같은 블록 안에 있다면, 모델만 바꾸려다가 프롬프트 구성 방식까지 건드리게 된다. 모델을 교체했을 때 결과가 달라진다면, 그 원인이 모델 자체에 있는지 프롬프트가 의도치 않게 변경된 것인지 파악하기 어려워진다.

독립 평가 가능성의 문제도 같은 구조에서 비롯된다. "프롬프트 구성기가 잘 작동하고 있는가"를 검증하고 싶을 때, 프롬프트 구성 로직이 LLM 호출과 얽혀 있다면 실제 API를 호출하지 않고서는 프롬프트 구성 결과를 확인할 수 없다. 평가 비용이 올라가고, 모델 응답의 불확실성이 프롬프트 품질 판단에 개입한다. 각 구성 요소가 독립된 단위로 분리되어 있을 때, 프롬프트 구성기는 LLM 호출 없이도 단독으로 테스트할 수 있다[^web_09].

### 실패 격리: 어디서 무너졌는지 알 수 있어야 한다

구성 요소가 뒤섞인 시스템에서 오류가 발생하면, 문제를 특정 단계에 귀속시키기가 어렵다. 사용자에게 잘못된 응답이 반환되었을 때, 그것이 입력 처리 과정의 전처리 오류인지, 프롬프트 구성이 문맥을 잘못 조합한 결과인지, 모델이 의도치 않은 출력을 생성한 것인지, 아니면 출력 파싱 단계에서 정보가 왜곡된 것인지를 구분할 수 없다. 로그를 아무리 들여다봐도 실패의 정확한 지점을 짚어내기 어렵다.

분리된 구조에서는 각 구성 요소의 입력과 출력이 명확하게 정의되어 있으므로, 어느 단계에서 값이 기대와 달라지기 시작했는지를 추적할 수 있다. 이것이 실패 격리가 주는 실질적인 가치다. 장애 대응 속도가 빨라지고, 재발 방지를 위한 수정이 전체 파이프라인이 아닌 특정 구성 요소에 집중될 수 있다.

소프트웨어 공학에서는 이를 관심사 분리(Separation of Concerns, SoC) 원칙으로 설명한다. 각 모듈이 하나의 명확한 책임만 갖도록 설계하는 것이다[^web_09]. LLM 파이프라인에서 이 원칙이 더욱 중요해지는 이유는, 파이프라인의 각 단계가 서로 다른 속도로 변화하기 때문이다. 모델은 빠르게 교체되고, 프롬프트는 운영 중에도 지속적으로 수정되며, 출력 스키마는 다운스트림 시스템의 요구에 따라 달라진다. 변화의 속도가 다른 것들을 하나로 묶어두면, 한 곳의 변경이 다른 곳에 예측하기 어려운 영향을 미친다.

### 분리는 시스템의 수명을 늘린다

PoC(개념 검증) 단계에서는 구성 요소가 뒤섞인 구조도 잘 작동하는 것처럼 보인다. 입력의 다양성이 제한적이고, 모델 교체나 스키마 변경이 없으며, 운영 중 실패를 경험하기 전이기 때문이다. 그 상태에서 빠르게 작동하는 코드를 만드는 것은 어렵지 않다.

그러나 시스템이 실제 운영으로 나아가는 순간, 모델을 업그레이드해야 하고, 프롬프트를 A/B 테스트해야 하며, 특정 단계의 성능을 측정하고 개선해야 하는 요구가 생긴다. 그때 구성 요소가 분리되지 않은 시스템은 변경 비용이 급격히 상승하고, 각 수정이 새로운 잠재적 오류를 남긴다. 결국 "잘 작동하지만 손댈 수 없는 시스템"이 된다.

구성 요소를 분리하는 것은 초기 설계 비용을 다소 높이는 선택이지만, 그 비용은 시스템이 변화에 노출되는 첫 순간부터 회수되기 시작한다. 이 장에서 다룬 여섯 개의 구성 요소는 그 분리의 경계를 어디에 그을지에 대한 하나의 기준점이다.

---

4장에서는 LLM 파이프라인을 이루는 구성 요소들을 역할 중심으로 분해하고, 각 구성 요소가 왜 독립된 설계 단위가 되어야 하는지를 살펴보았다. 5장에서는 이 구조 위에 검색 기능을 얹는 방식, 즉 RAG(Retrieval-Augmented Generation)를 단순한 기능 추가가 아니라 아키텍처적 선택으로 바라보는 관점을 다룬다. 파이프라인의 어느 구성 요소가 RAG 도입으로 확장되고, 어떤 새로운 설계 문제가 생겨나는지를 구체적으로 들여다볼 것이다.
