# LLM 파이프라인 기본 구성 요소

## 도입: 어디서부터 생각해야 하는가

챗봇이 어느 날부터 이상한 답변을 내놓기 시작했다고 상상해 보자. 어제까지는 분명히 잘 동작했는데, 오늘부터는 엉뚱한 내용을 반환하거나, 응답 형식이 뭉개지거나, 가끔은 아예 답변 자체를 거부한다. 첫 번째 시도는 프롬프트 수정이다. 지시문을 더 명확하게 고쳐 보고, 예시를 추가해 본다. 나아지는 것 같다가도 다시 무너진다. 두 번째 시도는 모델 교체다. 더 최신 버전으로 바꾸면 해결될 것이라는 기대로 API 엔드포인트를 변경해 본다. 증상이 조금 달라지긴 하지만, 근본적으로는 같은 문제가 다른 형태로 재현된다. 결국 어느 지점에서 막다른 벽에 부딪힌다. 문제가 어디에 있는지 특정할 수 없기 때문이다.

이 상황이 낯설지 않다면, 그것은 시스템이 분해되어 있지 않기 때문일 가능성이 높다. LLM을 호출하는 코드 한 줄과 사용자 입력을 받아 응답을 돌려주는 몇 개의 함수가 뒤섞인 채 하나의 덩어리처럼 동작할 때, 실패는 어디서든 발생할 수 있지만 어디서 발생했는지는 아무도 알 수 없다. 3장에서 살펴봤듯, 실패의 원인은 모델 자체가 아닌 시스템에 있는 경우가 많다.[^web_07] 그런데 그 시스템이 내부적으로 구분된 단계 없이 하나로 뭉쳐 있다면, 진단은 원리적으로 불가능해진다. 프롬프트를 고치는 것은 그 덩어리의 어느 한 부분을 건드리는 일이고, 모델을 바꾸는 것은 또 다른 부분에 손을 대는 일이지만, 어느 쪽이 실제로 문제였는지는 끝내 확인할 수 없다.

"프롬프트 엔지니어링만 잘하면 된다"는 인식은 이 덩어리 구조를 전제로 한다. 프롬프트를 시스템의 전부인 것처럼 다루는 접근은, 규모가 작고 요구사항이 단순한 초기 단계에서는 어느 정도 통할 수 있다. 하지만 입력의 다양성이 높아지고, 실패에 대한 허용 범위가 좁아지며, 유지보수가 필요해지는 시점부터 이 접근은 한계를 드러낸다.[^web_10] 프롬프트는 시스템의 구성 요소 중 하나일 뿐이다. 그것이 전부인 구조에서는, 프롬프트가 원인이 아닌 실패도 프롬프트를 탓하게 된다. 3장에서 분류한 실패 유형 중 하나를 다시 꺼내 보면, 같은 입력에서도 맥락이나 이력이 달라질 때 품질이 저하되는 경우가 있다. 이 실패는 파이프라인의 어느 단계에서 발생한 것인가—입력이 잘못 처리된 것인가, 프롬프트 구성이 불완전한 것인가, 아니면 이전 대화 이력이 잘못 관리된 것인가. 분리된 구조 없이는 이 질문 자체를 물을 수 없다.

1장에서 소개한 하네스(Harness) 개념을 다시 떠올려 보자. 그림 1-2는 모델 단독 실행과, 하네스가 모델을 감싸는 실행 구조를 대비해서 보여준다. 하네스는 입력을 통제하고, 출력을 평가하며, 실패를 처리하는 구조다. 그런데 하네스가 실제로 작동하려면, 그 내부가 역할 단위로 분리되어 있어야 한다. 입력을 처리하는 부분, 프롬프트를 구성하는 부분, 모델을 호출하는 부분, 응답을 처리하는 부분이 구분되어 있어야만, 각 단계에서 무엇이 들어오고 무엇이 나가는지를 추적할 수 있다. 그리고 그 추적이 가능해야만 실패가 어느 단계에서 발생했는지를 특정할 수 있다.

이것이 이 장의 출발점이다. LLM을 "호출"하는 것은 한 줄의 코드로 가능하지만, LLM 시스템을 "설계"한다는 것은 실패를 진단할 수 있는 구조를 갖추는 일이다. 진단 가능성(Diagnosability)—어느 단계에서 무엇이 잘못되었는지를 특정할 수 있는 능력—이 시스템 설계의 첫 번째 목표가 되어야 한다.[^web_07] 이 장에서는 그 구조를 구성하는 부품들을 하나씩 분해해서 살펴본다. 각 구성 요소가 어떤 역할을 하는지, 그것이 독립된 단위로 존재해야 하는 이유가 무엇인지, 그리고 그것이 실패할 때 어떤 증상이 나타나는지를 순서대로 정리한다.

그 전에 먼저 한 가지 관점을 정리할 필요가 있다. LLM 시스템을 파이프라인(Pipeline)이라는 틀로 바라보는 것이 왜 유용한지, 그리고 그 틀이 기존 소프트웨어 설계 경험과 어떻게 연결되는지부터 살펴보자.

[^web_07]: 실패 분류 논문. LLM 기반 시스템의 실패 원인을 분류하고, 이를 시스템 엔지니어링 문제로 프레이밍하는 학술적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 중심 접근의 산업적 한계를 다루며, 시스템 수준의 설계 전환 필요성을 논증한다.

## 파이프라인이라는 관점

앞에서 LLM 시스템의 실패가 "분해되지 않은 구조"에서 비롯된다는 점을 살펴봤다. 이제 그 분해를 어떤 방식으로 접근할 것인지, 그 틀부터 정리하자.

"파이프라인(Pipeline)"이라는 단어를 들으면 LangChain이나 LlamaIndex 같은 특정 프레임워크를 떠올리기 쉽다. 이 두 가지를 동일시하는 것은 자연스러운 연상이지만, 그 연상이 오히려 개념을 좁히는 함정이 된다. 파이프라인은 특정 도구나 라이브러리가 아니라, 순서(Sequence)와 역할(Role)이 명시된 처리 흐름이라는 개념적 구조다.[^web_01] LangChain은 파이프라인을 구현하는 한 가지 방법일 뿐이며, LangChain 없이는 파이프라인이 존재하지 않는다는 뜻이 아니다. 파이프라인이라는 관점은 특정 프레임워크를 도입하기 이전에, 시스템을 어떻게 생각할 것인가의 문제다.

소프트웨어 개발 경험이 있다면 이미 이 구조와 유사한 것을 다뤄본 적이 있을 것이다. 웹 서버의 미들웨어 체인(Middleware Chain)을 떠올려 보자. HTTP 요청이 들어오면, 먼저 인증 미들웨어가 요청의 유효성을 확인하고, 그다음 로깅 미들웨어가 요청 내용을 기록하며, 마지막으로 비즈니스 로직이 실제 처리를 수행한다. 각 미들웨어는 자신의 역할만 담당하고, 처리 결과를 다음 단계로 넘긴다. 어느 미들웨어에서 오류가 발생했는지는 즉시 특정할 수 있다. LLM 파이프라인의 단계 분리는 이와 동일한 원리로 작동한다.[^web_05] 입력 처리, 프롬프트 구성, 모델 호출, 출력 처리 각각이 독립된 책임을 가진 레이어로 존재한다. MVC 아키텍처에서 모델·뷰·컨트롤러를 분리하는 이유—각 관심사(Concern)를 격리하여 변경의 파급 범위를 제한하는 것—는 LLM 파이프라인의 단계 분리에도 동일하게 적용된다. 이를 소프트웨어 공학에서는 관심사의 분리(Separation of Concerns, SoC)라고 부른다.[^web_09]

파이프라인 관점의 핵심 이점은, 각 단계를 독립적으로 교체·테스트·진단할 수 있는 경계가 생긴다는 데 있다.[^web_02] 단계가 분리되어 있을 때, "프롬프트 구성 로직을 변경했는데 출력 형식이 달라졌다"는 사실을 추적할 수 있다. 반대로 단계가 뒤섞인 덩어리 구조에서는 어느 변경이 어떤 결과를 낳았는지 역추적하기가 어렵다. Chip Huyen이 지적하듯, LLM 시스템에서 각 단계는 독립적으로 테스트될 수 있어야 한다.[^web_02] 특정 단계만 격리하여 다른 입력을 넣어보고, 그 출력이 기대에 부합하는지 확인할 수 있어야 비로소 시스템 전체의 품질을 관리할 수 있게 된다.

> **[그림 4-1]** LLM 파이프라인 전체 흐름도
>
> *편집 지시: 입력 처리기 → 프롬프트 구성기 → LLM 호출기 → 출력 처리기 → 상태 관리자 (대화 이력 양방향 연결) → 오류 처리기 (전 단계와 연결)의 선형 흐름을 표시한다. 하네스(Harness)가 이 전체 파이프라인을 외곽에서 감싸는 형태로 표현한다. 각 단계 사이에 데이터 흐름 방향(→)을 명시한다. ch01 그림 1-2의 시각적 스타일과 연속성을 유지한다.*

그림 4-1은 이 장에서 다룰 LLM 파이프라인의 전체 흐름을 보여준다. 여섯 개의 구성 요소가 순서대로 데이터를 통과시키며, 하네스(Harness)가 이 전체 흐름을 외부에서 감싸고 있다. 1장에서 소개한 하네스의 역할—입력 통제, 출력 평가, 실패 처리—이 실제로 구현되려면 이 여섯 단계 각각이 독립된 경계를 가져야 한다. 하네스는 파이프라인 위에 얹히는 추가 구조가 아니라, 이 파이프라인 전체를 운영 가능한 단위로 묶는 외곽 프레임이다.

한 가지 더 짚어 두어야 할 것이 있다. "프롬프트 엔지니어링"은 이 파이프라인 전체가 아니라, 프롬프트 구성기(Prompt Builder)라는 한 단계에 해당한다. 프롬프트를 잘 작성하는 것은 중요하지만, 그것은 여섯 단계 중 하나를 개선하는 일이다. 나머지 다섯 단계가 제대로 설계되어 있지 않다면, 프롬프트 구성 단계만 아무리 정교하게 다듬어도 시스템 전체의 안정성은 보장되지 않는다.[^web_10] 이 사실을 구조로 보여주는 것이 파이프라인 관점이 가진 또 하나의 가치다.

이렇게 파이프라인의 개념과 그것이 왜 유용한지를 정리했다. 이제부터는 이 흐름을 구성하는 각 단계를 순서대로 살펴볼 차례다. 파이프라인의 첫 번째 관문은 사용자로부터 들어온 입력을 받아들이고, 그것이 다음 단계로 넘어가기에 적합한 형태인지를 판단하는 단계다. 이 단계에서 무엇이 일어나는지, 그리고 이 단계가 독립적으로 존재해야 하는 이유가 무엇인지를 다음 절에서 구체적으로 들여다보자.

## 입력 처리기: 시스템이 받아들이는 것을 통제하라

파이프라인이라는 관점이 왜 유용한지 이해했으니, 이제 그 흐름의 첫 번째 단계를 구체적으로 들여다보자. 파이프라인은 입력이 들어오는 지점에서 시작되고, 그 지점을 담당하는 것이 입력 처리기(Input Processor)다.

입력 처리기를 별도의 설계 단위로 두지 않으면 어떤 일이 생기는지부터 살펴보는 것이 빠르다. 사용자가 1만 자짜리 텍스트를 입력창에 붙여 넣었다고 가정하자. 이 텍스트는 아무런 검증 없이 프롬프트 구성 로직으로 넘어가고, 거기서 시스템 프롬프트와 대화 이력과 합쳐진 뒤 LLM 호출 단계에 도달한다. 그리고 바로 그 지점에서 컨텍스트 윈도우(Context Window) 초과 오류가 발생한다. 개발자 입장에서는 LLM 호출 단계가 실패했으니 그쪽을 먼저 살펴보게 된다. 호출 코드에는 이상이 없다. 모델 설정을 뒤지고, API 응답을 확인하고, 로그를 추적하다가 한참 뒤에야 원인이 사용자 입력의 길이였음을 발견한다. 실패의 원인은 파이프라인 입구에 있었지만, 오류는 출구 가까이에서 터졌다.[^web_01]

이 상황이 보여주는 문제는 두 가지다. 첫째, 처리되지 않은 입력이 파이프라인 전체를 오염시킨다. 둘째, 실패 지점과 실패 원인이 분리되어 디버깅 비용이 폭증한다. 입력 처리기가 제자리에 있었다면, 1만 자 입력은 파이프라인 입구에서 차단되거나 잘라졌을 것이고, 오류 메시지는 원인이 발생한 바로 그 지점에서 발생했을 것이다.

한 가지 더 짚어야 할 것이 있다. 컨텍스트 윈도우 초과는 입력 처리기가 없어서만 발생하는 문제가 아니다. 대화 이력을 관리하는 상태 관리자(State Manager), 시스템 프롬프트를 조합하는 프롬프트 구성기(Prompt Builder), RAG 검색 결과가 추가되는 단계까지, 복수의 단계에서 누적된 컨텍스트가 윈도우를 채운다. 입력 처리기의 역할은 이 누적을 줄이는 첫 번째 기여 단계다. 입력 처리기 하나로 컨텍스트 윈도우 초과를 완전히 막을 수 있다고 생각하면 곤란하다.[^web_03]

---

**역할(Role)**

입력 처리기의 역할은 사용자 입력을 정제·검증하여 파이프라인이 안전하게 처리할 수 있는 형태로 변환하는 것이다.[^web_01] 여기서 '안전하게'라는 표현은 두 가지 의미를 담고 있다. 하나는 구조적 안전성—형식이 맞고, 길이가 허용 범위 안에 있으며, 인코딩이 정규화된 상태. 다른 하나는 의미적 안전성—시스템이 처리해서는 안 되는 입력을 사전에 걸러내는 것이다.

1장에서 소개한 하네스의 세 축 가운데 '입력 통제'가 구체적인 컴포넌트로 구현되는 지점이 바로 여기다. 추상적인 설계 원칙이 코드 단위에서 실체를 갖게 된다.[^web_03]

---

**실패 시 증상**

입력 처리기가 없거나 불완전할 때 나타나는 증상은 대부분 하위 단계에서 발생한다는 공통점이 있다. 앞서 본 컨텍스트 윈도우 초과 사례처럼, 오류는 LLM 호출 단계에서 터지지만 원인은 입력 단계에 있다. 원인과 증상의 거리가 멀수록 진단 비용은 올라간다.

증상을 유형화하면 다음과 같다. 예상치 못한 입력 형식(예: 바이너리 데이터, 잘못된 문자 인코딩)이 프롬프트 구성 로직에서 처리되지 못하고 파싱 오류를 일으킨다. 지나치게 긴 입력이 컨텍스트를 초과하여 모델 호출이 실패한다. 악의적으로 구성된 입력이 파이프라인을 그대로 통과하여 프롬프트 구성 단계에 도달한다—이때 입력 처리기는 파이프라인 진입 시점에서 차단하는 첫 번째 방어 지점 중 하나가 된다. 다만 프롬프트 인젝션(Prompt Injection)에 대한 완전한 방어는 프롬프트 구성기의 구조화, 출력 처리기의 검증까지 포함하는 다층 방어(Defense in Depth)를 통해 이루어진다는 점은 명심하자.[^web_03]

---

**설계 시 고려사항**

입력 처리기를 설계할 때 최소한으로 명세해야 할 항목은 세 가지다. 입력 길이 제한, 인코딩 정규화, 허용 입력 유형 명세. 의사코드(pseudocode) 수준으로 표현하면 아래와 같다.

```
function process_input(raw_input):
    # 1. 허용 입력 유형 확인
    if not is_allowed_type(raw_input):
        raise InputTypeError

    # 2. 인코딩 정규화
    normalized = normalize_encoding(raw_input)

    # 3. 입력 길이 제한 적용
    if len(normalized) > MAX_INPUT_LENGTH:
        normalized = truncate(normalized, MAX_INPUT_LENGTH)
        log_warning("입력 길이 초과: 잘라냄 처리")

    # 4. 로깅 (첫 번째 기록 지점)
    log_input(normalized)

    return normalized
```

여기서 한 가지 설계 원칙을 짚어 두자. 입력 처리기는 시스템의 첫 번째 방어선이기도 하지만, 동시에 첫 번째 로깅 포인트(Logging Point)—시스템이 무엇을 받았는지를 기록하는 최초 지점—이기도 하다.[^web_01] 원본 입력을 가공 없이 기록해 두는 것이 이후 디버깅과 감사(Audit)에서 얼마나 중요한지는, 실제로 운영 중인 시스템에서 입력 로그가 없어 사후 분석이 불가능했던 경험을 해본 사람이라면 바로 이해할 것이다.

---

정리하면, 입력 처리기는 파이프라인이 "무엇을 받아들이는가"를 결정하는 단계다. 이 단계가 독립된 경계를 가질 때, 실패는 발생한 지점에서 포착되고, 로그는 원인을 담게 되며, 이후 단계는 정제된 입력만을 다루게 된다.

그런데 정제된 입력이 있다고 해서 곧바로 모델에 전달할 수 있는 것은 아니다. 사용자 입력 외에도 시스템 지시, 대화 이력, 외부에서 주입되는 컨텍스트 등 다양한 정보가 하나의 요청으로 조합되어야 비로소 모델이 처리할 수 있는 형태가 된다. 그 조합을 담당하는 것이 다음 단계, 프롬프트 구성기(Prompt Builder)다.

[^web_01]: a16z 레퍼런스 아키텍처. LLM 시스템을 단계화된 흐름으로 분류하는 산업계 기준점으로, 입력 처리 단계의 역할 범주화에 대한 권위 있는 근거를 제공한다.
[^web_03]: Anthropic. 입력 통제를 빌딩 블록 개념으로 설명하며, 다층 방어 구조와 컴포넌트 독립성의 설계 원칙을 뒷받침한다.

## 프롬프트 구성기(Prompt Builder): 맥락을 조립하는 단계

입력 처리기가 사용자 입력을 정제하고 검증하는 역할을 한다면, 그 이후에는 정제된 입력만으로는 부족하다는 현실을 마주하게 된다. 모델이 적절한 응답을 생성하려면 사용자 입력 외에도 시스템 지시(System Instruction), 대화 이력, 필요한 경우 외부에서 주입되는 참조 컨텍스트까지 하나의 일관된 요청으로 조합되어야 한다. 그 조합 작업을 전담하는 단계가 프롬프트 구성기(Prompt Builder)다.

"프롬프트만 잘 쓰면 된다"는 인식이 왜 불충분한지는, 실제 운영 환경에서 프롬프트가 어떻게 다루어지는지를 들여다보면 금방 드러난다. 많은 초기 구현에서 시스템 지시문, 사용자 입력 포맷팅, 비즈니스 로직이 하나의 함수 안에 뒤섞여 있다.[^web_22] 다음과 같은 구조를 흔히 마주치게 된다.

```
function handle_request(user_input, user_id):
    # 비즈니스 로직과 프롬프트 조합이 뒤섞임
    user_data = fetch_user_profile(user_id)
    if user_data.plan == "premium":
        system_msg = "당신은 전문 재무 어드바이저입니다..."
    else:
        system_msg = "당신은 일반 안내 어드바이저입니다..."

    prompt = system_msg + "\n\n"
    prompt += "사용자 이름: " + user_data.name + "\n"
    prompt += "잔고: " + str(user_data.balance) + "\n"
    prompt += "질문: " + user_input
    return call_llm(prompt)
```

이 구조에서 프롬프트 문구를 수정하려면 비즈니스 로직 함수 전체를 열어야 한다. 시스템 지시문을 변경하는 작업이 플랜 분기 로직, 사용자 데이터 포맷팅, LLM 호출과 같은 관심사가 뒤섞인 코드를 건드리게 된다. 이 상황에서는 회귀 테스트(Regression Test)가 사실상 불가능하다. 프롬프트를 조금 바꾸었을 때 무엇이 영향을 받는지 추적할 경계가 없기 때문이다.[^web_19] 프롬프트를 독립적으로 버전 관리하거나, A/B 테스트하거나, 특정 버전으로 롤백하는 것도 마찬가지로 어렵다.

이와 대비하여, 프롬프트 구성기가 독립된 단위로 분리된 구조는 아래와 같이 구성된다.

```
function build_prompt(system_instruction, context, user_turn, few_shot_examples=None):
    messages = []

    # 1. 시스템 지시 (System Instruction)
    messages.append({"role": "system", "content": system_instruction})

    # 2. 퓨샷 예시 (Few-shot Examples) — 선택적
    if few_shot_examples:
        for example in few_shot_examples:
            messages.append({"role": "user", "content": example.input})
            messages.append({"role": "assistant", "content": example.output})

    # 3. 컨텍스트 (외부 주입 정보)
    if context:
        messages.append({"role": "system", "content": context})

    # 4. 사용자 입력 (User Turn)
    messages.append({"role": "user", "content": user_turn})

    return messages
```

여기서 `build_prompt`는 어떤 비즈니스 로직도 담고 있지 않다. 사용자가 프리미엄 요금제인지 일반 요금제인지를 판단하는 것은 이 함수의 책임이 아니다. 그 판단 결과로 선택된 `system_instruction`을 받아서 조합하는 것이 이 함수의 전부다. 시스템 지시를 변경하고 싶다면 이 함수를 호출하는 쪽에서 전달값만 바꾸면 되고, 프롬프트 조합 로직 자체를 테스트하고 싶다면 이 함수만 격리하여 검증할 수 있다.[^web_04]

---

**역할(Role)**

프롬프트 구성기의 역할은 시스템 지시, 컨텍스트, 사용자 입력, 퓨샷 예시(Few-shot Example — 모델이 참고할 입출력 예시 쌍)를 조합하여 LLM에 전달할 최종 입력을 구성하는 것이다.[^web_10] 이 단계는 파이프라인에서 "모델이 무엇을 보는가"를 결정하는 단계이기도 하다. 입력 처리기가 "무엇을 받아들이는가"를 결정한다면, 프롬프트 구성기는 "그것을 어떤 맥락 안에 위치시키는가"를 결정한다.

---

**실패 시 증상**

프롬프트 구성기가 독립된 단위로 존재하지 않을 때 가장 자주 나타나는 증상은 프롬프트 드리프트(Prompt Drift)다. 프롬프트 드리프트란, 시스템의 다른 부분을 수정하는 과정에서 의도치 않게 프롬프트 내용이나 구조가 조금씩 변형되는 현상을 말한다.[^web_19] 비즈니스 로직과 프롬프트가 같은 함수 안에 있을 때, 로직을 수정하면서 옆에 있는 프롬프트 문자열이 함께 바뀌는 일이 생기고, 이 변경은 코드 리뷰에서 잘 포착되지 않는다. 이런 사소한 변경이 모델 응답의 형식이나 품질에 예기치 않은 영향을 미치는 것이 드리프트의 전형적인 패턴이다.

또한 프롬프트 변경과 비즈니스 로직 변경이 결합되어 있으면 회귀 테스트가 불가능해진다. "이번 배포에서 프롬프트를 건드렸는가, 아닌가"를 확인할 수 있는 경계가 없기 때문이다.[^web_22]

---

**설계 시 고려사항**

프롬프트 구성기를 설계할 때 가장 먼저 검토해야 할 것은 템플릿의 버전 관리다. 프롬프트 템플릿(Prompt Template — 변수가 포함된 재사용 가능한 프롬프트 구조)을 코드와 분리하여 별도 파일이나 저장소로 관리하면, 프롬프트 변경 이력을 독립적으로 추적할 수 있다. 시스템 지시와 사용자 입력의 명확한 분리 역시 설계 원칙으로 명세해 두어야 한다. 이 두 가지가 뒤섞이면 모델이 지시 우선순위를 잘못 해석하는 경우가 생길 수 있다.[^web_04]

한 가지 더 짚어둘 것이 있다. 3번 항목의 `context` 슬롯—외부에서 주입되는 참조 정보가 들어오는 자리—이 바로 5장에서 다룰 RAG(Retrieval-Augmented Generation, 검색 증강 생성)의 핵심 연결 지점이다. RAG는 별도의 아키텍처가 아니라, 프롬프트 구성기의 컨텍스트 슬롯에 검색된 외부 정보를 주입하는 구조로 이해할 수 있다. 이 단계에서 그 연결 구조를 미리 설계해 두면, RAG를 나중에 덧붙이는 것이 아니라 자연스럽게 확장하는 것이 가능해진다.

프롬프트 구성기가 최종 메시지 배열을 완성하면, 그것은 이제 실제로 모델에 전달될 준비가 된 상태다. 그 전달을 담당하는 것이 다음 단계인 LLM 호출기(LLM Caller)다. 어떤 모델을 사용할지, 어떤 파라미터로 호출할지, 호출이 실패했을 때 어떻게 처리할지—이런 결정들이 프롬프트 구성기와 분리된 별도의 단계로 존재해야 하는 이유는, 다음 절에서 살펴본다.

[^web_04]: 프롬프트 템플릿 논문. 프롬프트 구성의 구조화가 응답 품질에 미치는 실증적 영향을 분석하며, 템플릿 분리와 버전 관리의 설계적 근거를 제공한다.
[^web_10]: Beyond Prompt Engineering. 프롬프트 구성이 시스템 설계의 일부임을 보여주는 산업적 관점을 제시하며, 프롬프트 중심 접근의 한계를 논증한다.

## LLM 호출기: 모델 호출을 격리된 단계로 두어야 하는 이유

프롬프트 구성기가 시스템 지시, 컨텍스트, 사용자 입력을 조합하여 최종 메시지 배열을 완성하면, 그것은 이제 모델에 전달될 준비가 된 상태다. 그런데 그 전달을 어떻게 수행할 것인가—어떤 모델을 사용할지, 어떤 파라미터로 호출할지, 호출이 실패했을 때 어떻게 처리할지—는 프롬프트를 "만드는" 일과는 성격이 다른 문제다. 이 전달을 담당하는 단계가 LLM 호출기(LLM Caller)다.

처음 LLM 기반 시스템을 구성할 때 흔히 빠지는 오해가 있다. "LLM을 호출하는 것이 곧 LLM 시스템을 완성하는 것"이라는 인식이다. API 한 줄로 모델을 호출하고 응답을 받아오는 코드는 실제로 몇 분 안에 작성할 수 있다. 그런데 그 호출 코드가 비즈니스 로직 곳곳에 산재되어 있다면—사용자 요청을 처리하는 함수 안에서, 데이터를 저장하는 로직 옆에서, 응답을 포맷하는 코드와 나란히 놓여—모델을 교체하는 일이 전체 코드베이스를 수정하는 일이 된다. GPT-4 호출 코드를 Claude로 바꾸려 할 때, 그 호출이 20개의 함수에 흩어져 있다면 영향 범위를 추산하는 것조차 쉽지 않다.[^web_02]

---

**역할(Role)**

LLM 호출기의 역할은 프롬프트 구성기의 출력—완성된 메시지 배열—을 받아 모델 API를 실제로 호출하고, 원시 응답(Raw Response — 파싱이나 가공이 이루어지지 않은 모델의 직접 출력)을 반환하는 것이다.[^web_01] 이 단계는 "무엇을 모델에 보낼 것인가"는 이미 결정된 상태에서, 오직 "어떻게 보낼 것인가"만을 담당한다. 어떤 모델 공급자(OpenAI, Anthropic, Google 등)를 사용할지, 동기 호출인지 스트리밍(Streaming — 모델 응답을 토큰 단위로 순차 수신하는 방식) 호출인지, 타임아웃은 얼마로 설정할지—이 결정들이 모두 이 단계 안에 모여 있다.

```
# LLM 호출기 — 역할 분리 예시 (의사코드)

function call_llm(messages, model="gpt-4", timeout=30, stream=False):
    # 모델 공급자 추상화: 공급자가 바뀌어도 이 함수만 수정
    response = llm_provider.call(
        model=model,
        messages=messages,
        timeout=timeout,
        stream=stream
    )
    # 원시 응답 반환 — 파싱은 다음 단계(출력 처리기)의 책임
    return response.raw_output
```

여기서 `call_llm`은 응답 내용을 해석하거나 가공하지 않는다. JSON을 파싱하거나 오류 메시지를 사용자에게 맞게 변환하는 일은 이 함수의 책임 밖이다. 원시 응답을 그대로 다음 단계로 넘기는 것이 전부다.

---

**실패 시 증상**

LLM 호출기가 독립된 단위로 존재하지 않을 때 가장 먼저 나타나는 증상은 모델 교체 시 전체 코드베이스 수정이다.[^web_02] 호출 코드가 비즈니스 로직에 분산되어 있으면, 모델 공급자를 바꾸거나 새로운 모델 버전으로 업그레이드할 때 변경이 필요한 지점을 일일이 찾아야 한다. 누락된 하나의 호출 지점이 운영 환경에서 예기치 않은 오류로 이어질 수 있다.

두 번째 증상은 호출 비용과 지연(Latency) 측정 지점의 부재다.[^web_01] LLM API 호출은 시스템 내에서 가장 비용이 높고 응답 속도가 느린 단계다. 이 단계가 격리되어 있어야만 "이번 요청에서 모델 호출이 얼마나 걸렸는가", "토큰 소비량이 예상 범위 안에 있는가"를 정확하게 측정할 수 있다. 호출 코드가 흩어져 있으면 이런 측정 자체가 불가능해진다.

---

**설계 시 고려사항**

LLM 호출기를 설계할 때 가장 먼저 검토해야 할 것은 모델 공급자 추상화(Provider Abstraction)다. 특정 공급자의 SDK를 직접 호출하는 코드를 이 레이어 안에 캡슐화하면, 공급자를 교체할 때 변경 범위를 이 레이어로 한정할 수 있다.[^web_02] LangChain과 LlamaIndex는 각자 다른 방식으로 이 추상화를 구현하고 있지만, 두 프레임워크가 해결하려는 개념적 문제는 동일하다—호출 레이어를 나머지 로직과 분리하는 것이다.[^web_15][^web_16] 이는 "구성 요소는 특정 도구에 종속되지 않는다"는 원칙을 실증하는 사례이기도 하다.

| 구현 방식 | 추상화 접근 | 특징 |
|---|---|---|
| LangChain | `ChatOpenAI`, `ChatAnthropic` 등 공급자별 클래스, 공통 인터페이스 제공 | 공급자 전환 시 클래스만 교체 |
| LlamaIndex | `LLM` 베이스 클래스 상속, `complete()`·`chat()` 메서드 통일 | 쿼리 파이프라인과 호출 레이어 분리 강조 |
| 직접 구현 | 공급자별 API 래퍼 함수 작성 | 의존성 최소화, 유연성 높음 |

타임아웃 설정도 이 단계에서 명시적으로 관리해야 한다. LLM 호출은 네트워크 지연과 모델 추론 시간을 포함하기 때문에 수 초에서 수십 초까지 소요될 수 있다. 타임아웃 없이 호출이 무한 대기 상태에 빠지면 파이프라인 전체가 블로킹될 수 있다. 스트리밍 응답 처리 방식 역시 이 레이어에서 결정되어야 한다. 스트리밍을 사용할 때와 배치(Batch — 전체 응답을 한 번에 수신하는 방식) 호출을 사용할 때 나머지 파이프라인이 영향을 받지 않도록 인터페이스를 설계하는 것이 핵심이다.

---

LLM 호출기가 원시 응답을 반환하면, 파이프라인의 역할은 아직 끝나지 않았다. 모델이 돌려준 텍스트는 그 자체로는 시스템이 활용할 수 있는 형태가 아닐 수 있다. JSON 형식을 기대했는데 자연어 문장이 왔을 수도 있고, 필요한 필드가 누락되었을 수도 있으며, 응답이 지정된 길이를 초과했을 수도 있다. 이 원시 응답을 시스템이 처리할 수 있는 구조화된 형태로 변환하는 것, 그리고 예상치 못한 형식에 대응하는 것—이것이 다음 단계인 출력 처리기(Output Processor)의 책임이다.

[^web_16]: 프레임워크 비교 자료. LlamaIndex가 LLM 호출 단계를 추상화하는 방식을 분석하며, LangChain과 다른 접근 방식으로 동일한 개념적 문제를 해결하고 있음을 보여준다.

## 출력 처리기: 모델 응답을 시스템이 사용할 수 있는 형태로

LLM 호출기가 원시 응답(Raw Response)을 반환하는 것으로 파이프라인의 역할이 끝나지 않는다는 점은 앞 섹션 말미에서 이미 예고했다. 모델이 돌려준 텍스트는 그 자체로는 시스템이 활용하기 어려운 경우가 많다. 출력 처리기(Output Processor)는 바로 이 지점—모델의 원시 응답과 나머지 시스템 사이—에 위치한다.

---

**역할(Role)**

출력 처리기의 역할은 모델의 원시 응답을 파싱(Parsing — 구조화되지 않은 텍스트에서 의미 단위를 추출하는 과정)·정제·변환하여, 다운스트림 컴포넌트(Downstream Component — 파이프라인에서 현재 단계 이후에 위치한 구성 요소)가 사용할 수 있는 구조화된 형태로 만드는 것이다.[^web_17] 1장에서 소개한 하네스의 세 축 중 "출력 평가"가 실제 파이프라인 컴포넌트로 구현되는 것이 바로 이 단계다. 모델이 무엇을 반환했는지를 검증하고, 기대하는 형식과 일치하는지를 확인하며, 이후 단계에서 소비할 수 있는 형식으로 변환하는 일이 여기서 이루어진다.

이 단계를 "모델과 나머지 시스템 사이의 계약(Contract)"으로 바라보는 시각이 있다.[^web_18] 출력 처리기는 모델이 무엇을 반환해야 하는지를 명시한 스키마(Output Schema — 출력의 구조, 타입, 필드 명세)를 기준으로, 실제 응답이 그 계약을 이행했는지를 검사한다. 모델은 확률적으로 동작하기 때문에 같은 프롬프트에도 매번 동일한 형식의 응답이 보장되지 않는다. 출력 처리기는 이 비결정성(Non-determinism)의 영향이 시스템 나머지 부분으로 전파되지 않도록 막는 경계선이다.

---

**실패 시 증상**

출력 처리기가 독립된 단위로 존재하지 않을 때 발생하는 대표적인 증상은, 파싱 오류가 엉뚱한 단계의 문제로 오인되는 것이다.[^web_17] 예를 들어, 다음과 같은 상황을 생각해 보자. 시스템이 모델에 JSON 형식의 응답을 요청했다. 모델은 실제로 올바른 JSON을 생성했지만, 마크다운 코드블록(` ```json ... ``` `)으로 감싸서 반환했다. 출력을 그대로 `json.parse()`에 넘기면 파싱 오류가 발생한다. 이 오류의 실제 원인은 마크다운 래핑(Wrapping)이라는 출력 형식 불일치인데, 출력 처리 레이어가 없는 구조에서는 이 오류가 LLM 호출기 혹은 프롬프트 구성기의 문제처럼 보인다. 엔지니어는 프롬프트를 수정하거나 모델을 교체하는 방향으로 대응하지만, 문제의 실제 위치는 그곳이 아니다.

두 번째 증상은 비정형 출력의 다운스트림 전파다. 출력 처리기가 없으면, 모델이 예상과 다른 구조의 응답을 반환할 때 그 응답이 그대로 다음 단계로 넘어간다. 비즈니스 로직이 특정 필드를 기대하고 있는데 그 필드가 없거나, 타입이 달라지거나, 중첩 구조가 바뀌어 있을 수 있다. 이 경우 오류는 출력 처리 시점이 아니라 비즈니스 로직 내부에서 발생하며, 원인 추적이 어려워진다.[^web_21]

---

**설계 시 고려사항**

출력 처리기를 설계할 때 가장 먼저 명시해야 할 것은 출력 스키마(Output Schema)다. 어떤 필드가 필수인지, 각 필드의 타입은 무엇인지, 허용 가능한 값의 범위는 어떻게 되는지를 코드 수준에서 정의해 두면, 모델 응답의 형식 변화를 즉시 탐지할 수 있다. 스키마 자체도 버전으로 관리해야 한다. 모델 버전이 바뀌거나 프롬프트가 변경되면 응답 구조가 달라질 수 있으므로, 스키마 변경 이력을 추적하는 것이 운영 단계에서 중요해진다.[^web_18]

파싱 실패 탐지와 오류 처리기(Error Handler)와의 연계도 이 단계에서 설계해야 한다.[^web_21] 출력 처리기가 파싱에 실패했을 때, 그 실패를 즉시 상위로 전파할 것인지, 아니면 재시도(Retry)를 트리거할 것인지를 미리 결정해 두어야 한다. 재시도를 선택한다면, 파싱 실패 원인을 프롬프트에 피드백하여 모델이 올바른 형식으로 다시 응답하도록 유도하는 방식을 고려할 수 있다. 이 흐름—출력 처리기의 실패 신호 → 오류 처리기의 재시도 트리거—이 명확하게 설계되어 있을 때, 파이프라인은 일시적인 형식 불일치를 스스로 복구할 수 있는 탄력성을 갖는다.

```
# 출력 처리기 — 파싱 실패 처리 예시 (의사코드)

function process_output(raw_response, schema):
    # 마크다운 코드블록 제거 등 전처리
    cleaned = strip_markdown_wrapper(raw_response)
    
    try:
        parsed = json.parse(cleaned)
        validate_against_schema(parsed, schema)  # 스키마 검증
        return parsed
    except ParseError as e:
        # 파싱 실패 → 오류 처리기로 전달 (재시도 트리거)
        raise OutputParseFailure(raw=raw_response, reason=str(e))
```

출력 처리기가 정상적으로 동작하면 구조화된 결과물이 다음 단계로 전달된다. 그런데 이 결과물은 단순히 "처리된 응답" 이상의 의미를 가진다. 대화형 시스템에서는 이 응답이 다음 턴의 컨텍스트가 되고, 멀티스텝 태스크에서는 다음 단계의 입력이 된다. 즉, 출력 처리기를 거친 결과를 어딘가에 유지하고, 필요한 시점에 다시 꺼내 쓸 수 있어야 한다. 이 역할을 담당하는 것이 다음에 살펴볼 상태 관리자(State Manager)다. LLM은 본질적으로 무상태(Stateless — 각 호출이 이전 호출의 맥락을 기억하지 못하는 특성)이기 때문에, 상태 관리자 없이는 대화의 연속성을 유지하는 일 자체가 불가능하다.

[^web_17]: 출력 처리 심화 자료. 출력 파싱 오류의 유형과 처리 레이어 구조를 분석하며, 출력 처리기가 독립된 레이어로 존재해야 하는 근거를 제공한다.
[^web_18]: data contract 관련 자료. 출력 처리기를 시스템 간 계약으로 보는 관점과 출력 스키마 버전 관리의 필요성을 논거로 제공한다.
[^web_21]: 파싱 오류 처리 자료. 출력 처리기와 오류 처리기의 협력 관계, 파싱 실패 탐지 및 재시도 트리거 설계를 다룬다.

## 상태 관리자(State Manager): LLM은 기억하지 못한다

출력 처리기가 모델의 원시 응답을 구조화된 형태로 변환하는 과정을 살펴봤다. 그런데 여기서 한 가지 질문이 생긴다. 그렇게 처리된 결과물은 어디로 가는가? 단발성 질의응답이라면 그 결과를 즉시 사용자에게 반환하는 것으로 충분하다. 하지만 대화가 여러 번 오고 가는 멀티턴(Multi-turn — 사용자와 시스템이 여러 차례 교환하는 대화 방식) 시스템이라면, 이전 대화의 맥락이 다음 요청에 반영되어야 한다. 이 역할을 담당하는 것이 상태 관리자(State Manager)다.

---

**역할(Role)**

LLM은 본질적으로 무상태(Stateless — 각 호출이 이전 호출의 맥락을 기억하지 못하는 특성)다.[^web_08] 모델에 API 요청을 보낼 때마다, 모델은 그 요청에 담긴 내용만을 처리한다. 이전에 어떤 대화가 오갔는지, 사용자가 누구인지, 직전 응답에서 어떤 결정을 내렸는지—이 모든 것은 요청 본문에 명시적으로 포함되지 않는 한 모델에게 존재하지 않는 것과 같다. Andrej Karpathy는 이를 두고 "LLM은 매번 태어나서 죽는다"고 표현한 바 있다.[^web_08] 각 호출은 그 자체로 완결된 독립 사건이다.

이 무상태성은 API 단계에서는 문제가 없다. 문제는 무상태인 모델 위에 대화 연속성이 있는 시스템을 올릴 때 발생한다. 상태 관리자의 역할은 바로 여기서 정의된다. 파이프라인 실행 간에 유지되어야 하는 정보—대화 이력, 세션 식별자, 사용자 맥락—를 관리하고, 각 요청이 처리될 때 필요한 상태를 꺼내 프롬프트 구성기에 공급하는 것이다.[^web_03]

---

**실패 시 증상**

상태 관리자가 독립된 컴포넌트로 존재하지 않을 때의 증상은 명확하다. 다음 상황을 생각해 보자. 사용자가 고객 지원 챗봇과 대화를 나누고 있다. 첫 번째 메시지에서 "이번 달 구매 내역 중 환불 가능한 항목이 있나요?"라고 물었고, 두 번째 메시지에서 "그중에서 가장 금액이 큰 건요?"라고 이어갔다. 그리고 세 번째 메시지에서 "**아까 말한 그 조건으로** 환불 신청해 주세요"라고 요청했다. 상태 관리자가 없는 구조에서, 세 번째 메시지를 받은 모델은 "아까 말한 그 조건"이 무엇인지 알 방법이 없다. 이전 두 메시지가 컨텍스트에 포함되어 있지 않기 때문이다. 결과적으로 모델은 엉뚱한 응답을 반환하거나, 사용자에게 다시 설명을 요구하게 된다.

이보다 더 조용하게 발생하는 실패도 있다. 상태 관리를 애플리케이션 레이어에 임시로 위임한 경우, 상태가 존재하기는 하지만 파이프라인과 분리되어 있어 컨텍스트 윈도우 예산(Context Window Budget — 모델에 전달할 수 있는 토큰의 최대 한도)을 고려하지 않은 채 이력 전체가 프롬프트에 붙여 넣어지는 일이 발생한다. 대화가 길어질수록 프롬프트 크기가 비제어적으로 커지고, 어느 순간 컨텍스트 윈도우 초과로 요청이 실패하거나, 초과를 막기 위해 이력을 무작위로 잘라내면서 중요한 맥락이 소실된다.[^web_08]

---

**설계 시 고려사항**

상태 관리자를 설계할 때 먼저 상태의 유형을 구분하는 것이 유용하다. 크게 세 가지로 나눌 수 있다.

- **단기 메모리(Short-term Memory)**: 현재 진행 중인 대화 이력. 세션이 종료되면 폐기될 수 있는 일시적 상태다.
- **장기 메모리(Long-term Memory)**: 사용자 프로필, 선호 설정, 누적된 상호작용 패턴처럼 세션을 넘어 지속되는 상태다.
- **세션 경계(Session Boundary)**: 하나의 대화 단위가 어디서 시작되고 어디서 끝나는지를 정의하는 논리적 구분이다.

이 세 유형을 명시적으로 구분하지 않으면, 단기 이력과 장기 프로필이 뒤섞여 컨텍스트 윈도우를 불필요하게 소모하거나, 세션이 달라졌는데도 이전 세션의 맥락이 잘못 유입되는 문제가 생긴다.[^web_20]

상태 저장소는 파이프라인 내부가 아닌 독립된 외부 저장소로 분리하는 것이 권장된다.[^web_20] 상태가 파이프라인 코드 내부에 인메모리 변수 형태로 존재하면, 서버 재시작이나 스케일 아웃 시 상태가 소실된다. 상태 관리자가 외부 저장소(데이터베이스, 캐시 레이어 등)와 인터페이스하는 독립 컴포넌트로 설계될 때, 파이프라인은 수평 확장과 장애 복구를 보다 안정적으로 지원할 수 있다.

컨텍스트 윈도우 예산 관리도 이 단계의 책임이다. 대화 이력을 그대로 쌓기보다는, 오래된 메시지를 요약하거나 중요도 기반으로 선별하여 윈도우 내에서 가장 유효한 컨텍스트를 유지하는 전략이 필요하다.[^web_08] 상태 관리자가 이 선별 로직을 담당할 때, 프롬프트 구성기는 "어떤 상태를 쓸 것인가"를 결정하는 대신 "전달받은 상태를 어떻게 배치할 것인가"에만 집중할 수 있다.

---

상태 관리자까지 갖추면 파이프라인은 대화 연속성을 유지할 수 있는 구조를 갖추게 된다. 그러나 아직 한 가지 질문이 남는다. 이 파이프라인이 실행 중에 실패하면 어떻게 되는가? 모델 API가 응답하지 않거나, 출력 파싱이 실패하거나, 외부 도구 호출이 타임아웃되는 상황에서 시스템은 어떻게 반응해야 하는가. 상태가 잘 유지되더라도, 실패를 처리하는 레이어가 없다면 파이프라인은 단 한 번의 장애에도 멈춰 버린다. 다음 섹션에서는 파이프라인 전체에서 실패를 포착하고 복구하는 역할을 담당하는 오류 처리기(Error Handler)를 살펴본다.

[^web_08]: 컨텍스트 관리 가이드. stateless LLM의 본질에서 상태 관리 필요성을 도출하는 논거와 Karpathy 인용을 포함한다.
[^web_03]: Anthropic 설계 원칙. 메모리와 상태를 LLM 시스템의 빌딩 블록으로 다루는 관점을 제시한다.
[^web_20]: Google ADK 사례. 상태 관리와 컨텍스트 구성의 분리를 실제 Big Tech 설계 결정으로 뒷받침한다.

## 오류 처리기(Error Handler): 실패를 예외가 아닌 설계 대상으로

상태 관리자까지 갖추면 파이프라인은 대화 연속성을 유지할 수 있는 구조를 갖추게 된다. 그러나 아직 한 가지 질문이 남아 있다. 이 파이프라인이 실행 도중 실패하면 어떻게 되는가? 모델 API가 응답을 돌려주지 않거나, 출력 파싱이 실패하거나, 외부 서비스 호출이 타임아웃되는 상황에서 시스템은 어떻게 반응해야 하는가. 상태가 아무리 잘 관리되더라도, 실패를 처리하는 레이어가 없다면 파이프라인은 단 한 번의 장애에도 완전히 멈춰 버린다. LLM 시스템에서 오류는 예외적 상황이 아니다. 오류는 정상적인 운영의 일부다.[^web_06]

---

**역할(Role)**

오류 처리기의 역할은 파이프라인 각 단계에서 발생하는 실패를 탐지하고, 그 원인 유형에 맞는 대응—재시도, 폴백, 알림—을 실행하는 것이다.[^web_06] 여기서 핵심은 "원인 유형에 맞는"이라는 조건이다. 파이프라인의 각 단계는 서로 다른 종류의 실패를 겪는다. 입력 처리기의 실패는 대부분 형식 오류(Malformed Input)이며, 이는 재시도로 해결되지 않는다. 잘못 구성된 입력을 그대로 다시 전달해도 결과는 같다. 반면 LLM 호출기의 실패는 API 타임아웃이나 일시적 서비스 불가(HTTP 503) 같은 순간적 문제일 수 있으며, 이 경우 지수 백오프(Exponential Backoff — 재시도 간격을 점차 늘려가는 방식)를 적용한 재시도가 효과적이다. 출력 처리기의 실패는 모델이 예상된 JSON 스키마를 따르지 않은 파싱 실패일 가능성이 높으며, 이 경우 파싱 오류를 LLM 호출기로 다시 피드백하여 재생성을 요청하는 방식이 적합하다.[^web_21]

이처럼 실패 유형이 단계마다 다르다면, 오류 처리 로직도 단계에 특화되어 설계되어야 한다. 모든 실패를 동일하게 처리하는 단일 try-catch 블록은 이 요구를 충족하지 못한다. 오류 처리기가 파이프라인의 독립된 구성 요소로 존재해야 하는 근본적인 이유가 여기에 있다.[^web_07]

다음은 단계별 실패 유형과 적합한 대응을 간략히 비교한 것이다.

| 단계 | 실패 유형 예시 | 적합한 대응 |
|---|---|---|
| 입력 처리기 | 형식 오류, 필수 필드 누락 | 즉시 거부 및 사용자 알림 (재시도 불필요) |
| LLM 호출기 | API 타임아웃, 503 서비스 불가 | 지수 백오프 재시도, 서킷 브레이커 |
| 출력 처리기 | JSON 파싱 실패, 스키마 불일치 | 재생성 요청, 폴백 출력 |

---

**서킷 브레이커 패턴**

LLM 호출기의 연속적 실패에 대응하는 방식으로 서킷 브레이커(Circuit Breaker — 연속 실패가 임계값을 초과하면 호출을 일시 중단하는 패턴)가 자주 언급된다.[^web_06] 전기 회로의 차단기를 떠올리면 이해가 쉽다. 과부하가 감지되면 차단기가 회로를 끊어 전체 시스템을 보호하듯, 서킷 브레이커는 모델 API의 연속 실패가 일정 횟수를 넘으면 추가 호출을 일시 차단하고 폴백(Fallback — 주 경로가 실패했을 때 대체 경로로 전환하는 전략) 응답을 반환한다. 이는 이미 장애 상태인 외부 서비스에 계속 요청을 보내 파이프라인 전체가 블로킹되는 상황을 막는다.

---

**실패 시 증상**

오류 처리기가 독립 컴포넌트로 존재하지 않을 때 가장 먼저 나타나는 증상은, 한 단계의 오류가 파이프라인 전체를 중단시킨다는 점이다. 모델 API 호출이 일시적으로 실패했을 때, 재시도 로직 없이 즉시 예외가 상위로 전파되어 사용자에게 오류 화면을 노출하게 된다. 더 조용하고 위험한 증상도 있다. 실패 원인 유형에 관계없이 획일적인 처리가 적용되는 경우다. 예를 들어 출력 처리기의 파싱 실패를 LLM 호출기의 타임아웃과 동일한 재시도 로직으로 처리하면, 아무리 재시도해도 파싱 실패는 해소되지 않는다. 모델이 잘못된 형식의 응답을 계속 돌려줄 가능성이 높기 때문이다. 실패 원인을 구분하지 않는 오류 처리는, 오히려 실패를 반복적으로 증폭시키는 결과를 낳는다.[^web_07]

---

**설계 시 고려사항**

오류 처리기를 설계할 때는 세 가지 축을 먼저 정리하는 것이 유용하다.

첫째, 재시도 조건과 횟수 제한이다. 모든 실패가 재시도 대상이 아니며, 재시도가 적합한 경우에도 무제한 반복은 장애를 증폭시킨다. 재시도 가능(Retryable) 실패와 재시도 불가(Non-retryable) 실패를 명시적으로 구분하고, 재시도 최대 횟수와 간격 전략을 설정해야 한다.

둘째, 폴백 정책이다. 재시도가 모두 소진된 이후 시스템이 어떻게 동작해야 하는지를 미리 정의해 두는 것이 필요하다. 기본값(Default Response)을 반환하거나, 더 단순한 모델로 전환하거나, 사용자에게 명확한 안내 메시지를 제공하는 방식이 모두 폴백 정책의 범주에 포함된다.[^web_06]

셋째, 실패 로깅과 알림이다. 오류 처리기가 단순히 실패를 감추는 역할에 그쳐서는 안 된다. 1장에서 정의한 하네스의 세 축 중 "실패 처리"가 이 단계에서 구체적 컴포넌트로 구현되는 지점임을 상기하자. 실패의 유형, 발생 단계, 빈도를 구조화된 형태로 기록할 때, 운영 중 파이프라인의 어느 단계가 가장 취약한지를 지속적으로 파악할 수 있다. 진단 가능성은 오류 처리기의 로깅 품질에 크게 달려 있다.[^web_07]

---

여섯 개의 구성 요소를 이제 모두 살펴봤다. 입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기—각각이 어떤 역할을 하고, 실패하면 어떤 증상이 나타나며, 설계할 때 무엇을 고려해야 하는지를 정리했다. 이제 한 가지 질문이 남는다. 왜 이것들을 굳이 분리해야 하는가? 하나의 큰 함수로 작성해도 동작은 한다. 그렇다면 분리에는 어떤 설계적 이유가 있는가. 다음 섹션에서는 바로 그 질문을 다룬다.

[^web_21]: 파싱 오류 처리 자료. 출력 처리기의 실패가 오류 처리기와 협력하는 계층 구조를 다룬다.

## 왜 분리하는가: 교체 가능성과 평가 가능성

지금까지 여섯 개의 구성 요소를 하나씩 살펴봤다. 입력 처리기부터 오류 처리기까지, 각 단계가 어떤 역할을 하고 실패하면 어떤 증상이 나타나며 설계할 때 무엇을 고려해야 하는지를 정리했다. 그런데 여기서 한 걸음 물러서면 당연한 질문이 떠오른다. 굳이 이렇게 나눠야 하는가? 입력 검증부터 오류 재시도까지를 하나의 함수에 담아도 동작은 한다. 실제로 초기 프로토타입은 대부분 그런 방식으로 시작된다. 그렇다면 분리는 코드 정리 취향의 문제인가, 아니면 더 본질적인 이유가 있는가.

답은 분리가 설계 목표—교체 가능성(Replaceability), 독립 평가 가능성(Independent Evaluability), 실패 격리(Failure Isolation)—를 달성하기 위한 의도적 선택이라는 것이다.[^web_09] 이 세 가지가 왜 중요한지는, 분리하지 않았을 때 어떤 결과가 생기는지를 보면 역설적으로 분명해진다.

먼저 교체 가능성을 생각해 보자. LLM 파이프라인에서 가장 자주 교체되는 구성 요소는 무엇인가. 모델이다. GPT-4에서 Claude로, 혹은 더 저렴한 오픈소스 모델로 전환하는 결정은 비용, 성능, 정책 등 다양한 이유로 발생한다. 이때 LLM 호출기가 독립된 레이어로 분리되어 있다면, 모델 교체는 그 레이어 내부의 변경으로 국한된다. 프롬프트 구성 로직은 그대로 유지되고, 출력 파싱 스키마도 건드릴 필요가 없다. 반면 모델 호출 코드가 프롬프트 구성 로직, 응답 파싱 코드와 한 덩어리로 뒤섞여 있다면, 모델 교체는 그 덩어리 전체에 영향을 미친다. 어디까지 수정해야 하는지 파악하기 어렵고, 수정 후 어느 부분이 변경된 것인지 추적하기도 어렵다. 변경의 파급 범위를 제어할 수 없게 되는 것이다.[^web_09]

독립 평가 가능성은 운영 단계에서 더 뚜렷하게 체감된다. 프롬프트 구성 로직을 개선했다고 가정하자. 그 변경이 실제로 출력 품질에 영향을 주었는지 측정하려면, 다른 단계의 변수를 고정한 상태에서 프롬프트 구성기만을 대상으로 평가를 실행해야 한다. 구성 요소가 분리되어 있을 때 이것이 가능하다. 특정 단계에 테스트 입력을 주입하고, 그 단계의 출력만을 측정하는 방식으로 평가를 설계할 수 있다.[^web_02] 하지만 모든 단계가 하나의 흐름으로 엮여 있다면, 프롬프트 구성 로직의 효과를 다른 단계의 영향과 분리해서 측정하는 것이 원리적으로 불가능해진다. 무엇을 바꿨을 때 무엇이 나아졌는지 알 수 없는 상태—이것이 평가 가능성이 없는 시스템의 전형적인 증상이다.

실패 격리는 오류 처리기를 다룬 앞 섹션에서 이미 구체적으로 살펴봤다. 한 단계의 실패가 파이프라인 전체를 중단시키지 않으려면, 각 단계의 경계가 명확해야 한다. 경계가 있어야 "이 단계가 실패했다"는 사실을 탐지할 수 있고, 나머지 단계의 실행을 보호할 수 있다. 경계 없는 구조에서는 실패가 어디서 발생했는지 특정할 수 없으며, 결국 전체를 다시 시도하거나 전체를 중단하는 것 외에 선택지가 없어진다.

소프트웨어 설계에서 관심사의 분리(Separation of Concerns, SoC)는 오래된 원칙이다. LLM 파이프라인에서 구성 요소를 나누는 것은 이 원칙의 구체적 적용이다.[^web_09] 단, LLM 시스템에서 이 원칙은 단순한 코드 정리 이상의 의미를 갖는다. 모델의 비결정성과 외부 API 의존성이 조합되면, 파이프라인의 어느 지점에서든 예상치 못한 실패가 발생할 수 있다. 그 실패를 진단하고, 고치고, 개선 효과를 검증하는 전 과정이 분리된 구조를 전제로 한다.

분리는 완성된 시스템에서만 유효한 것이 아니다. 프로토타입을 만들 때부터 각 구성 요소의 경계를 의식하는 것이 중요하다. 처음부터 하나의 덩어리로 만들고 나중에 분리하려고 하면, 단계 간 의존이 깊어져 분리 비용이 급격히 높아진다. 반대로, 처음부터 역할 단위로 경계를 그어 두면 각 단계를 독립적으로 발전시킬 수 있는 여지가 생긴다.

---

이것으로 4장의 논의를 마무리한다. 파이프라인이라는 관점에서 시작해, 입력 처리기·프롬프트 구성기·LLM 호출기·출력 처리기·상태 관리자·오류 처리기 여섯 개의 구성 요소를 각각의 역할과 실패 증상, 설계 고려사항의 축으로 살펴봤다. 그리고 그것들이 왜 독립된 단위로 존재해야 하는지—교체 가능성, 독립 평가 가능성, 실패 격리—를 정리했다. 이 여섯 개의 구성 요소는 LLM 시스템을 설계하는 가장 기본적인 어휘다. 5장에서는 이 어휘 위에서 RAG(검색 증강 생성)가 어떤 구성 요소를 확장하고, 어떤 새로운 설계 선택을 요구하는지를 다룬다. 지금 정리한 파이프라인 구조가 RAG를 기능이 아닌 아키텍처로 이해하는 출발점이 될 것이다.
