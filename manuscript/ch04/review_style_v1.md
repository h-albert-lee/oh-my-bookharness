# Style Review: ch04

## Score

- tone_fit: **3/5**
- reader_friendliness: **4/5**
- redundancy: **2/5**

## Must Fix

- 선언문 스타일 반복 위반: '~해야 한다' 형태가 도입부와 본문 전반에 걸쳐 과도하게 반복됨. 예: '진단 가능성(Diagnosability)이 시스템 설계의 첫 번째 목표가 되어야 한다', '분리되어 있어야 한다', '완결되어야 이후 단계들이…', '보존되어 있어야 한다' 등. tone_guide의 '~해야 한다'의 반복적 나열 금지 원칙에 직접 저촉됨. 각 문장을 서술형(~된다, ~한다, ~이다) 또는 설명형으로 전환 필요.
- 자기 고양적 선언 표현 존재: '진단 가능성(Diagnosability)—어느 단계에서 무엇이 잘못되었는지를 특정할 수 있는 능력—이 시스템 설계의 첫 번째 목표가 되어야 한다'는 문장은 필자의 주장을 원칙처럼 선언하는 형태로, 한빛미디어 기술서 톤에서 지양하는 권위적 서술임. '이 장에서는 진단 가능성을 기준으로 구성 요소를 분해한다' 수준으로 낮출 것.
- 볼드 강조 남용: '실패는 하위 단계에서 발생하지만, 원인은 상위 단계에 있다.' 등 문장 전체를 굵게 처리한 사례가 다수 존재함. 기술서에서 볼드는 핵심 용어·개념 단위에 제한적으로 사용해야 하며, 문장 전체 강조는 레이아웃 품질과 독자 집중도를 저하시킴.

## Should Fix

- 도입부 길이 과잉: '도입: 어디서부터 생각해야 하는가' 섹션이 6개 단락으로 구성되어 있으며, 같은 논지(덩어리 구조 → 진단 불가 → 파이프라인 필요)가 3회 이상 반복 순환됨. 독자 동기부여 목적의 도입은 2~3단락으로 압축하고, 나머지 논거는 본문 섹션에 분산 배치 권장.
- 각주 설명 방식의 일관성 부재: 각주가 '레퍼런스 설명'과 '논증 근거 서술'이 혼재된 형태로 작성됨. 예: '[^web_07]: 실패 분류 논문. LLM 기반 시스템의 실패 원인을 분류하고…'는 문헌 정보가 아닌 편집자 메모에 가까움. 실제 출판 기준에 맞는 인용 형식(저자, 제목, 출처)으로 통일하거나, 현재 형태가 집필 초안용 임시 메모임을 명시할 것.
- 용어 병기 원칙 부분 준수 불일치: 'Diagnosability'는 한글(영문) 병기 후 한글 사용 원칙에 따라 '진단 가능성(Diagnosability)'으로 첫 등장 처리된 것은 적절하나, 이후에도 '진단 가능성(Diagnosability)'으로 반복 병기되는 사례가 있음. 첫 등장 이후에는 '진단 가능성'으로만 사용할 것.
- '파이프라인이라는 관점' 섹션 내 중복 논증 정리: '단계가 분리되어 있을 때 추적 가능하다 → 덩어리 구조에서는 인과관계 파악 어렵다'는 논지가 이미 도입부에서 충분히 서술됨. 이 섹션은 파이프라인의 구조적 정의와 기존 SW 아키텍처 비유(미들웨어, MVC)를 중심으로 재편하고, 반복 논증은 삭제 권장.
- 의사코드 사용의 적합성 재검토: blueprint에 '코드 비중 최소화, 이론 중심'이 명시되어 있음. 입력 처리기 섹션의 의사코드는 개념 이해에 유용하나, 실제 구현 코드처럼 보이는 구체적 함수 구조는 이론서 성격과 다소 긴장 관계에 있음. 플로우차트 또는 단계 목록(텍스트 기반)으로 대체하거나, '개념 수준의 예시'임을 본문에서 명확히 안내할 것.
- 프롬프트 구성기 섹션 미완성: 해당 섹션이 코드 블록 중간에서 잘려 있음('# 비즈니스' 이후 내용 없음). 리뷰 범위 밖이지만, 미완성 섹션이 포함된 초안이 검토 대상으로 제출된 점을 편집 프로세스 측면에서 확인 필요.

## Nice to Have

- 도입부 시나리오의 독자 공감 강화: '챗봇이 갑자기 이상한 답변을 내놓기 시작했다'는 시나리오는 실무 경험과 잘 연결되는 좋은 도입임. 다만 '프롬프트를 고쳤다 → 모델을 바꿨다 → 막다른 벽'으로 이어지는 전개가 조금 빠르게 진행됨. 각 시도에서 독자가 느끼는 좌절감을 한 문장씩 더 구체화하면 공감도 상승 가능.
- 그림 참조 설명의 독립성 보완: '그림 4-1은 이 장에서 다룰 LLM 파이프라인의 전체 흐름을 보여준다'는 문장이 그림 없이 본문만 읽는 독자에게 맥락을 충분히 제공하지 못함. 그림 설명 직후에 '그림이 없는 상태에서도 이해할 수 있도록' 구성 요소를 간략히 나열하는 1~2문장 보완 고려.
- 섹션 간 전환 문장의 패턴 다양화: 각 섹션 말미의 전환 문장이 '다음 섹션의 주제다', '시작하자', '살펴볼 차례다' 등 유사한 형태로 반복됨. 전환 패턴을 섹션별로 다양화하면 독자의 읽기 흐름이 단조롭지 않게 유지됨.
- 소제목 서술 방식 통일 검토: '입력 처리기: 시스템이 받아들이는 것을 통제하라'는 명령형이며, '파이프라인이라는 관점'은 명사형, '프롬프트 구성기(Prompt Builder): 맥락을 조립하는 단계'는 서술형임. 장 전체 소제목의 어미 형태를 통일하면 구조적 일관성이 높아짐.
