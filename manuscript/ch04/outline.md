# outline.md — ch04: LLM 파이프라인 기본 구성 요소

## 장 제목

**4장. LLM 파이프라인 기본 구성 요소**

---

## 장 전체 Argument Flow

ch03이 "실패의 원인은 시스템에 있다"고 진단했다면, ch04는 그 시스템이 실제로 무엇으로 구성되는지를 분해해 보인다. 이 장의 논리적 흐름은 하나의 질문에서 출발한다: *LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가?* 그 차이는 "단일 API 호출"과 "역할이 분리된 처리 흐름"의 차이다. 장은 먼저 파이프라인이라는 관점 자체가 왜 필요한지를 설득한 뒤, 여섯 개의 핵심 구성 요소(입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기)를 각각 "이것이 없으면 어떤 일이 벌어지는가"라는 역순 서술 방식으로 정의한다. 이어서 구성 요소들을 분리하는 행위 자체가 교체 가능성과 평가 가능성이라는 설계 목표를 달성하기 위한 의도적 선택임을 보여주고, 데이터가 각 단계를 통과하며 어떻게 변형되는지를 정보 흐름(information flow) 개념으로 통합한다. 마지막으로 ch01에서 도입한 하네스가 이 파이프라인 전체를 감싸는 외부 구조임을 재확인하며 장을 닫는다. 이 장을 읽고 나면 독자는 RAG(5장)가 프롬프트 구성기에 외부 정보를 주입하는 확장이고, 에이전트(6장)가 이 파이프라인을 동적으로 반복 실행하는 구조임을 예측할 수 있어야 한다.

### Section 0. 도입: 어디서부터 생각해야 하는가

**제목:** API 호출 한 줄로는 충분하지 않은 이유

**목적:**
ch03에서 남긴 질문("그렇다면 시스템은 무엇으로 구성되는가")을 이 장의 출발점으로 명시적으로 환기한다. 독자가 이미 겪었을 법한 구체적인 상황—"작동하던 챗봇이 어느 날 이상한 답변을 내놓기 시작했는데 어디가 문제인지 모르겠다"—을 제시하고, 그 상황이 진단 불가능한 이유가 시스템이 분해되어 있지 않기 때문임을 보여준다. 선언이나 개요 나열이 아니라 **문제 상황 묘사**로 장을 연다.

**사용 source:**
- web_07 (실패 분류 논문): ch03의 "실패는 시스템 문제"라는 진단을 이 장으로 연결하는 학술적 근거
- web_10 (Beyond Prompt Engineering): "프롬프트만으로는 충분하지 않다"는 관점, 진단 불가능 상황의 산업적 맥락
- ch01 summary (하네스 개념, 그림 1-2 언급)

**포함 예시:**
- 단일 LLM 호출로 구성된 챗봇이 특정 시점 이후 품질 저하를 보이는 가상 시나리오: 프롬프트를 수정해도, 모델을 교체해도 문제가 재현됨 → 어디가 원인인지 특정 불가
- ch03에서 등장한 실패 유형 중 1~2개를 "이 실패가 파이프라인의 어느 단계에서 발생했는가"라는 질문으로 재해석

**핵심 요점:**
- LLM을 "호출"하는 것은 한 줄의 코드로 가능하지만, LLM 시스템을 "설계"하는 것은 실패를 어느 단계에서 발생한 것인지 특정할 수 있는 구조를 갖추는 일이다
- 진단 가능성(diagnosability)이 시스템 설계의 첫 번째 목표
- 이 장은 그 구조를 부품 단위로 분해해 보인다

**앞뒤 연결:**
- 앞(ch03): "실패 원인이 시스템에 있다"는 진단 → "그렇다면 시스템은 무엇인가"로 전환
- 뒤(Section 1): 파이프라인이라는 관점 자체를 소개하기 위한 정지 작업

### Section 1. 파이프라인이라는 관점

**제목:** 순서와 역할이 있는 처리 흐름으로 LLM 시스템 보기

**목적:**
"파이프라인"이라는 단어를 LangChain 같은 특정 프레임워크와 동일시하는 오해를 먼저 해소한다. 파이프라인은 개념적 구조이며, 소프트웨어 개발자가 이미 알고 있는 HTTP 요청 처리 흐름(미들웨어 체인, MVC 레이어)과 동일한 추상화 수준임을 보여준다. 그런 다음 LLM 시스템을 파이프라인으로 바라보는 것이 왜 유익한지—교체, 테스트, 진단의 단위가 생기기 때문임—를 논증한다.

**사용 source:**
- web_01 (a16z 레퍼런스 아키텍처): LLM 시스템을 단계화된 흐름으로 분류하는 산업계 기준점
- web_05 (Craig Risi): 기존 SW 아키텍처(MVC, 미들웨어)와 LLM 파이프라인의 구조적 유사성 및 차이
- web_09 (모듈형 파이프라인): SoC(관심사의 분리) 원칙을 LLM에 적용하는 실용 가이드
- web_02 (Chip Huyen): "각 단계를 독립적으로 테스트해야 한다"는 설계 원칙

**포함 예시:**
- 비유: 웹 서버의 미들웨어 체인 — 인증 미들웨어, 로깅 미들웨어, 비즈니스 로직이 분리된 구조가 LLM 파이프라인의 단계 분리와 동일한 원리임을 보여주는 1단락 비교
- 그림 4-1 (필수): LLM 파이프라인 전체 흐름도 — 6개 구성 요소가 데이터를 어떻게 통과시키는지, 하네스가 이 전체를 감싸는 위치에 있음을 함께 표시. ch01 그림 1-2와 시각적 연속성 유지

**핵심 요점:**
- 파이프라인 = 순서(sequence)와 역할(role)이 명시된 처리 흐름. 특정 도구나 프레임워크가 아니다
- 파이프라인 관점의 이점: 각 단계를 독립적으로 교체·테스트·진단할 수 있는 경계가 생긴다
- 프롬프트 엔지니어링은 파이프라인 전체가 아니라 한 단계(프롬프트 구성기)에 해당한다
- 하네스(ch01)는 이 파이프라인 전체를 외부에서 감싸는 구조임을 재확인

**앞뒤 연결:**
- 앞(Section 0): 진단 불가능 문제의 원인이 "분해되지 않은 구조"라고 했다 → 파이프라인이 그 분해의 방식
- 뒤(Section 2~7): 각 구성 요소를 순서대로 정의하기 위한 틀 제공

### Section 2. 입력 처리기

**제목:** 입력 처리기(Input Processor): 시스템이 받아들이는 것을 통제하라

**목적:**
입력 처리기가 없을 때 발생하는 구체적 문제(악성 입력 통과, 예외 처리 불가, 디버깅 불가)를 먼저 보여준 뒤 역할을 정의한다. 역할(Role) / 실패 시 증상 / 설계 시 고려사항 3요소 서술 구조를 이 섹션에서 처음 도입하고 이후 구성 요소 섹션에서 반복 적용한다.

**사용 source:**
- web_01 (a16z): 입력 처리 단계의 역할 범주화
- web_03 (Anthropic): 입력 통제를 빌딩 블록 개념으로 설명하는 맥락
- ch01 summary: "입력 통제"가 하네스 설계의 첫 번째 축임을 재연결

**포함 예시:**
- 실패 시 증상 사례: 사용자가 입력한 1만 자 텍스트가 프롬프트 구성기로 그대로 넘어가 컨텍스트 윈도우를 초과하는 상황 → 오류 메시지가 입력 단계가 아닌 LLM 호출 단계에서 발생하여 원인 파악 지연
- 설계 고려사항 예시: 입력 길이 제한, 인코딩 정규화, 허용 입력 유형 명세 — 의사코드 수준 1개

**핵심 요점:**
- 역할: 사용자 입력을 정제·검증하여 파이프라인이 처리할 수 있는 형태로 변환
- 실패 시 증상: 하위 단계에서 예측 불가능한 오류 발생, 실패 원인 추적 불가
- 설계 고려사항: 입력 처리기는 시스템의 첫 번째 방어선이자 첫 번째 기록 지점(로깅 포인트)
- ch01의 "입력 통제" 개념이 구체적 컴포넌트로 구현되는 지점임을 명시

**앞뒤 연결:**
- 앞(Section 1): 파이프라인의 첫 번째 단계로 자연스럽게 진입
- 뒤(Section 3): 정제된 입력이 프롬프트 구성기에 전달되는 흐름으로 연결

### Section 3. 프롬프트 구성기

**제목:** 프롬프트 구성기(Prompt Builder): 맥락을 조립하는 단계

**목적:**
"프롬프트만 잘 쓰면 된다"는 오해를 정면으로 해소한다. 프롬프트 구성은 파이프라인의 한 단계이며, 동시에 가장 많은 설계 결정이 집중되는 단계임을 보여준다. 프롬프트가 코드 로직과 뒤섞였을 때 발생하는 유지보수 문제를 구체적으로 서술하고, 프롬프트 구성기가 독립된 단위여야 하는 운영적 근거를 제시한다.

**사용 source:**
- web_04 (프롬프트 템플릿 논문): 프롬프트 구성의 구조화가 품질에 미치는 실증적 영향
- web_10 (Beyond Prompt Engineering): 프롬프트 구성이 시스템 설계의 일부임을 보여주는 산업적 관점
- web_19 (프롬프트 드리프트 분석): 프롬프트 구성기가 독립 단위여야 하는 운영적 근거
- web_22 (운영 실패 사례): 프롬프트와 로직이 섞인 코드의 실제 유지보수 실패 맥락

**포함 예시:**
- 나쁜 설계 사례: 비즈니스 로직, 사용자 데이터 포맷팅, 시스템 지시문이 하나의 함수 안에 뒤섞인 의사코드 → 프롬프트 수정 시 로직 전체가 영향 받는 구조
- 좋은 설계 사례: 시스템 지시(system instruction), 맥락(context), 사용자 입력(user turn)이 분리된 조립 구조 — 의사코드 수준 비교

**핵심 요점:**
- 역할: 시스템 지시, 컨텍스트, 사용자 입력, 예시(few-shot)를 조합하여 LLM에 전달할 최종 입력 구성
- 실패 시 증상: 프롬프트 변경이 비즈니스 로직 변경과 결합되어 회귀 테스트 불가, 프롬프트 드리프트 발생
- 설계 고려사항: 프롬프트 템플릿의 버전 관리, 시스템 지시와 사용자 입력의 명확한 분리
- 5장(RAG) 포석: "프롬프트 구성기에 외부 정보를 주입하는 구조"가 RAG의 핵심임을 이 단계에서 예고

**앞뒤 연결:**
- 앞(Section 2): 입력 처리기가 정제한 데이터를 받아 조립하는 단계
- 뒤(Section 4): 조립된 프롬프트가 LLM 호출기로 전달되는 흐름

### Section 4. LLM 호출기

**제목:** LLM 호출기(LLM Caller): 모델 호출을 격리된 단계로 두어야 하는 이유

**목적:**
LLM 호출 자체는 파이프라인의 한 단계이지 전부가 아님을 확인한다. 이 단계를 나머지 로직과 분리했을 때 얻는 이점—모델 교체의 용이성, 호출 방식(스트리밍/배치) 변경의 격리, 지연(latency)·비용 측정 지점 확보—을 중심으로 설명한다.

**사용 source:**
- web_01 (a16z): LLM 호출을 격리된 단계로 다루는 아키텍처 패턴
- web_02 (Chip Huyen): 모델 교체 가능성(model swappability)을 위한 호출 레이어 분리 원칙
- web_15, web_16 (프레임워크 비교): LangChain/LlamaIndex가 LLM 호출을 추상화하는 방식이 서로 다르지만 동일한 개념 문제를 해결하고 있음을 보여주는 비교

**포함 예시:**
- 비교 예시: LLM 호출 코드가 비즈니스 로직 전반에 산재된 구조 vs. 호출기 레이어로 격리된 구조 — 모델을 GPT-4에서 Claude로 교체할 때 영향 범위 비교
- 프레임워크 비교 표(소규모): LangChain, LlamaIndex, 직접 구현이 LLM 호출 단계를 어떻게 추상화하는지 — "구성 요소는 특정 도구에 종속되지 않는다"는 주장의 실증

**핵심 요점:**
- 역할: 프롬프트 구성기의 출력을 받아 모델 API를 호출하고 원시 응답(raw response)을 반환
- 실패 시 증상: 모델 교체 시 전체 코드베이스 수정 필요, 호출 비용·지연 측정 지점 부재
- 설계 고려사항: 모델 공급자 추상화, 타임아웃 설정, 스트리밍 응답 처리 방식
- "LLM 호출 = LLM 시스템 완성"이라는 오해를 이 단계에서 명시적으로 반박

**앞뒤 연결:**
- 앞(Section 3): 프롬프트 구성기의 출력을 입력으로 받음
- 뒤(Section 5): 모델의 원시 응답을 출력 처리기가 받아 처리하는 흐름

### Section 5. 출력 처리기

**제목:** 출력 처리기(Output Processor): 모델 응답을 시스템이 사용할 수 있는 형태로

**목적:**
모델 응답이 항상 기대한 형태로 오지 않는다는 현실을 출발점으로 삼는다. 출력 처리기가 없을 때 발생하는 파싱 오류, 형식 불일치, 다운스트림 컴포넌트 장애를 보여준 뒤, 출력 처리기가 "모델과 나머지 시스템 사이의 계약"을 구현하는 단계임을 정의한다. ch01의 "출력 평가" 개념이 이 단계와 연결됨을 명시한다.

**사용 source:**
- web_17 (출력 처리 심화): 출력 파싱 오류 유형과 처리 레이어 구조
- web_18 (data contract): 출력 처리기를 시스템 간 계약으로 보는 관점
- web_21 (파싱 오류 처리): 출력 처리기와 오류 처리기의 협력 관계

**포함 예시:**
- 실패 시 증상 사례: JSON 형식을 요청했지만 모델이 마크다운 코드블록으로 감싼 JSON을 반환 → 파싱 실패가 LLM 호출 단계의 문제로 오인되는 상황
- 설계 고려사항 예시: 출력 스키마 명세(output schema), 파싱 실패 시 재시도 트리거 조건 — 오류 처리기와의 연계 지점

**핵심 요점:**
- 역할: 모델의 원시 응답을 파싱·정제·변환하여 다운스트림 컴포넌트가 사용할 수 있는 구조화된 형태로 변환
- 실패 시 증상: 파싱 오류가 상위 단계 문제로 오인, 비정형 출력이 다운스트림에 전파
- 설계 고려사항: 출력 스키마 버전 관리, 파싱 실패 탐지 및 오류 처리기 연동
- ch01의 "출력 평가" 개념이 이 단계에서 구체적 컴포넌트로 구현됨을 명시

**앞뒤 연결:**
- 앞(Section 4): LLM 호출기의 원시 응답을 받음
- 뒤(Section 6): 정상 처리된 출력은 상태 관리자에 기록되거나 최종 반환됨; 실패는 오류 처리기로 전달됨

### Section 6. 상태 관리자

**제목:** 상태 관리자(State Manager): LLM은 기억하지 못한다

**목적:**
LLM이 본질적으로 무상태(stateless)임을 먼저 확인한 뒤, 그것이 대화 연속성·세션 유지·멀티턴 처리에 왜 문제가 되는지를 보여준다. 상태 관리를 파이프라인 외부(애플리케이션 레이어)에 위임했을 때와 명시적 컴포넌트로 분리했을 때의 차이를 대비한다. 6장(에이전트)에서 상태 관리자가 핵심이 된다는 포석을 여기서 심는다.

**사용 source:**
- web_08 (컨텍스트 관리): stateless LLM의 본질에서 상태 관리 필요성을 도출하는 논거, Karpathy 인용 포함
- web_20 (Google ADK 사례): 상태 관리와 컨텍스트 구성 분리를 실제 Big Tech 결정으로 뒷받침
- web_03 (Anthropic): 메모리와 상태를 빌딩 블록으로 다루는 관점

**포함 예시:**
- 무상태 LLM의 구체적 한계: 사용자가 세 번째 메시지에서 "아까 말한 그 조건으로"라고 했을 때 상태 관리자 없이는 이전 대화를 복원할 방법이 없는 상황
- 설계 고려사항 예시: 단기 메모리(대화 이력), 장기 메모리(사용자 프로필), 세션 경계 — 세 가지 유형의 상태를 구분하는 간단한 분류

**핵심 요점:**
- 역할: 대화 이력, 세션 정보, 사용자 맥락 등 파이프라인 실행 간에 유지되어야 하는 상태를 관리
- 실패 시 증상: 멀티턴 대화에서 맥락 단절, 세션별 개인화 불가, 컨텍스트 윈도우 비효율적 사용
- 설계 고려사항: 상태의 유형(단기/장기) 분리, 상태 저장소의 독립성, 컨텍스트 윈도우 예산(budget) 관리
- 6장(에이전트) 포석: 에이전트가 파이프라인을 여러 번 실행할 때 상태 관리자가 실행 간 정보를 연결하는 핵심 구조임을 예고

**앞뒤 연결:**
- 앞(Section 5): 출력 처리기가 생성한 결과가 상태로 기록되는 흐름
- 뒤(Section 7): 상태 관리와 함께 파이프라인 전체에서 실패를 처리하는 오류 처리기로 이동

### Section 7. 오류 처리기

**제목:** 오류 처리기(Error Handler): 실패를 예외가 아닌 설계 대상으로

**목적:**
LLM 시스템에서 오류는 예외적 상황이 아니라 정상적 운영의 일부임을 선언한다. 오류 처리기가 파이프라인의 독립적 구성 요소여야 하는 이유—각 단계의 실패 유형이 다르고, 적절한 대응(재시도·폴백·서킷 브레이커)이 실패 원인에 따라 달라지기 때문—를 논증한다. ch01의 "실패 처리" 개념이 이 단계에서 구체화됨을 명시한다.

**사용 source:**
- web_06 (Maxim AI): 재시도·폴백·서킷 브레이커의 독립 레이어화 논거
- web_21 (파싱 오류 처리): 출력 처리기 실패가 오류 처리기와 협력하는 계층 구조
- web_07 (실패 분류 논문): 파이프라인 각 단계별 실패 유형 분류의 학술적 근거

**포함 예시:**
- 단계별 실패 유형 분류: 입력 처리기 실패(형식 오류) vs. LLM 호출기 실패(타임아웃, API 오류) vs. 출력 처리기 실패(파싱 실패) — 각각에 적합한 대응이 다름을 보여주는 간단한 비교
- 서킷 브레이커 비유: 전기 회로의 차단기가 과부하를 감지해 전체 시스템을 보호하는 것처럼, LLM 호출기의 연속 실패가 감지되면 호출을 일시 중단하는 패턴

**핵심 요점:**
- 역할: 파이프라인 각 단계에서 발생하는 실패를 탐지하고, 재시도·폴백·알림 등 적절한 대응을 실행
- 실패 시 증상: 한 단계의 오류가 파이프라인 전체를 중단시킴, 실패 원인 유형에 무관한 획일적 처리
- 설계 고려사항: 재시도 조건과 횟수 제한, 폴백 정책(fallback policy), 실패 로깅과 알림
- ch01의 "실패 처리" 개념이 구체적 컴포넌트로 구현되는 지점임을 명시

**앞뒤 연결:**
- 앞(Section 6): 상태 관리자와 오류 처리기가 파이프라인의 "운영 지지대" 역할을 함께 담당함을 연결
- 뒤(Section 8): 여섯 구성 요소를 각각 정의한 뒤, "왜 분리해야 하는가"라는 설계 논리 섹션으로 전환

### Section 8. 구성 요소 분리의 설계적 이유

**제목:** 왜 분리하는가: 교체 가능성과 평가 가능성

**목적:**
앞의 여섯 섹션이 "무엇인가"를 다뤘다면, 이 섹션은 "왜 분리해야 하는가"를 다룬다. 분리는 코드 정리 취향이 아니라 설계 목표(교체 가능성, 독립 평가 가능성, 실패 격리)를 달성하기 위한 의도적 선택임을 논증한다. 분리하지 않았을 때의 구체적 결과를 보여줌으로써 역설적으로 분리의 가치를 드러낸다.

- web_09 (모듈형 파이프라인): SoC 원칙의
