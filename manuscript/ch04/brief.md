# Chapter Brief: ch04
## LLM 파이프라인 기본 구성 요소

---

## 1. 이 장의 목표

ch03에서 "문제는 모델이 아니라 시스템이다"라는 진단을 내렸다면, ch04는 그 시스템이 실제로 어떤 부품들로 구성되는지를 분해해서 보여주는 장이다.

LLM 기반 시스템을 처음 설계하는 독자가 "어디서부터 생각해야 하는가"를 답할 수 있도록, 파이프라인의 각 구성 요소를 역할 중심으로 정의하고, 각 요소가 왜 독립된 설계 단위가 되어야 하는지를 구조적으로 설명한다.

**이 장은 튜토리얼이 아니다.** 특정 프레임워크나 라이브러리 사용법이 아니라, "LLM 시스템을 구성하는 개념적 단위"를 독자가 내면화하도록 돕는 것이 목적이다.

---

## 2. 독자가 얻을 것

- LLM 파이프라인을 블랙박스가 아닌, 역할이 구분된 구성 요소들의 연결로 인식하는 관점
- 각 구성 요소(입력 처리, 프롬프트 구성, LLM 호출, 출력 처리, 상태 관리 등)가 왜 분리되어야 하는지에 대한 설계적 근거
- 나중에 RAG(5장)와 에이전트(6장)를 배울 때 "이 구조의 어느 부분이 확장되는가"를 파악할 수 있는 기반 지식
- 파이프라인의 어느 지점에서 무엇이 실패할 수 있는지를 예측하는 사고 습관

---

## 3. 장의 핵심 질문

> LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가?

부속 질문:

- 파이프라인의 각 단계는 어떤 역할을 수행하며, 그 경계는 어디인가?
- 각 구성 요소는 독립적으로 교체하거나 평가할 수 있는가?
- 구성 요소 간의 정보 흐름은 어떻게 설계되어야 하는가?

---

## 4. 반드시 들어갈 개념

### 4-1. 파이프라인(Pipeline)으로서의 LLM 시스템

단일 API 호출이 아닌, 순서와 역할이 있는 처리 흐름으로서의 LLM 시스템 개념.
"프롬프트 엔지니어링"이 파이프라인 전체 중 한 단계에 불과하다는 점을 명확히 한다.

### 4-2. 핵심 구성 요소 (최소 5개 이상, 각각 역할 정의)

| 구성 요소 | 역할 요약 |
|---|---|
| 입력 처리기(Input Processor) | 사용자 입력을 정제·검증하는 단계 |
| 프롬프트 구성기(Prompt Builder) | 컨텍스트, 시스템 지시, 사용자 입력을 조합하는 단계 |
| LLM 호출기(LLM Caller) | 모델 API를 실제로 호출하고 응답을 받는 단계 |
| 출력 처리기(Output Processor) | 모델 응답을 파싱·정제·변환하는 단계 |
| 상태 관리자(State Manager) | 대화 이력·세션·메모리를 유지하는 단계 |
| 오류 처리기(Error Handler) | 실패·재시도·폴백을 담당하는 단계 |

> **편집 지시**: 각 구성 요소를 '역할(Role) / 실패 시 증상 / 설계 시 고려사항' 3요소로 서술. 표보다 서술 중심으로, 그림(파이프라인 흐름도) 1개 필수 삽입.

### 4-3. 구성 요소 분리의 설계적 이유

각 단계를 분리하지 않았을 때 어떤 문제가 생기는가를 구체적 상황으로 제시.
"프롬프트와 로직이 섞인 코드"가 왜 유지보수 불가능 상태로 귀결되는지.

### 4-4. 정보 흐름(Information Flow) 개념 도입

파이프라인에서 데이터가 어떻게 전달되고 변형되는지.
각 단계의 입력과 출력을 명시적으로 정의하는 것이 왜 중요한가.
→ 5장 RAG의 정보 흐름 통제 논의로 직결되는 개념 포석.

### 4-5. 하네스와의 연결

ch01에서 도입한 하네스 개념이 파이프라인 구조와 어떻게 연결되는가.
하네스는 파이프라인 전체를 감싸는 외부 구조임을 재확인.

## 5. 피해야 할 오해

| 오해 | 이 장의 입장 |
|---|---|
| "파이프라인 = LangChain 같은 프레임워크" | 파이프라인은 개념적 구조다. 특정 도구에 종속되지 않는다. |
| "프롬프트만 잘 쓰면 된다" | 프롬프트 구성은 파이프라인의 한 단계일 뿐이며, 다른 단계의 실패를 보완하지 못한다. |
| "구성 요소가 많을수록 복잡하고 나쁘다" | 분리는 교체 가능성과 평가 가능성을 확보하기 위한 설계 선택이다. |
| "LLM 호출 = LLM 시스템 완성" | LLM 호출 전후의 처리 구조가 시스템 품질을 결정한다. |
| "이 구조는 RAG/에이전트에는 해당 안 된다" | RAG와 에이전트는 이 기본 파이프라인의 확장이다(5·6장에서 다룸). |

## 6. 앞 장과의 연결

**ch03 연결 포인트:**

ch03은 "실패의 원인이 모델이 아니라 시스템 설계에 있다"고 주장했다. 그 주장은 "그렇다면 시스템은 무엇으로 구성되는가"라는 질문을 남긴다. ch04는 그 질문에 직접 답한다.

ch03에서 사용한 사례나 논거(예: 운영 환경에서의 실패 유형)를 ch04 도입부에서 짧게 환기시킨 뒤, "이 실패들이 파이프라인의 어느 단계에서 발생하는가"로 전환하는 흐름이 자연스럽다.

**ch01 연결 포인트:**

하네스 개념을 ch04에서 파이프라인 구조와 명시적으로 연결. "하네스는 파이프라인을 감싸는 구조"라는 문장으로 ch01의 핵심 개념을 이 장의 맥락에 재배치한다.

---

## 7. 뒤 장으로의 연결

**ch05 (RAG):** ch04에서 정의한 프롬프트 구성기와 정보 흐름 개념이 RAG에서 어떻게 확장되는지로 연결. "프롬프트 구성기에 외부 정보를 주입하는 구조"가 RAG의 핵심임을 ch04 말미에 예고.

**ch06 (에이전트):** ch04의 파이프라인이 단일 실행 흐름이라면, 에이전트는 이 파이프라인을 여러 번, 동적으로 실행하는 구조임을 암시. 상태 관리자(State Manager)가 에이전트 구조에서 핵심이 된다는 포석을 남긴다.

## 8. 원하는 톤 메모

- "구성 요소를 나열하는 목록 장"이 되어서는 안 된다. 각 요소가 왜 존재해야 하는지의 **설계 논리**를 중심에 두고, 나열은 그 논리를 뒷받침하는 방식으로 구성한다.
- 독자가 기존에 접한 백엔드 API 설계, MVC 패턴 같은 소프트웨어 구조화 경험과 연결하는 비유를 활용할 것. LLM 파이프라인은 전혀 새로운 개념이 아니라 소프트웨어 아키텍처 원칙의 연장선임을 느끼도록.
- 코드 예시는 최소화. 불가피하게 사용할 경우 의사 코드(pseudocode) 수준으로 한정.
- 각 구성 요소 설명에서 "이 단계가 없으면 어떻게 되는가"를 먼저 제시한 뒤 정의를 내리는 역순 서술 방식을 권장. 독자의 문제 인식을 먼저 자극한다.
