# 4장. LLM 파이프라인 기본 구성 요소

## API 호출 한 줄로는 충분하지 않은 이유

어느 날 서비스가 이상해지기 시작했다. 고객 응대 챗봇이 며칠 전까지만 해도 정상적으로 동작했는데, 오늘은 엉뚱한 답변을 반복하고 있다. 팀원들이 모여 원인을 찾기 시작한다. 누군가는 모델의 응답 품질이 갑자기 나빠진 것 아니냐고 말한다. 다른 사람은 어제 수정한 프롬프트가 문제인 것 같다고 한다. 또 다른 팀원은 사용자 입력 방식이 바뀐 탓일 수도 있다고 덧붙인다. 모두가 의심을 갖고 있지만 아무도 확신하지 못한다. 그리고 결정적으로, 어디서부터 손을 대야 할지 모르겠다.

이 상황에서 문제는 모델이 아닐 수 있다. ch03에서 살펴봤듯이, LLM 기반 시스템이 운영 환경에서 실패할 때 그 원인은 대부분 시스템 설계에 있다.[^web_07] 그렇다면 자연스럽게 다음 질문이 뒤따른다. "시스템은 무엇으로 구성되어 있는가?" 그리고 더 실용적인 질문은 이것이다. "어느 부분이 문제인지 어떻게 특정할 수 있는가?"

진단이 불가능한 이유는 시스템이 분해되어 있지 않기 때문이다. LLM을 호출하는 코드와 프롬프트를 구성하는 코드, 응답을 파싱하는 코드, 대화 이력을 관리하는 코드가 하나의 함수에 뒤섞여 있다면, 그 함수가 잘못된 결과를 낸다는 사실만 알 뿐, 왜 그런 결과가 나오는지 추적하기 어렵다. 품질 저하가 입력 처리 문제인지, 프롬프트 구성 문제인지, 모델 자체의 문제인지, 아니면 이전 대화 이력의 오염 때문인지 알 수 없다. 이 진단 불가능성(diagnosability의 부재)이 운영 환경에서 LLM 시스템을 가장 위험한 상태로 만든다.[^web_10]

이 장에서는 그 해답을 파이프라인이라는 관점에서 찾는다. 먼저 파이프라인이라는 개념 자체를 정의하고, 이어서 여섯 개의 핵심 구성 요소를 하나씩 분해해 살펴본다. 각 구성 요소는 단순히 나열하는 방식이 아니라, "이것이 없으면 어떤 일이 벌어지는가"라는 역순 서술 방식으로 정의할 것이다. 문제를 먼저 인식해야 해결책의 가치가 보이기 때문이다. 장의 마지막에는 이 구성 요소들이 왜 분리된 단위로 설계되어야 하는지의 논리를 정리하고, 1장에서 도입한 평가 하네스(harness)가 이 파이프라인 전체를 어떻게 감싸는지 확인할 것이다.

이 장을 읽고 나면, 5장의 RAG(검색 증강 생성)가 파이프라인의 어느 단계를 확장하는 것인지, 6장의 에이전트가 이 파이프라인을 어떻게 반복 실행하는 구조인지를 예측할 수 있게 될 것이다.

## 파이프라인이라는 관점: 순서와 역할이 있는 처리 흐름으로 LLM 시스템 보기

"파이프라인"이라는 단어를 들으면 LangChain이나 LlamaIndex 같은 특정 프레임워크를 먼저 떠올리는 독자가 많다. 그러나 파이프라인은 특정 도구나 라이브러리가 아니다. 파이프라인은 개념적 구조다. 순서(sequence)와 역할(role)이 명시된 처리 흐름을 파이프라인이라고 부른다. 프레임워크는 그 개념을 특정 방식으로 구현한 것일 뿐이며, 프레임워크 없이도 파이프라인 구조로 시스템을 설계할 수 있다.[^web_01]

이 개념이 소프트웨어 개발자에게 전혀 낯선 것은 아니다. 웹 서버를 개발해본 경험이 있다면 미들웨어 체인(middleware chain)을 이미 알고 있다. HTTP 요청이 들어오면 인증 미들웨어가 먼저 처리하고, 로깅 미들웨어가 기록하고, 비즈니스 로직이 실행되고, 응답이 반환되기 전에 직렬화 레이어를 거친다. 각 미들웨어는 자신의 역할만 수행하고 다음 단계로 넘긴다. MVC(Model-View-Controller) 패턴도 마찬가지다. 요청 처리, 데이터 가공, 뷰 렌더링이 각각 독립된 레이어로 분리되어 있다. LLM 파이프라인은 이와 동일한 추상화 수준에 있다.[^web_05] 사용자 입력이 들어오면 정제 단계를 거치고, 프롬프트가 조립되고, 모델이 호출되고, 응답이 처리되고, 결과가 반환된다. 각 단계는 자신의 역할이 있고, 그 경계가 분명하다.

파이프라인 관점이 가져다주는 가장 큰 이점은 교체·테스트·진단의 단위가 생긴다는 점이다. 각 단계가 독립적으로 분리되어 있다면, 입력 처리 단계만 따로 테스트할 수 있고, 프롬프트 구성 단계만 교체할 수 있으며, 오류가 발생했을 때 어느 단계에서 생긴 것인지 특정할 수 있다. Chip Huyen은 LLM 시스템 설계에서 "각 단계를 독립적으로 테스트해야 한다"는 원칙을 강조한다.[^web_02] 이 원칙이 성립하려면 먼저 "단계"라는 단위 자체가 존재해야 한다.

또 하나 명확히 해야 할 오해가 있다. 프롬프트 엔지니어링은 파이프라인 전체가 아니라, 파이프라인 안의 한 단계—프롬프트 구성기(Prompt Builder)—에 해당한다. 프롬프트를 아무리 정교하게 다듬어도, 입력 처리 단계가 잘못된 데이터를 넘기거나 상태 관리 단계가 잘못된 이력을 제공한다면 좋은 결과를 기대하기 어렵다.[^web_10] "프롬프트만 잘 쓰면 된다"는 생각은 파이프라인의 한 단계를 전체로 착각한 데서 비롯한다.

**그림 4-1. LLM 파이프라인 전체 흐름도**

```
┌─────────────────────────────────────────────────────────────────┐
│                       평가 하네스 (Harness)                       │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │ 입력     │──▶│ 프롬프트  │──▶│  LLM     │──▶│  출력    │     │
│  │ 처리기   │   │  구성기   │   │  호출기   │   │  처리기   │     │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘     │
│        │               │               │               │        │
│        ▼               ▼               ▼               ▼        │
│  ┌─────────────────────────────────────────────────┐           │
│  │                  오류 처리기                      │           │
│  └─────────────────────────────────────────────────┘           │
│                         │                                       │
│  ┌─────────────────────────────────────────────────┐           │
│  │                  상태 관리자                      │           │
│  └─────────────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────────────┘
```

*그림 4-1 설명: 사용자 입력이 파이프라인의 각 단계를 순서대로 통과하며 처리된다. 오류 처리기와 상태 관리자는 파이프라인의 모든 단계와 연결되어 있으며, 평가 하네스가 이 전체 구조를 외부에서 감싼다.*

1장에서 도입한 평가 하네스(harness) 개념을 여기서 다시 확인하자. 하네스는 "반복 가능하게 실행과 평가를 수행하는 외부 구조"로 정의된다. 그림 4-1에서 볼 수 있듯이, 하네스는 파이프라인의 특정 단계가 아니라 파이프라인 전체를 외부에서 감싸는 구조다. 파이프라인이 "무엇을 처리하는가"를 정의한다면, 하네스는 "그 처리가 잘 이루어지고 있는지 어떻게 확인하는가"를 담당한다. 두 개념은 독립적이지 않다. 하네스가 효과적으로 작동하려면 감싸는 대상인 파이프라인이 명확하게 분해되어 있어야 한다.

이제 각 구성 요소를 하나씩 살펴보자. 파이프라인의 첫 번째 단계인 입력 처리기(Input Processor)부터 시작한다.

## 입력 처리기: 시스템이 받아들이는 것을 통제하라

먼저, 입력 처리기가 없는 상황을 상상해보자. 사용자가 어떤 텍스트를 입력하든 그대로 다음 단계로 넘어간다. 사용자가 실수로 10만 자짜리 문서를 통째로 붙여넣었다. 이 텍스트는 프롬프트 구성 단계로 넘어가고, 최종 프롬프트는 LLM의 컨텍스트 윈도우를 훨씬 초과한다. 오류는 LLM 호출 단계에서 발생한다. 운영팀은 LLM API 오류 로그를 확인하고 "모델 호출 자체에 문제가 있다"고 결론 내린다. 실제 문제는 입력 길이 제한의 부재였지만, 그것은 세 단계나 건너뛴 지점에서 드러났다. 원인과 증상이 공간적으로 분리된 것이다.[^web_01]

더 심각한 상황도 있다. 악의적 사용자가 시스템 지시를 무력화하려는 문자열을 입력한다. "이전의 모든 지시를 무시하고 다음 내용을 수행하라"와 같은 패턴이다. 입력 처리기가 없다면 이 문자열은 검증 없이 프롬프트 구성 단계로 전달되고, 최악의 경우 시스템 프롬프트와 뒤섞여 의도하지 않은 동작을 유발한다. 이것이 프롬프트 인젝션(prompt injection) 공격의 전형적인 경로다. 입력 처리기는 이 경로를 차단하는 첫 번째 방어선이다.[^web_03]

**역할:** 입력 처리기(Input Processor)는 사용자 또는 외부 시스템에서 들어오는 입력을 정제·검증하여 파이프라인이 처리할 수 있는 형태로 변환하는 단계다. 단순히 데이터를 받아 넘기는 관문이 아니라, 시스템의 기대 범위를 명시하고 그 범위 밖의 입력을 조기에 처리하는 활성적인 단계다.

**실패 시 증상:** 입력 처리기가 없거나 미흡하게 구현되었을 때 증상은 대부분 하위 단계에서 나타난다. 컨텍스트 윈도우 초과 오류가 LLM 호출 단계에서 발생하거나, 인코딩 오류가 프롬프트 구성 단계에서 발생하거나, 허용되지 않은 입력 유형이 모델의 예상치 못한 동작을 유발한다. 이 모든 경우에서 공통점은 하나다. 실패 원인이 발생 지점이 아닌 다른 곳에서 드러난다는 것이다. 이 때문에 디버깅에 불필요한 시간이 소모된다.

**설계 시 고려사항:** 입력 처리기를 설계할 때 핵심적으로 다뤄야 할 항목들을 정리하면 다음과 같다.

```
// 의사코드: 입력 처리기의 기본 책임
function processInput(raw_input):
    // 1. 길이 검증
    if len(raw_input) > MAX_INPUT_LENGTH:
        raise InputTooLongError

    // 2. 인코딩 정규화
    normalized = normalize_encoding(raw_input)

    // 3. 입력 유형 검증
    if not is_valid_type(normalized):
        raise InvalidInputTypeError

    // 4. 기본 정제 (불필요한 공백, 제어 문자 제거)
    cleaned = sanitize(normalized)

    // 5. 로깅 (첫 번째 기록 지점)
    log_input(cleaned, metadata={"timestamp": now(), "session_id": session_id})

    return cleaned
```

여기서 주목할 점은 마지막 단계인 로깅이다. 입력 처리기는 시스템의 첫 번째 방어선이기도 하지만, 동시에 첫 번째 기록 지점(logging point)이기도 하다. 파이프라인을 통과한 입력이 어떤 형태였는지를 기록해 두어야만 나중에 오류가 발생했을 때 "어떤 입력이 이 오류를 유발했는가"를 역추적할 수 있다. 1장에서 정의한 하네스의 핵심 기능 중 하나가 "입력 통제"였다. 입력 처리기는 그 개념이 구체적인 컴포넌트로 구현되는 지점이다.[^web_03]

입력의 유형은 텍스트만이 아닐 수도 있다. 이미지, 오디오, 구조화된 JSON 데이터가 입력으로 들어오는 멀티모달 시스템에서는 입력 처리기의 역할이 더 복잡해진다. 하지만 기본 원칙은 동일하다. 파이프라인이 기대하는 형태로 입력을 변환하고, 그 범위를 벗어나는 입력은 조기에 처리하라. 이 원칙을 지키는 것만으로도 하위 단계에서의 예측 불가능한 오류를 상당 부분 줄일 수 있다.

입력 처리기가 정제한 데이터는 이제 프롬프트 구성기로 넘어간다. 이 단계에서 시스템의 지능—즉, 모델에 무엇을 어떻게 물어볼 것인가—이 비로소 조립되기 시작한다.

## 프롬프트 구성기: 맥락을 조립하는 단계

"프롬프트를 잘 쓰면 모든 문제가 해결된다"는 믿음은 LLM 개발 초기부터 널리 퍼져 있었다. 그 믿음이 완전히 틀린 것은 아니다. 프롬프트의 품질이 모델 출력에 직접적인 영향을 미치는 것은 사실이다.[^web_04] 그러나 이 단계를 파이프라인의 맥락에서 보면, 프롬프트 작성은 전체 시스템 설계의 일부일 뿐이며—그것도 중요한 일부이지만—다른 단계의 실패를 프롬프트 수준에서 보완할 수는 없다.

더 큰 문제는 다른 곳에 있다. 프롬프트 구성 로직이 비즈니스 코드 전반에 산재되어 있을 때 어떤 일이 벌어지는지 살펴보자. 다음 의사코드는 실제 프로덕션 코드에서 흔히 볼 수 있는 구조다.

```
// 나쁜 설계: 프롬프트, 비즈니스 로직, 데이터 포맷팅이 뒤섞인 구조
function handleCustomerQuery(user_id, user_message):
    // 비즈니스 로직: 사용자 등급 조회
    user_tier = db.getUserTier(user_id)

    // 데이터 포맷팅과 프롬프트 조립이 섞임
    if user_tier == "premium":
        prompt = "당신은 프리미엄 고객 전담 상담원입니다. "
                 + "사용자 메시지: " + user_message
                 + " 반드시 JSON 형태로 응답하세요."
    else:
        prompt = "당신은 일반 고객 상담원입니다. "
                 + "사용자 메시지: " + user_message

    return llm.call(prompt)
```

이 구조에서 "JSON 형태로 응답하세요"라는 지시를 수정해야 한다면? 코드 전체를 뒤져야 한다. 시스템 지시를 A/B 테스트하고 싶다면? 비즈니스 로직을 건드리지 않고는 불가능하다. 프롬프트가 드리프트(drift)—즉, 배포된 프롬프트와 실제 동작하는 프롬프트가 달라지는 현상—가 발생해도 어느 코드 경로에서 어떤 프롬프트가 조립되고 있는지 추적하기 어렵다.[^web_19] 이런 구조는 유지보수가 불가능한 상태로 귀결된다.[^web_22]

**역할:** 프롬프트 구성기(Prompt Builder)는 시스템 지시(system instruction), 컨텍스트(context), 사용자 입력(user turn), 몇 가지 예시(few-shot examples) 등을 조합하여 LLM에 전달할 최종 입력을 구성하는 단계다. 이 단계의 핵심은 조합의 논리를 코드 전반에 분산시키지 않고 한 곳에 집중시키는 것이다.

좋은 설계는 이렇게 보인다.

```
// 좋은 설계: 프롬프트 구성기가 독립된 단위로 분리된 구조
class PromptBuilder:
    def __init__(self, template_version="v2.1"):
        self.system_instruction = load_template("system", template_version)
        self.few_shot_examples = load_examples(template_version)

    def build(self, context, user_input):
        return {
            "system": self.system_instruction,
            "examples": self.few_shot_examples,
            "context": context,
            "user": user_input
        }
```

이 구조에서는 시스템 지시의 버전을 관리할 수 있고, few-shot 예시를 독립적으로 교체할 수 있으며, 전체 프롬프트 조립 방식을 비즈니스 로직에 영향을 주지 않고 변경할 수 있다. 테스트도 명확해진다. 특정 입력과 컨텍스트가 주어졌을 때 프롬프트 구성기가 기대한 형태의 프롬프트를 생성하는지를 독립적으로 검증할 수 있다.

**실패 시 증상:** 프롬프트 구성 로직이 비즈니스 코드와 뒤섞여 있을 때의 증상은 주로 회귀(regression)로 나타난다. 로직을 수정했더니 프롬프트가 바뀌었고, 바뀐 프롬프트가 모델 출력을 변경했으며, 결과적으로 다운스트림 시스템이 오류를 낸다. 이 연쇄를 추적하는 것은 극도로 어렵다. 프롬프트 드리프트—의도하지 않은 프롬프트 변경이 누적되는 현상—가 발생해도 명확한 로그가 없다면 언제, 어떻게 프롬프트가 바뀌었는지 알 수 없다.[^web_19]

**설계 시 고려사항:** 프롬프트 구성기를 설계할 때 가장 먼저 해야 할 일은 시스템 지시와 사용자 입력을 명확하게 분리하는 것이다. 시스템 지시는 서비스의 행동 방침을 정의하고, 사용자 입력은 개별 요청을 담는다. 이 두 가지가 뒤섞이면 하나를 수정할 때 다른 하나가 의도치 않게 영향을 받는다. 프롬프트 템플릿의 버전 관리도 중요하다. 어떤 버전의 프롬프트가 어느 시점에 어떤 결과를 냈는지 추적할 수 있어야 나중에 품질 저하가 발생했을 때 "프롬프트 변경이 원인인가, 다른 요소가 원인인가"를 구분할 수 있다.

5장에서 다룰 RAG(검색 증강 생성, Retrieval-Augmented Generation)는 본질적으로 이 프롬프트 구성기에 외부 정보를 주입하는 구조다. 데이터베이스나 문서 저장소에서 검색된 관련 정보를 컨텍스트로 추가하여 프롬프트를 구성하는 방식이다. 이 확장이 가능하려면 프롬프트 구성기가 독립된 단위로 분리되어 있어야 한다. 프롬프트 구성 로직이 비즈니스 코드 전반에 산재되어 있다면 RAG를 적용하는 일 자체가 대규모 리팩터링을 요구하게 된다.

프롬프트 구성기가 최종 입력을 완성하면, 그것은 이제 LLM 호출기로 넘어간다.

## LLM 호출기: 모델 호출을 격리된 단계로 두어야 하는 이유

많은 초기 구현에서 LLM 호출 코드는 비즈니스 로직 전반에 흩어져 있다. 고객 응대 함수에도 `openai.chat.completions.create()`가 있고, 문서 요약 함수에도 같은 코드가 있으며, 분류 함수에도 그 코드가 있다. 이 구조가 특정 일 서비스 모델을 다른 공급사 모델로 교체해야 하는 상황과 만나면 어떻게 되는가? 비즈니스 코드 전체를 뒤지며 API 호출 코드를 하나씩 수정해야 한다. 비용 측정을 위해 토큰 사용량을 집계하려면? 각 호출 지점마다 별도 코드를 추가해야 한다. 타임아웃을 일괄 변경하려면? 다시 전체를 수정해야 한다. 이것은 유지보수 문제이기도 하지만, 더 근본적으로는 설계의 문제다.[^web_02]

LLM 호출이 실제로 무엇을 하는지 명확히 하자. 호출기는 프롬프트를 받아 모델 API를 통해 전송하고, 원시 응답(raw response)을 반환하는 일을 한다. 이것이 전부다. 이 역할만을 담당하는 독립적 단위가 존재할 때, 나머지 시스템은 "어떤 모델을 어떻게 호출하는가"에 대해 알 필요가 없다. 비즈니스 로직은 단지 프롬프트를 넘기고 응답을 받을 뿐이다.

**역할:** LLM 호출기(LLM Caller)는 프롬프트 구성기의 출력을 받아 모델 API를 실제로 호출하고, 모델의 원시 응답을 반환하는 단계다. 이 단계의 핵심 가치는 모델 공급자(vendor)와의 의존성을 한 곳에 집중시키는 것이다.

**실패 시 증상:** LLM 호출 코드가 비즈니스 로직 전반에 산재되어 있을 때 가장 분명한 증상은 교체 비용이다. GPT-4 대신 Claude를 사용하기로 결정하는 일이 단순한 설정 변경이 아닌 전면 코드 수정을 요구한다면, 그 시스템은 특정 모델 공급자에 암묵적으로 종속되어 있는 것이다. 또한 호출 비용과 지연 시간(latency)을 측정하는 지점이 명확하지 않아, 운영 비용 최적화나 성능 병목 분석이 어렵다.

```
// 의사코드: 호출기를 독립 단위로 분리한 구조
class LLMCaller:
    def __init__(self, provider="openai", model="gpt-4o", timeout=30):
        self.provider = provider
        self.model = model
        self.timeout = timeout

    def call(self, prompt, stream=False):
        start_time = now()
        try:
            response = api_client.send(
                provider=self.provider,
                model=self.model,
                prompt=prompt,
                stream=stream,
                timeout=self.timeout
            )
            log_metrics(latency=now()-start_time, tokens=response.usage)
            return response.raw_text
        except TimeoutError:
            raise LLMTimeoutError(latency=now()-start_time)
        except APIError as e:
            raise LLMAPIError(code=e.code, message=e.message)
```

이 구조에서 모델 공급자를 변경하려면 `provider` 와 `model` 파라미터만 바꾸면 된다. 타임아웃 정책을 조정하려면 `timeout` 파라미터를 수정하면 된다. 지연 시간과 토큰 사용량 측정은 이 단계 안에서 일관되게 이루어진다. 비즈니스 로직은 이 모든 내부 결정으로부터 격리되어 있다.

**설계 시 고려사항:** 모델 공급자 추상화가 첫 번째 고려사항이다. OpenAI, Anthropic, Google, 오픈소스 모델 등 다양한 공급자를 동일한 인터페이스로 호출할 수 있도록 추상화 레이어를 두는 것이 좋다.[^web_01] 두 번째는 스트리밍 응답 처리다. 실시간으로 토큰이 생성되는 스트리밍 방식과 완료 후 전체를 반환하는 방식은 처리 로직이 다르다. 이 차이를 호출기 내부에서 처리하면 하위 단계는 어떤 방식으로 호출했는지 알 필요가 없다. 세 번째는 타임아웃과 기본 오류 처리다. 네트워크 장애, API 속도 제한(rate limit), 서버 오류 등은 LLM 호출에서 일상적으로 발생하는 오류다. 이 오류들을 호출기 단계에서 감지하고 표준화된 예외 형태로 변환하면, 오류 처리기가 오류 유형에 따라 적절한 대응을 취할 수 있다.

LangChain과 LlamaIndex 같은 프레임워크가 LLM 호출을 어떻게 추상화하는지는 서로 다르다.[^web_15][^web_16] 하지만 둘 다 동일한 개념 문제를 해결하고 있다. 모델 호출 코드를 나머지 로직으로부터 분리하라는 것이다. 프레임워크를 사용하든 직접 구현하든, 이 설계 원칙은 동일하게 적용된다. 특정 프레임워크가 없어도 파이프라인 구조로 LLM 시스템을 설계할 수 있다는 점을 다시 강조한다.

"LLM을 호출했다"는 것은 시스템을 완성한 것이 아니라, 파이프라인의 중간 단계를 통과한 것이다. 호출기가 반환한 원시 응답은 이제 출력 처리기로 전달된다.

## 출력 처리기: 모델 응답을 시스템이 사용할 수 있는 형태로

모델이 JSON을 반환하라는 지시를 받았다. 그런데 실제로 반환된 응답은 이렇게 생겼다.

```
물론이죠! 요청하신 JSON입니다:

```json
{
  "status": "approved",
  "score": 87
}
```

도움이 되셨으면 좋겠습니다!
```

JSON이 맞다. 하지만 마크다운 코드블록으로 감싸여 있고, 앞뒤에 자연어 문장이 붙어 있다. `json.parse(response)`는 실패한다. 시스템은 오류를 반환하고, 운영팀은 LLM 호출 단계의 문제로 오해한다. 모델 API 로그를 확인하면 모델은 정상적으로 응답했다. 오류의 원인은 출력 처리 단계의 부재였다.[^web_17]

이런 상황은 특이한 경우가 아니다. LLM은 지시에 엄격하게 따르도록 훈련되어 있지만, 출력 형식에 대한 일관성은 항상 보장되지 않는다. 모델 버전이 업데이트되면 이전에 잘 파싱되던 응답 형식이 미묘하게 달라지기도 한다. 프롬프트가 변경되면 응답 형식에 영향을 줄 수 있다. 출력 처리기가 없다면 이 모든 불일치가 그대로 다운스트림 컴포넌트로 전파된다.

**역할:** 출력 처리기(Output Processor)는 LLM 호출기가 반환한 원시 응답을 파싱·정제·변환하여 다운스트림 컴포넌트가 사용할 수 있는 구조화된 형태로 변환하는 단계다. 모델과 나머지 시스템 사이의 계약(contract)을 실행하는 레이어라고 볼 수 있다.[^web_18] "이 파이프라인은 항상 특정 스키마의 데이터를 반환한다"는 약속을 지키는 것이 이 단계의 책임이다.

**실패 시 증상:** 출력 처리기가 없거나 불충분할 때의 증상은 파싱 오류가 상위 단계 문제로 오인되는 것이다. 더 심각한 경우는 파싱에는 성공했지만 잘못된 형식의 데이터가 다운스트림으로 전파되는 경우다. 예를 들어, 신뢰도 점수를 0에서 1 사이의 float으로 기대하는 컴포넌트에 "87점"이라는 문자열이 전달된다면, 그 오류는 훨씬 나중에, 전혀 예상치 못한 지점에서 발생할 수 있다.

```
// 의사코드: 출력 처리기의 기본 책임
class OutputProcessor:
    def __init__(self, output_schema):
        self.schema = output_schema

    def process(self, raw_response):
        // 1. 마크다운 코드블록 등 불필요한 감싸기 제거
        cleaned = strip_formatting(raw_response)

        // 2. 스키마에 맞게 파싱 시도
        try:
            parsed = parse_to_schema(cleaned, self.schema)
        except ParseError as e:
            // 3. 파싱 실패 시 오류 처리기에 위임
            raise OutputParseError(raw=raw_response, reason=e)

        // 4. 스키마 검증
        validate(parsed, self.schema)

        return parsed
```

**설계 시 고려사항:** 출력 스키마(output schema)를 명시적으로 정의하고 버전으로 관리하는 것이 핵심이다.[^web_17] 스키마는 이 파이프라인이 외부에 약속하는 계약이다. 스키마가 변경되면 의존하는 컴포넌트에 영향을 준다. 파싱 실패를 감지하는 즉시 오류 처리기와 협력하는 구조도 필요하다. 파싱에 실패했을 때 재시도를 할 것인가, 다른 프롬프트로 재질의할 것인가, 폴백 값을 반환할 것인가—이 결정은 출력 처리기 단독으로 내릴 수 없다. 오류 처리기와의 명확한 협력 경계가 설계되어 있어야 한다.[^web_21]

1장에서 하네스의 세 가지 핵심 기능으로 "입력 통제, 출력 평가, 실패 처리"를 꼽았다. 출력 처리기는 "출력 평가"가 구체적인 컴포넌트로 구현되는 지점이다. 모델의 응답이 기대한 형태인지 검증하고, 기대에 맞지 않으면 적절한 처리를 트리거한다. 이 단계가 잘 설계되어 있을수록 파이프라인의 출력에 대한 신뢰도가 높아진다.

정상적으로 처리된 출력은 이제 최종 반환되거나, 대화 이력으로 상태 관리자에 기록된다. 그리고 시스템은 다음 요청을 기다린다. 하지만 여기서 중요한 질문이 등장한다. "다음 요청이 왔을 때, 이전 대화의 맥락을 어떻게 유지하는가?"

## 상태 관리자: LLM은 기억하지 못한다

LLM은 본질적으로 무상태(stateless)다. 각 API 호출은 독립적이다. 모델은 이전 호출에서 무슨 일이 있었는지, 같은 사용자와 어떤 대화를 나눴는지, 어떤 결정을 내렸는지 전혀 알지 못한다.[^web_08] 이것은 결함이 아니라 설계적 특성이다. 모델은 받은 입력에만 반응한다. "기억"처럼 보이는 동작은 모두 이전 대화 내용이 현재 입력에 포함되어 있기 때문에 가능한 것이다.

이 특성이 실제 서비스에서 어떤 문제를 만드는지 생각해보자. 사용자가 세 번째 메시지에서 이렇게 말한다. "아까 말한 그 조건으로 다시 계산해줘." 상태 관리자가 없다면, "아까 말한 그 조건"이 무엇인지를 알 방법이 없다. 파이프라인은 이 요청만 받았을 뿐, 이전 두 개의 메시지를 알지 못한다. 사용자 입장에서는 대화가 연속적으로 느껴지지만, 시스템 입장에서는 매 메시지가 새로운 요청이다. 이 간극을 메우는 것이 상태 관리자(State Manager)의 역할이다.

**역할:** 상태 관리자는 대화 이력, 세션 정보, 사용자 맥락 등 파이프라인 실행 간에 유지되어야 하는 상태를 관리하는 단계다. "이전 대화의 내용"을 현재 프롬프트 구성기에 제공하는 것이 가장 기본적인 기능이다.

상태의 유형을 구분하는 것이 중요하다. 크게 세 가지로 나눌 수 있다. 첫째는 단기 메모리(short-term memory)로, 현재 진행 중인 대화의 이력이다. 가장 최근의 N개 메시지나 현재 세션 내의 전체 대화 내용이 여기에 해당한다. 둘째는 장기 메모리(long-term memory)로, 사용자의 선호, 이전 세션의 요약, 반복적으로 언급되는 정보 등이다. 셋째는 운영 상태(operational state)로, 현재 진행 중인 태스크의 단계, 부분적으로 완성된 작업의 중간 결과 등이다.[^web_03] 에이전트 시스템에서 이 세 번째 유형이 특히 중요해지는데, 이는 6장에서 자세히 다룬다.

**실패 시 증상:** 상태 관리가 없거나 불충분할 때의 증상은 주로 대화 품질로 나타난다. 멀티턴 대화에서 맥락이 끊기고, 사용자는 같은 정보를 반복해서 제공해야 한다. 더 미묘한 문제는 컨텍스트 윈도우의 비효율적 사용이다. 대화 이력 전체를 무조건 프롬프트에 포함시키면, 대화가 길어질수록 컨텍스트 윈도우를 빠르게 소진한다. 상태 관리자는 어떤 이력을 얼마나 포함시킬지를 결정하는 컨텍스트 윈도우 예산(context window budget) 관리도 담당해야 한다.[^web_08]

**설계 시 고려사항:** 상태를 파이프라인 외부(애플리케이션 레이어)에 위임하는 방식과 명시적인 컴포넌트로 분리하는 방식을 비교해보자. 상태를 애플리케이션 전역 변수나 세션 스토리지에 임시로 저장하는 방식은 단기적으로는 동작하지만, 동시 사용자가 많아지거나 서버 인스턴스가 여럿이 되면 상태 일관성을 보장하기 어렵다. Google의 Agent Development Kit(ADK)는 상태 관리와 컨텍스트 구성을 분리하는 방식을 채택했는데, 이는 "어떤 상태가 있는가"와 "그 상태를 어떻게 프롬프트에 포함시킬 것인가"가 서로 독립된 설계 결정임을 보여준다.[^web_20]

6장에서 다룰 에이전트는 이 파이프라인을 여러 번, 동적으로 실행하는 구조다. 에이전트가 복잡한 태스크를 처리하기 위해 파이프라인을 반복 실행할 때, 각 실행의 결과와 상태를 연결하는 것이 상태 관리자의 역할이다. 단순 대화 시스템에서도 중요한 상태 관리자가 에이전트 시스템에서 더욱 핵심적인 구성 요소가 되는 이유가 바로 여기에 있다.

상태 관리자와 함께 파이프라인의 "운영 지지대" 역할을 하는 또 하나의 구성 요소가 있다. 오류 처리기다.

## 오류 처리기: 실패를 예외가 아닌 설계 대상으로

LLM 시스템에서 오류는 예외적 상황이 아니다. 네트워크 지연으로 API 호출이 타임아웃되고, 속도 제한(rate limit)에 걸려 호출이 거부되며, 모델이 기대한 형식으로 응답하지 않는 일은 충분한 규모의 운영 환경에서 일상적으로 발생한다. 이것을 비정상적인 상황으로 취급하고 예외 처리 코드를 그때그때 추가하는 방식으로는 시스템을 안정적으로 운영하기 어렵다.[^web_06]

오류 처리를 파이프라인의 독립적 구성 요소로 두어야 하는 핵심 이유는 각 단계의 실패 유형이 다르고, 적절한 대응 방식도 다르기 때문이다. 다음 비교를 보자.

| 단계 | 실패 유형 예시 | 적절한 대응 |
|------|--------------|-------------|
| 입력 처리기 | 입력 길이 초과, 허용되지 않은 문자 | 즉시 오류 반환, 재시도 불필요 |
| LLM 호출기 | 네트워크 타임아웃 | 지수 백오프(exponential backoff)로 재시도 |
| LLM 호출기 | API 속도 제한(429 오류) | 대기 후 재시도, 또는 다른 모델로 폴백 |
| 출력 처리기 | 파싱 실패 | 재질의(re-query) 시도, 또는 단순화된 프롬프트로 폴백 |
| 출력 처리기 | 스키마 검증 실패 | 로깅 후 기본값 반환, 또는 오류 전파 |

이 표에서 보듯이, 동일한 "오류"라도 발생 단계와 유형에 따라 최적의 대응이 완전히 다르다. 모든 오류에 단순 재시도를 적용하면, 파싱 실패에도 재시도를 하게 된다. 프롬프트가 잘못되어 파싱에 실패하는 경우라면, 같은 프롬프트로 재시도해도 같은 결과가 나올 가능성이 높다. 이런 상황에서 재시도는 오류를 해결하는 것이 아니라 비용을 낭비하고 지연을 증가시키는 것에 불과하다.[^web_07]

**역할:** 오류 처리기(Error Handler)는 파이프라인 각 단계에서 발생하는 실패를 탐지하고, 오류 유형에 따라 재시도(retry)·폴백(fallback)·알림(alert) 등 적절한 대응을 실행하는 단계다. 각 단계가 독립적으로 분리되어 있을 때만 이 단계별 맞춤 대응이 가능하다.

서킷 브레이커(circuit breaker) 패턴을 생각해보자. 전기 회로의 차단기는 과부하나 단락이 감지되면 회로 전체를 차단해 더 큰 손상을 막는다. LLM 파이프라인에서도 마찬가지다. LLM 호출기가 연속으로 N번 실패한다면, 이는 해당 API 자체에 문제가 있을 가능성이 높다. 이 상황에서 계속 호출을 시도하는 것은 의미 없는 비용 소모다. 서킷 브레이커는 연속 실패가 임계값을 넘으면 일정 시간 동안 호출을 중단하고, 그 기간이 지나면 다시 시도한다. 이 패턴을 구현하려면 오류 처리기가 파이프라인의 독립적인 구성 요소로서 각 단계의 상태를 모니터링할 수 있어야 한다.[^web_06]

**실패 시 증상:** 오류 처리기가 없거나 미흡할 때의 가장 직접적인 증상은 한 단계의 오류가 파이프라인 전체를 중단시키는 것이다. 사용자는 예상치 못한 오류 메시지를 받고, 운영팀은 사후에야 문제를 인지한다. 더 나쁜 경우는 오류가 조용히 넘어가는 경우다. 파싱에 실패했지만 빈 값이 그대로 다운스트림으로 전달되어 나중에 전혀 다른 지점에서 이상한 결과를 낸다.

**설계 시 고려사항:** 재시도 조건과 횟수 제한이 우선이다. 모든 오류에 재시도를 적용하지 않는다. 네트워크 오류나 일시적 API 오류처럼 재시도가 의미 있는 경우에만 적용하고, 횟수를 제한한다. 폴백 정책(fallback policy)도 명확하게 정의해야 한다. "재시도가 N번 모두 실패하면 무엇을 반환하는가?"에 대한 답이 미리 준비되어 있어야 한다. 단순화된 응답을 반환할 수도 있고, 오류를 상위 레이어로 전파할 수도 있으며, 캐시된 이전 응답을 반환할 수도 있다. 어떤 선택이 적합한지는 서비스의 성격에 따라 다르지만, 선택이 명시적으로 이루어져 있어야 한다는 것이 중요하다.

1장에서 하네스의 핵심 기능으로 꼽은 "실패 처리"가 구체적인 컴포넌트로 구현되는 지점이 오류 처리기다. 실패 처리가 각 비즈니스 로직 함수에 분산되어 있지 않고 전담 컴포넌트에 집중되어 있을 때, 실패 정책의 일관성을 유지하고 변경하기 쉬워진다.

여섯 개의 구성 요소를 각각 살펴봤다. 이제 한 걸음 물러서서 더 근본적인 질문을 다뤄야 할 시간이다. 왜 이것들을 분리해야 하는가?

## 구성 요소 분리의 설계적 이유: 교체 가능성과 평가 가능성

앞에서 여섯 개의 구성 요소를 설명하면서 각각의 역할과 실패 증상을 다뤘다. 그런데 반문할 수 있다. "이 모든 기능을 하나의 클래스 또는 하나의 함수에 구현하면 안 되는가?" 코드 줄 수는 비슷할 것이고, 처음에는 더 단순해 보일 수도 있다. 이 반문에 답하는 것이 이 섹션의 목적이다.

구성 요소를 분리하는 것은 코드 정리 취향의 문제가 아니다. 세 가지 설계 목표를 달성하기 위한 의도적 선택이다. 첫 번째는 교체 가능성(replaceability), 두 번째는 독립 평가 가능성(independent evaluability), 세 번째는 실패 격리(failure isolation)다.

**교체 가능성:** 소프트웨어 설계에서 "단일 책임 원칙(Single Responsibility Principle)"은 하나의 모듈이 하나의 이유로만 변경되어야 한다고 말한다. LLM 파이프라인에서 이 원칙이 의미하는 바는 명확하다. 모델을 교체하고 싶을 때는 LLM 호출기만 수정하면 된다. 프롬프트 전략을 변경하고 싶을 때는 프롬프트 구성기만 수정하면 된다. 출력 스키마를 바꾸고 싶을 때는 출력 처리기만 수정하면 된다. 구성 요소가 분리되어 있지 않다면, 이 변경들 중 하나가 다른 변경과 예상치 못하게 결합되어 회귀를 유발한다.[^web_09]

관심사의 분리(Separation of Concerns, SoC) 원칙을 LLM 시스템에 적용하면 이렇게 된다.[^web_09] 입력의 형태, 프롬프트의 내용, 모델의 선택, 응답의 형식, 상태의 관리, 오류의 처리—이 여섯 가지 관심사는 서로 독립적으로 변경될 수 있다. 하나의 모듈에 이 모든 관심사가 뒤섞여 있다면, 사실상 어떤 변경도 격리된 범위 안에서 이루어지지 않는다.

**독립 평가 가능성:** 이 책의 핵심 주제 중 하나는 평가 주도 개발(EDD, Evaluation-Driven Development)이다. EDD가 성립하려면 각 구성 요소를 독립적으로 평가할 수 있어야 한다. 프롬프트 구성기가 올바른 형태의 프롬프트를 조립하는지, 출력 처리기가 파싱을 정확히 수행하는지, 상태 관리자가 올바른 이력을 제공하는지—이것들을 독립적으로 검증할 수 있어야 한다.[^web_02] 구성 요소들이 하나로 뒤섞여 있다면, 전체 파이프라인의 최종 출력만을 평가할 수 있다. 최종 출력이 나쁘다는 것은 알지만 어느 단계가 원인인지는 알 수 없는 상태가 된다. 이것은 진단이 불가능한 상태와 다르지 않다.

**실패 격리:** 세 번째 이유는 실패의 영향 범위를 제한하는 것이다. 출력 처리기에서 파싱 실패가 발생했을 때, 그 실패는 출력 처리기 수준에서 처리되어야 한다. LLM 호출기나 프롬프트 구성기에는 영향을 주어서는 안 된다. 구성 요소가 명확하게 분리되어 있을 때만 이 격리가 가능하다. 실패 격리는 단순히 코드 안정성의 문제가 아니라 운영 가시성(observability)의 문제이기도 하다. 각 단계의 성공률, 지연 시간, 오류 유형을 독립적으로 모니터링할 수 있을 때만 시스템의 어느 부분이 병목인지, 어느 부분이 불안정한지를 파악할 수 있다.[^web_07]

분리하지 않았을 때의 결과를 다시 한번 구체적으로 상상해보자. 프롬프트 구성 로직, LLM 호출, 응답 파싱, 대화 이력 관리가 모두 하나의 `handle_request()` 함수에 들어 있다고 하자. 이 함수가 잘못된 결과를 반환한다. 원인은 무엇인가? 컨텍스트 윈도우를 초과했을 수도 있고, 프롬프트 형식이 바뀌었을 수도 있으며, 모델 API가 일시적으로 불안정했을 수도 있고, 파싱 로직이 새로운 응답 형식에 대응하지 못했을 수도 있다. 이 모든 가능성을 하나의 함수 안에서 구분하는 것은 극도로 어렵다. 반면 여섯 개의 구성 요소가 분리되어 있다면, 각 단계의 입출력 로그를 확인하는 것만으로도 어느 단계에서 문제가 발생했는지 빠르게 특정할 수 있다.

구성 요소가 많을수록 복잡하고 나쁘다는 직관적 반응이 있을 수 있다. 하지만 진정한 복잡성은 분리되지 않은 결합(coupling)에서 온다. 분리는 각 단위의 복잡성을 낮추고, 변경의 영향 범위를 제한하며, 테스트와 평가의 단위를 만든다. 이것은 LLM 시스템에 특유한 원칙이 아니라 소프트웨어 공학의 기본 원칙이 LLM 시스템에 그대로 적용되는 것이다.[^web_05]

## 정보 흐름: 데이터가 파이프라인을 통과하며 변형되는 방식

여섯 개의 구성 요소를 정의하고 분리의 이유를 살펴봤다. 이제 한 가지 더 생각해야 할 것이 있다. 각 구성 요소가 독립적으로 잘 설계되어 있더라도, 구성 요소 사이에서 데이터가 어떻게 전달되고 변형되는지가 명확하지 않으면 전체 파이프라인은 여전히 불투명하다. 이것이 정보 흐름(information flow) 개념이다.

각 단계의 입력과 출력을 명시적으로 정의하는 것이 왜 중요한가? 파이프라인의 각 단계는 데이터를 받아서 변환하고 다음 단계로 넘기는 변환기(transformer)다. 입력과 출력의 형태를 명확하게 정의하면, 두 가지 이점이 생긴다. 첫째, 각 단계를 독립적으로 테스트할 수 있다. "이 입력을 주면 저 출력이 나와야 한다"는 단언(assertion)이 가능해진다. 둘째, 두 단계 사이에서 데이터가 어떻게 변환되었는지를 추적할 수 있다. 최종 출력이 기대와 다를 때, 어느 변환 단계에서 데이터가 의도치 않게 바뀌었는지 추적할 수 있다.[^web_13]

파이프라인의 각 단계별 입출력을 정리하면 다음과 같다.

```
사용자 입력(raw text)
        ↓
[입력 처리기]
입력: 원시 사용자 텍스트
출력: 정제·검증된 텍스트 + 메타데이터
        ↓
[프롬프트 구성기]
입력: 정제된 텍스트 + 시스템 지시 + 컨텍스트(상태 관리자 제공)
출력: 완성된 프롬프트 구조 (system/user/examples)
        ↓
[LLM 호출기]
입력: 완성된 프롬프트 구조
출력: 모델의 원시 응답 (raw text)
        ↓
[출력 처리기]
입력: 원시 응답
출력: 파싱·검증된 구조화 데이터 (스키마 준수)
        ↓
최종 결과 반환 / 상태 관리자에 기록
```

각 화살표는 데이터의 변환을 나타낸다. 이 변환의 성격을 파악하면 파이프라인 전체의 동작을 이해하기 쉬워진다. 입력 처리기는 형태를 정규화한다. 프롬프트 구성기는 여러 소스의 정보를 조합한다. LLM 호출기는 구조화된 프롬프트를 자연어 응답으로 변환한다. 출력 처리기는 자연어 응답을 구조화된 데이터로 변환한다.

이 중 LLM 호출 단계의 변환이 다른 단계와 본질적으로 다르다는 점을 주목하자. 다른 단계의 변환은 결정론적(deterministic)이다. 같은 입력이 들어오면 같은 출력이 나온다. 그러나 LLM 호출 단계는 비결정적(nondeterministic)이다. 같은 프롬프트를 두 번 호출해도 다른 응답이 나올 수 있다. 이 비결정성(nondeterminism)이 LLM 파이프라인을 일반적인 소프트웨어 파이프라인과 다르게 만드는 핵심 특성이다. 나머지 다섯 개의 구성 요소는 이 비결정성을 관리하고 제한하기 위해 존재한다고도 볼 수 있다. 입력을 통제하고, 프롬프트를 구조화하고, 출력을 검증하고, 상태를 관리하고, 실패를 처리하는 것—이 모든 활동이 비결정적인 핵심 단계를 예측 가능한 시스템 안에 가두기 위한 구조다.

정보 흐름을 명시적으로 설계하는 것은 5장 RAG를 이해하는 데도 직접적으로 연결된다. RAG는 프롬프트 구성기의 입력에 새로운 소스—외부 데이터베이스에서 검색된 정보—를 추가하는 방식으로 이 파이프라인을 확장한다. 이 확장이 가능하려면 프롬프트 구성기의 입력 인터페이스가 명확하게 정의되어 있어야 한다. 정보 흐름을 명시적으로 관리하는 것이 이러한 확장을 용이하게 만드는 이유다.[^web_17][^web_18]

## 하네스와 파이프라인: 두 개념의 관계 재확인

이 장 전체를 통해 파이프라인의 여섯 개 구성 요소를 분해하고, 각 구성 요소의 역할과 설계 논리를 살펴봤다. 마지막으로, 1장에서 도입한 평가 하네스(harness)가 이 파이프라인과 어떤 관계에 있는지 명확히 정리하자.

하네스는 "반복 가능하게 실행과 평가를 수행하는 외부 구조"로 정의된다. 1장에서 그림 1-2가 "AI 모델 단독 실행"과 "하네스가 감싸는 실행"을 비교했다면, 이 장의 그림 4-1은 그 "하네스가 감싸는 구조" 안에 무엇이 있는지를 보여준다. 하네스는 파이프라인의 특정 단계가 아니다. 파이프라인 전체를 외부에서 감싸는 구조다.

이 관계를 구체적으로 설명하면 이렇다. 파이프라인은 "무엇을 어떻게 처리하는가"를 정의한다. 입력이 들어오면 여섯 개의 단계를 순서대로 통과하여 출력이 나온다. 하네스는 "그 처리가 올바르게 이루어지고 있는가"를 외부에서 확인하고 보장하는 구조다. 하네스는 파이프라인을 다양한 입력으로 반복 실행하고, 각 실행의 결과를 평가하며, 실패 패턴을 기록하고, 회귀를 감지한다.

1장에서 하네스의 세 가지 핵심 기능으로 "입력 통제, 출력 평가, 실패 처리"를 제시했다. 이제 이 세 기능이 파이프라인의 어떤 구성 요소와 대응하는지 분명해졌다. 입력 통제는 입력 처리기에서 구현된다. 출력 평가는 출력 처리기에서 구현된다. 실패 처리는 오류 처리기에서 구현된다. 하네스는 이 구성 요소들이 올바르게 작동하는지를 파이프라인 외부에서 반복적으로 검증하는 프레임이다.

하네스가 효과적으로 작동하려면 감싸는 대상인 파이프라인이 충분히 분해되어 있어야 한다. 파이프라인이 하나의 덩어리로 뭉쳐 있다면, 하네스는 최종 출력의 옳고 그름만 판단할 수 있다. 파이프라인이 여섯 개의 독립된 단계로 분리되어 있다면, 하네스는 각 단계의 동작을 독립적으로 평가할 수 있다. "프롬프트 구성기가 이 입력에 대해 올바른 프롬프트를 생성하는가?", "출력 처리기가 이 응답을 올바르게 파싱하는가?"—이런 단계별 평가가 가능해질 때 시스템의 품질을 세밀하게 관리할 수 있다.

## 장 마무리: 파이프라인이라는 언어로 시스템을 설계한다

이 장에서는 LLM을 호출하는 것과 LLM 시스템을 설계하는 것의 차이를 파이프라인이라는 관점으로 설명했다. 핵심 내용을 정리하면 다음과 같다.

**여섯 개의 구성 요소와 각각의 역할:**

| 구성 요소 | 역할 | ch01 대응 개념 |
|----------|------|--------------|
| 입력 처리기 | 사용자 입력 정제·검증 | 입력 통제 |
| 프롬프트 구성기 | 컨텍스트·지시·입력 조합 | — |
| LLM 호출기 | 모델 API 호출, 원시 응답 반환 | — |
| 출력 처리기 | 응답 파싱·검증·변환 | 출력 평가 |
| 상태 관리자 | 대화 이력·세션 관리 | — |
| 오류 처리기 | 실패 탐지·재시도·폴백 | 실패 처리 |

**구성 요소 분리의 세 가지 설계 목표:** 교체 가능성, 독립 평가 가능성, 실패 격리. 이 세 목표는 LLM 시스템에만 적용되는 특수한 원칙이 아니라 소프트웨어 설계의 기본 원칙—관심사의 분리, 단일 책임 원칙—이 LLM 시스템에 적용된 것이다.

**정보 흐름의 명시성:** 각 단계의 입력과 출력을 명확하게 정의하면 테스트와 디버깅의 단위가 생기고, 파이프라인의 어느 변환 단계에서 문제가 발생했는지 추적할 수 있다. LLM 호출 단계의 비결정성(nondeterminism)은 나머지 다섯 단계가 관리하고 제한하는 설계적 과제다.

**하네스와 파이프라인의 관계:** 파이프라인은 처리 구조이고, 하네스는 그 처리를 외부에서 반복 실행하고 평가하는 구조다. 두 개념은 상호 보완적이며, 하네스가 효과적으로 작동하려면 감싸는 파이프라인이 충분히 분해되어 있어야 한다.

다음 장에서는 이 파이프라인의 구체적인 확장을 다룬다. 5장의 주제는 RAG(검색 증강 생성)다. RAG는 새로운 아키텍처 개념이 아니라, 이 장에서 정의한 파이프라인—특히 프롬프트 구성기—에 외부 정보 소스를 연결하는 방식으로 이해할 수 있다. 프롬프트 구성기가 독립된 단위로 설계되어 있을 때, 그 단계에 외부 검색 결과를 주입하는 것이 얼마나 자연스럽게 가능한지를 5장에서 확인하게 될 것이다. 이 장에서 파이프라인의 구조를 내면화했다면, RAG는 그 구조의 논리적 확장으로 느껴질 것이다.
