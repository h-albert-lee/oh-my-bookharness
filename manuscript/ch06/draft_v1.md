# 6장. AI 에이전트는 단순 프로그램이 아니라 시스템이다

## 6.1 에이전트 내부를 컴포넌트로 읽기

"여행 일정을 짜 줘"라는 요청 하나를 에이전트에 입력했다고 하자. 잠시 후 도쿄 3박 4일 일정이 깔끔하게 출력된다. 출력이 정확하고 빠르다면 내부에서 무슨 일이 일어났는지 굳이 살펴볼 이유가 없어 보인다. 그러나 일정이 어긋나거나, 예산 조건을 무시하거나, 이전에 말했던 선호도가 반영되지 않는다면 어떻게 해야 할까? 이때 개발자가 가장 먼저 마주치는 문제는 '어디를 고쳐야 하는가'다.

에이전트를 "모델 호출 하나"로 인식하고 있다면 이 질문에 답하기 어렵다. 모델 프롬프트를 수정하고, 다시 실행하고, 결과가 다를 때 이 개선이 무엇 때문인지 설명하지 못한다. 이 장에서는 그 평면적 인식에서 벗어나기 위한 분해의 언어를 먼저 정리한다. 에이전트를 Planning(계획), Memory(메모리), Tool Use(도구 사용) 세 컴포넌트와 그것들을 조율하는 모델 레이어로 나누어 읽는 것이 출발점이다.[^web_01] ch01부터 ch05까지 각 장에서 개별적으로 다루어진 모델, 컨텍스트, 도구, 메모리, 평가는 이 분해 구조 위에 다시 배치될 때 서로의 관계가 비로소 선명해진다.

### 6.1.1 컴포넌트 분해의 기준: 책임 범위와 교체 가능성

컴포넌트를 나누는 기준은 복잡도가 아니다. 무엇에 책임지는가, 그리고 나머지 부분을 건드리지 않고 교체할 수 있는가가 기준이다. 이 두 기준을 함께 적용했을 때 에이전트 내부에서 자연스럽게 분리되는 네 개의 레이어가 있다.[^web_01]

**그림 6-1. 에이전트 컴포넌트 구조 — Planning·Memory·Tool Use와 모델 레이어의 위치 관계**

```
┌───────────────────────────────────────────────┐
│                    에이전트                     │
│                                               │
│   ┌─────────────┐     ┌─────────────────────┐ │
│   │  Planning   │────▶│     Model Layer     │ │
│   │  (계획)     │     │  (추론 · 생성 · 조율) │ │
│   └─────────────┘     └──────────┬──────────┘ │
│                                  │            │
│   ┌─────────────┐     ┌──────────▼──────────┐ │
│   │   Memory    │◀───▶│      Tool Use       │ │
│   │  (기억·맥락) │     │   (외부 연산·검색)   │ │
│   └─────────────┘     └─────────────────────┘ │
└───────────────────────────────────────────────┘
```

- **Planning 레이어**: 목표를 하위 태스크로 분해하고 실행 순서를 결정한다. 어떤 도구를 언제 호출할지, 결과가 불충분할 때 다시 시도할지도 여기서 결정된다.
- **Memory 레이어**: 현재 세션 내 컨텍스트(단기 기억), 과거 세션이나 외부 저장소의 정보(장기 기억), 그리고 검색 증강(RAG)으로 끌어오는 외부 지식을 포함한다. ch04에서 다룬 메모리 전략이 이 레이어에 해당한다.
- **Tool Use 레이어**: 검색 API, 코드 실행기, 데이터베이스 조회처럼 모델 외부에서 이루어지는 연산 전체를 담당한다. ch03의 도구 개념이 여기에 위치한다.
- **Model 레이어**: 위 세 컴포넌트의 입력을 받아 추론하고, 다음 행동을 결정하거나 최종 응답을 생성한다. ch01과 ch02에서 다룬 모델과 컨텍스트 어셈블리가 이 레이어에서 작동한다.

::: note ch01–ch05 개념과 컴포넌트의 대응 관계
이전 장에서 다룬 개념들은 이 분해 구조 안에서 다음과 같이 배치된다.

| 이전 장 개념 | 해당 컴포넌트 레이어 |
|---|---|
| 모델 추론, 토큰 생성 (ch01) | Model Layer |
| 컨텍스트 윈도우, 시스템 프롬프트 (ch02) | Model Layer / Memory |
| 도구 정의, API 연결 (ch03) | Tool Use |
| 단기·장기 메모리, RAG (ch04) | Memory |
| 평가 설계, 하네스 (ch05) | 시스템 외부 (harness) |

평가 하네스(harness)는 어느 컴포넌트에도 속하지 않는다. 시스템 외부에서 시스템 전체를 반복 실행하고 관찰하는 구조이기 때문에, 특정 레이어 안에 두면 하네스의 독립성이 훼손된다. 이 배치 원칙은 6.4절에서 더 자세히 다룬다.
:::

"여행 일정을 짜 줘" 요청이 들어왔을 때 내부 흐름을 이 레이어 단위로 추적하면 다음과 같다.

1. **Model → Planning**: 모델이 요청을 해석하고 "항공편 검색 → 숙소 검색 → 일정 조합" 순서를 계획한다.
2. **Planning → Tool Use**: 계획에 따라 항공편 검색 API를 호출한다.
3. **Tool Use → Memory**: 검색 결과를 세션 컨텍스트에 저장한다.
4. **Memory → Model**: 사용자의 이전 선호도(창가 좌석 선호, 4성급 이상 숙소)를 컨텍스트에 포함해 모델에 전달한다.
5. **Model**: 모든 입력을 종합해 최종 일정을 생성한다.

이 흐름에서 응답이 어긋났다면 어디서 발생한 문제인지 컴포넌트 단위로 좁힐 수 있다. 항공편 정보가 틀렸다면 Tool Use 레이어의 API 호출이나 결과 파싱을 확인해야 하고, 선호도가 반영되지 않았다면 Memory 레이어의 컨텍스트 어셈블리를 점검해야 한다. "프롬프트를 고쳐야 한다"는 단일한 처방 대신 책임 범위가 분리된다.

단순해 보이는 에이전트도 마찬가지다. 단일 Q&A 에이전트를 예로 들어 보자. 사용자의 질문을 받아 한 번에 답을 돌려주는 구조다. 겉으로는 모델 호출 하나처럼 보이지만, 내부에는 이미 컴포넌트가 존재한다. 시스템 프롬프트를 구성하는 작업(컨텍스트 어셈블리), 대화 이력을 관리하는 작업(Memory), 모델 출력을 파싱해 응답 형식으로 변환하는 작업(Tool Use 또는 파싱 레이어)이 모두 포함되어 있다. Anthropic의 에이전트 가이드는 이를 명시적으로 지적한다. 워크플로우와 에이전트를 아키텍처적으로 구분할 때, 핵심 기준은 복잡도가 아니라 모델이 자체적으로 실행 흐름을 제어하는지 여부다.[^web_02] 실행 흐름을 제어하는 순간 이미 시스템이다.

컴포넌트 분해가 교체 가능성 측면에서 실무적으로 의미 있는 이유도 여기에 있다. Planning 로직을 규칙 기반에서 LLM 기반으로 바꾸거나, Memory 백엔드를 인메모리에서 벡터 DB로 전환할 때, 분해 구조가 없으면 변경 범위를 국소화할 방법이 없다. 반대로 레이어가 명확히 구분된 시스템에서는 하나의 컴포넌트를 교체해도 인접 레이어의 계약(contract)이 유지되는 한 전체 시스템이 영향을 받지 않는다.

### 6.1.2 Thought→Action→Observation 루프: 내부 흐름의 정형화

컴포넌트를 나누는 것만으로는 충분하지 않다. 실행 시점에 이 컴포넌트들이 어떤 순서로 호출되는지, 그 흐름이 반복되는 구조인지 일회성인지도 파악해야 한다. 이를 위한 정형화된 표현이 ReAct 사이클의 Thought→Action→Observation 루프다.[^web_03]

**그림 6-2. Thought→Action→Observation 반복 사이클과 컴포넌트 호출 순서**

```
         ┌──────────────────────────────────────────┐
         │                                          │
         ▼                                          │
   ┌───────────┐    컴포넌트: Model + Planning       │
   │  Thought  │  "다음에 무엇을 해야 하는가?"         │
   └─────┬─────┘                                    │
         │                                          │
         ▼                                          │
   ┌───────────┐    컴포넌트: Tool Use               │
   │  Action   │  "도구를 호출하거나 응답을 생성한다"    │
   └─────┬─────┘                                    │
         │                                          │
         ▼                                          │
   ┌─────────────┐  컴포넌트: Memory                 │
   │ Observation │  "결과를 받아 컨텍스트에 기록한다"   │
   └─────┬───────┘                                  │
         │                                          │
         ├── 목표 달성? ──── Yes ──▶ [최종 출력]      │
         │                                          │
         └── No ──────────────────────────────────▶┘
```

- **Thought**: 모델이 현재 상태를 보고 다음 행동을 추론한다. Planning 레이어와 Model 레이어가 함께 작동하는 단계다.
- **Action**: 추론 결과에 따라 도구를 호출하거나, 더 이상 외부 정보가 필요 없다면 최종 응답을 생성한다. Tool Use 레이어가 활성화된다.
- **Observation**: 도구 호출의 결과를 받아 메모리에 기록하고 다음 Thought의 입력으로 만든다. Memory 레이어가 관여하는 단계다.

이 루프는 복잡한 멀티스텝 에이전트에서만 나타나는 구조가 아니다. 단일 Q&A 에이전트에서도 루프는 존재한다. 다만 단 한 번 순환하고 종료될 뿐이다. Thought 단계에서 "이 질문은 외부 정보가 필요 없다"고 판단하면 Action은 바로 최종 응답 생성이 되고, Observation은 사용자에게 전달되는 텍스트다. 구조는 동일하고, 반복 횟수만 다르다.

::: box 워크플로우와 에이전트의 경계 — Anthropic 가이드의 아키텍처 구분
Anthropic의 에이전트 구축 가이드는 워크플로우(workflow)와 에이전트(agent)를 다음과 같이 구분한다.[^web_02]

- **워크플로우**: 실행 흐름이 코드로 미리 정의된다. 모델은 각 단계의 처리를 담당하지만, 어떤 단계를 거칠지는 개발자가 결정한다.
- **에이전트**: 모델이 스스로 실행 흐름을 제어한다. 어떤 도구를 언제 사용할지, 언제 멈출지를 모델이 동적으로 결정한다.

이 구분에서 핵심은 제어 흐름의 소재다. 에이전트는 모델이 자신의 Thought 단계에서 다음 Action을 결정하기 때문에, 실행 전에 전체 경로를 예측하기 어렵다. 가이드는 이 예측 불가능성이 에이전트를 강력하게 만드는 동시에 취약하게 만든다고 지적한다. 복잡성은 필요할 때만 도입하라는 원칙도 같은 이유에서 나온다.

이 구분은 컴포넌트 분해와 직결된다. 워크플로우에서는 Planning이 코드로 구현되어 있지만, 에이전트에서는 Planning이 모델의 추론 능력에 위임된다. 어떤 방식이든 Planning 레이어는 존재한다. 단지 그것이 하드코딩된 규칙인지 모델이 동적으로 생성하는 계획인지의 차이다.
:::

이 루프 구조를 명확히 이해하면 한 가지 중요한 사실이 드러난다. 컴포넌트들은 순서 있게 협력하며 서로의 출력에 의존한다는 점이다. Thought의 품질은 Observation이 얼마나 정확한 정보를 제공했는지에 달려 있고, Action의 성패는 Thought가 올바른 판단을 내렸는지에 달려 있다. 이 의존 관계는 단순히 흐름도를 그리는 것을 넘어, 한 컴포넌트의 실패가 다음 컴포넌트로 어떻게 전파되는지를 예측해야 한다는 설계적 함의를 갖는다.

바로 이 지점에서 다음으로 살펴봐야 할 문제가 생긴다. 컴포넌트가 무엇인지는 이제 알았다. 그러나 컴포넌트들이 서로 어떤 약속으로 연결되어 있는지, 그리고 그 약속이 어긋났을 때 어떤 일이 일어나는지는 아직 정리되지 않았다. 컴포넌트를 나누는 것만으로는 부족하다. 컴포넌트 사이의 경계, 즉 계약(contract)이 어떻게 정의되고, 하나의 컴포넌트에서 발생한 실패가 인접 컴포넌트를 거쳐 시스템 전체로 번지는 경로를 추적할 수 있어야 한다. 6.2절에서는 그 계약과 실패 전파 메커니즘을 살펴본다.

## 6.2 컴포넌트 간 계약과 실패의 전파

6.1절에서는 AI 에이전트를 구성하는 컴포넌트들을 분해하고, 각 컴포넌트가 Thought→Action→Observation 루프 안에서 어떤 역할을 맡는지 살펴보았다. 그런데 컴포넌트를 열거하는 것만으로는 충분하지 않다. 컴포넌트들은 고립된 섬이 아니라 서로 데이터를 주고받으며 협력한다. 그 연결선에는 암묵적이든 명시적이든 어떤 약속이 존재한다. 이 절에서는 그 약속, 즉 컴포넌트 간 계약(contract)이 무엇을 의미하는지 정리하고, 그 계약이 어긋났을 때 실패가 시스템 전체로 어떻게 번지는지를 추적한다.

### 6.2.1 컴포넌트 간 인터페이스 계약: 무엇이 명시되어야 하는가

컴포넌트 경계에서 데이터가 오갈 때, 주고받는 쪽 모두가 동의하는 약속을 **인터페이스 계약(interface contract)**이라 부른다. 계약은 세 가지 차원에서 정의된다. 첫째는 **형식(format)**: 어떤 구조의 데이터를 넘길 것인가. 둘째는 **타이밍(timing)**: 언제 데이터가 준비되고, 언제 소비될 것인가. 셋째는 **예외 처리(exception handling)**: 데이터가 없거나 잘못된 경우 어떻게 행동할 것인가.

Lin & Zhang은 이 문제를 'models propose, architectures dispose'라는 명제로 압축한다.[^web_05] 모델이 아무리 좋은 추론 결과를 내놓아도, 그 결과를 받아서 다음 단계로 전달하는 아키텍처 — 즉 인터페이스 계약 — 가 명시되어 있지 않으면 그 품질은 시스템 수준에서 보존되지 않는다. 모델의 출력 품질과 시스템의 신뢰성은 별개의 문제라는 것이다.

::: note 계약(contract)의 의미 — 인터페이스 명세와 런타임 기대치
소프트웨어 공학에서 '계약'은 함수나 모듈이 호출자에게 보장하는 동작 명세를 가리킨다. 에이전트 시스템에서의 계약은 이보다 넓다. 컴포넌트 간 계약은 호출 형식(입력 스키마)뿐 아니라 런타임에 성립해야 할 전제조건(precondition)과 사후조건(postcondition)을 포함한다. 예를 들어 "메모리 컴포넌트는 질의와 관련된 컨텍스트를 토큰 한도 이내로 반환한다"는 런타임 기대치는 형식 명세가 아니지만, 계약의 일부다.
:::

에이전트 시스템을 구성하는 주요 컴포넌트 간의 계약을 구체적으로 정리하면 다음과 같다.

**표 6-1. 컴포넌트별 입력·출력·예외 계약 요약**

| 컴포넌트 | 입력 | 출력 | 예외 처리 기대치 |
|---|---|---|---|
| Memory | 질의(query), 현재 컨텍스트 | 관련 컨텍스트 텍스트, 메타데이터 | 검색 결과 없음 → 빈 결과 반환, 오류 아님 |
| Planning | 목표, 현재 컨텍스트, 이전 관찰 | 다음 행동 계획 (단계 목록 또는 단일 지시) | 목표 불명확 → 명확화 요청 또는 폴백 계획 |
| Tool Use | 도구 이름, 호출 인자 (JSON 등) | 도구 실행 결과 (형식 지정) | 도구 오류 → 오류 코드와 메시지 포함 응답 |
| Model | 프롬프트 (컨텍스트 포함) | 자연어 텍스트 또는 구조화 출력 | 출력 형식 불일치 → 재시도 또는 파싱 오류 |
| Reflection | 행동 결과, 목표 | 자기 평가 결과, 수정 제안 | 평가 불가 → 생략 또는 중립 평가 |

이 표에서 눈여겨볼 점은 예외 처리 기대치 열이다. 많은 에이전트 구현이 정상 경로(happy path)만을 명세하고, 예외 경로는 암묵적으로 남겨 둔다. 이 암묵성이 실패 원인을 특정하기 어렵게 만드는 주된 원인이다.

3장에서 다루었던 도구 호출을 이 맥락에서 재해석해 보자. 도구 호출은 단순한 함수 호출이 아니라 **시스템 경계를 넘는 외부 의존(external dependency)**이다.[^web_05] 도구가 반환하는 응답 형식은 에이전트 내부에서 통제할 수 없다. API 버전이 바뀌거나, 외부 서비스가 예상과 다른 스키마로 응답하면, 그 순간 Tool Use 컴포넌트와 Planning 컴포넌트 사이의 계약이 깨진다. 계약이 명시되어 있지 않다면, 이 깨짐은 오류로 감지되지 않고 잘못된 데이터가 다음 컴포넌트로 조용히 흘러 들어간다.

2장에서 다루었던 컨텍스트(context)도 같은 시각으로 볼 수 있다. 컨텍스트는 컴포넌트 간 인터페이스를 통해 흐르는 가장 중요한 데이터 스트림이다. 메모리 컴포넌트가 잘못된 컨텍스트를 꺼내 모델에 넘기면, 그 컨텍스트는 계약상 "유효한 입력"으로 취급되지만 내용상으로는 오염된 상태다. 형식 계약은 지켜졌어도 의미적 계약은 위반된 것이다.

### 6.2.2 실패 전파 메커니즘: 어디서 시작해서 어디까지 가는가

계약의 의미를 이해했다면, 이제 계약이 어긋났을 때 무슨 일이 생기는지를 추적해 보자. 실패는 발생한 컴포넌트에서 끝나지 않는다. 에이전트 루프의 구조상, 한 컴포넌트의 출력은 다음 컴포넌트의 입력이 되기 때문에 실패는 경로를 따라 전파된다.

Zhu et al.은 단일 에이전트 내부의 오류 전파를 메모리(memory) → 반영(reflection) → 계획(planning) → 행동(action) → 시스템 출력(system output) 순서로 분석한다.[^web_08] 각 단계는 이전 단계의 출력을 전제로 작동하기 때문에, 앞 단계에서 발생한 오류는 뒤 단계에서 증폭되거나 변형되어 나타난다.

**그림 6-3. 오류 전파 경로 — 컴포넌트 단계별 실패 확산 구조**

```
  ┌──────────┐   잘못된 컨텍스트    ┌────────────┐  오류 평가     ┌──────────┐
  │  Memory  │ ─────────────────▶ │ Reflection │ ───────────▶ │ Planning │
  │ (검색 오류)│                   │ (오염된 기반)│              │(잘못된 계획)│
  └──────────┘                   └────────────┘              └────┬─────┘
                                                                   │
                                                          잘못된 지시
                                                                   ▼
                                                           ┌──────────┐   잘못된 결과  ┌───────────────┐
                                                           │  Action  │ ────────────▶ │ System Output │
                                                           │(오류 실행) │              │  (최종 오진)   │
                                                           └──────────┘              └───────────────┘
```

이 구조에서 주목해야 할 점은 **최종 출력의 오류가 메모리 컴포넌트에서 시작되었다**는 사실이 시스템 외부에서는 전혀 보이지 않는다는 것이다. 사용자나 운영자는 Action 단계의 잘못된 실행이나 System Output의 부정확한 결과만을 관찰한다. 증상은 맨 끝에 나타나지만, 원인은 맨 처음에 있다.

MAST(Cemri et al.)는 멀티에이전트 벤치마크에서 수집한 실패 사례들을 세 범주로 분류한다.[^web_04]

- **시스템 설계 이슈(system design issues)**: 컴포넌트 간 인터페이스 계약이 불충분하거나 폴트 격리가 구현되지 않아 발생하는 실패
- **에이전트 간 불일치(inter-agent misalignment)**: 에이전트들이 서로 다른 전제나 포맷을 기대해 발생하는 계약 위반
- **태스크 검증 실패(task verification failure)**: 행동의 결과를 검증하는 메커니즘이 없어 잘못된 결과가 다음 단계로 그대로 전달되는 실패

이 세 범주는 멀티에이전트 시스템을 위한 분류이지만, 단일 에이전트 내부의 컴포넌트 간 관계에도 그대로 적용된다. 컴포넌트를 독립된 에이전트로 치환해 읽으면 된다.

아래 예시는 동일한 증상이 서로 다른 컴포넌트의 실패에서 비롯될 수 있음을 보여 준다.

::: box 동일 증상, 다른 원인 — 진단 예시
에이전트가 "파일을 저장했다"고 보고했지만 실제 파일이 존재하지 않는다. 이 증상의 원인은 세 곳 중 하나일 수 있다.

1. **Tool Use 계약 위반**: 도구 호출 응답 포맷이 예상과 달라(예: `"status": "ok"` 대신 `"result": "success"`) 에이전트가 성공으로 잘못 해석했다.
2. **Memory 오염**: 이전 실행에서 성공했던 경로가 메모리에 저장되어 있고, 현재 실행에서 그 캐시된 결과가 관찰(Observation)로 주입되었다.
3. **Planning 전제 오류**: Reflection 단계에서 "저장이 필요 없다"는 잘못된 평가를 내렸고, Planning이 그 전제 위에서 저장 단계를 건너뛰었다.

세 경우 모두 최종 증상은 같다. 그러나 원인이 다르면 수정 방향도 완전히 다르다. 계약이 명시되어 있지 않으면, 증상만 보고 원인을 특정할 수 없다.
:::

::: box Replit 사례: 모델 교체로 해결하려 했던 문제가 시스템 경계 설계 문제였던 경위
실배포 에이전트 시스템의 실패 사례를 분석한 자료에 따르면, Replit의 AI 코딩 에이전트에서 발생한 반복적 오류는 초기에 모델 성능 부족으로 진단되었다.[^web_07] 팀은 더 강력한 모델로 교체하는 방향을 검토했다. 그러나 실제 원인은 도구 호출 결과를 메모리에 기록하는 방식의 문제였다. 이전 실행에서 발생한 오류 메시지가 다음 실행의 컨텍스트에 그대로 포함되어, 모델이 오류 상태를 정상 상태로 오해하고 계획을 수립했던 것이다.

모델을 교체했다면 더 나은 추론 능력이 이 오류를 일정 수준 보완했을 수도 있다. 그러나 근본 원인 — 메모리 컴포넌트와 Planning 컴포넌트 사이의 계약, 즉 "어떤 상태를 유효한 컨텍스트로 넘길 것인가" — 은 그대로 남아 다른 형태의 실패로 재발했을 가능성이 높다. 이 사례는 'models propose, architectures dispose'[^web_05]의 실례다.
:::

실패 전파를 이해하면 진단의 방향이 바뀐다. 증상이 나타난 컴포넌트가 아니라, 전파 경로를 거슬러 올라가 계약이 처음 어긋난 지점을 찾아야 한다. 이것이 가능하려면 한 가지 전제가 필요하다. 각 컴포넌트에서 무슨 일이 일어났는지를 사후에 볼 수 있어야 한다는 것이다. 실패 전파 경로를 설계 단계에서 파악하는 것은 중요하지만, 운영 중인 시스템에서 실패가 어디서 시작되었는지 추적하려면 시스템 내부 상태가 외부에서 관찰 가능해야 한다. 계약과 전파 경로를 아는 것만으로는 충분하지 않다. 그 경로 위에서 실제로 무슨 데이터가 흘렀는지 기록되고 검사될 수 있어야 한다. 6.3절에서는 이 **관찰 가능성(observability)**의 개념과 시스템 경계를 어떻게 정의할 것인지를 다룬다.

## 6.3 시스템 경계와 관찰 가능성

앞 절에서 실패가 어떻게 컴포넌트를 넘어 전파되는지를 살펴보았다. 그런데 그 전파 경로를 추적하는 작업 자체가 가능하려면 두 가지가 먼저 성립해야 한다. 에이전트가 어디서 시작하고 어디서 끝나는지, 즉 **시스템 경계(system boundary)**가 정의되어 있어야 하고, 그 경계 안에서 무슨 일이 일어났는지를 경계 밖에서 볼 수 있어야 한다. 이 두 가지를 갖추지 않으면, 실패 원인을 아는 것과 실패 원인을 찾는 것은 전혀 다른 문제가 된다. 6.3절에서는 이 시스템 경계와 관찰 가능성을 설계 수준에서 어떻게 다룰 것인지를 정리한다.

---

### 6.3.1 시스템 경계 정의: 에이전트는 어디서 시작하고 어디서 끝나는가

에이전트 시스템을 실제로 운영하다 보면 이런 상황이 반복된다. 에이전트가 외부 API를 호출했는데 응답이 돌아오지 않는다. 타임아웃이 발생했고, 에이전트는 재시도 없이 작업을 중단한다. 이것은 에이전트의 버그인가, 아니면 외부 서비스의 장애인가?

이 질문에 즉시 답하지 못한다면, 시스템 경계가 정의되어 있지 않은 것이다.

::: note 시스템 경계(system boundary)
에이전트 시스템에서 시스템 경계란 내부 컴포넌트와 외부 의존의 구분선이다. 경계 안에 있는 것은 설계와 구현의 책임 범위이고, 경계 밖에 있는 것은 인터페이스 계약을 통해 다루어야 할 외부 의존이다. 경계가 명시되지 않으면 실패의 책임 소재를 분석할 기준이 없어진다.
:::

시스템 경계는 물리적 경계가 아니라 책임의 경계다. 에이전트 코드가 외부 API를 직접 호출하더라도, 그 API의 가용성은 에이전트 시스템의 책임 범위 밖이다. 반면 API 응답을 어떻게 파싱하고, 어떤 오류를 재시도할 것인지, 타임아웃을 어떻게 처리할 것인지는 경계 안에서 결정되어야 한다.

ch03에서 다루었던 도구 호출(tool use)은 이 맥락에서 다시 읽을 수 있다. 도구는 에이전트가 외부 세계와 상호작용하는 수단이다. 도구 호출은 시스템 경계를 넘는 행위이고, 그 결과가 경계 안으로 돌아오는 것이 Observation이다. 도구 자체(외부 API, 데이터베이스, 파일 시스템)는 경계 밖에 있고, 도구 호출의 포맷과 결과 해석은 경계 안에 있다.

**그림 6-4. 에이전트 시스템 경계 — 내부 컴포넌트와 외부 의존의 구분**

```
┌─────────────────────────────────────────────────────────────┐
│                    에이전트 시스템 경계 안                      │
│                                                             │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐             │
│   │ Planning │───▶│ Memory   │───▶│ Reflection│            │
│   └────┬─────┘    └──────────┘    └──────────┘             │
│        │                                                    │
│        ▼                                                    │
│   ┌──────────┐   도구 호출(경계 통과)                         │
│   │ Action   │ ─────────────────────────────────────────▶  │
│   └──────────┘ ◀─────────────────────────────────────────  │
│                   결과 반환(경계 통과)                         │
└─────────────────────────────────────────────────────────────┘
                                        │
                 경계 밖                 ▼
         ┌────────────┬─────────────────────────┐
         │ 외부 API   │ 데이터베이스              │ 파일 시스템 │
         └────────────┴─────────────────────────┘
```

경계가 불명확할 때 발생하는 실질적인 혼란은 오진(misdiagnosis)이다. 외부 API가 비정상적인 응답 포맷을 돌려줬는데, 이것을 에이전트의 파싱 로직 버그로 분류하는 경우가 있다. 반대로 에이전트 내부에서 컨텍스트 윈도우가 초과되어 Planning이 잘못된 결론을 내렸는데, 이것을 "API가 이상한 응답을 준 것"으로 귀속시키기도 한다. 두 경우 모두 경계가 어디에 있는지 합의가 없어서 발생하는 책임 소재의 혼란이다.

Anthropic의 운영 가이드는 에이전트 시스템을 배포하기 전에 시스템 경계를 문서화하도록 권고한다.[^web_02] 어떤 컴포넌트가 경계 안에 있고, 각 외부 의존은 어떤 인터페이스 계약으로 다루어지며, 그 계약이 위반되었을 때 경계 안에서 어떻게 처리할 것인지가 명시되어야 한다는 것이다. 이 작업은 실패가 발생한 후에 하면 늦다. 경계 정의가 없는 상태에서의 사후 분석은 증거가 아니라 추정에 의존하게 된다.

---

### 6.3.2 관찰 가능성의 설계: 내부 상태를 외부에서 읽는 구조

시스템 경계를 정의했다고 해서 경계 안의 일을 알 수 있는 것은 아니다. 경계 안에서 어떤 데이터가 어느 컴포넌트를 거쳐 흘렀는지를 사후에 재구성할 수 있어야 비로소 진단이 가능해진다. 이것이 **관찰 가능성(observability)**의 문제다.

관찰 가능성은 성능 최적화를 위한 부가 기능이 아니다. 시스템이 의도한 대로 작동하고 있는지 확인하기 위한 최소 조건이다. 관찰 가능성이 없는 시스템은 같은 입력에 다른 출력이 나왔을 때 그 이유를 설명할 수 없다. 재현 가능한지조차 확인할 수 없다.

::: box 관찰 가능성이 없을 때의 디버깅 — 갈림점을 찾지 못하는 상황
에이전트가 어제는 올바른 요약을 생성했고 오늘은 부정확한 요약을 생성했다. 입력 문서는 동일하다.

내부 상태 기록이 없다면 분석 가능한 경우의 수는 다음과 같다.

- 모델 자체의 비결정적 출력(temperature > 0이라면 충분히 가능)
- Memory 컴포넌트에 어제의 실행 결과가 남아 있고, 오늘 그것이 컨텍스트에 포함됨
- Reflection 단계에서 평가 기준이 달라짐 (예: 시스템 프롬프트가 환경 변수 참조를 포함하고 있고 그 값이 바뀜)
- 외부 도구(예: 검색 API)의 응답이 달라짐

이 네 가지 가능성을 구분하려면 각 컴포넌트의 입력과 출력이 기록되어 있어야 한다. 기록이 없다면 네 가지 가능성 모두 배제할 수 없고, 진단은 추측에 의존할 수밖에 없다. 관찰 가능성은 이 구분을 가능하게 하는 구조다.
:::

관찰 가능성을 구현하는 기술적 수단으로는 **로그(log)**, **트레이스(trace)**, **스팬(span)**이 있다. 로그는 특정 시점에 발생한 이벤트를 기록한다. 트레이스는 하나의 요청이 여러 컴포넌트를 거치는 전체 흐름을 단일 식별자로 묶어 추적한다. 스팬은 트레이스 안에서 각 컴포넌트가 처리한 단위 작업을 표현한다. 에이전트 시스템에서는 Planning 스팬, Tool Call 스팬, Reflection 스팬처럼 ReAct 사이클의 각 단계를 스팬 단위로 기록하는 방식이 실무에서 자주 사용된다.[^web_03]

Böckeler는 에이전트 시스템의 하네스(harness)를 설계할 때 두 방향의 흐름을 구분하도록 권고한다.[^web_06] 하나는 **guides(피드포워드)**, 다른 하나는 **sensors(피드백)**다.

- **guides**: 에이전트에게 행동 방향을 주입하는 구조. 시스템 프롬프트, 도구 명세, 제약 조건이 여기에 해당한다. 에이전트가 무엇을 해야 하는지를 정의하는 입력 방향의 흐름이다.
- **sensors**: 에이전트의 행동 결과를 수집하는 구조. 로그, 트레이스, 중간 결과 기록이 여기에 해당한다. 에이전트가 실제로 무엇을 했는지를 관찰하는 출력 방향의 흐름이다.

이 구분은 단순한 어휘 정리가 아니라 설계 결정을 위한 지침이다. 시스템 프롬프트를 수정하는 것은 guides 쪽의 변경이다. 로그 수집 위치를 바꾸는 것은 sensors 쪽의 변경이다. 두 방향을 혼동하면, 예를 들어 "에이전트가 이상하게 동작하니 시스템 프롬프트를 바꿔보자"는 접근이 실제로는 sensors 부재로 인해 원인이 파악되지 않은 상황을 우회하는 시도임을 인지하지 못하게 된다.

**표 6-1. guides와 sensors의 구분**

| 구분 | 방향 | 역할 | 예시 |
|------|------|------|------|
| guides (피드포워드) | 시스템 → 에이전트 | 행동 방향 주입 | 시스템 프롬프트, 도구 명세, 제약 조건 |
| sensors (피드백) | 에이전트 → 시스템 | 행동 결과 수집 | 로그, 트레이스, 중간 결과 기록 |

::: tip 관찰 가능성 최소 구현 — 어떤 지점에서 무엇을 기록해야 하는가
관찰 가능성을 처음 구현할 때 모든 내부 상태를 기록하려는 시도는 오히려 노이즈를 늘린다. 최소한 다음 세 지점의 기록으로 시작하는 것이 실용적이다.

1. **도구 호출의 입력과 출력**: 경계를 넘는 모든 호출에서 요청 파라미터와 응답 전문을 기록한다. 외부 의존의 계약 위반을 탐지하는 가장 직접적인 방법이다.
2. **Planning 단계의 최종 결론**: 에이전트가 다음에 무엇을 하기로 결정했는지를 기록한다. Thought 전체를 저장하는 것보다 결론 부분만 구조화해 기록하면 사후 분석이 수월해진다.
3. **Memory 읽기 결과**: 어떤 이전 상태가 현재 실행의 컨텍스트로 주입되었는지를 기록한다. 반복 실행 간 상태 오염 여부를 확인하는 핵심 지점이다.

Lin & Zhang은 컴포넌트 간 인터페이스 계약이 명시되고 폴트 격리가 구현된 시스템에서 관찰 가능성의 효용이 극대화된다고 분석한다.[^web_05] 계약이 없는 상태에서 기록만 늘리면, 무엇이 정상이고 무엇이 이상인지를 판단하는 기준 자체가 없어진다.
:::

MAST가 분류한 태스크 검증 실패(task verification failure)는 관찰 가능성 부재와 직접적으로 연결된다.[^web_04] 에이전트가 어떤 행동을 취했는지, 그 결과가 의도와 일치하는지를 검증하는 메커니즘이 없으면, 잘못된 결과가 다음 단계의 입력으로 조용히 전달된다. 이 실패는 외부에서 보면 마치 에이전트가 "정상적으로 실행을 완료했는데 결과가 이상한" 것처럼 나타난다. 실제 원인은 검증 단계의 부재이지만, 관찰 가능성이 없으면 이 원인을 식별할 단서조차 남지 않는다.

시스템 경계를 정의하고 관찰 가능성을 갖추면, 에이전트 시스템의 진단 가능성(diagnosability)이 근본적으로 달라진다. 그런데 이 관찰 가능성을 구현하는 구조, 즉 에이전트의 실행을 반복적으로 실행하고 그 결과를 수집·평가하는 외부 구조를 어디에 위치시켜야 하는가라는 질문이 남는다. 관찰하는 구조가 관찰 대상과 같은 경계 안에 있다면, 시스템 자체의 오류가 관찰 결과에 영향을 줄 수 있다. 다음 절에서는 평가 하네스가 왜 반드시 시스템 경계 밖에 놓여야 하는지를, 여기서 확립한 경계와 관찰 가능성의 개념을 바탕으로 살펴본다.

## 6.4 평가 하네스는 왜 시스템 외부에 있어야 하는가

앞 절에서는 관찰 가능성이 에이전트 시스템의 설계 안에 내재되어야 한다는 점을 살펴보았다. guides와 sensors라는 두 방향의 흐름을 구분하고, 각 컴포넌트의 경계에서 무엇을 기록해야 하는지를 정리했다. 이제 자연스럽게 다음 질문이 따라온다. 그 관찰 결과를 수집하고 판단하는 구조 자체는 어디에 위치해야 하는가? 관찰하는 구조가 관찰 대상과 같은 경계 안에 있다면, 무언가 문제가 생기지 않을까?

이 절에서는 그 질문에 답하기 위해 **평가 하네스(harness)**의 개념과 위치를 구체적으로 다룬다.

::: note harness (평가 하네스)
concept_dictionary 기준 정의: 반복 가능하게 실행과 평가를 수행하는 외부 구조.

하네스는 에이전트 시스템을 반복적으로 실행하고, 각 실행의 결과를 수집하며, 사전에 정의된 기준에 따라 그 결과를 평가하는 구조 전체를 가리킨다. 특정 언어나 프레임워크로 작성된 테스트 코드 하나가 아니라, 실행 환경의 격리, 입력 주입, 출력 수집, 판단 적용까지의 전체 파이프라인이 하네스에 해당한다.
:::

### 6.4.1 하네스의 두 기능: 피드포워드와 피드백 채널

Böckeler는 에이전트 시스템을 'Agent = Model + Harness'라는 공식으로 표현한다.[^web_06] 이 공식이 의미하는 바는, 에이전트의 행동을 결정하는 모델과 그 모델이 작동하는 조건과 환경을 구성하는 하네스가 서로 구분되는 두 요소라는 것이다. 하네스 없이 모델만 있으면 에이전트는 동작할 수 있을지 모르지만, 그 동작을 반복하고 평가하고 개선하는 루프는 존재하지 않는다.

하네스는 크게 두 가지 기능 채널로 구성된다.

- **guides (피드포워드 채널)**: 에이전트에게 행동 지침을 주입하는 방향의 흐름이다. 시스템 프롬프트, 도구 명세, 허용 행동의 제약 조건이 여기에 해당한다. 에이전트가 어떤 도구를 사용할 수 있는지, 어떤 형식으로 응답해야 하는지, 어떤 경우에 멈춰야 하는지를 정의하는 것이 guides의 역할이다.
- **sensors (피드백 채널)**: 에이전트의 행동 결과를 수집하는 방향의 흐름이다. 실행 로그, 도구 호출 결과, 중간 판단 기록, 최종 출력이 여기에 해당한다. 에이전트가 실제로 어떤 행동을 취했고 그 결과가 무엇인지를 외부에서 관찰하는 것이 sensors의 역할이다.[^web_06]

**그림 6-5. 'Agent = Model + Harness' — 하네스의 위치와 두 기능 채널**

```
┌─────────────────────────────────────────────────┐
│                    Harness (외부)                │
│                                                 │
│   guides (피드포워드)          sensors (피드백)   │
│   ─────────────────→          ←───────────────  │
│   시스템 프롬프트               실행 로그         │
│   도구 명세                     출력 기록         │
│   제약 조건                     판단 결과         │
│         │                           ↑           │
│         ▼                           │           │
│   ┌──────────────────────────────────────────┐  │
│   │           Agent (내부)                   │  │
│   │   Model ←──→ Memory ←──→ Tools          │  │
│   └──────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

여기서 주목해야 할 점은 두 채널 모두 에이전트 시스템의 경계 *바깥*에서 연결된다는 것이다. guides는 외부에서 에이전트 안으로 흘러 들어가고, sensors는 에이전트 안에서 외부로 흘러 나온다. 하네스는 이 두 채널을 잇는 외부 구조다. 만약 이 구조가 에이전트 내부에 있다면, guides가 에이전트의 현재 상태에 의존해 변형될 수 있고, sensors가 수집하는 데이터가 에이전트의 내부 오류에 의해 오염될 수 있다.

평가의 독립성은 바로 이 외부 위치에서 나온다. 외부에 있는 하네스는 에이전트가 어떤 상태에 있든 동일한 입력을 주입하고, 동일한 방식으로 출력을 수집하며, 동일한 기준으로 결과를 판단한다. 이 일관성이 반복 가능성의 조건이다.

### 6.4.2 하네스 위치가 평가 신뢰도에 미치는 영향

하네스가 시스템 내부에 결합되어 있을 때 생기는 문제를 구체적인 상황으로 살펴보자.

동일한 입력으로 에이전트를 열 번 실행했을 때 결과가 다섯 가지 서로 다른 형태로 나온다고 가정한다. 이것이 모델의 비결정성 때문인지, 메모리에 이전 실행 결과가 누적되어 있기 때문인지, 외부 도구의 응답이 달라졌기 때문인지 판별해야 한다. 하네스가 내부에 결합된 경우, 평가 로직 자체가 메모리 컴포넌트와 같은 저장소를 공유하거나, 도구 호출 결과를 캐시하는 레이어를 거칠 수 있다. 이 경우 평가 결과는 에이전트의 현재 상태를 반영하지만, 격리된 반복 실행의 결과를 반영하지 않는다.

반면 외부 하네스가 있는 경우, 각 실행 전에 메모리 상태를 초기화하고, 도구 호출을 모의(mock)하거나 실제 API와의 연결을 제어하며, 각 실행의 입력과 출력을 독립적으로 기록한다. 열 번의 실행이 서로 오염되지 않는 독립적인 실험으로 처리된다. 결과의 변동이 발생했을 때 원인을 가설 형태로 분리해 검증할 수 있는 환경이 갖춰진다.

**표 6-2. 하네스 위치(내부/외부)에 따른 평가 속성 비교**

| 평가 속성 | 내부 결합 하네스 | 외부 독립 하네스 |
|----------|----------------|----------------|
| **독립성** | 시스템 상태가 평가에 개입 가능 | 평가가 시스템 상태로부터 격리됨 |
| **반복 가능성** | 실행 간 상태 오염 발생 가능 | 동일 조건 반복 실행 보장 |
| **격리성** | 컴포넌트 간 경계가 흐려짐 | 각 컴포넌트 단위로 교체·대조 가능 |
| **디버깅 효율** | 원인 추적이 추측에 의존 | 입출력 기록 기반 원인 분리 가능 |
| **평가 일관성** | 실행 환경에 따라 기준이 흔들림 | 사전 정의된 기준이 일관되게 적용됨 |

Lin & Zhang은 컴포넌트 간 인터페이스 계약이 명시되고 폴트 격리(fault isolation)가 구현된 구조에서 외부 평가의 신뢰도가 담보된다고 분석한다.[^web_05] 폴트 격리란 한 컴포넌트의 오류가 다른 컴포넌트의 상태를 오염시키지 않도록 경계를 유지하는 설계 원칙이다. 하네스가 외부에 있어야 이 격리가 평가 단계까지 유지된다. 내부에 있으면, 평가 단계가 폴트의 영향 안에 놓이게 된다.

MAST 연구에서 확인된 태스크 검증 실패(task verification failure)도 이 문제와 연결된다.[^web_04] 에이전트가 수행한 행동의 결과가 의도와 일치하는지를 검증하는 구조가 에이전트와 같은 경계 안에 있다면, 에이전트가 잘못된 방향으로 수렴하면서도 내부 검증은 통과하는 상황이 발생할 수 있다. 검증 기준이 검증 대상과 같은 상태를 공유하기 때문이다. 외부 하네스는 이 문제를 구조적으로 차단한다.

::: box harness ≠ 테스트 코드 — 평가 하네스와 테스트 프레임워크의 구조적 차이
"하네스를 만든다"는 말을 들으면 pytest나 unittest 같은 테스트 프레임워크를 떠올리는 경우가 많다. 그러나 테스트 코드는 하네스의 일부일 수 있지만, 하네스와 동일하지 않다.

테스트 프레임워크는 주어진 함수가 주어진 입력에 대해 기대한 출력을 반환하는지를 단위 수준에서 검증한다. 반면 평가 하네스는 에이전트 전체를 반복적으로 실행하는 환경 구성, 각 실행 전후의 상태 초기화, 실행 결과의 수집과 구조화, 그리고 사전 정의된 판단 기준의 적용까지를 포함한다.

구체적으로 다르게 나타나는 지점은 다음과 같다.

- **실행 단위**: 테스트 코드는 함수 또는 메서드 단위. 하네스는 에이전트 루프 전체의 한 사이클 또는 태스크 단위.
- **상태 관리**: 테스트 코드는 픽스처(fixture)로 입력을 준비한다. 하네스는 메모리, 도구 응답, 외부 API 상태까지 포함한 실행 환경 전체를 관리한다.
- **판단 기준**: 테스트 코드는 assert 조건으로 통과/실패를 판정한다. 하네스는 정성적 판단 기준, 부분 점수, 여러 평가자의 결과를 통합하는 방식까지 포함할 수 있다.
- **반복 목적**: 테스트 코드는 회귀 방지가 주목적이다. 하네스는 에이전트 동작의 반복 가능성 확인, 비결정성 분석, 컴포넌트 교체 효과 측정 등 더 넓은 목적을 갖는다.
:::

5장에서 다룬 평가 방법론 — 기준 기반 평가, 모델 기반 판단, 인간 평가 루프 — 은 모두 평가 주체가 평가 대상으로부터 독립되어 있다는 전제 위에서 작동한다. 외부 하네스가 없다면, 이 방법론들은 개념으로는 존재하지만 실제로 일관되게 적용할 구조가 없는 상태가 된다. 하네스는 평가 방법론이 실제 에이전트 시스템에 닿을 수 있는 인터페이스다.

::: tip 하네스 외부화의 최소 조건 — 어떤 인터페이스를 열어 두어야 하는가
하네스를 외부에 두려면 에이전트 시스템이 다음 세 가지 인터페이스를 열어 두어야 한다.

1. **상태 초기화 인터페이스**: 메모리 컴포넌트를 특정 상태로 설정하거나 초기화할 수 있는 진입점. 이것이 없으면 각 실행이 이전 실행의 영향을 받아 반복 가능성이 깨진다.
2. **도구 응답 제어 인터페이스**: 외부 도구나 API 호출을 모의(mock)하거나 실제 응답으로 대체할 수 있는 레이어. 실행 결과의 변동 원인이 에이전트 내부인지 외부 의존인지를 분리하는 데 필수적이다.
3. **실행 결과 추출 인터페이스**: 에이전트의 중간 상태와 최종 출력을 구조화된 형태로 꺼낼 수 있는 출구. 로그 파일 파싱에만 의존하면 구조 변경 시 하네스도 함께 깨지는 취약성이 생긴다.

이 세 인터페이스가 갖춰지면, 하네스는 에이전트 시스템과 계약 기반으로 연결된다. 에이전트 내부 구현이 바뀌어도 인터페이스가 유지되는 한 하네스는 독립적으로 작동할 수 있다.
:::

하네스의 위치가 확정되면, 설계 판단의 범위가 달라진다. 어떤 컴포넌트를 교체하거나 확장할 때 그 변경이 다른 컴포넌트와 하네스에 어떤 영향을 미치는지를 구조적으로 예측할 수 있게 된다. 이 예측 가능성이 시스템 관점에서 설계한다는 것의 실질적인 의미다. 다음 절에서는 이 시스템 관점의 설계가 실제 판단 장면에서 어떻게 다른 결과를 만들어내는지를, 컴포넌트 교체를 중심으로 구체적으로 살펴본다.

## 6.5 시스템으로 설계할 때 달라지는 판단들

앞 절에서 평가 하네스(harness)가 시스템 외부에 위치해야 하는 이유를 살펴보았다. 하네스의 위치가 정해지면, 그것은 단순히 평가 방식의 문제가 아니라 설계 전반의 판단 방식을 바꾼다. 이 절에서는 6.1부터 6.4까지 다룬 시스템 분해, 컴포넌트 계약, 실패 경로, 관찰 가능성, 하네스의 논의를 하나로 수렴시킨다. 시스템 관점으로 설계한다는 것이 실제 판단 장면에서 어떻게 다른 결과를 만들어내는지, 구체적인 시나리오를 통해 정리하자.

---

### 6.5.1 컴포넌트 교체 가능성: 무엇을 고정하고 무엇을 바꾸는가

시스템 관점으로 설계된 에이전트에서 모델을 교체하는 상황을 생각해 보자. 모델 컴포넌트만 교체한다면 무엇이 바뀌고 무엇이 그대로여야 하는가.

바뀌는 것은 하나다. 모델과 외부 세계가 맺는 인터페이스 계약, 즉 어떤 형식의 입력을 받아 어떤 형식의 출력을 돌려주는가라는 약속이다. GPT-4o에서 Claude 3.5 Sonnet으로 모델을 바꾼다면, 프롬프트 구조, 응답 파싱 로직, 함수 호출 포맷 등 모델과 직접 맞닿은 계층이 조정 대상이 된다. 반면 메모리 컴포넌트가 어떤 구조로 컨텍스트를 관리하는지, 도구 컴포넌트가 어떤 입출력 계약을 갖는지, 하네스가 어떤 인터페이스로 시스템을 반복 실행하는지는 모델 교체의 영향 범위 바깥에 있어야 한다.

이 분리가 가능하려면 인터페이스 계약이 사전에 명시되어 있어야 한다. Lin & Zhang은 컴포넌트화(componentization)와 명시적 인터페이스가 없는 시스템에서는 한 컴포넌트의 변경이 예상치 못한 경로로 다른 컴포넌트에 전파된다고 지적한다.[^web_05] 이 전파를 추적하지 못하면, 모델 교체 후 발생한 오류가 모델 자체의 문제인지 인터페이스 불일치인지 메모리 상태의 충돌인지를 구분할 수 없게 된다.

도구 추가 시나리오도 비슷한 구조를 갖는다. 새로운 외부 API를 도구로 통합할 때, 시스템 관점에서 먼저 해야 할 것은 이 도구가 시스템 경계 내에서 어떤 책임을 갖는지를 정의하는 일이다. 도구가 어떤 입력을 받아 어떤 형식의 응답을 돌려주는지, 실패했을 때 어떤 신호를 반환하는지, 응답 지연이 발생했을 때 에이전트 루프가 어떻게 처리하는지를 도구를 추가하기 전에 먼저 명시해야 한다. 이 명시 없이 도구를 연결하면, 실제 운영에서 실패가 발생했을 때 원인이 도구의 외부 API인지, 도구 래퍼 레이어인지, 에이전트의 도구 호출 로직인지를 사후에 추적해야 하는 상황이 된다. Böckeler의 표현을 빌리면, 실패 경로는 도구를 추가하는 순간 함께 설계되어야 한다.[^web_06]

**표 6-3. 컴포넌트 교체 시나리오별 변경 범위**

| 교체 대상 | 변하는 것 | 고정되어야 하는 것 |
|------------|-----------|-------------------|
| **모델** | 프롬프트 포맷, 응답 파싱, 함수 호출 스키마 | 메모리 구조, 도구 인터페이스, 하네스 실행 계약 |
| **메모리** | 컨텍스트 구성 방식, 검색 로직 | 모델 입력 형식, 도구 호출 흐름, 하네스 상태 초기화 인터페이스 |
| **도구** | 외부 API 연결, 응답 파싱, 실패 신호 형식 | 에이전트 루프의 도구 호출 계약, 하네스의 모의 응답 인터페이스 |
| **하네스** | 평가 기준, 반복 실행 방식, 결과 수집 형식 | 에이전트 시스템이 노출하는 세 인터페이스(상태 초기화·도구 제어·결과 추출) |

이 표에서 주목할 것은 "고정되어야 하는 것" 열이다. 교체 대상이 무엇이든 간에, 인접 컴포넌트와 맺은 인터페이스 계약과 하네스와의 연결 구조는 고정 지점으로 유지된다. 이 고정 지점이 있기 때문에 변경의 영향 범위를 예측할 수 있고, 예측 가능하기 때문에 교체 이후의 평가가 의미를 갖는다.

시스템 관점은 미래의 변경에 대비하기 위한 것이 아니다. 현재의 판단을 명확히 하기 위한 것이다. 지금 이 컴포넌트를 이렇게 연결할 때, 변경이 어디까지 전파될 수 있는가를 설계 시점에 추적하는 것이 목적이다. 이 추적이 가능해지면, "이 변경이 왜 이 결과를 만들었는가"라는 질문에 가설을 세우고 검증하는 구조가 갖춰진다.

::: tip 교체 가능성 설계의 실용적 기준 — 인터페이스를 먼저 쓰고 구현을 나중에 채우기
컴포넌트의 교체 가능성을 설계할 때 실용적인 순서는 구현보다 인터페이스를 먼저 정의하는 것이다.

모델 컴포넌트를 추가하기 전에 "이 컴포넌트는 어떤 입력을 받아 어떤 출력을 돌려주는가, 실패하면 어떤 신호를 내보내는가"를 문서화하거나 코드의 타입 시그니처 수준으로 명시한다. 구현 없이 인터페이스만 있는 상태에서 인접 컴포넌트와의 연결을 상상해 보면, 계약 위반이 발생할 지점이 구현 전에 드러난다.

Anthropic의 설계 가이드는 이와 유사한 원칙을 다른 방식으로 표현한다: 복잡성을 추가하기 전에 단순한 인터페이스가 요구사항을 충족하는지 먼저 확인하라.[^web_02] 인터페이스를 먼저 정의하는 습관은 불필요한 복잡성을 추가하는 것에 대한 자연스러운 제동 장치가 된다.
:::

### 6.5.2 시스템 경계의 확장: 멀티 에이전트와 운영 환경으로의 열린 질문

6.1부터 6.4까지 다룬 모든 논의는 단일 에이전트 시스템을 전제로 했다. 그러나 실제 AI 응용 시스템은 점점 더 복수의 에이전트가 협력하거나 경쟁하는 구조로 나아간다. 단일 에이전트 수준에서 확립한 시스템 관점은 이 방향으로의 확장에서 무엇을 시사하는가.

Lilian Weng은 에이전트들이 협력할 때 시스템 경계를 재정의하지 않으면 컴포넌트 간 계약이 충돌하고 관찰 가능성이 소실된다고 지적한다.[^web_01] 단일 에이전트에서 "메모리 컴포넌트와 모델 컴포넌트가 어떤 계약으로 연결되는가"였던 질문이, 멀티 에이전트 환경에서는 "에이전트 A의 출력이 에이전트 B의 어떤 입력 인터페이스와 연결되는가, 그 계약이 명시되어 있는가"로 확장된다. 각 에이전트가 하나의 컴포넌트처럼 작동하는 더 큰 시스템이 생기는 것이다.

이때 단일 에이전트 수준에서 습득한 설계 습관이 그대로 적용된다. 에이전트 간 인터페이스 계약을 명시하고, 한 에이전트의 실패가 다른 에이전트에 어떻게 전파되는지를 추적하며, 전체 멀티 에이전트 시스템을 바깥에서 관찰하고 반복 실행할 수 있는 하네스를 별도로 구성한다. 규모가 커질수록 이 습관의 필요성은 줄어드는 것이 아니라 오히려 늘어난다. MAST 연구에서 확인된 에이전트 간 불일치(inter-agent inconsistency)는 멀티 에이전트 구성에서 발생하는 실패 중 상당 부분이 단일 에이전트 수준의 인터페이스 계약 부재에서 비롯됨을 보여준다.[^web_04]

운영 환경이라는 방향에서도 질문이 열려 있다. 개발 환경에서 작동하던 에이전트가 실제 트래픽을 처리하는 환경으로 옮겨지면, 관찰 가능성의 요구사항이 달라진다. Böckeler는 운영 환경에서의 하네스가 개발 환경과 달리 실시간 센서(sensor) 역할까지 수행해야 한다고 설명한다.[^web_06] 개발 환경에서는 사후 분석을 위해 실행 결과를 기록하는 것으로 충분하지만, 운영 환경에서는 현재 실행 중인 루프의 상태를 외부에서 실시간으로 읽을 수 있어야 한다. 이는 에이전트 시스템이 내부 상태를 구조화된 형태로 노출하는 관찰 가능성 인터페이스를 갖추어야 한다는 것을 의미한다.

::: box 이 장의 핵심 질문 재방문 — "원인은 모델에 있는가, 시스템에 있는가?"에 이제 어떻게 답하는가
이 장은 "에이전트가 잘못된 결과를 냈을 때, 그 원인은 모델에 있는가, 시스템에 있는가?"라는 질문으로 시작했다.

6.1에서 살펴본 것처럼, 이 질문에 답하려면 먼저 시스템을 분해할 수 있어야 한다. 어떤 컴포넌트가 있고, 각 컴포넌트가 어떤 책임을 가지며, 컴포넌트 간에 어떤 계약으로 연결되어 있는지를 파악할 수 없다면 원인의 소재를 구분하는 것 자체가 불가능하다.

6.2에서 정리한 컴포넌트 간 인터페이스 계약이 있으면, "이 결과는 모델이 잘못된 추론을 한 것인가, 아니면 메모리 컴포넌트가 잘못된 컨텍스트를 모델에 전달한 것인가"를 계약 위반 여부로 추적할 수 있다.

6.3에서 다룬 실패 경로 분석은 이 추적을 전파 단계별로 구체화한다. Zhu et al.의 분석처럼 memory→planning→action으로 이어지는 오류 전파 경로를 사전에 그려 놓았다면, 관찰된 실패를 경로 위의 어느 지점에 위치시킬 수 있다.[^web_08]

6.4에서 확립한 외부 하네스가 있으면, 이 추적을 반복 가능한 실험으로 전환할 수 있다. 메모리 상태를 바꾸고 다시 실행해보거나, 도구 응답을 모의 값으로 대체하고 결과가 달라지는지 확인하는 것이 구조적으로 가능해진다.

이 모든 것이 갖춰진 다음에야 "원인은 모델에 있는가, 시스템에 있는가"라는 질문이 추측이 아닌 검증의 대상이 된다.
:::

**그림 6-6. 6장 전체 논증 구조: 분해에서 판단까지**

```
시스템 분해 (6.1)
  → 컴포넌트: 모델 / 메모리 / 도구
  → 시스템 경계 정의

       ↓

인터페이스 계약 (6.2)
  → 컴포넌트 간 입출력 명세
  → 폴트 격리 조건 확보

       ↓

실패 경로 추적 (6.3)
  → 오류 전파 경로 사전 정의
  → 원인 소재 분리 가능

       ↓

관찰 가능성 + 하네스 (6.4)
  → 외부 독립 구조로 반복 실행
  → 평가가 시스템 상태에서 격리

       ↓

컴포넌트 교체 판단 (6.5)
  → 변하는 것 vs. 고정되어야 하는 것
  → 영향 범위를 계약 단위로 추적

       ↓

확장 방향으로 열린 질문
  → 멀티 에이전트: 에이전트 간 계약 재정의
  → 운영 환경: 관찰 가능성 요구사항 상승
```

---

이 장에서 강조하고자 한 것은 두 가지다. 첫째, 시스템 관점은 복잡성을 추가하기 위한 것이 아니라 복잡성이 어디에 있는지를 보기 위한 것이다. Anthropic의 설계 가이드가 "복잡성은 필요할 때만"이라는 원칙을 반복하는 이유가 여기에 있다.[^web_02] 단순한 워크플로우로 충분한 태스크에 에이전트 루프를 도입하는 것, 명확한 규칙 기반으로 처리할 수 있는 분기에 LLM 추론을 투입하는 것은 시스템의 복잡성을 늘리는 동시에 관찰 가능성을 낮추는 선택이다. 시스템 관점으로 설계한다는 것은 단순함을 유지하는 근거도 된다.

둘째, "모델이 좋으면 시스템은 따라온다"는 인식은 구조적으로 틀렸다. Lin & Zhang의 표현처럼, 모델은 제안하고 아키텍처는 결정한다.[^web_05] 모델의 출력이 신뢰 가능한 결과로 이어지려면, 모델에 어떤 컨텍스트가 전달되는지, 모델의 출력이 어떤 도구 호출로 이어지는지, 그 결과가 어떻게 검증되는지를 설계가 보장해야 한다. 이 보장이 없는 시스템에서는 더 좋은 모델을 쓴다고 해도 실패 경로가 사라지지 않는다.

다음 장에서는 이 시스템 관점의 논의를 복수의 에이전트가 협력하는 구조로 확장한다. 단일 에이전트에서 컴포넌트였던 것이 에이전트 하나가 되고, 에이전트들이 서로 계약을 맺고 실패를 격리해야 하는 상황에서 이 장의 설계 습관이 어떻게 적용되는지를 살펴볼 것이다.
