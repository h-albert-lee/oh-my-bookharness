# 6장. AI 에이전트는 단순 프로그램이 아니라 시스템이다

## 6.1 에이전트 내부를 컴포넌트로 읽기

"여행 일정을 짜 줘"라는 요청 하나를 에이전트에 입력했다고 하자. 잠시 후 도쿄 3박 4일 일정이 깔끔하게 출력된다. 출력이 정확하고 빠르다면 내부에서 무슨 일이 일어났는지 굳이 살펴볼 이유가 없어 보인다. 그러나 일정이 어긋나거나, 예산 조건을 무시하거나, 이전에 말했던 선호도가 반영되지 않는다면 어떻게 해야 할까? 이때 개발자가 가장 먼저 마주치는 문제는 '어디를 고쳐야 하는가'다.

에이전트를 "모델 호출 하나"로 인식하고 있다면 이 질문에 답하기 어렵다. 모델 프롬프트를 수정하고, 다시 실행하고, 결과가 달라졌을 때 그 이유를 설명하지 못한다. 이 절에서는 그 평면적 인식에서 벗어나기 위한 분해의 언어를 정리한다. 에이전트를 Planning(계획), Memory(메모리), Tool Use(도구 사용) 세 컴포넌트와 그것들을 조율하는 모델 레이어로 나누어 읽는 것이 출발점이다.[^web_01] 1장부터 5장까지 각각 다루어진 모델, 컨텍스트, 도구, 메모리, 평가는 이 분해 구조 위에 다시 배치될 때 서로의 관계가 비로소 선명해진다.

---

### 6.1.1 컴포넌트 분해의 기준: 책임 범위와 교체 가능성

컴포넌트를 나누는 기준은 복잡도가 아니다. 무엇에 책임지는가, 그리고 나머지 부분을 건드리지 않고 교체할 수 있는가가 기준이다. 이 두 기준을 함께 적용했을 때 에이전트 내부에서 자연스럽게 분리되는 세 개의 컴포넌트와 하나의 모델 레이어가 있다.[^web_01]

**그림 6-1. 에이전트 컴포넌트 구조 — Planning·Memory·Tool Use와 모델 레이어의 위치 관계**

```
┌──────────────────────────────────────────────────┐
│                     에이전트                      │
│                                                  │
│   ┌─────────────┐      ┌──────────────────────┐  │
│   │  Planning   │─────▶│     Model Layer      │  │
│   │   (계획)    │      │  (추론 · 생성 · 조율)  │  │
│   └─────────────┘      └──────────┬───────────┘  │
│          ▲                        │              │
│          │                        ▼              │
│   ┌──────┴──────┐      ┌──────────────────────┐  │
│   │   Memory    │      │      Tool Use        │  │
│   │  (기억·맥락) │      │   (외부 연산·검색)    │  │
│   └─────────────┘      └──────────────────────┘  │
└──────────────────────────────────────────────────┘
```

- **Planning 레이어**: 목표를 하위 태스크로 분해하고 실행 순서를 결정한다. 어떤 도구를 언제 호출할지, 결과가 불충분할 때 다시 시도할지도 여기서 결정된다.
- **Memory 레이어**: 현재 세션 내 컨텍스트(단기 기억), 과거 세션이나 외부 저장소의 정보(장기 기억), 그리고 검색 증강(RAG)으로 끌어오는 외부 지식을 포함한다. 4장에서 다룬 메모리 전략이 이 레이어에 해당한다.
- **Tool Use 레이어**: 검색 API, 코드 실행기, 데이터베이스 조회처럼 모델 외부에서 이루어지는 연산 전체를 담당한다. 3장의 도구 개념이 여기에 위치한다.
- **Model 레이어**: 위 세 컴포넌트의 입력을 받아 추론하고, 다음 행동을 결정하거나 최종 응답을 생성한다. 1장과 2장에서 다룬 모델과 컨텍스트 어셈블리가 이 레이어에서 작동한다.

::: note 1–5장 개념과 컴포넌트의 대응 관계
이전 장에서 다룬 개념들은 이 분해 구조 안에서 다음과 같이 배치된다.

**표 6-1. 이전 장 개념과 에이전트 컴포넌트 레이어 대응**

| 이전 장 개념 | 해당 컴포넌트 레이어 |
|---|---|
| 모델 추론, 토큰 생성 (1장) | Model Layer |
| 컨텍스트 윈도우, 시스템 프롬프트 (2장) | Model Layer / Memory |
| 도구 정의, API 연결 (3장) | Tool Use |
| 단기·장기 메모리, RAG (4장) | Memory |
| 평가 설계, 하네스 (5장) | 시스템 외부 (harness) |

평가 하네스(harness)는 어느 컴포넌트에도 속하지 않는다. 시스템 외부에서 시스템 전체를 반복 실행하고 관찰하는 구조이기 때문에, 특정 레이어 안에 두면 하네스의 독립성이 훼손된다. 이 배치 원칙은 6.4절에서 더 자세히 다룬다.
:::

컴포넌트 분해가 가져오는 실질적인 이점은 진단 범위를 좁혀 준다는 데 있다. "여행 일정을 짜 줘" 요청이 들어왔을 때 내부 흐름을 이 레이어 단위로 추적하면 다음과 같다.

1. **Model → Planning**: 모델이 요청을 해석하고 "항공편 검색 → 숙소 검색 → 일정 조합" 순서를 계획한다.
2. **Planning → Tool Use**: 계획에 따라 항공편 검색 API를 호출한다.
3. **Tool Use → Memory**: 검색 결과를 세션 컨텍스트에 저장한다.
4. **Memory → Model**: 사용자의 이전 선호도(창가 좌석 선호, 4성급 이상 숙소)를 컨텍스트에 포함해 모델에 전달한다.
5. **Model**: 모든 입력을 종합해 최종 일정을 생성한다.

이 흐름에서 응답이 어긋났다면 어디서 발생한 문제인지 컴포넌트 단위로 좁힐 수 있다. 항공편 정보가 틀렸다면 Tool Use 레이어의 API 호출이나 결과 파싱을 확인해야 하고, 선호도가 반영되지 않았다면 Memory 레이어의 컨텍스트 어셈블리를 점검해야 한다. "프롬프트를 고쳐야 한다"는 단일한 처방 대신 책임 범위가 분리된다.

::: tip 컴포넌트 분해는 설계 도구이기도 하다
분해가 진단에만 쓰이는 것은 아니다. 항공편 검색 API를 다른 공급사의 API로 교체해야 할 때, 교체가 Tool Use 레이어 안에 국소화될 수 있다면 Planning이나 Memory에 손댈 필요가 없다. 컴포넌트 경계가 교체 범위도 결정한다.
:::

### 6.1.2 Thought→Action→Observation 루프: 내부 흐름의 정형화

컴포넌트를 나열하는 것만으로는 에이전트가 어떻게 움직이는지 보이지 않는다. 컴포넌트들이 어떤 순서로, 어떤 신호를 주고받으며 작동하는지를 이해하려면 내부 실행 흐름을 정형화한 모델이 필요하다. ReAct(Reasoning + Acting) 프레임워크는 이 흐름을 Thought(사고) → Action(행동) → Observation(관찰) 세 단계의 반복 사이클로 포착한다.[^web_03]

**그림 6-2. Thought→Action→Observation 반복 사이클과 컴포넌트 호출 순서**

```
         ┌─────────────────────────────────────────┐
         │                                         │
         ▼                                         │
   ┌───────────┐    Planning + Model    
   │  Thought  │◀──────────────────────────────┐   │
   │  (사고)   │  목표 분해, 다음 행동 결정     │   │
   └─────┬─────┘                               │   │
         │                                     │   │
         ▼                                     │   │
   ┌───────────┐    Tool Use                   │   │
   │  Action   │  외부 API 호출, 코드 실행      │   │
   │  (행동)   │                               │   │
   └─────┬─────┘                               │   │
         │                                     │   │
         ▼                                     │   │
   ┌───────────┐    Memory                     │   │
   │Observation│  결과를 컨텍스트에 저장        │   │
   │  (관찰)   │─────────────────────────────▶─┘   │
   └───────────┘                                   │
         │              목표 달성 시               │
         └─────────────────────────────────────────┘
                       최종 응답 생성
```

각 단계가 어떤 컴포넌트와 맞닿아 있는지 살펴보자. **Thought** 단계에서는 Model 레이어가 현재까지의 컨텍스트를 바탕으로 다음에 무엇을 해야 할지 추론한다. 이 추론이 Planning 레이어의 입력이 되어 구체적인 행동 계획으로 변환된다. **Action** 단계에서는 Tool Use 레이어가 외부 시스템을 호출한다. 검색 API를 실행하거나, 코드를 실행하거나, 데이터베이스에 질의한다. **Observation** 단계에서는 그 결과가 Memory 레이어에 저장되고, 다음 Thought 단계의 입력으로 흘러간다. 이 루프가 목표 달성 조건을 만족할 때까지 반복되며, 조건이 충족되면 Model 레이어가 최종 응답을 생성한다.[^web_03]

단순해 보이는 에이전트에서도 이 루프는 존재한다. 단일 Q&A 에이전트를 예로 들어 보자. 사용자의 질문을 받아 한 번에 답을 돌려주는 구조처럼 보이지만, 내부적으로는 다음과 같은 하위 단계가 있다.[^web_02]

- **컨텍스트 어셈블리**: 시스템 프롬프트와 사용자 메시지를 결합해 모델 입력을 구성한다. (Memory + Model)
- **추론 및 생성**: 모델이 입력을 받아 응답 토큰을 생성한다. (Model)
- **응답 파싱**: 생성된 텍스트를 구조화된 형식으로 변환하거나 후처리한다. (Tool Use의 경량 형태)

루프가 한 번만 돌고 끝날 뿐, 세 단계의 구조는 그대로다. Thought→Action→Observation이 한 번의 사이클로 완결되는 것이지, 루프가 없는 것이 아니다.

::: box 워크플로우와 에이전트의 경계 — Anthropic 가이드의 아키텍처 구분
Anthropic의 에이전트 설계 가이드는 시스템을 크게 두 유형으로 구분한다.[^web_02]

**워크플로우(Workflow)**: 실행 경로가 미리 정해진 시스템. 모델이 각 단계에서 판단을 내리지만, 어떤 단계를 거칠지는 개발자가 사전에 정의한다. 조건 분기도 명시적으로 코딩된다.

**에이전트(Agent)**: 실행 경로가 모델의 판단에 따라 동적으로 결정되는 시스템. 어떤 도구를 몇 번 호출할지, 언제 멈출지를 모델이 스스로 결정한다.

이 구분이 중요한 이유는 두 시스템에서 실패가 발생하는 위치가 다르기 때문이다. 워크플로우에서는 실행 경로 자체가 잘못 설계되었을 때 실패한다. 에이전트에서는 모델의 판단이 불충분할 때, 또는 루프가 종료 조건을 찾지 못할 때 실패한다. Anthropic 가이드는 "복잡도는 필요할 때만 추가하라"고 권고하는데, 이는 에이전트 형태가 항상 더 낫다는 뜻이 아니라 두 유형의 실패 경로가 다르다는 것을 인식하고 선택하라는 의미다.
:::

이처럼 Thought→Action→Observation 루프는 에이전트의 규모나 복잡도와 무관하게 내부 흐름의 기본 단위로 적용된다. 중요한 것은 이 루프가 각 컴포넌트를 어떤 계약(contract, 컴포넌트 간에 서로 기대하는 입출력의 형식과 조건)으로 연결하느냐에 있다. 컴포넌트를 분리하는 것만으로 충분하지 않다. 분리된 컴포넌트들이 어떤 인터페이스로 연결되는지, 그리고 한 컴포넌트의 실패가 다른 컴포넌트로 어떻게 전파되는지가 시스템의 실제 동작을 결정한다. 다음 절에서는 바로 이 연결의 문제를 다룬다.

## 6.2 컴포넌트 간 계약과 실패의 전파

앞에서 에이전트 내부를 Planning·Memory·Tool Use·Model 레이어로 분해하고, Thought→Action→Observation 루프가 어떻게 작동하는지 살펴보았다. 이제 그 컴포넌트들 사이를 잇는 연결선을 들여다볼 차례다. 컴포넌트를 나누는 것만으로는 충분하지 않다. 분리된 컴포넌트들이 어떤 약속으로 데이터를 주고받는지, 그리고 한 컴포넌트에서 발생한 문제가 나머지 컴포넌트로 어떻게 번져 나가는지를 이해해야 "원인은 모델에 있는가, 시스템에 있는가"라는 질문에 구조적으로 답할 수 있다.

---

### 6.2.1 컴포넌트 간 인터페이스 계약: 무엇이 명시되어야 하는가

컴포넌트 사이에서 데이터가 오가는 경계를 **인터페이스 계약(interface contract)** 이라고 부른다. 여기서 '계약'이란 법적 의미의 계약이 아니라, 두 컴포넌트가 서로에게 기대하는 입출력의 형식·타이밍·예외 처리 방식을 명시한 약속이다. 인터페이스 계약이 명시되어 있으면 한쪽이 기대와 다른 값을 내보냈을 때 즉시 감지할 수 있다. 명시되어 있지 않으면 문제가 발생해도 어느 쪽의 책임인지 구분하기 어렵다.[^web_05]

::: note 계약(contract)의 의미 — 인터페이스 명세와 런타임 기대치
소프트웨어 공학에서 '계약'은 컴포넌트 간에 사전에 합의된 명세(specification)를 뜻한다. 여기서 계약의 구성 요소는 세 가지다.

- **입력 명세(precondition)**: 이 컴포넌트가 받을 수 있는 데이터의 형식과 값의 범위
- **출력 명세(postcondition)**: 정상 처리 후 반환하는 데이터의 형식과 보장 범위
- **예외 명세(exception contract)**: 입력이 기대에 벗어났을 때 어떻게 처리하는가 — 오류를 반환하는가, 기본값을 쓰는가, 중단하는가

AI 에이전트에서 이 계약이 종종 암묵적으로 처리된다. 모델이 반환한 JSON을 파싱하는 코드가 특정 키의 존재를 가정하거나, 도구 호출 결과의 길이 제한을 상정하지 않는 식이다. 이런 암묵적 가정이 계약 위반의 씨앗이 된다.
:::

에이전트를 구성하는 각 컴포넌트 경계에서 어떤 계약이 필요한지 정리하면 다음과 같다.

**표 6-1. 컴포넌트별 입력·출력·예외 계약 요약**

| 컴포넌트 경계 | 입력 명세 | 출력 명세 | 예외 처리 |
|---|---|---|---|
| Model → Planning | 목표 설명, 현재 컨텍스트 (토큰 수 제한 내) | 실행 가능한 하위 태스크 목록, 실행 순서 | 파싱 불가 출력 시 재시도 또는 폴백 계획 |
| Planning → Tool Use | 도구 이름, 파라미터 (스키마 정의 내) | 도구 실행 결과 (지정 포맷) | 타임아웃, 오류 코드, 빈 결과 처리 방식 |
| Tool Use → Memory | 실행 결과 (직렬화 가능한 형식) | 저장 완료 확인, 검색 시 관련도 정렬 결과 | 저장 용량 초과, 검색 결과 없음 처리 |
| Memory → Model | 관련 컨텍스트 청크 (우선순위 정렬) | — (모델 입력으로 조립됨) | 검색 실패 시 빈 컨텍스트 vs. 중단 결정 |

3장에서 도구 호출을 다룰 때 언급했듯이, Tool Use 레이어는 에이전트의 시스템 경계를 넘어 외부 의존으로 향하는 지점이다.[^web_05] 외부 API가 반환하는 데이터 형식은 에이전트 내부 설계자가 통제할 수 없다. 따라서 Planning → Tool Use 경계에서의 계약은 특히 엄밀하게 정의되어야 한다. "도구가 예상과 다른 형식을 반환할 경우 어떻게 처리하는가"를 명시하지 않으면, 외부 API의 사소한 변경이 에이전트 전체를 중단시킬 수 있다.

2장에서 다룬 컨텍스트 역시 이 관점에서 재해석할 수 있다. 컨텍스트는 단순히 모델에게 전달하는 텍스트 덩어리가 아니라, Memory → Model 경계를 통해 흐르는 구조화된 데이터다. 어떤 정보를 얼마나 담아 보낼지, 우선순위는 어떻게 정하는지가 이 경계의 계약 내용이다. 계약이 느슨하면 컨텍스트가 너무 많거나 너무 적게 채워지고, 모델은 엉뚱한 전제 위에서 추론을 시작하게 된다.

---

### 6.2.2 실패 전파 메커니즘: 어디서 시작해서 어디까지 가는가

계약이 깨졌을 때 어떤 일이 생기는지를 이해하려면 실패가 단일 컴포넌트 안에서 머물지 않는다는 사실을 먼저 받아들여야 한다. 한 컴포넌트의 오작동은 그 출력에 의존하는 다음 컴포넌트로 흘러가고, 거기서 다시 다음 단계로 번져나간다. Zhu et al.의 분석은 이 전파 경로를 memory → reflection → planning → action → system 순서로 추적한다.[^web_08] 오류는 시작점에서 끝나지 않고, 파이프라인을 따라 증폭된다.

**그림 6-3. 오류 전파 경로 — 컴포넌트 단계별 실패 확산 구조**

```
  ┌──────────┐   잘못된 컨텍스트   ┌────────────┐   왜곡된 반영   ┌──────────┐
  │  Memory  │──────────────────▶│ Reflection │──────────────▶│ Planning │
  │(검색 오류)│                   │(오염된 평가)│               │(잘못된 계획)│
  └──────────┘                   └────────────┘               └────┬─────┘
                                                                   │ 잘못된 지시
                                                                   ▼
                                                             ┌──────────┐   최종 오류
                                                             │  Action  │──────────▶ 시스템 출력
                                                             │(오작동)  │
                                                             └──────────┘
```

전파 경로에서 주목할 점은 최종 출력에서 관찰되는 증상이 실제 오류 발생 지점과 멀리 떨어져 있을 수 있다는 것이다. Memory 레이어에서 잘못된 정보를 검색했다고 하자. 이 오염된 컨텍스트는 Reflection(이전 행동에 대한 평가와 갱신)을 거치면서 에이전트가 자신의 상황을 잘못 판단하게 만들고, 그 판단이 Planning에 반영되어 잘못된 계획이 생성되며, 최종적으로 Action 단계에서 엉뚱한 도구가 호출되거나 잘못된 파라미터가 전달된다. 개발자가 관찰하는 것은 마지막 Action의 오작동뿐이다.

MAST 프레임워크는 이런 실패를 분류하는 세 가지 범주를 제시한다.[^web_04]

- **시스템 설계 이슈**: 컴포넌트 간 계약 자체가 잘못 설계된 경우. 예를 들어 도구 호출 응답 포맷이 에이전트 내부 파서의 기대와 다르게 정의된 경우.
- **에이전트 간 불일치**: 다중 에이전트 환경에서 에이전트들이 서로 다른 전제를 가지고 동작하는 경우. 단일 에이전트 수준에서는 컴포넌트 간 상태 불일치로 해석할 수 있다.
- **태스크 검증 실패**: 특정 단계의 출력이 다음 단계의 입력 조건을 충족하는지 확인하지 않는 경우.

이 세 범주를 앞서 살펴본 컴포넌트 계약과 연결하면 실패 원인을 진단하는 구체적인 언어가 생긴다. 동일한 증상이 서로 다른 컴포넌트의 실패에서 비롯될 수 있다는 점이 핵심이다.

예를 들어 에이전트가 "항공편을 찾지 못했다"는 결과를 반환했다고 하자. 이 증상의 원인은 여러 위치에 있을 수 있다.

- Tool Use 레이어에서 API 호출 파라미터가 잘못 조립된 경우 → 시스템 설계 이슈
- Memory 레이어가 이미 검색된 결과를 잘못 저장해 Planning이 불필요한 재검색을 생략한 경우 → 상태 오염에 의한 전파
- Planning 레이어가 날짜 파싱 오류를 검증하지 않고 Tool Use에 전달한 경우 → 태스크 검증 실패

증상만 보고 진단하면 Tool Use 레이어의 API 연결을 수정하려 할 것이다. 그러나 원인이 Memory나 Planning에 있다면, API를 아무리 수정해도 같은 증상이 반복된다.

::: box Replit 사례: 모델 교체로 해결하려 했던 문제가 시스템 경계 설계 문제였던 경위
실배포 환경에서 이 진단 오류가 어떤 결과를 낳는지 보여주는 사례가 있다. Replit의 에이전트 개발 과정에서 나타난 실패 사례는 초기에 모델 성능 문제로 분류되었다.[^web_07] 더 강력한 모델로 교체하면 문제가 해결될 것이라는 가정이었다.

그러나 모델 교체 이후에도 같은 유형의 실패가 반복되었다. 추적 결과 문제는 컴포넌트 간 경계 설계에 있었다. 에이전트가 도구 호출 결과를 컨텍스트에 누적하는 방식이 명시되어 있지 않았고, 누적된 컨텍스트가 일정 크기를 넘어서면 초기 지시가 잘려나가는 상황이 반복되고 있었다. 이는 Memory → Model 경계의 계약 부재에서 비롯된 시스템 설계 이슈였다.

'models propose, architectures dispose'라는 명제가 실무에서 의미하는 바는 이것이다.[^web_05] 모델의 출력 품질과 시스템의 신뢰성은 별개의 문제다. 모델이 개별 추론을 잘 수행하더라도, 그 추론의 전제가 되는 컨텍스트가 오염되어 있거나 컴포넌트 간 계약이 지켜지지 않으면 시스템 전체는 신뢰할 수 없는 결과를 낸다.
:::

실패 전파를 이해하면 자연스럽게 하나의 질문이 뒤따른다. 전파 경로를 사후에 추적하려면 각 컴포넌트가 어떤 상태에 있었는지를 외부에서 볼 수 있어야 한다. 어느 컴포넌트가 무엇을 받고, 무엇을 내보냈는지를 기록하지 않으면, 발생한 실패의 경로를 소급해 재구성하는 것은 불가능에 가깝다. 이 필요성이 시스템 관찰 가능성(observability)을 요구하는 근거다. 다음 절에서는 에이전트 시스템의 경계를 어떻게 정의하고, 그 내부 상태를 어떻게 외부에서 관찰 가능하게 만들 것인지를 살펴본다.

## 6.3 시스템 경계와 관찰 가능성

앞 절에서는 실패가 하나의 컴포넌트에서 발생해 다른 컴포넌트로 전파되는 경로를 살펴보았다. 그런데 그 경로를 추적하는 작업 자체가 가능하려면 두 가지 전제가 충족되어야 한다. 첫째, 에이전트 시스템이 어디서 시작하고 어디서 끝나는지가 명확하게 정의되어 있어야 한다. 둘째, 시스템 내부에서 무슨 일이 일어났는지를 외부에서 볼 수 있어야 한다. 이 두 조건을 각각 시스템 경계(system boundary)와 관찰 가능성(observability)이라 부른다. 이 절에서는 두 개념을 순서대로 살펴보고, 이 둘이 왜 분리된 설계 문제가 아니라 하나의 쌍을 이루는 원칙인지를 정리한다.

---

### 6.3.1 시스템 경계 정의: 에이전트는 어디서 시작하고 어디서 끝나는가

시스템 경계란 에이전트 내부 컴포넌트와 외부 의존 요소를 가르는 선이다. 이 경계가 불명확하면 실패의 책임 소재가 모호해진다.

::: note 시스템 경계(system boundary)
에이전트 시스템이 직접 통제하는 컴포넌트(내부)와 그 바깥에 위치한 외부 서비스·저장소·사용자 인터페이스(외부) 사이의 구분선. 이 경계를 명시하는 것은 책임 소재 분석의 전제 조건이다.[^web_02]
:::

**그림 6-4. 에이전트 시스템 경계 — 내부 컴포넌트와 외부 의존의 구분**

```
┌──────────────────────────────────────────────────────┐
│                  에이전트 시스템 (내부)                 │
│                                                      │
│  ┌────────────┐   ┌────────────┐   ┌─────────────┐  │
│  │  Planning  │──▶│   Model    │──▶│   Memory    │  │
│  └────────────┘   └─────┬──────┘   └─────────────┘  │
│                         │                           │
│                   ┌─────▼──────┐                    │
│                   │  Tool Use  │                    │
│                   └─────┬──────┘                    │
└─────────────────────────┼────────────────────────────┘
          시스템 경계       │
        ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
                           │
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────▼─────┐   ┌──────▼──────┐  ┌─────▼──────┐
    │ 외부 API  │   │  데이터베이스  │  │  사용자 UI │
    └───────────┘   └─────────────┘  └────────────┘
```

이 경계가 무너지면 어떤 일이 생기는지 두 가지 시나리오로 살펴보자.

**시나리오 A — 외부 오류가 내부 오류로 잘못 분류되는 경우.** 항공편 검색 API가 타임아웃을 반환했다. 에이전트는 "항공편을 찾지 못했다"는 결과를 출력한다. 시스템 경계가 명시되어 있지 않으면 개발자는 이것을 에이전트의 Planning 또는 Tool Use 레이어 문제로 볼 가능성이 높다. 프롬프트를 수정하거나 도구 호출 로직을 수정하지만 문제는 재현된다. 원인은 외부 API 가용성에 있었는데, 경계가 흐릿하여 내부 컴포넌트를 반복 점검하는 데 시간을 낭비한 것이다.

**시나리오 B — 내부 상태 오염이 외부 입력 문제로 오진되는 경우.** 에이전트가 사용자의 이전 선호도를 무시하는 것처럼 보인다. Memory 레이어가 이전 세션의 잔여 상태를 그대로 유지하면서 새 세션의 입력을 덮어쓰고 있는 상황이다. 그러나 시스템 경계가 명시되어 있지 않으면 이것을 사용자 입력 파싱 문제로 오해할 수 있다. 외부 입력 처리 로직을 수정하지만 증상은 반복된다.

3장에서 다룬 도구 호출을 이 관점에서 다시 위치 지정하면, 도구 호출은 "시스템 경계를 넘는 외부 의존"이다. 에이전트가 API를 호출하는 순간, 제어권은 일시적으로 외부로 넘어간다. 외부 서비스의 응답 형식, 지연, 가용성은 에이전트 내부에서 통제할 수 없다. 이 사실을 설계 시점에 인식하지 못하면, 외부 의존의 실패를 내부 컴포넌트의 문제로 혼동하게 된다.[^web_02]

Anthropic의 운영 가이드는 에이전트 배포 전에 시스템 경계를 명시적으로 문서화할 것을 권장한다.[^web_02] 이는 단순한 아키텍처 기술이 아니라, 실패 대응 시 어디부터 점검할지를 사전에 합의해 두는 작업이다.

---

### 6.3.2 관찰 가능성의 설계: 내부 상태를 외부에서 읽는 구조

시스템 경계를 정의했다면, 다음 질문은 경계 내부에서 무슨 일이 일어나는지를 어떻게 알 수 있는가다. 관찰 가능성은 시스템 내부 상태를 외부에서 읽을 수 있도록 만드는 설계 속성이다. 이것은 성능 최적화를 위한 부가 기능이 아니라, 시스템이 의도대로 작동하고 있는지 확인하는 최소 조건이다.[^web_05]

::: box 관찰 가능성이 없었을 때의 디버깅 — 컴포넌트 간 갈림점을 찾지 못하는 상황

같은 입력에서 두 번의 실행이 서로 다른 출력을 냈다고 하자. 내부 상태를 전혀 기록하지 않은 시스템에서는 다음과 같은 질문에 답할 수 없다.

- Memory 레이어가 두 실행에서 동일한 컨텍스트를 조립했는가?
- Planning 레이어가 두 번 모두 같은 하위 태스크 순서를 생성했는가?
- Tool Use 레이어의 API 응답이 두 실행 간에 달랐는가?

이 질문 중 하나에라도 답하지 못하면 갈림점을 특정할 수 없다. 개발자가 할 수 있는 것은 전체 시스템을 재실행하며 운에 맡기는 것뿐이다. 결과의 비결정성(non-determinism)이 모델 때문인지, 외부 API 응답 변동 때문인지, 메모리 상태 차이 때문인지를 구분하지 못하면 수정의 방향도 없다.
:::

관찰 가능성을 구현하는 기술적 수단으로는 크게 세 가지가 있다. **로그(log)**는 특정 시점의 이벤트와 상태를 텍스트로 기록한다. **트레이스(trace)**는 요청 하나가 시스템을 통과하는 전체 경로를 추적한다. **스팬(span)**은 트레이스 내에서 개별 컴포넌트의 시작과 종료 시점 및 내부 속성을 기록하는 단위다. 세 가지 중 어느 하나만 있어도 없는 것보다 낫지만, 컴포넌트 경계를 따라 전파되는 실패를 추적하려면 트레이스와 스팬의 조합이 실질적으로 필요하다.

이 구조를 Böckeler의 harness 모델에서 나온 어휘로 정리하면 더 명확해진다. 에이전트에게 행동 방향을 공급하는 구조를 **guides(피드포워드)**, 에이전트의 행동 결과를 수집하는 구조를 **sensors(피드백)**라 구분한다.[^web_06] 시스템 프롬프트, 도구 스펙, 메모리에서 끌어온 컨텍스트는 guides에 해당한다. 로그, 트레이스, 스팬은 sensors에 해당한다. 이 구분이 유용한 이유는, 시스템에 문제가 생겼을 때 guides 쪽을 고쳐야 하는지 sensors 쪽을 보강해야 하는지를 먼저 분리해서 생각하게 만들기 때문이다.

**표 6-3. guides와 sensors의 구분**

| 방향 | 역할 | 구체적 수단 |
|---|---|---|
| guides (피드포워드) | 에이전트에게 행동 방향 공급 | 시스템 프롬프트, 도구 스펙, 메모리 컨텍스트 |
| sensors (피드백) | 에이전트 행동 결과 수집 | 로그, 트레이스, 스팬, 오류 이벤트 |

폴트 격리(fault isolation)와 관찰 가능성의 관계도 짚어 둘 필요가 있다. 폴트 격리란 한 컴포넌트의 실패가 다른 컴포넌트로 확산되지 않도록 경계를 강화하는 설계 방식이다. 그런데 폴트 격리가 얼마나 잘 작동하고 있는지는 관찰 가능성이 없으면 확인 자체가 불가능하다.[^web_05] MAST의 태스크 검증 범주, 즉 특정 단계의 출력이 다음 단계의 입력 조건을 충족하는지 확인하지 않아 발생하는 실패가 드러나지 않는 경우도 마찬가지다. 중간 단계의 출력을 기록하지 않으면, 태스크 검증 실패는 최종 출력의 이상 증상으로만 나타나고 그 원인은 보이지 않는다.[^web_04]

::: tip 관찰 가능성 최소 구현 — 어떤 지점에서 무엇을 기록해야 하는가
처음부터 완전한 관찰 가능성 인프라를 갖추기 어렵다면, 다음 세 지점을 최우선 기록 대상으로 삼자.

1. **컴포넌트 경계 진입·이탈 시**: 각 컴포넌트가 무엇을 입력으로 받았고 무엇을 출력으로 냈는지를 기록한다. 이것만 있어도 갈림점 탐색이 가능하다.
2. **외부 의존 호출 시**: 도구 호출이나 외부 API 요청의 파라미터와 응답을 기록한다. 내부 오류와 외부 오류를 분리하는 데 필수적이다.
3. **상태 변경 시**: Memory 레이어에 무언가가 쓰이거나 읽힐 때 기록한다. 상태 오염에 의한 전파를 추적하는 기반이 된다.
:::

시스템 경계와 관찰 가능성, 이 두 설계 원칙은 독립적으로 존재하지 않는다. 경계를 정의해야 어디에 sensors를 배치할지가 결정되고, sensors가 제자리에 있어야 경계가 제대로 작동하는지 확인할 수 있다. 어느 한쪽만 갖추어진 시스템은 절반만 통제 가능한 시스템이다.

이제 관찰 가능성이 확립되었다고 가정해 보자. 그러면 자연스럽게 다음 질문이 생긴다. 이 관찰 장치, 즉 sensors 전체를 아우르는 구조를 시스템 내부에 두어야 하는가, 외부에 두어야 하는가. 이 질문이 평가 하네스의 설계 위치를 결정하는 핵심 논거로 이어진다. 다음 절에서는 왜 평가 하네스가 시스템 외부에 놓여야 하는지를 구체적으로 살펴본다.

## 6.4 평가 하네스는 왜 시스템 외부에 있어야 하는가

6.3에서는 시스템 내부 상태를 외부에서 읽을 수 있도록 만드는 관찰 가능성을 살펴보았다. sensors가 제자리에 배치되어야 컴포넌트 경계를 따라 전파되는 실패를 추적할 수 있다는 것이 그 핵심이었다. 그런데 여기서 한 가지 질문이 남는다. 이 sensors, 관찰 장치 전체를 아우르는 구조를 누가, 어디서 운영해야 하는가. 이 절에서는 그 구조의 이름인 **평가 하네스(harness)**를 정의하고, 그것이 왜 시스템 바깥에 위치해야 하는지를 구조적으로 살펴본다.

::: note 평가 하네스(harness)
반복 가능하게 실행과 평가를 수행하는 외부 구조. 단순한 테스트 코드가 아니라, 시스템 전체를 반복 실행하고 그 결과를 관찰·판단하는 구조 전체를 가리킨다.
:::

---

### 6.4.1 하네스의 두 기능: 피드포워드와 피드백 채널

Böckeler는 에이전트를 'Agent = Model + Harness'로 표현한다.[^web_06] 이 공식에서 모델이 추론의 주체라면, 하네스는 모델이 올바른 방향으로 작동하도록 입력을 공급하고 그 결과를 수집하는 구조 전체다. 하네스는 두 채널로 나뉜다.

첫 번째는 **guides(피드포워드)** 채널이다. 에이전트에게 행동 지침을 주입하는 모든 수단이 여기에 해당한다. 시스템 프롬프트, 도구 스펙(tool specification), 제약 조건, 그리고 평가 시나리오에서 에이전트에게 제시하는 초기 입력이 guides를 통해 흘러들어간다. 에이전트가 어떤 태스크를 수행해야 하는지, 어떤 도구를 사용할 수 있는지, 어떤 제약 안에서 행동해야 하는지를 외부에서 규정하는 것이 guides의 역할이다.

두 번째는 **sensors(피드백)** 채널이다. 에이전트가 실제로 무엇을 했는지를 기록하고 수집하는 구조다. 실행 로그, 출력 기록, 중간 판단 결과, 도구 호출 이력이 여기에 해당한다. 6.3절에서 다룬 로그·트레이스·스팬은 모두 sensors의 구체적 구현이다.

**그림 6-5. 'Agent = Model + Harness' — 하네스의 위치와 두 기능 채널**

```
┌──────────────────────────────────────────────────────────┐
│                    평가 하네스 (Harness)                   │
│                                                          │
│  guides (피드포워드)          sensors (피드백)             │
│  ┌──────────────────┐        ┌──────────────────┐        │
│  │ 시스템 프롬프트   │        │ 실행 로그         │        │
│  │ 도구 스펙        │        │ 출력 기록         │        │
│  │ 제약 조건        │        │ 판단 결과         │        │
│  └────────┬─────────┘        └────────▲──────────┘        │
│           │                          │                   │
└───────────┼──────────────────────────┼───────────────────┘
            │                          │
            ▼                          │
┌──────────────────────────────────────────────────────────┐
│                       에이전트 (Agent)                    │
│                                                          │
│   Planning ──▶ Model Layer ──▶ Tool Use ──▶ Memory       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

이 두 채널이 시스템 외부에서 연결될 때 평가의 독립성이 확보된다. 만약 guides와 sensors가 시스템 내부 어딘가에 묶여 있다면, 에이전트의 내부 상태가 평가 입력이나 관찰 결과에 개입할 수 있다. 어떤 실행에서 Memory 레이어에 이전 결과가 남아 있다면, 다음 실행의 guides가 그 상태를 모르는 채로 동일한 입력을 주입하더라도 결과는 달라진다. 평가가 시스템 상태에 오염되는 것이다.

---

### 6.4.2 하네스 위치가 평가 신뢰도에 미치는 영향

하네스가 시스템 내부에 결합되면 평가 결과의 신뢰도가 구조적으로 떨어진다. 이 문제를 **내부 결합 하네스(internal coupling harness)**의 실패 경로로 구체화해 보자.

동일한 입력을 열 번 실행했더니 결과가 세 가지 서로 다른 답으로 갈렸다고 하자. 원인을 찾으려 할 때 내부 결합 하네스 상황에서는 다음과 같은 일이 벌어진다.

- 각 실행이 Memory 레이어에 남긴 상태를 초기화하지 않으면, 두 번째 실행부터는 첫 번째 실행의 잔여 상태를 이어받아 실행된다.
- sensors가 에이전트 내부 모듈과 같은 프로세스에서 동작하면, 에이전트가 오류로 종료될 때 sensors도 함께 멈추어 정작 필요한 오류 직전의 상태 기록이 남지 않는다.
- guides가 시스템 내부 설정 파일에 묶여 있으면, 실험 조건을 바꾸기 위해 시스템 자체를 재배포해야 한다.

반면 외부에 독립된 하네스가 있다면 이 세 문제는 다르게 전개된다. 각 실행 전에 하네스가 Memory 레이어를 지정된 초기 상태로 리셋하고, 에이전트가 비정상 종료되어도 하네스는 그 이벤트를 기록하고 다음 실행을 계속한다. 실험 조건을 바꾸는 것은 시스템 재배포가 아니라 하네스 설정 변경으로 충분하다.

**표 6-2. 하네스 위치(내부/외부)에 따른 평가 속성 비교**

| 평가 속성 | 내부 결합 하네스 | 외부 독립 하네스 |
|---|---|---|
| **독립성** | 시스템 상태가 평가 결과에 개입 가능 | 시스템 상태와 평가 결과가 분리됨 |
| **반복 가능성** | 동일 입력이라도 잔여 상태에 따라 결과 달라짐 | 초기 상태 리셋 후 동일 조건 재현 가능 |
| **격리성** | 에이전트 오류 시 관찰 기록도 손실 위험 | 에이전트 오류와 하네스 기록이 분리됨 |
| **실험 조건 변경** | 시스템 재배포 또는 내부 코드 수정 필요 | 하네스 설정 변경으로 충분 |

MAST의 태스크 검증 범주, 즉 특정 단계의 출력이 다음 단계의 입력 조건을 충족하는지 확인하지 않아 발생하는 실패는 외부 하네스 없이는 체계적으로 검출하기 어렵다.[^web_04] 태스크 검증 실패는 최종 출력에서 이상 증상으로만 나타나기 때문에, 중간 단계의 출력을 외부에서 독립적으로 기록하고 판단하는 구조가 없으면 어느 단계에서 검증이 깨졌는지 역추적하지 못한다. 폴트 격리가 작동하고 있는지 확인하는 것도 마찬가지다. 외부에서 반복 실행하며 격리 경계가 유지되는지를 체계적으로 관찰해야 격리 설계가 실제로 의도대로 작동하는지 알 수 있다.[^web_05]

5장에서 다룬 평가 방법론(단일 평가자, 모델 기반 평가, 참조 데이터와의 비교 등)이 일관되게 적용되려면 외부 하네스 구조가 전제되어야 한다. 평가 방법론은 "무엇을 어떻게 측정하는가"를 정의하지만, 그 측정이 매번 동일한 조건에서 이루어지도록 보장하는 것은 하네스의 몫이다. 하네스 없이 평가 방법론만 있다면, 방법론이 훌륭하더라도 결과의 비교 가능성이 담보되지 않는다.

::: box harness ≠ 테스트 코드 — 테스트 프레임워크와 평가 하네스의 구조적 차이

"하네스를 만들어야 한다"는 말을 들으면 pytest나 unittest 같은 테스트 프레임워크를 떠올리기 쉽다. 그러나 평가 하네스는 테스트 코드의 상위 개념이다.

테스트 코드는 특정 함수나 모듈이 주어진 입력에 대해 기대 출력을 반환하는지 확인한다. 입력과 기대값이 고정되어 있고, 실행은 단발적이다.

평가 하네스는 다르다. 에이전트 시스템 전체를 반복적으로 실행하고, 각 실행의 초기 상태를 제어하며, 실행 중 모든 채널(guides·sensors)을 관리하고, 결과를 수집·판단하는 구조다. 테스트 코드가 하네스의 일부로 포함될 수는 있지만, 하네스는 테스트 코드보다 훨씬 넓은 범위를 아우른다.

단순한 비유로 정리하면: 테스트 코드가 '악기 하나를 조율하는 행위'라면, 평가 하네스는 '오케스트라 리허설 전체를 설계하고 운영하는 구조'에 가깝다.
:::

::: tip 하네스 외부화의 최소 조건
시스템에 외부 하네스를 붙이기 위해 처음부터 완전한 구조를 갖출 필요는 없다. 다음 세 가지 인터페이스를 열어 두는 것이 최소 조건이다.

1. **초기 상태 주입 인터페이스**: 각 실행 전에 Memory 레이어 및 세션 상태를 외부에서 지정할 수 있어야 한다. 이것이 없으면 반복 가능성을 보장할 수 없다.
2. **실행 결과 수집 인터페이스**: 최종 출력뿐 아니라 중간 단계 출력과 도구 호출 이력을 외부에서 수집할 수 있어야 한다. 이것이 없으면 태스크 검증 실패를 역추적하지 못한다.
3. **guides 주입 인터페이스**: 시스템 프롬프트와 도구 스펙을 외부에서 교체할 수 있어야 한다. 이것이 없으면 실험 조건을 바꿀 때마다 시스템 코드를 수정해야 한다.
:::

하네스가 시스템 외부에 위치해야 한다는 원칙은 추상적인 설계 철학이 아니다. 이것은 평가 결과를 신뢰할 수 있는가, 그리고 문제가 생겼을 때 원인을 특정할 수 있는가라는 실용적 질문에 대한 구조적 답변이다.

이제 하네스의 위치가 확정되었다면, 다음 질문은 자연스럽게 이어진다. 경계를 명확히 하고, 관찰 가능성을 갖추고, 평가 하네스를 외부에 두는 방식으로 시스템을 설계하면, 실제 컴포넌트 교체나 확장 결정을 내릴 때 무엇이 달라지는가. 그 판단의 차이를 다음 절에서 구체적으로 들여다보자.

## 6.5 시스템으로 설계할 때 달라지는 판단들

6.4절에서는 평가 하네스가 왜 시스템 외부에 위치해야 하는지, 그리고 그 위치가 평가 신뢰도에 어떤 구조적 차이를 만드는지 살펴보았다. 이제 6.1절부터 6.4절까지 다루어 온 분해·계약·관찰·평가 하네스라는 네 가지 시스템 관점이 실제 설계 판단에서 어떻게 쓰이는지로 수렴해 보자. 이 절은 그 논증들이 만나는 지점이다.

시스템 관점으로 설계한다는 것이 막연하게 들릴 수 있다. 그러나 이것은 실제로 두 가지 매우 구체적인 판단 능력으로 나타난다. 하나는 컴포넌트를 교체하거나 확장할 때 변경의 영향 범위를 미리 읽는 능력이고, 다른 하나는 복잡성을 추가해야 할 시점과 단순함을 유지해야 할 시점을 구분하는 능력이다. 두 판단 모두 시스템을 분해하지 않으면 내릴 수 없다.

---

### 6.5.1 컴포넌트 교체 가능성: 무엇을 고정하고 무엇을 바꾸는가

시스템 관점이 가장 즉각적으로 효용을 드러내는 순간은 "이 모델을 다른 것으로 바꾸고 싶다"거나 "새 API를 도구로 연결하고 싶다"는 요청이 들어올 때다. 이 요청을 받았을 때 "코드 어디를 건드려야 하는가"라는 질문 대신 "어느 계약이 변하고, 어느 계약은 고정되어야 하는가"를 먼저 묻는 것이 시스템 관점 설계의 실천 방식이다.

#### 모델 교체 시나리오

GPT-4o 계열 모델을 Claude 계열로 바꾼다고 가정하자. 시스템 관점으로 설계된 에이전트에서 변하는 것은 Model 레이어의 API 호출 방식과 응답 포맷 계약뿐이다. Memory 레이어가 저장하고 조회하는 방식, Tool Use 레이어의 도구 스펙, 평가 하네스(harness)가 시스템에 주입하는 초기 상태와 실험 조건 — 이 모두는 고정된 채 남아 있어야 한다.

반대로 시스템 경계가 명확하지 않은 구현에서는 모델 교체가 연쇄 수정을 일으킨다. 모델 응답을 직접 파싱하던 Tool Use 로직이 모델 교체와 함께 깨지고, Memory에 저장하던 포맷이 달라지고, 평가 스크립트가 새 응답 스키마를 인식하지 못한다. 컴포넌트 경계가 없으니 변경이 어디까지 퍼지는지 사전에 알 수 없다.

Lin & Zhang이 제시하는 'models propose, architectures dispose'라는 명제는 이 지점에서 실용적 의미를 갖는다.[^web_05] 모델이 제안하는 것과, 시스템이 그 제안을 어떻게 처리하는가는 별개의 책임 영역이다. 모델 품질이 올라간다고 시스템이 자동으로 더 안정적으로 동작하지 않는 이유가 여기에 있다. 시스템 설계가 모델의 출력을 신뢰 가능한 방식으로 처리하도록 구성되어 있지 않으면, 더 좋은 모델을 연결해도 전체 결과의 신뢰성은 올라가지 않는다.

#### 도구 추가 시나리오

새로운 외부 결제 API를 도구로 추가한다고 하자. 여기서 시스템 관점으로 설계한다는 것은 코드를 먼저 작성하는 것이 아니라, 시스템 경계를 먼저 재정의하는 것을 의미한다. 이 도구가 실패하면 어떤 실패 경로가 새로 열리는가? 결제 API 타임아웃은 Planning 레이어의 재시도 로직에 어떤 영향을 주는가? 이 도구의 출력이 Memory에 저장되는 방식은 기존 도구들과 계약이 일치하는가?

Anthropic 가이드는 도구가 추가될수록 시스템 복잡성이 증가하며, 그 복잡성은 반드시 정당한 필요에 의해서만 도입해야 한다고 강조한다.[^web_02] 도구를 추가하면 실패 경로도 추가된다. 새로운 실패 경로를 추적하지 않은 채 도구만 붙이는 것은 시스템을 설계하는 것이 아니라 시스템을 확대하는 것이다.

**표 6-3. 컴포넌트 교체 시나리오별 변경 범위**

| 교체·추가 대상 | 변하는 것 | 고정되어야 하는 것 |
|---|---|---|
| **모델 교체** | API 호출 방식, 응답 포맷 계약 | Memory 조회·저장 방식, 도구 스펙, 하네스 구조 |
| **도구 추가** | Tool Use 레이어 인터페이스 계약, 실패 경로 목록 | Planning 재시도 로직 계약, Memory 저장 포맷, 평가 하네스 실험 조건 |
| **메모리 교체** | Memory 레이어 내부 저장 구현 | Memory 조회 인터페이스 계약, Model·Tool Use와의 연결 방식 |

시스템 관점 설계가 "미래의 변경에 대비하기 위한 선투자"처럼 들릴 수 있다. 그러나 이 표가 보여주는 것은 다르다. 인터페이스 계약이 명확할 때 현재의 변경 결정이 즉시 명확해진다는 점이다. "이 변경이 어디까지 전파되는가?"라는 질문에 계약 단위로 답할 수 있으면, 변경을 승인할지 보류할지 판단하는 데 드는 비용이 줄어든다.

::: tip 교체 가능성 설계의 실용적 기준 — 인터페이스를 먼저 쓰고 구현을 나중에 채우기

컴포넌트의 교체 가능성을 확보하는 가장 직접적인 방법은 구현보다 인터페이스를 먼저 정의하는 것이다. 다음 순서를 참고하자.

1. **이 컴포넌트가 받아야 하는 입력과 돌려주어야 하는 출력을 먼저 적는다.** 구현 방식은 이 계약이 확정된 이후에 결정한다.
2. **이 컴포넌트가 실패했을 때 시스템이 받아야 할 신호를 정의한다.** 예외 유형, 타임아웃 한계, 재시도 가능 여부를 명시한다.
3. **계약을 만족하는 가장 단순한 구현으로 시작한다.** 교체 가능성은 처음부터 정교한 구현을 요구하지 않는다. 계약이 고정되어 있으면 나중에 구현을 교체해도 나머지 시스템은 영향을 받지 않는다.
:::

### 6.5.2 시스템 경계의 확장: 멀티 에이전트와 운영 환경으로의 열린 질문

6.1절부터 6.5.1절까지는 단일 에이전트를 시스템으로 읽는 방식을 중심으로 논의를 전개했다. 그러나 이 관점은 두 방향으로 자연스럽게 확장된다. 하나는 에이전트가 복수가 될 때, 다른 하나는 설계된 시스템이 실제 운영 환경에 올라갈 때다.

#### 멀티 에이전트 방향

에이전트가 복수가 되면 시스템 경계는 단일 에이전트의 경계에서 에이전트 간 경계로 확장된다. 오케스트레이터(orchestrator) 에이전트가 서브에이전트에게 태스크를 위임할 때, 그 위임은 컴포넌트 간 계약과 동일한 구조적 성격을 갖는다. 서브에이전트는 어떤 입력을 받아 어떤 출력을 돌려주는가? 서브에이전트가 실패했을 때 오케스트레이터는 어떤 신호를 받는가?

Lilian Weng의 프레임은 멀티 에이전트 구성에서도 Planning·Memory·Tool Use 컴포넌트 분류가 각 에이전트 수준과 시스템 전체 수준에서 동시에 적용된다는 점을 보여준다.[^web_01] 단일 에이전트에서 확립한 시스템 경계 사고가 없으면, 에이전트를 추가하는 것이 시스템을 확장하는 것인지 복잡성을 증가시키는 것인지 구분하지 못한다.

Anthropic 가이드는 여기서 유용한 판단 기준을 제공한다. 단순 워크플로우(워크플로우는 실행 흐름이 미리 정해진 구조를 가리킨다)로 충분한 태스크에 에이전트 구조를 도입하면 불필요한 시스템 의존이 쌓인다.[^web_02] 멀티 에이전트 구성도 마찬가지다. 에이전트를 추가하는 결정은 "이 태스크가 단일 에이전트로 해결할 수 없는가"가 아니라 "이 추가가 시스템 경계를 명확하게 재정의할 수 있는가"로 판단해야 한다.

#### 운영 환경 방향

운영 환경에 에이전트 시스템을 올리면 관찰 가능성(observability) 요구사항이 달라진다. 개발 환경에서는 개발자가 직접 실행 흐름을 따라가며 확인할 수 있지만, 운영 환경에서는 그 흐름이 자동으로 기록되고 이상 시 알림이 가야 한다. Böckeler가 설명하는 sensors(피드백 채널)의 역할이 운영 환경에서 더 중요해지는 이유다.[^web_06] 개발 단계에서 관찰 가능성 인터페이스를 열어 두지 않으면, 운영 단계에서 이를 추가하기 위해 시스템 내부를 다시 수정해야 한다.

::: box 이 장의 핵심 질문 재방문 — "원인은 모델에 있는가, 시스템에 있는가?"

이 장은 "에이전트가 잘못된 결과를 냈을 때, 그 원인은 모델에 있는가, 시스템에 있는가?"라는 질문으로 시작했다. 이제 이 질문에 좀 더 구조적으로 답할 수 있다.

원인을 모델과 시스템으로 구분해서 추적하려면 먼저 시스템이 분해되어 있어야 한다. 컴포넌트가 나뉘지 않으면 실패가 어느 경계에서 발생했는지 특정할 수 없고, 인터페이스 계약이 없으면 모델 출력이 시스템의 기대와 어긋났는지 판단할 기준이 없다. 평가 하네스가 외부에 없으면 동일한 조건으로 재현해서 확인할 수도 없다.

이 장이 제시한 분해·계약·관찰·평가 하네스 네 가지는 결국 이 질문에 답할 수 있는 조건을 갖추는 과정이다. 원인이 모델에 있는지 시스템에 있는지를 알 수 있다는 것 자체가 시스템 관점 설계의 성과다.
:::

**그림 6-6. 이 장 전체의 논증 구조 — 분해에서 판단으로**

```
┌─────────────────────────────────────────────────────────┐
│               에이전트를 시스템으로 읽기                     │
│                                                         │
│  6.1 컴포넌트 분해           → 책임 범위를 분리한다          │
│  6.2 컴포넌트 간 계약        → 의존의 방향을 명확히 한다      │
│  6.3 실패 경로 추적          → 어디서 깨지는지 추적한다       │
│  6.4 평가 하네스 외부화      → 관찰의 독립성을 확보한다       │
│                │                                        │
│                ▼                                        │
│  6.5 설계 판단의 변화        → 교체 영향 범위를 읽는다        │
│                │                                        │
│        ┌───────┴────────┐                               │
│        ▼                ▼                               │
│  멀티 에이전트로 확장    운영 환경으로 이동                   │
│  (시스템 경계 재정의)    (관찰 가능성 요구 변화)              │
└─────────────────────────────────────────────────────────┘
```

이 장이 확립한 시스템 관점은 멀티 에이전트 구성과 운영 환경이라는 두 방향 모두의 공통 전제가 된다. 에이전트를 몇 개로 늘리든, 어떤 인프라에 올리든, 단일 에이전트를 시스템으로 읽는 습관이 없으면 그 확장은 복잡성을 축적하는 방향으로 흐를 가능성이 높다.

복잡성은 설계의 목표가 아니다. 시스템 관점은 복잡한 것을 만드는 도구가 아니라, 필요한 복잡성과 불필요한 복잡성을 구분하는 근거가 된다. 단순한 워크플로우로 충분한 상황에서 에이전트 구조를 도입하지 않을 판단도, 새로운 도구 추가가 시스템 경계를 어떻게 바꾸는지를 먼저 확인하는 습관도, 모두 이 관점에서 나온다.

다음 장에서는 에이전트가 복수가 될 때 시스템 경계가 어떻게 달라지는지를 본격적으로 다룬다. 이 장에서 정착시킨 컴포넌트 분해와 계약 사고가 멀티 에이전트 구성에서 어떻게 적용되고, 어떤 새로운 질문을 열어 놓는지 살펴볼 것이다.
