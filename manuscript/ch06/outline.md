# 6장. AI 에이전트는 단순 프로그램이 아니라 시스템이다

---

## 장 전체 Argument Flow

단일 모델 호출처럼 보이는 에이전트도, 그 안을 들여다보면 모델·컨텍스트·도구·메모리가 계약을 맺고 연결된 컴포넌트들의 집합이다. 이 장은 "에이전트가 잘못된 결과를 냈을 때 원인은 모델에 있는가, 시스템에 있는가?"라는 질문을 중심으로, 에이전트를 시스템으로 분해하는 어휘와 프레임을 쌓아 올린다. 먼저 에이전트 내부 흐름을 컴포넌트 단위로 해부하여 각각의 책임 범위를 정의하고(6.1), 컴포넌트들이 어떤 계약으로 연결되며 그 계약이 깨질 때 실패가 어떻게 전파되는지를 추적한다(6.2). 이어서 실패를 사전에 막는 것이 아니라 설계 안에서 격리하고 드러낼 수 있어야 한다는 관찰 가능성 원칙을 다루고(6.3), 그 연장선에서 평가 하네스가 시스템 외부에 위치해야 하는 구조적 이유를 제시한다(6.4). 마지막으로 이 시스템 관점이 컴포넌트 교체·확장 시 어디가 변하고 어디가 고정되어야 하는지에 대한 판단 기준을 제공하고, 멀티 에이전트와 운영 환경으로의 확장 방향을 열어 두며 마무리한다(6.5).

### Section 6.1 에이전트 내부를 컴포넌트로 읽기

#### 목적
독자가 "에이전트 = 모델 호출 하나"라는 평면적 인식에서 벗어나, 에이전트를 Planning·Memory·Tool Use 세 컴포넌트와 그것을 조율하는 모델 레이어로 분해하는 어휘를 획득하게 한다. ch01–ch05에서 개별적으로 소개된 개념들이 이 분해 구조 위에 재배치된다는 것을 보여 주는 것이 이 절의 주임무다.

#### 사용 Source
- `web_01` (Lilian Weng): Planning·Memory·Tool Use 세 컴포넌트 분류 및 도입 어휘
- `web_02` (Anthropic 가이드): 워크플로우와 에이전트의 아키텍처적 구분, '단순한 에이전트도 시스템이다' 근거
- `web_03` (ReAct): Thought→Action→Observation 반복 사이클로 내부 흐름 시각화

#### 포함 예시
- **진단용 시나리오**: 사용자가 "여행 일정을 짜 줘"라고 요청했을 때 응답이 나오기까지 내부에서 어떤 흐름이 일어나는지 — 계획(Planning), 외부 검색(Tool Use), 이전 선호도 참조(Memory), 최종 생성(모델)이 어떤 순서로 개입하는지를 단계별로 서술
- **"단순해 보이는" 에이전트 반례**: 단일 Q&A 에이전트도 컨텍스트 어셈블리·시스템 프롬프트 관리·응답 파싱이라는 하위 컴포넌트를 가진다는 점을 보여 주는 짧은 예

#### 핵심 요점
- 컴포넌트 분해는 복잡도가 아니라 책임 범위를 명확히 하기 위한 인식 도구다
- 단순한 에이전트도 시스템이다: 복잡도와 시스템 여부는 무관하다
- ch01–ch05의 각 개념(모델, 컨텍스트, 도구, 메모리, 평가)은 이 분해 구조 안에서 각기 다른 위치를 점한다

#### 앞뒤 연결
- **앞**: 장 도입부에서 핵심 질문("원인은 모델인가, 시스템인가?")을 제시하고, 그 질문에 답하려면 시스템을 분해하는 언어가 먼저 필요하다는 흐름으로 진입
- **뒤**: 컴포넌트 분해가 완료된 후, 이제 컴포넌트들이 어떻게 연결되는지(계약, 인터페이스)를 다루는 6.2로 이행

### Section 6.2 컴포넌트 간 계약과 실패의 전파

#### 목적
컴포넌트들이 어떤 계약(인터페이스)으로 연결되는지를 명시하고, 그 계약이 깨질 때 실패가 시스템 전체로 어떻게 전파되는지를 추적한다. "원인은 모델에 있는가, 시스템에 있는가?"라는 핵심 질문에 구조적으로 답하기 위한 분석 언어를 여기서 확립한다.

#### 사용 Source
- `web_05` (Lin & Zhang, 2512.09458): 'models propose, architectures dispose' 명제, 인터페이스 계약·폴트 격리·컴포넌트화 처방
- `web_04` (MAST, Cemri et al.): 실패 분류 — 시스템 설계 이슈·에이전트 간 불일치·태스크 검증 세 범주
- `web_08` (Zhu et al.): memory→reflection→planning→action→system 순서의 오류 전파 분석
- `web_07` (AI Agent Reliability): Replit 사례 등 실배포 실패 사례

#### 포함 예시
- **MAST 실패 분류 적용**: 도구 호출 응답 포맷이 예상과 다를 때(계약 위반), 메모리에서 잘못된 컨텍스트가 흘러들어올 때(상태 오염), 계획 단계가 잘못된 전제를 받을 때(전파) — 각각 어느 컴포넌트의 실패인지 분류해 보는 진단 예시
- **Replit 사례 환기**: 실배포에서 발생한 실패가 모델 성능이 아닌 시스템 경계 설계 문제였음을 간결하게 제시
- **오류 전파 경로 추적**: 메모리 컴포넌트의 검색 오류가 반영(reflection) → 계획(planning) → 행동(action)을 거쳐 최종 출력에 나타나기까지의 경로

#### 핵심 요점
- 컴포넌트 간 계약(인터페이스)이 명시되지 않으면 실패 원인의 위치를 특정할 수 없다
- 실패는 발생 지점에서 끝나지 않는다: 전파 경로를 모르면 증상만 보고 원인을 오진한다
- 'models propose, architectures dispose': 모델의 출력 품질과 시스템의 신뢰성은 별개의 문제다

#### 앞뒤 연결
- **앞**: 6.1에서 컴포넌트 분해를 마쳤으므로, 이제 그 컴포넌트들 사이의 연결선을 검토하는 흐름으로 자연스럽게 진입
- **뒤**: 실패 전파를 알면, 시스템 내부 상태를 "외부에서 볼 수 있어야" 한다는 관찰 가능성 필요성이 자연스럽게 도출됨(6.3)

#### 중제목 구조

##### #### 6.2.1 컴포넌트 간 인터페이스 계약: 무엇이 명시되어야 하는가
- 각 컴포넌트 경계에서 주고받는 데이터의 형식·타이밍·예외 처리를 계약으로 정의하는 방식
- 도구 호출이 "시스템 경계를 넘는 외부 의존"임을 명시(ch03 연결)
- 컨텍스트가 컴포넌트 간 인터페이스를 통해 흐르는 데이터임을 재해석(ch02 연결)

### Section 6.3 시스템 경계와 관찰 가능성

#### 목적
에이전트가 어디서 시작하고 어디서 끝나는지(시스템 경계)를 정의하지 않으면 책임 소재 분석이 불가능하다는 것을 먼저 확립하고, 그 경계 내부 상태를 외부에서 볼 수 있어야(관찰 가능성) 디버깅과 평가가 가능하다는 설계 원칙으로 이어간다. 이 절은 "시스템을 설계하는 것"과 "시스템을 통제하는 것"이 별개의 문제임을 명확히 하는 데 집중한다.

#### 사용 Source
- `web_06` (Böckeler, martinfowler.com): 'Agent = Model + Harness' 공식, guides(피드포워드)와 sensors(피드백) 구분
- `web_05` (Lin & Zhang): 폴트 격리와 관찰 가능성의 관계
- `web_02` (Anthropic 가이드): 운영 관점에서의 시스템 경계 정의 필요성
- `web_04` (MAST): 태스크 검증 범주 — 시스템 외부에서의 관찰이 없었을 때 드러나지 않는 실패 유형

#### 포함 예시
- **시스템 경계 정의 실패 시나리오**: 외부 API 오류가 에이전트 내부 오류로 잘못 분류되거나, 반대로 에이전트 내부 상태 오염이 외부 입력 문제로 오진되는 경우
- **관찰 가능성 부재의 결과**: 같은 입력에 다른 출력이 나왔을 때, 내부 상태를 기록하지 않으면 어떤 컴포넌트에서 갈라졌는지 사후 추적이 불가능한 상황
- **guides vs sensors 구분**: 에이전트에게 행동 방향을 주입하는 구조(시스템 프롬프트, 툴 스펙)와 에이전트의 행동 결과를 수집하는 구조(로그, 트레이스)를 설계 상 구분하는 예

#### 핵심 요점
- 시스템 경계를 정의하지 않으면 실패의 책임 소재가 모호해진다
- 관찰 가능성은 성능 최적화가 아니라 시스템이 작동하는지 확인하는 최소 조건이다
- guides(피드포워드)와 sensors(피드백)는 하네스의 두 기능적 방향을 구분하는 유용한 어휘다

#### 앞뒤 연결
- **앞**: 6.2에서 실패 전파 경로를 추적했지만, 그 추적 자체가 가능하려면 시스템 경계와 내부 상태 가시성이 전제되어야 한다는 논리적 필요에서 이 절이 시작
- **뒤**: 관찰 가능성이 확립되면, 평가 하네스가 왜 시스템 외부에 놓여야 하는지의 설계 근거가 완성됨(6.4)

#### 중제목 구조

##### #### 6.3.1 시스템 경계 정의: 에이전트는 어디서 시작하고 어디서 끝나는가

- 경계가 불명확할 때 발생하는 책임 소재 혼란 사례
- 도구 호출(ch03)을 "경계를 넘는 외부 의존"으로 재위치 지정

### Section 6.4 평가 하네스는 왜 시스템 외부에 있어야 하는가

#### 목적
concept_dictionary의 harness 정의("반복 가능하게 실행과 평가를 수행하는 외부 구조")를 이 장의 설계 원칙으로 확립한다. 하네스가 시스템 내부에 얽히면 평가 결과가 시스템 상태에 의존하게 되어 반복 가능성과 독립성이 무너진다는 구조적 근거를 제시한다. "harness = 테스트 코드"라는 축소된 인식을 반박하는 것이 이 절의 핵심 임무다.

#### 사용 Source
- `web_06` (Böckeler, martinfowler.com): 'Agent = Model + Harness' 공식, harness의 두 기능(guides/sensors)
- `web_05` (Lin & Zhang): 폴트 격리와 외부 평가 구조의 관계
- `web_04` (MAST): 태스크 검증이 시스템 외부에서 수행되어야 하는 이유
- ch05 평가 개념 연결 (이전 장 승인 요약 부재 시 brief 연결 지점 기준으로 서술)

#### 포함 예시
- **하네스 위치에 따른 평가 신뢰도 비교**: 하네스가 시스템 내부에 있을 때(시스템 상태가 평가에 영향)와 외부에 있을 때(독립적 반복 실행 가능)의 차이를 구조적으로 대비
- **harness ≠ 테스트 코드 반례**: 동일한 입력으로 열 번 실행했을 때 결과가 다르게 나오는 상황에서, 외부 하네스가 있을 때와 없을 때 디버깅 능력의 차이
- **guides/sensors 구분의 실제 활용**: 시스템 프롬프트·툴 스펙을 피드포워드 채널로, 로그·판단 결과를 피드백 채널로 분리하는 하네스 설계 스케치

#### 핵심 요점
- 하네스는 시스템을 반복 실행하고 관찰하는 구조 전체이며, 테스트 코드의 상위 개념이다
- 하네스가 시스템 외부에 있어야 평가의 독립성과 반복 가능성이 보장된다
- ch05에서 다룬 평가 방법론은 이 외부 하네스 구조가 있을 때 비로소 일관되게 적용될 수 있다

#### 앞뒤 연결
- **앞**: 6.3에서 관찰 가능성(sensors)이 설계 안에 있어야 한다고 했다면, 6.4는 그 관찰 결과를 수집·판단하는 구조(하네스) 자체가 시스템 바깥에 위치해야 한다는 논리적 연장
- **뒤**: 하네스의 위치가 확정되면, 이제 "시스템 관점에서 설계했을 때 컴포넌트 교체가 어떻게 국소화되는가"라는 실제 설계 판단 문제로 이행(6.5)

#### 중제목 구조

##### #### 6.4.1 하네스의 두 기능: 피드포워드와 피드백 채널
- guides(피드포워드): 에이전트에게 행동 지침을 주입하는 채널 — 시스템 프롬프트, 툴 스펙, 제약 조건
- sensors(피드백): 에이전트 행동 결과를 수집하는 채널 — 실행 로그, 출력 기록, 판단 결과
- 두 채널이 시스템 내부가 아닌 외부에서 연결될 때 평가의 독립성이 확보되는 이유

### Section 6.5 시스템으로 설계할 때 달라지는 판단들

#### 목적
앞 절들에서 확립한 시스템 관점을 실제 설계 판단으로 전환한다. 컴포넌트 교체·확장 시 어디가 변하고 어디가 고정되어야 하는지를 기준으로 정리하고, 이 장 전체의 논증을 통해 독자가 가져가야 할 설계 습관을 명시한다. 동시에 멀티 에이전트 구성과 운영 환경이라는 두 방향으로 열린 채 마무리한다.

#### 사용 Source
- `web_05` (Lin & Zhang): 컴포넌트화와 교체 가능성의 설계 처방
- `web_02` (Anthropic 가이드): '복잡성은 필요할 때만'이라는 설계 원칙, 워크플로우→에이전트 전환 기준
- `web_01` (Lilian Weng): 멀티 에이전트 방향으로의 확장 시 시스템 경계 재정의 필요성
- `web_06` (Böckeler): 운영 환경에서 하네스의 관찰 가능성 요구사항 변화

#### 포함 예시
- **모델 교체 시나리오**: 시스템 관점으로 설계된 에이전트에서 모델 컴포넌트를 교체할 때 변하는 것(모델 인터페이스 계약)과 변하지 않는 것(메모리·도구·하네스 구조)의 분리
- **도구 추가 시나리오**: 새로운 외부 API를 도구로 추가할 때, 시스템 경계를 먼저 재정의하고 실패 경로를 새로 추적해야 하는 이유
- **복잡성 기준 판단**: 단순 워크플로우로 충분한 태스크에 에이전트 구조를 도입했을 때 발생하는 불필요한 시스템 의존

#### 핵심 요점
- 시스템 관점으로 설계하면 컴포넌트 교체의 영향 범위가 명확해진다: "이 변경이 어디까지 전파되는가?"를 계약 단위로 추적할 수 있다
- "모델이 좋으면 시스템은 따라온다"가 아니라, 시스템 설계가 모델의 출력을 신뢰 가능하게 만든다
- 복잡성은 설계 결과물이지 목표가 아니다: 시스템 관점은 단순함을 유지하는 근거도 된다

#### 앞뒤 연결
- **앞**: 6.1–6.4를 통해 분해·계약·관찰·평가 하네스를 모두 다루었으므로, 이 절은 그 모든 것이 실제 설계 판단에서 어떻게 쓰이는지를 보여 주는 수렴 절
- **뒤**: 멀티 에이전트와 운영 환경이라는 두 방향의 열린 질문으로 후속 장 연결을 준비

#### 중제목 구조
