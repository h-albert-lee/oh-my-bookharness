# Argument Map

## 핵심 논제

- AI 프로젝트가 PoC에서는 성공하지만 운영에서 실패하는 것은 모델이 아니라 시스템 설계의 문제다.
- 전통적 TDD는 확률론적 AI 시스템에서 무력화된다. 평가 주도 개발(EDD)이 대안이다.
- AI 시스템의 성공은 모델 지능이 아니라 '실행-평가-교정'을 자동화하는 하네스(Harness) 아키텍처에 달려있다.
- RAG와 멀티 에이전트는 기술 트렌드가 아니라 정보 흐름과 역할을 분리하는 구조적 설계 선택이다.
- 안전성과 안정성은 선언적 개념이 아니라 설계 가능한 목표로 재정의되어야 한다.

## PART별 논증 흐름

### PART 1 (1-3장): 문제 정의
- 데모 환경 vs 운영 환경의 괴리 → 하네스 부재가 원인
- TDD의 Assert Equal이 확률론적 출력에서 붕괴 → EDD 도입 필요성
- 실패 원인을 모델/개인에 귀속하는 관점 비판 → 시스템 설계로 전환

### PART 2 (4-6장): 구조 이해
- LLM 기본 파이프라인의 구성 요소 정리
- RAG를 검색 기능이 아닌 정보 흐름 통제 아키텍처로 재해석
- 에이전트를 자동화 프로그램이 아닌 역할·책임 분리 시스템으로 설명

### PART 3 (7-9장): 설계 원칙
- '잘 동작하는 AI' ≠ '신뢰할 수 있는 AI 시스템' 구분
- 환각, 프롬프트 인젝션 등을 설계 가능한 문제로 재정의
- 가드레일과 정책 기반 설계로 안정성 확보

### PART 4 (10-12장): 평가 중심 개발
- 테스트만으로 AI 품질 보장 불가능 → 평가 필요성
- 평가 기준, 데이터, 자동 평가 구조 설계
- 운영 환경 평가 루프 (Human-in-the-loop 포함)

### PART 5 (13-15장): 설계 패턴
- 평가 가능한 시스템 패턴 (출력-판단 분리, 실패 가시화)
- Observability 패턴 (로그, 트레이싱, 메트릭)
- 도메인 특화 아키텍처 (금융·규제 환경 사례)
