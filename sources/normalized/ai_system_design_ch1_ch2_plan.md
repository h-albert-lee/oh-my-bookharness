## 1. 핵심 주장

이 source는 외부 연구 문헌이 아니라 **본 책 자체의 기획 문서**다. 따라서 '주장'은 책의 논지 설계 자체다.

핵심 설계 논제는 다음 세 층위로 구조화되어 있다:

**층위 1 — 진단 (1장 영역)**
- AI 프로젝트의 운영 실패 원인은 모델 성능이 아니라 '평가 환경(Harness)'의 부재다.
- 코딩 에이전트(Devin 등)의 성공은 모델 지능이 아니라, 실행-평가-교정을 자동화하는 샌드박스 구조 덕분이다.
- "프롬프트를 고치는 것은 시스템을 고치는 것이 아니다."

**층위 2 — 논증 (2장 영역)**
- 전통적 SW는 결정론적 세계를 전제하므로, Assert Equal 기반 TDD가 유효하다.
- AI 시스템은 확률론적 세계이므로, 동일 입력에도 출력이 달라진다 → TDD의 기반 전제가 붕괴된다.
- 무한한 자연어 입력 공간(Input Space) 앞에서 경계값 테스트·단위 테스트는 무력화된다.

**층위 3 — 선언 (2장 귀결)**
- TDD를 대체하는 새 패러다임: **EDD(Evaluation-Driven Development, 평가 주도 개발)**.
- 설계 순서의 역전: "테스트 코드 먼저"가 아니라 "평가자(Evaluator)와 하네스를 먼저 설계"한다.
- 품질 기준의 이동: "맞다/틀리다(binary)" → "충분히 좋다(good enough, gradable)".

## 2. 책에 활용 가능한 포인트

### 2-1. 비유 자산 (즉시 사용 가능)

| 비유 | 설명 | 적용 위치 |
|------|------|----------|
| **자동차 주행 테스트 트랙** | 하네스 = 엔진(LLM)이 아니라 트랙(평가 환경) | 1장 도입부, 독자 진입점 |
| **코딩 에이전트의 샌드박스** | Devin 등의 성공 = 모델 지능이 아니라 컴파일 에러 피드백 루프 | 1장 3절, 하네스 개념 설명 |
| **Assert Equal의 붕괴** | TDD 무력화의 핵심 메커니즘을 단 한 줄로 포착 | 2장 2절 제목 또는 소제목 |

### 2-2. 구조적 대비 프레임 (논증 기둥)

- **Software 1.0 vs Software 2.0** 대비: 입력→동일 출력 vs 입력→확률 분포 출력
  - 2장 전체 논증의 뼈대로 사용 가능
- **TDD vs EDD 비교표**: source가 명시적으로 요청한 필수 도식. 2장에 삽입 시 독자 이해도 핵심 기여 아이템

### 2-3. 개념 도입 순서 설계 (챕터 간 연결)

- **비결정성(nondeterminism)**: 1장에서 도입 → 2장에서 TDD 무력화 논거로 승계
- **하네스(Harness)**: 1장에서 비유로 도입 → 이후 장(4~6장, 13장)에서 아키텍처 구성요소로 심화
- **평가자(Evaluator)**: 2장에서 EDD 구성요소로 첫 언급 → 10~12장에서 상세 설계로 전개

### 2-4. 리서치 의뢰 항목 (집필 전 확보 필요)

| 항목 | 용도 |
|------|------|
| SWE-bench 구조 및 다이어그램 | 1장 3절: 코딩 에이전트 하네스 실증 근거 |
| Regression 장애 실사례 | 1장 4절: 평가 없는 프롬프트 변경의 위험 실증 |
| Ragas/TruLens/LangSmith 통합 방식 | 2장 3절: EDD 도구 ecosystem 소개 |
| Andrej Karpathy 관련 인용구 | 2장 인용문 (Software 2.0 개념 출처 확보 필수) |
| TDD vs EDD 비교표 초안 | 2장 필수 도식 |

## 3. 이 장과의 관련성

### ch01 관련성

- 이 source는 **1장의 완전한 설계도**다.
  - 도입부 비유(자동차 트랙), 절 구성(데모 환상 → 실패 원인 → 코딩 에이전트 사례 → 결론), 핵심 메시지("기능 개발을 멈추고 평가 시스템부터 설계하라") 모두 여기서 나온다.
  - 집필 시 이 source를 **마스터 아웃라인**으로 사용하되, 각 절의 논증을 별도 source(SWE-bench, 실패 사례 문헌 등)로 채워야 한다.

### ch02 관련성

- 이 source는 **2장의 핵심 논증 구조와 개념 도입 시퀀스**를 명시한다.
  - "선언보다 논증이 먼저"라는 설계 지시가 명확하다 → 2장은 EDD를 선언하기 전에 TDD 무력화를 논리적으로 증명하는 순서를 지켜야 한다.
  - Chapter Brief의 독자 학습 목표 표(이해/구분/용어/관점/위치 감각)는 **2장 도입부 설계에 직접 활용** 가능하다.
  - "핵심 질문 → 보조 질문" 구조는 2장 각 절의 소제목 후보로 전환 가능하다.

## 4. 주의할 점 / 한계

### 4-1. 내부 기획 문서 특성상의 한계

- 이 source는 **기획안(plan)**이지 완성된 원고나 검증된 이론이 아니다.
- 모든 개념(EDD, 하네스)은 **이 책이 처음 공식 정의하는 것**이다. 즉, 외부 인용 근거가 없다.
  - EDD라는 용어 자체는 학술적으로 정립된 개념이 아님. 집필 과정에서 독자적 정의임을 명시해야 한다.
  - 하네스(Harness)도 이 책의 아키텍처 메타포로 재정의한 것이므로, 기존 SW 테스팅 용어("test harness")와의 혼동 가능성을 관리해야 한다.

### 4-2. 미확보 상태인 실증 근거

- SWE-bench 구조 분석, 회귀 장애 사례, Karpathy 인용 모두 **리서치 예정 항목**으로만 기재되어 있다.
- 집필 전에 이 항목들이 실제로 확보되었는지 확인 후 논거로 사용해야 한다. 특히 Karpathy 인용은 정확한 출처(강연, 블로그, 논문) 확인 필수.

### 4-3. 개념 연쇄의 단절 위험

- 비결정성 개념이 1장에서 도입된다고 기획되어 있으나, ch02 Brief의 "이전 장에서 이어받는 개념" 섹션이 원문에서 미완성 상태(문장 중간에 끊김)다.
- 1장 원고 확정 전에 2장을 집필하면, 개념 연쇄 지점에서 중복 설명이나 공백이 생길 수 있다. **1장 완고 후 2장 착수** 순서를 지킬 것.

### 4-4. 논증 밀도 vs 분량 긴장

- 2장 목표가 "선언보다 논증 먼저"인데, 예상 분량은 20쪽/20,000자다.
- TDD 무력화 논증 + EDD 정의 + 아키텍처 맛보기를 20쪽에 담으려면 각 절의 논증을 상당히 압축해야 한다. 절간 비중 배분(논증 vs 선언 vs 미리보기) 사전 결정 필요.
