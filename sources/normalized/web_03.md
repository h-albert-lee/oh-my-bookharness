## 1. 핵심 주장

### 명제 ①: LLM 시스템의 실패는 모델이 아닌 시스템 구조에서 기인하며, 유형 분류가 가능하다

"Building Effective Agents"(추정: Anthropic 기술 블로그 혹은 공식 가이드)의 맥락과 분석 본문의 'System-Level Taxonomy' 논지가 공유하는 핵심은 동일하다. LLM 실패는 **무작위 사고가 아니라 구조적으로 예측 가능한 패턴**이며, 실패 유형의 체계적 분류가 올바른 처방의 전제조건이다.

- **System-Level이라는 단위**: 실패 분석의 대상이 모델 파라미터나 추론 성능이 아니라, 파이프라인 연결·출력 검증·피드백 루프·오케스트레이션 등 **시스템 구조적 요소**임을 전제한다.
- **Failure Mode 프레임**: 공학적 의미의 Failure Mode(어떤 방식으로 망가지는가의 유형화)를 LLM 시스템에 적용한다는 것은, AI 실패를 **엔지니어링 가능한 설계 문제**로 재정의하는 언어적 선택이다.
- **Reliable AI Applications**: 신뢰성(Reliability)이 목표 상태이며, 이를 달성하기 위한 전제가 실패 유형의 사전 파악임을 제목 구조 자체가 내포한다.

### 명제 ②: 모델 앞·뒤와 연결 구조 각각에서 독립적인 실패 유형이 발생한다

입력 처리, 프롬프트 구성, 출력 검증, 오케스트레이션, 배포 환경의 각 지점에서 서로 다른 실패 유형이 존재한다. 이는 **실패 귀인의 정확성**과 직결된다. 모델 외부 지점의 실패를 모델 탓으로 귀인하면, 모델을 교체해도 동일한 실패가 반복된다.

### 명제 ③: 에이전트 시스템의 신뢰성은 모델 지능이 아닌 설계 구조에 달려 있다

"Building Effective Agents"라는 제목이 함의하는 바는, 에이전트의 효과성이 모델 능력의 함수가 아니라 **오케스트레이션, 도구 설계, 피드백 구조, 실패 복구 메커니즘** 등 시스템 설계 변수의 함수라는 주장이다. 이는 Blueprint의 핵심 논제("AI 시스템의 성공은 모델 지능이 아니라 하네스 아키텍처에 달려있다")와 직접 대응된다.

## 3. 이 장과의 관련성

### ch03 (Primary — 최고 관련성)

ch03의 핵심 논제("실패는 모델이 아니라 시스템의 문제")와 web_03의 핵심 주장("LLM 실패는 System-Level에서 분류 가능하다")은 동일한 명제를 서로 다른 방향에서 지지한다. ch03이 독자를 설득하는 논증이라면, web_03은 그 논증의 외부 기술 문헌 기반이다.

**ch03 논증 흐름 내 배치 시나리오:**
```
STEP 1. "모델을 바꿔도 문제가 반복된다" — 도입 사례 제시
STEP 2. 귀인 오류의 구조 설명 (ch01 표 1-2 계승)
STEP 3. web_03 인용: "LLM 실패는 System-Level에서 분류된다"
         → 외부 기술 문헌이 동일한 구조를 확인한다는 논증 강화
STEP 4. ch03의 5개 실패 지점 분류 제시
         → web_03의 분류 체계와 대응 관계 명시
```

### ch04 (Secondary — 높은 관련성)

ch04가 제시하는 파이프라인 구성 요소 분류(입력 처리기, 프롬프트 구성기, LLM 호출기, 출력 처리기, 상태 관리자, 오류 처리기)는 web_03의 System-Level Taxonomy와 구조적으로 대응된다. 각 구성 요소가 독립된 설계 단위가 되어야 하는 이유를, web_03의 실패 유형 분류 논리가 외부에서 지지한다.

**ch04 배치 포인트:**
- §구성 요소 분리의 설계적 이유 — "각 구성 요소는 독립된 실패 지점이다"라는 문장의 기술 문헌 근거
- 파이프라인 흐름도 하단 주석 또는 참조 문헌으로 배치

### ch06 (Secondary — 직접 관련)

"Building Effective Agents"라는 제목이 원문이라면, ch06("AI 에이전트는 프로그램이 아니라 시스템이다")의 핵심 주장과 제목 수준에서 직접 대응한다. ch06 도입부에서 "에이전트를 효과적으로 만든다는 것은 무엇을 의미하는가"라는 질문을 제기할 때, web_03의 제목 자체를 논의 진입점으로 활용할 수 있다.

### ch08 (Tertiary — 복선 기능)

"실패 유형의 사전 분류 = 위험의 설계화의 전제"라는 명제를 ch03에서 web_03 인용과 함께 심어두면, ch08로의 이행 시 독자가 이미 준비된 인식 틀을 갖게 된다.

## 4. 주의할 점 / 한계

### 🚨 Critical — 인용 전 반드시 해결해야 할 문제

1. **제목 불일치**: Source 메타데이터의 "Building Effective Agents"와 분석 본문의 "Failure Modes in LLM Systems: A System-Level Taxonomy for Reliable AI Applications"는 서로 다른 문헌이다. 어느 것이 실제 원문인지 확인 없이는 어떤 논지도 귀속 불가.

2. **원문 미확보**: 이 note 전체가 제목 추론과 Chapter Brief 논지의 역방향 재구성이다. 실제 원문의 세부 분류 체계, 사례, 정의가 이 note의 추론과 다를 수 있다.

3. **authority: unknown**: 저자·출처·발행 연도 미확인. 인용 신뢰도 확보 불가 상태.

### 🟡 구조적 한계

- **Taxonomy의 실제 분류 항목 미확인**: 분석 본문이 5개 실패 지점(모델 자체/앞/뒤/연결 구조/운영 환경)을 제시하지만, 이것이 원문의 실제 분류인지 ch03의 독자적 분류인지 불분명하다. 원문 분류와 ch03 분류가 다를 경우, "외부 기술 문헌이 동일 구조를 지지한다"는 논증이 무너진다.

- **"Building Effective Agents"의 에이전트 정의 범위**: 원문이 Anthropic의 에이전트 가이드라면, 다루는 에이전트 개념이 책의 ch06 범위(멀티 에이전트, 오케스트레이션)와 일치하는지 별도 확인 필요.

- **발행 시점 이슈**: LLM 분야는 2023~2025년 사이 기술 맥락이 급변했다. 원문 발행 시점에 따라 현재 기준으로 유효하지 않은 분류나 권고가 포함될 수 있다.

### 🟢 활용 범위 제한

- 원문 확보 전까지는 **직접 인용 불가**. 제목 수준의 간접 참조("이 문제를 다루는 기술 문헌들은 실패를 System-Level에서 분류하는 접근을 취한다")로만 활용 권장.
- ch03의 분류 체계를 원문의 것과 동일시하는 서술 불가.
