## 1. 핵심 주장

이 문서는 source가 아니라 **이 책 자체의 설계도**다. 따라서 "주장"이 아닌 **프로젝트의 설계 원칙과 집필 계약**으로 읽어야 한다. 다음 세 가지가 이 문서의 실질적 핵심이다.

### 1-1. 책의 중심 논제 (Thesis)
- AI 프로젝트가 PoC에서는 성공하고 운영에서 실패하는 것은 **모델의 문제가 아니라 시스템 설계의 문제**다.
- 전통적 TDD는 확률론적 AI 시스템에서 무력화된다 → **EDD(Evaluation-Driven Development)** 가 대안이다.
- 하네스(Harness) 아키텍처: 실행-평가-교정의 자동화 루프가 AI 시스템 성공의 핵심이다.
- RAG·멀티 에이전트는 기술 트렌드가 아니라 **구조적 설계 선택**이다.

### 1-2. 책의 구조 원칙
- **5파트 15장** 구성. Part 1(실패 원인) → Part 2(시스템 구조) → Part 3(설계 원칙) → Part 4(평가 중심 개발) → Part 5(설계 패턴) 의 논리 흐름.
- 코드 비중 최소화, 이론 중심, 개념적 내면화 지향.

### 1-3. 저자 포지셔닝
- 이한울(에임인텔리전스 부사장, 前 신한투자증권 AI Lead / NCSOFT AI Researcher): **금융·보안·컴플라이언스 환경에서의 실전 구현 경험**이 저자 권위의 근거.

## 2. 책에 활용 가능한 포인트

### 2-1. 논지 전개의 뼈대 (전 장 공통)
- "데모는 되는데 운영에서 실패한다"는 문제 설정은 **모든 장의 도입부 후크**로 재활용 가능하다.
  - 각 장마다 "이 구성 요소/개념의 부재가 어떻게 운영 실패로 연결되는가"를 역추적하는 서술 구조로 응용할 것.
- TDD → EDD 패러다임 전환 프레임은 **Part 4** 의 핵심이지만, **2장(소프트웨어 패러다임의 전환)** 에서 개념 정착 후 이후 장에서 반복 참조하는 방식으로 활용 가능.

### 2-2. 독자층 정의의 활용
- 메인 독자(AI/ML 엔지니어, LLM 서비스 개발자)는 **기존 SW 개발 경험을 보유**한다 → 설명 시 "기존 소프트웨어 개발에서는 X였지만, LLM 시스템에서는 Y가 된다" 대조 구조가 유효.
- 서브 독자(PM, 아키텍트, 금융·보안 실무자)를 위한 **사이드바나 박스 콘텐츠** 삽입 지점을 각 장에서 명시적으로 설계할 것.

### 2-3. ch04 전용 포인트
- ch04의 위치: Part 2의 첫 장. **ch03의 진단("문제는 시스템이다")을 받아서 "그 시스템이 실제로 무엇으로 구성되는가"를 해부**하는 전환점.
- 핵심 질문 "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"는 **ch04의 개장(開章) 후크**로 그대로 사용 가능.
- 구성 요소 6종(Input Processor / Prompt Builder / LLM Caller / Output Processor / State Manager / Error Handler)은 각각 **역할 / 실패 시 증상 / 설계 시 고려사항** 3요소 서술 구조로 전개하도록 편집 지시가 이미 내려져 있음.
- ch04는 ch05(RAG)·ch06(에이전트)의 **개념적 선행 조건**이다 → 각 구성 요소 설명 말미에 "5장에서 이 부분이 어떻게 확장되는가"를 한 문장씩 예고하는 방식이 유효.

### 2-4. 하네스(Harness) 개념
- 기획안에서 **1장과 핵심 차별점**으로 명시. "코딩 에이전트의 샌드박스 = 하네스" 비유가 tone guide에서 예시로 제시됨 → 비유 설명의 레퍼런스 포인트로 활용.

## 4. 주의할 점 / 한계

### 4-1. 기획안은 의도이지 완성된 논리가 아니다
- argument_map의 세 번째·네 번째 논제는 문서에서 **문장이 잘려 있다** ("RAG와 멀티 에이전트는 기술 트렌드가 아니라" 이후 미완성). 집필 과정에서 저자와 논지를 **반드시 확인하고 보완**해야 한다.
- Part 5(설계 패턴)의 장별 내용(13–15장)은 기획안에 제목만 있고 Chapter Brief가 없다 → 집필 전 별도 브리프 작성 필요.

### 4-2. 독자층 설정과 난이도의 긴장
- 메인 독자(AI/ML 엔지니어)와 서브 독자(PM, 아키텍트) 간 기술 이해 수준 차이가 크다. 기획안은 이 긴장을 해소하는 구체적 편집 방침을 제시하지 않는다. **각 장에서 독자 수준을 어디에 맞출지** 집필 전 결정 필요.

### 4-3. "이론 중심 / 코드 최소화" 지침의 실행 난이도
- LLM 파이프라인 구성 요소(ch04)나 RAG 아키텍처(ch05)는 코드 없이 구조적으로만 설명할 때 **추상성이 지나쳐 독자가 실무 적용을 못하는** 위험이 있다. 코드 대신 어떤 시각적·개념적 대체물(다이어그램, 의사코드, 사례 시나리오)을 쓸지 장별로 명시적으로 설계할 것.

### 4-4. 일정 압박
- 1차 마감(2026-04-30)까지 약 7주. 300쪽 분량을 고려하면 매우 타이트하다. **초반 장(ch01–03)의 논지가 흔들리면 후반 장 전체가 재작업**될 위험이 있으므로, argument_map을 먼저 확정하고 집필을 시작해야 한다.

### 4-5. EDD 개념의 독창성 검증 필요
- TDD → EDD 전환을 책의 핵심 차별점으로 내세우지만, "EDD(Evaluation-Driven Development)"라는 용어와 프레임이 **기존 학술·실무 커뮤니티에서 어떻게 사용되고 있는지** 선행 문헌 조사가 필요하다. 저자 고유 개념임을 명확히 하거나, 기존 용법과의 차이를 서문에서 정의해야 한다.

## 5. 인용 시 주의 표현

이 문서는 **외부 인용 대상이 아니라 집필 계약 및 기준 문서**다. 따라서 일반적인 source citation 규칙이 아닌 다음 방식으로 참조한다.

- **편집 결정 근거로 참조 시**: "기획안 §Chapter Brief ch04 편집 지시에 따라 3요소 서술 구조 적용"
- **논지 방향 확인 시**: "argument_map §핵심 논제 1항 — '운영 실패는 시스템 설계의 문제'"
- **저자 소개·권위 근거로 참조 시**: "기획안 §저자 소개 — 신한투자증권 AI Lead, 금융·보안 환경 RAG 실전 구현 경험"
- **⚠️ 주의**: argument_map은 미완성 문장을 포함하므로, 해당 논제를 집필에 직접 반영하기 전에 저자 확인을 거쳐야 한다. 미완성 논제를 임의로 완성하지 말 것.
- **⚠️ 주의**: 기획안의 예상 판매 부수·손익분기 등 상업적 수치는 독자에게 노출되는 본문에 인용하지 않는다.
