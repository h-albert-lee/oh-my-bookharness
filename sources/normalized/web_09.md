## 1. 핵심 주장

### 추론된 핵심 명제

| 명제 | ch04 적용 해석 |
|---|---|
| ML 시스템의 운영 준비도는 정량적으로 측정 가능하다 | 파이프라인 각 구성 요소도 독립적으로 평가 가능하도록 설계되어야 함을 함의 |
| 테스트 통과 ≠ 운영 준비 완료 | 구성 요소 각각이 통과해도 파이프라인 전체 흐름이 실패할 수 있음의 선행 근거 |
| 기술 부채는 모델이 아니라 시스템 설계에 누적된다 | 파이프라인 구성 요소를 분리하지 않을 때 어느 단계에서 부채가 발생하는지 예측 가능 |
| 운영 준비도는 이진(pass/fail)이 아닌 연속 척도다 | 구성 요소별 성숙도 평가의 근거 — "충분히 동작하는 파이프라인" 개념과 호응 |

### 추론된 4축 구조 (동명 문헌 기준)

```
데이터(Data) ─── 모델(Model)
      │                  │
인프라(Infrastructure) ─ 모니터링(Monitoring)
```

이 4축 각각이 ch04의 파이프라인 구성 요소 분류와 **구조적으로 대응**된다:

- **데이터 축** → 입력 처리기(Input Processor) + 상태 관리자(State Manager)
- **모델 축** → 프롬프트 구성기(Prompt Builder) + LLM 호출기(LLM Caller)
- **인프라 축** → 오류 처리기(Error Handler) + LLM 호출기의 재시도·폴백 설계
- **모니터링 축** → 출력 처리기(Output Processor)의 검증 단계 + PART 4 평가 루프 복선

## 2. 책에 활용 가능한 포인트

### ✅ Must-Use (원문 확보 시 즉시 인용)

**① ch04 §4-3: 구성 요소 분리의 설계적 이유 — 산업 수준 앵커**

- "왜 구성 요소를 분리해야 하는가"라는 질문에 대해, ML 산업이 이미 4축으로 실패 영역을 구획했다는 사실이 외부 근거가 된다.
- "분리하지 않으면 어느 축이 무너졌는지 알 수 없다"는 논지를 뒷받침하는 산업 사례로 제시 가능하다.
- **활용 위치**: §4-3 "구성 요소를 분리하지 않았을 때 생기는 문제" 도입 단락 또는 각주

**② ch04 §4-2: 구성 요소 분류의 외부 정당화**

- ch04가 제안하는 6개 구성 요소 분류가 저자의 임의적 분류가 아님을 보강할 때 유효하다.
- "이 분류 방식은 ML 시스템 운영 준비도 평가 체계(ML Test Score)의 4축 구조와 방향이 일치한다"는 한 문장으로 처리 가능하다.
- **활용 위치**: 구성 요소 분류 소개 직후 설계 근거 설명 단락

**③ ch04 → ch10/11 복선 심기**

- ML Test Score = "파이프라인 각 구성 요소를 독립 평가 가능하도록 설계해야 한다"는 PART 4 논지의 전신으로 포지셔닝 가능하다.
- ch04에서 "각 구성 요소가 독립 교체 가능해야 한다"고 설명할 때, "이것이 나중에 ch11에서 다룰 평가 설계의 전제가 된다"는 방식으로 독자 동기를 유발할 수 있다.
- **활용 위치**: §4-2 마지막 단락 또는 "더 읽기" 박스

## 3. 이 장(ch04)과의 관련성

ch04의 핵심 질문은 "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"이다. 이 문헌은 다음 세 지점에서 이 질문에 직접 호응한다.

**첫째, "시스템 = 구성 요소의 집합"이라는 관점의 산업 수준 선례.**
ML Test Score가 ML 시스템을 4개 독립 축으로 분해하여 평가한다는 사실 자체가, ch04가 LLM 파이프라인을 6개 구성 요소로 분해하는 방식의 선행 사례로 기능한다. 독자에게 "이런 분해 방식은 ML 산업이 이미 실천하고 있다"는 신뢰를 줄 수 있다.

**둘째, 구성 요소 분리의 실패-감지 기능.**
4축 구조의 핵심 전제는 "축이 분리되어야 어느 축이 무너졌는지 알 수 있다"는 것이다. ch04 §4-3에서 "구성 요소를 분리하지 않으면 실패 지점을 특정할 수 없다"는 논지와 정확히 호응한다. 이 문헌을 근거로 "분리는 편의가 아니라 진단 가능성의 전제 조건"이라는 설계 원칙을 강화할 수 있다.

**셋째, PART 4 평가 루프의 전제 조건 구축.**
각 구성 요소가 독립적으로 평가 가능해야 한다는 ch04의 설계 요구사항은, ML Test Score가 각 축을 독립 점수화한다는 구조와 동형이다. ch04에서 이 관계를 짧게 언급해두면, PART 4(ch10~12)에서 평가 설계를 다룰 때 "ch04에서 예고한 그 구조"로 자연스럽게 귀환할 수 있다.

**서사 흐름에서의 역할:**
```
ch03 (시스템 실패 진단) 
  → [ML Test Score: 실패는 4축에 분산된다]  ← 본 문헌의 위치
  → ch04 (파이프라인 구성 요소 분해) 
  → ch10~12 (각 구성 요소의 평가 설계)
```

## 5. 인용 시 주의 표현

### 원문 확보 전 — 사용 가능한 최소한의 언급 형식

> *(각주 처리 권장)*
> Breck et al.은 ML 시스템의 운영 준비도를 데이터, 모델, 인프라, 모니터링의 4개 축으로 분류하고, 각 축을 독립적으로 점수화하는 루브릭을 제안했다. 이 분류 방식은 ML 시스템이 단일 모델이 아닌 다층 구성 요소의 집합임을 전제한다.
> — *Breck et al., "The ML Test Score: A Rubric for ML Production Readiness and Technical Debt Reduction," ICSE 2017 (원문 검증 중)*

### 원문 확보 후 — 본문 삽입 가능한 형식 (안)

**형식 A: 설계 근거 제시용 (§4-3)**
> ML 시스템의 운영 준비도를 측정하려 했던 Breck et al.(2017)은 시스템을 네 개의 독립 축으로 분리하는 루브릭을 제안했다. 그 전제는 단순하다. 축이 분리되어 있어야 어느 축이 무너졌는지 알 수 있다는 것이다. 파이프라인의 구성 요소를 나누는 이유도 다르지 않다.

**형식 B: 각주 한 줄용 (§4-2 구성 요소 분류 근거)**
> 이 분류 방식은 Breck et al.(2017)이 ML 운영 준비도 평가를 위해 제안한 4축(데이터·모델·인프라·모니터링) 구조와 방향이 일치한다.

**형식 C: ch03→ch04 연결 가교용 (ch04 도입부)**
> 지난 장에서 기술 부채가 모델이 아니라 시스템 설계에 누적된다고 진단했다. 그렇다면 그 시스템은 무엇으로 이루어져 있는가. ML 산업은 이미 이 질문에 답하려 했다. Breck et al.(2017)은 ML 시스템을 네 개의 독립 영역으로 분해하고, 각각을 별도로 평가할 수 있는 루브릭을 만들었다. 이 장은 같은 질문을 LLM 파이프라인에 던진다.

---

> **편집 메모**: 형식 A는 §4-3의 설계 원칙 논증에 가장 자연스럽게 통합된다. 형식 C는 장 도입부의 서사 에너지를 높이는 데 유효하나, 원문 확보 없이는 과잉 주장의 위험이 있으므로 반드시 각주에 "원문 검증 중" 표시를 병기할 것.
