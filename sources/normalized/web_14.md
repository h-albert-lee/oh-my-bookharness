## 1. 핵심 주장

> ⚠️ **이하 분석은 제목 및 도메인 맥락 기반 추정 전체. 직접 인용 전 원문 확인 필수.**

**[주장 1] LLM 에이전트의 신뢰성은 모델 역량이 아니라 아키텍처 설계로 확보된다**
- "Resilient(회복 탄력적)"를 제목에 명시한 것은 에이전트 실패가 예외적 사건이 아닌 정상 상태임을 전제한다
- 회복 탄력성은 모델을 더 좋게 만드는 방식이 아니라, 실패를 수용하는 구조적 설계로 달성된다는 관점

**[주장 2] Plan-then-Execute 패턴은 LLM 에이전트의 구조적 분해 방식 중 하나다**
- 단일 호출(Monolithic Call) 대신 '계획(Plan) 단계'와 '실행(Execute) 단계'를 구조적으로 분리
- 이 분리가 각 단계의 독립적 검증·교체·오류 처리를 가능하게 한다는 설계 논리
- ch04의 '구성 요소 분리의 설계적 이유(4-3절)'와 직접 정렬되는 외부 근거

**[주장 3] 보안(Secure) 구현이 아키텍처 수준에서 다루어진다**
- 보안을 사후 검토가 아닌 설계 단계의 구성 요소로 다룬다는 시각
- 오류 처리기(Error Handler), 입력 검증(Input Validation) 등 ch04의 구성 요소들과 연결 가능

## 2. 책에 활용 가능한 포인트

### ✅ Must-Cite Points

**[포인트 A] Plan-then-Execute 패턴이 파이프라인 분리 설계의 구체적 사례로 기능**
- ch04의 핵심 메시지("LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 다르다")를 에이전트 아키텍처 수준에서 외부 확인
- "계획과 실행의 분리"는 ch04에서 소개하는 구성 요소 분리 원칙의 실제 적용 형태로 제시 가능
- **활용 위치**: 4-3절(구성 요소 분리의 설계적 이유) 내 사례 단락
- **활용 방식**: "이 분리 원칙은 에이전트 설계에서 Plan-then-Execute 패턴으로 구현된다. 계획 단계의 출력이 실행 단계의 입력이 되는 구조는, 각 단계를 독립적으로 평가하고 교체할 수 있게 한다."

**[포인트 B] '회복 탄력성(Resilience)'이 모델이 아닌 아키텍처의 속성이라는 프레이밍**
- ch04 전체 논지—"파이프라인의 각 구성 요소는 왜 독립된 설계 단위가 되어야 하는가"—의 동기를 외부 언어로 보강
- "신뢰할 수 있는 시스템"이 모델 성능이 아니라 구조적 설계의 결과라는 PART 3(7장, 8장) 논지의 씨앗을 ch04에서 미리 심는 데 활용 가능
- **활용 위치**: 4-1절(파이프라인으로서의 LLM 시스템) 도입부 또는 장 말미 요약
- **활용 방식**: "에이전트 아키텍처 설계 논의에서 '회복 탄력성'은 더 나은 모델을 쓰는 방식이 아니라, 실패를 수용하는 구조를 만드는 방식으로 달성된다고 기술된다."

**[포인트 C] 오류 처리기(Error Handler)와 보안 레이어를 아키텍처 수준에서 설계하는 관점**
- ch04의 오류 처리기(Error Handler) 구성 요소 서술—"실패·재시도·폴백을 담당하는 단계"—의 외부 근거
- "보안은 사후 검토가 아니라 설계 단계의 구성 요소"라는 시각이 ch04의 구성 요소 분리 논리와 정렬
- **활용 위치**: 오류 처리기 서술 단락 또는 각주
- **활용 방식**: "이 단계를 사후에 붙이는 방식은 실패 원인을 구조화하는 데 실패한다는 지적은 에이전트 아키텍처 설계 논의에서도 반복적으로 등장한다."

### ⚠️ 역용(Counterpoint) 포인트

**[역용 포인트 1] Plan-then-Execute가 모든 LLM 시스템에 적용 가능한 보편 패턴이 아닐 수 있다**
- ch04는 에이전트(6장)가 아닌 기본 파이프라인을 다루는 장이므로, Plan-then-Execute를 ch04의 보편 설계 원칙인 것처럼 제시하면 범위 과잉이 발생
- **활용 제한**: 이 패턴은 ch04에서 "에이전트로 확장될 때 어떤 구조가 나타나는가"의 예고 형태로만 사용. ch06에서 본격 전개.
- **회피 방식**: "이 분리 원칙이 에이전트 수준에서 어떻게 구체화되는지는 6장에서 다룬다"로 명시적으로 범위 한정

**[역용 포인트 2] 'Secure' 강조가 ch04의 초점(구조 이해)보다 ch08·9장(가드레일·보안)과 더 정렬**
- 보안 구현 논의를 ch04에서 과도하게 끌어오면 이 장의 핵심 목표("어디서부터 생각해야 하는가")가 흐려질 위험
- **회피 방식**: 보안 관련 언급은 "이 구성 요소가 보안 설계와 연결된다는 점은 8장에서 다룬다"로 전방 참조 처리
