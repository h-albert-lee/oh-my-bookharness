## 2. 책에 활용 가능한 포인트

### 🔴 Must-Cite Items

**[포인트 A] ML 시스템의 컴포넌트 구조 — "파이프라인으로서의 LLM 시스템" 개념의 외부 근거**

- **최적 배치**: ch04 도입부(4-1절). "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"라는 핵심 질문을 제기한 직후
- **ch04에서의 역할**:
  독자가 LLM 시스템을 단일 API 호출로 인식하는 한, 파이프라인 설명 전체가 불필요한 복잡성으로 느껴진다. 이 포인트는 "산업 현장의 실제 ML 시스템은 처음부터 컴포넌트 집합으로 설계되어왔다"는 역사적·구조적 사실을 2015년 구글 연구로 확립한다. LLM 파이프라인은 새로운 발명이 아니라, ML 시스템 설계의 연장선임을 보여주는 맥락 설정 역할
- **ch03과의 중복 처리**:
  ch03에서 같은 자료의 "빙산의 일각" 다이어그램을 이미 썼다면, ch04에서는 다이어그램 재인용 없이 "3장에서 살펴본 것처럼, ML 시스템의 대부분은 모델 바깥에 있다. 이 장에서는 그 바깥이 구체적으로 무엇으로 이루어지는지를 분해한다"는 연결 문장으로 처리. 자료의 재소환은 최소화하고 새 논증에 집중
- **활용 예시 문장**:
  > *"2015년 구글 연구팀이 실제 ML 시스템을 해부했을 때 발견한 것은, 누구도 설계한 적 없지만 어느새 시스템 전체를 떠받치고 있는 수십 개의 역할 단위들이었다. 데이터를 긁어오는 코드, 피처를 가공하는 코드, 설정값을 관리하는 코드. 각각이 독립된 역할을 했지만, 그 경계가 불분명했기 때문에 어디서 문제가 생겼는지 아무도 말할 수 없었다. LLM 시스템도 다르지 않다. 처음부터 역할을 분리해서 설계하지 않으면, 같은 일이 반복된다."*

---

**[포인트 B] 컴포넌트 경계 불명확의 비용 — "구성 요소 분리의 설계적 이유"(4-3절)의 부정적 증거**

- **최적 배치**: ch04의 4-3절 "구성 요소 분리의 설계적 이유". 각 구성 요소를 설명한 후, 분리하지 않았을 때의 구체적 비용을 서술하는 대목
- **ch04에서의 역할**:
  4-3절의 핵심 논증은 "왜 귀찮게 분리해야 하는가"에 대한 답이다. 긍정적 근거(분리하면 교체·평가·테스트가 쉬워진다)만으로는 독자 설득이 약하다. Sculley et al.의 자료는 "분리하지 않았을 때 실제로 어떤 일이 벌어지는가"를 산업 규모의 사례로 보여주는 부정적 증거를 제공한다. 기술 부채가 컴포넌트 경계에 축적되고, 그 부채는 코드 리뷰나 단위 테스트로 포착되지 않는다는 점이 핵심
- **활용 예시 문장**:
  > *"경계가 없는 파이프라인은 어디서 문제가 생겼는지를 추적할 수 없다. Sculley 외 연구자들은 이 상태를 '은닉 기술 부채'라고 불렀다. 드러나지 않기 때문에 측정도, 제거도 되지 않는다. 입력 처리와 프롬프트 구성과 LLM 호출이 한 덩어리로 뭉쳐 있으면, 출력이 이상해졌을 때 어느 단계가 원인인지 알 수 없다. 분리는 편의가 아니라 가시성의 문제다."*

---

**[포인트 C] CACE를 역전시킨 설계 논리 — "각 구성 요소는 독립적으로 교체 가능해야 한다"의 구조적 근거**

- **최적 배치**: ch04의 각 구성 요소 설명부(4-2절) 서두. 6개 구성 요소를 나열하기 전, 왜 이것들을 독립 단위로 다루는가의 설계 철학을 먼저 제시하는 위치
- **ch04에서의 역할**:
  ch03에서 CACE는 "무엇을 바꿔도 모든 것이 영향받는다"는 위협으로 등장했다. ch04에서는 같은 원칙을 설계 논리로 역전시킨다. "무엇을 바꿔도 모든 것이 영향받기 때문에, 역으로 각 요소를 독립적으로 통제하고 관찰할 수 있는 구조가 필요하다." CACE가 위협에서 설계 요건으로 전환되는 순간이다
- **ch03과의 연결 처리**:
  > *"3장에서 CACE는 실패의 원인이 왜 구조 전체에 퍼져 있는지를 설명했다. 여기서는 같은 원칙이 설계 요건으로 변환된다. 무엇을 바꿔도 모든 것이 영향받는다면, 각 구성 요소는 독립적으로 통제 가능해야 한다."*
- **활용 예시 문장**:
  > *"어떤 단계를 수정해도 다른 단계에 영향이 미친다면, 설계자가 할 수 있는 것은 두 가지다. 모든 변경을 포기하거나, 영향이 전파되는 경로를 구조적으로 제어하거나. LLM 파이프라인을 독립된 구성 요소로 분리하는 이유는 바로 이것이다. 통제 가능한 단위를 만드는 것."*

## 4. 주의할 점 / 한계

### 4-1. 자료 적용 범위 한계

Sculley et al.(2015)는 전통적 ML 시스템(데이터 파이프라인, 피처 엔지니어링, 배치 학습)을 대상으로 한다. LLM 파이프라인에는 **프롬프트 구성기, 상태 관리자** 같은 LLM 특유의 구성 요소가 존재하며, 이들은 2015년 논문에서 명시적으로 다루어지지 않는다. 자료를 "구조적 원칙의 근거"로만 사용하고, "LLM 파이프라인 설계의 직접 지침"으로 제시하지 않도록 주의.

### 4-2. ch03 소진 이후의 독자 피로 관리

ch03에서 이 자료를 충분히 활용했다면, ch04에서 같은 자료를 다시 전면 인용하면 독자에게 "이 책은 논문 하나로 다 설명하려 한다"는 인상을 줄 수 있다. ch04에서는 **간략 참조("3장에서 소개한 그 연구")로 처리하고**, 자료에서 파생된 설계 논리를 저자 자신의 언어로 전개하는 비중을 높여야 한다.

### 4-3. 2015년 자료의 시효 문제

10년 전 논문이다. 독자 중 일부는 "LLM 시대에 왜 2015년 ML 논문을?"이라는 반응을 보일 수 있다. 활용 시 **원칙의 시효와 기술의 시효를 분리**하는 문장이 필요하다. "구체적 기술 환경은 달라졌지만, 시스템 설계 원칙은 달라지지 않았다"는 명시적 프레임 삽입 권장.

### 4-4. 원문 확인 미완료

등록 제목과 발췌 내용의 불일치로 인해 인용 정확도 검증이 완료되지 않았다. 특히 "CACE"와 "Configuration Debt" 개념의 정확한 출처 표현과 맥락을 원문에서 재확인해야 한다.
