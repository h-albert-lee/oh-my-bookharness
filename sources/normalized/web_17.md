# Source Note: web_17
**LLMs are facing a QA crisis: Here's how we could solve it**

---

## 1. 핵심 주장

### 제목에서 추론되는 이중 구조

이 아티클은 두 명제를 전제로 삼는다.

**명제 1 — LLM QA는 위기 상태다**

"crisis"는 점진적 불편함이 아니라 **기존 방법론의 구조적 붕괴**를 선언하는 단어다. LLM 출력의 비결정성(nondeterminism)으로 인해, 전통적 QA의 핵심 메커니즘—"입력 고정 → 출력 검증"—이 원리상 성립하지 않음을 전제한다.

**명제 2 — 다른 방법으로는 해결 가능하다**

"Here's how we could solve it"에서 "could"는 확정된 해법이 아닌 **제안적·실험적 접근**임을 시사한다. 기존 QA를 개선하는 것이 아니라, **평가 기준과 방법론 자체를 재설계**해야 한다는 방향을 암시한다.

### 추론 가능한 세부 주장

| 예상 주장 | 추론 근거 |
|---|---|
| 단위 테스트·Assert Equal은 LLM 출력에 무의미하다 | 비결정성 출력의 구조적 특성 |
| 품질 기준을 "정확(correct)"에서 "충분히 좋음(good enough)"으로 재정의해야 한다 | 정답이 없는 생성형 출력의 본질 |
| 평가자(Evaluator) 또는 메트릭 기반 체계가 필요하다 | QA 위기 해법의 산업적 수렴 방향 |
| LLM-as-judge 접근이 실질적 대안으로 부상한다 | 2023~2024년 LLM QA 담론의 주류 흐름 |

> **⚠ 분석 전체가 제목 기반 추론임을 명기한다. 원문 미확인 상태.**

### 🔧 ch04에서의 제한적 보조 활용

이 source를 ch04에서 직접 인용하기는 어렵다. 그러나 다음 방식의 **간접 활용**은 가능하다.

**파이프라인 설계 동기 부여 맥락**: ch04는 "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"를 핵심 질문으로 삼는다. 이 질문에 답하기 전에, "단순 호출로 끝나는 구조가 왜 위험한가"를 설명하는 배경 근거로 QA 위기 담론을 1~2문장으로 참조할 수 있다.

**출력 처리기(Output Processor) 설계 근거**: LLM 출력이 비결정적이라는 사실—QA 위기의 근본 원인—은, 출력 처리 단계를 별도 구성 요소로 분리해야 하는 설계적 이유와 직결된다. "출력을 그냥 받아 쓰지 않고, 왜 파싱·정제·검증 단계를 두어야 하는가"의 동기 설명에 활용 가능.

**오류 처리기(Error Handler)와 비결정성**: 동일 입력에 다른 출력이 나올 수 있다는 사실은, 재시도(retry)·폴백(fallback) 전략을 단순 예외처리가 아니라 **설계 단계에서 내재화**해야 하는 이유가 된다. ch04의 오류 처리기 섹션 도입에 간접 연결 가능.

### 🔮 후속 장 예비 연결

- **ch10~12**: "어떻게 해결할 수 있는가"에 해당하는 해법 방향(LLM-as-judge, 루브릭 기반 평가)은 ch10~12에서 본격 활용. ch04에서는 위기의 실재성만 참조하고, 해법의 구체성은 유보.

### 논지 대응 구조

```
ch04의 핵심 질문:
  "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가?"
                      ↓
web_17이 간접 제공하는 것:
  "단순 호출 구조(QA 불가 상태)를 넘어서려면
   각 처리 단계를 구조적으로 분리하고 검증 가능하게 만들어야 한다."
```

### ch04 개념과의 연결 지점

| ch04 구성 요소 | web_17 간접 연결 포인트 |
|---|---|
| **출력 처리기(Output Processor)** | LLM 출력의 비결정성 → 파싱·정제 단계 분리 필요의 근거 |
| **오류 처리기(Error Handler)** | 동일 입력 → 다른 출력 가능성 → 재시도·폴백의 설계 내재화 필요 |
| **파이프라인 설계 철학** | 단일 API 호출로는 QA가 불가능하다는 진단 → 단계 분리 설계의 필요성 |
| **구성 요소 분리의 이유** | 어느 단계에서 실패했는지 추적하려면, 각 단계가 독립적이어야 함 |

### 연결 가능한 문장 구조 예시 (ch04 도입부 활용)

> *"LLM을 단순히 API로 호출하고 응답을 바로 사용하는 구조는, 개발 단계에서는 작동하는 것처럼 보인다. 그러나 이 구조가 운영 환경에 나가는 순간, '어디서 잘못되었는가'를 추적하기 어려워진다. 어떤 입력이 어떤 프롬프트를 거쳐 어떤 출력을 냈는지가 뒤엉켜 있기 때문이다. 파이프라인의 각 단계를 분리하는 것은 코드를 깔끔하게 만들기 위한 취향이 아니다. 그것은 시스템을 관찰 가능하고, 교체 가능하며, 평가 가능하게 만드는 구조적 전제다."*

### ⚠ ch04 적용 시 구조적 한계

- 이 source의 핵심 논지는 **"QA 방법론의 위기"**이지, **"파이프라인 설계"**가 아니다. ch04에서 직접 인용하면 논지 이탈이 발생할 수 있다.
