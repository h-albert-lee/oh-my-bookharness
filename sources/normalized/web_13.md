## 1. 핵심 주장

> ⚠️ **원문 미등록 상태** — 제목과 프로젝트 맥락 기반 역추론. 실제 원문 확인 전 본문 직접 인용 불가. 아래 모든 주장은 추정(inferred)으로 표기한다.

### 주장 1 — 비결정성은 단일 원인이 아닌 다층적 현상이다 [추정]

"defeating"이라는 동사 선택이 핵심 단서다. "eliminating(제거)"이나 "avoiding(회피)"이 아니라 **격파(defeating)**라는 표현은, 비결정성을 파라미터 조정 하나로 해결하는 기술적 문제가 아니라 **다수의 원인이 계층적으로 누적되는 구조적 현상**으로 전제한다. 원인 레이어별로 각각 다른 전략이 필요하다는 논증 구조를 암시한다.

### 주장 2 — temperature=0 설정은 완전한 결정론을 보장하지 않는다 [추정]

제목이 "inference" 단계를 명시한다는 점에서, 이 문서는 sampling 전략(temperature, top-p) **외부**의 비결정성 원인 — GPU 부동소수점 연산의 비결정성, CUDA 비결정적 커널, 배치 처리 순서, 병렬 연산 누적 오차 — 을 다룰 가능성이 높다. "temperature를 0으로 설정하면 동일한 출력이 나온다"는 실무 현장의 통념을 정면 반박하는 구조다.

### 주장 3 — 비결정성 제어는 완전 제거가 아니라 전략적 관리의 문제다 [추정]

"defeating"이 완전 소멸을 의미하지 않는다면, 이 문서의 결론은 seed 고정, deterministic 연산 모드 설정, 배치 크기 제어 등 **복수의 전술 조합으로 비결정성을 허용 가능한 범위 안으로 통제하는 전략**을 제시하는 방향일 것으로 추정된다. 이는 기술적 제어의 천장이 존재함을 간접적으로 인정하는 논조이며, 평가 구조 도입의 당위성과 연결된다.

### 🟡 보조 활용 포인트

- **ch04 LLM 호출기(LLM Caller) 설계 논거**: ch04의 핵심 구성 요소 중 "LLM 호출기"를 단순한 API wrapper로 정의하지 않아야 하는 이유를 기술적으로 뒷받침한다. 비결정성이 inference 레벨에 내재해 있다면, LLM 호출기는 응답을 받는 것 이상으로 **재시도 전략, 응답 일관성 검증, 오류 처리**를 포함해야 한다는 설계 근거로 기능한다.

- **ch04 오류 처리기(Error Handler) 설계 강화**: "같은 입력, 다른 출력"이 inference 인프라에서 기인한다는 사실은, 오류 처리기가 단순한 exception catch 이상의 역할 — 응답 편차 감지, 폴백 로직, 결과 유효성 검증 — 을 설계에 포함해야 함을 정당화한다.

- **독자 공감 진입점**: ch04 도입부에서 "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"라는 핵심 질문을 제기할 때, 비결정성의 inference-level 기원을 사례로 활용 가능. 독자가 이미 겪은 "왜 같은 코드에서 다른 결과가 나오는가"라는 경험을 기술적으로 설명해주는 역할.

- **ch10 재소환 근거**: 동일 논거를 ch10 "왜 테스트로는 부족한가" 절에서 재활용 가능. 비결정성이 단위 테스트의 반복 가능성 전제 자체를 붕괴시킨다는 논지의 기술적 배경으로 기능. 1회성 인용이 아니라 책 전반의 논증 기반으로 설계.

- **Failure Topology와의 연결**: ch03 §실패 지점 지도에서 "모델 자체 레이어"의 실패가 모델 알고리즘의 문제가 아니라 inference 인프라의 문제일 수 있음을 구체화하는 사례로 배치 가능.

### 직접 연결 지점

**연결 지점 1 — LLM 호출기(LLM Caller) 설계의 근거 강화**

ch04의 구성 요소 중 LLM 호출기를 설명할 때, "API 호출은 결정론적이지 않다"는 명제가 필요하다. web_13은 이 명제의 기술적 근거를 제공한다. temperature=0으로 설정하더라도 GPU 연산 레벨에서 비결정성이 잔존할 수 있다면, LLM 호출기는 단순히 API를 호출하고 응답을 반환하는 wrapper가 아니라, **응답 일관성을 고려한 설계 단위**로 정의되어야 한다. 이는 "왜 각 구성 요소가 독립된 설계 단위여야 하는가"라는 ch04의 핵심 질문에 대한 구체적 답변을 제공한다.

**연결 지점 2 — "역할·실패 시 증상·설계 고려사항" 3요소 서술 강화**

ch04 편집 지시에서 각 구성 요소를 '역할 / 실패 시 증상 / 설계 시 고려사항'으로 서술하도록 명시되어 있다. web_13은 LLM 호출기와 오류 처리기의 "실패 시 증상" 항목을 풍부하게 채우는 기술적 근거로 기능한다. inference 레벨 비결정성 → 동일 입력 다른 출력 → 증상이 intermittent하여 재현이 어려움 → 이 특성이 설계 시 고려사항을 결정한다는 서술 흐름이 가능하다.

**연결 지점 3 — ch03과의 연결 고리 유지**

ch04는 ch03("문제는 모델이 아니라 시스템이다")의 후속 장으로, ch03의 진단 위에서 "그렇다면 시스템은 어떤 구조인가"를 제시한다. web_13의 논거는 ch03에서 이미 사용된 논거를 ch04에서 구성 요소 설계 수준으로 구체화하는 다리 역할을 한다. 독자 입장에서는 "ch03에서 들었던 그 비결정성 문제가, ch04의 파이프라인 설계에서 이렇게 반영된다"는 학습 연속성이 확보된다.
