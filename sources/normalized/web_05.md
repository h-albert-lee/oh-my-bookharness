## 2. 책에 활용 가능한 포인트

### ✅ ch04 도입부: "왜 구성 요소를 분리해야 하는가"의 외부 정당성 앵커

ch04의 핵심 질문은 "LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가"이다. 이 질문은 독자에게 직관적으로 와닿지 않을 수 있다. "API 호출 잘 되면 됐지, 뭘 더 설계해야 하나?"는 생각이 가장 흔한 저항이다.

Gartner 수치는 이 저항을 무너뜨리는 서사적 입구로 기능할 수 있다. "30%가 폐기됐다 → 이 프로젝트들은 API 호출에는 성공했다 → 그런데 왜 실패했는가 → 시스템 수준에서 붕괴가 일어났다"는 흐름을 통해, 독자 스스로 구성 요소 분리의 필요성을 납득하도록 이끌 수 있다.

> **활용 문장 초안:**
> "Gartner는 생성형 AI 프로젝트의 약 30%가 PoC를 넘지 못하고 폐기될 것이라 예측했다. 이 프로젝트들의 공통점이 있다. LLM 호출은 성공했다는 것이다. 모델은 응답을 돌려줬고, 데모는 인상적이었다. 그런데 운영에서 무너졌다. 이유는 API 호출이 시스템 설계가 아니기 때문이다."

이 흐름은 ch03의 "문제는 모델이 아니라 시스템이다"를 이어받아, ch04에서 "그렇다면 시스템은 어떤 부품으로 구성되는가"로 자연스럽게 전환하는 브리지 역할을 한다.

---

### ✅ ch04 구성 요소 분리 논증의 '사전 동기 레이어'로 활용

ch04의 4-3항("구성 요소 분리의 설계적 이유")은 분리의 필요성을 구조적으로 설명하는 섹션이다. 그러나 독자는 이 설명을 읽기 전에 이미 "왜 이게 중요한가"를 납득해야 한다. Gartner 수치는 그 납득의 선제 조건을 충족시키는 역할을 한다.

**논리 흐름 설계 예시:**
1. Gartner 수치 제시 → "이 실패는 API 수준의 문제가 아니다"
2. 독자 질문 유도 → "그렇다면 어디서 무너지는가?"
3. 저자 논지 전환 → "파이프라인의 각 단계가 독립된 실패 단위가 아닐 때 무너진다. 분리하지 않으면 진단도, 교체도, 개선도 불가능하다."

---

### ✅ ch04 각 구성 요소의 '실패 시 증상' 서술의 서사적 배경 제공

ch04의 편집 지시는 각 구성 요소를 "역할 / 실패 시 증상 / 설계 시 고려사항" 3요소로 서술하도록 되어 있다. Gartner 수치는 이 "실패 시 증상" 서술의 배경 현실감을 높이는 데 활용 가능하다.

각 구성 요소를 설명할 때, "이런 실패가 실제로 프로젝트 폐기로 이어진다"는 맥락을 한 줄씩 연결하면 독자는 개념 학습을 단순 지식 습득이 아닌 실전 위험 예방으로 받아들이게 된다.

> **활용 구조 초안 (출력 처리기 예시):**
> "출력 처리기(Output Processor)가 없으면 모델 응답은 그대로 사용자에게 전달된다. 모델은 때로 JSON 대신 설명문을 돌려주고, 때로 필드 이름을 바꾸고, 때로 아무것도 돌려주지 않는다. 이 한 단계의 부재가 시스템 전체를 불안정하게 만든다. 30%의 폐기된 프로젝트 중 상당수는 이런 지점에서 붕괴했을 것이다."

---

### ✅ ch05·ch06 연결부에서 "구성 요소 확장" 논증의 기반 수치로 재활용

ch04는 ch05(RAG)와 ch06(에이전트)을 배울 때 "구조의 어느 부분이 확장되는가"를 파악하는 기반 지식을 제공하는 장이다. Gartner 수치는 ch04 소결에서 재활용해, "기본 파이프라인을 이해하지 못한 채 RAG나 에이전트를 추가하면 이 30%의 대열에 합류한다"는 경고성 전환 문장으로 활용할 수 있다.

> **활용 문장 초안 (ch04 소결):**
> "RAG를 붙이고 에이전트를 얹기 전에, 기본 파이프라인이 각 역할별로 분리되어 있는지를 먼저 물어야 한다. 그 질문을 건너뛴 프로젝트들이 Gartner가 예측한 30%를 채운다."

## 4. 주의할 점 / 한계

### ⚠️ 원문 미확인 — 모든 수치는 검증 전 사용 불가

"30%"라는 수치는 현재 source note의 추정이다. 원문 Gartner 보고서에 접근해 다음을 반드시 확인해야 한다:

- 정확한 수치 (30%인지, 혹은 유사 표현인지)
- "폐기"의 조작적 정의 (포기? 축소? 전환?)
- PoC의 기준 (무엇을 PoC로 정의했는가)
- 보고서 발행 연도 및 조사 방법론
- 예측 근거 (설문 기반인지, 모델 기반인지)

### ⚠️ ch03과 ch04에서 동일 수치 중복 사용 시 서사 소진 위험

web_05가 ch03 도입부에서 이미 사용된 경우, ch04에서 동일 수치를 재인용하면 독자가 "또 이 얘기"로 받아들일 수 있다. 재활용 시에는 관점을 전환해야 한다:

- ch03에서의 용도: "모델 탓이라는 귀인 오류의 증거"
- ch04에서의 용도: "API 호출과 시스템 설계의 차이를 보여주는 현상 기술"

동일 수치를 다른 논증 레이어에서 재사용하는 것은 가능하나, 문장 표현과 맥락이 명확히 달라야 한다.

### ⚠️ 기술적 원인 분류는 보고서 범위 밖

Gartner 보고서는 "왜 어느 구성 요소에서 실패하는가"를 다루지 않는다. ch04의 각 구성 요소별 실패 증상 서술은 저자가 직접 구성해야 하며, Gartner 수치를 그 근거로 사용하면 논리적 비약이 된다.

수치는 "현상 기술"에만 사용하고, "기술적 분류"의 근거로는 사용하지 않는다.

### ⚠️ Source 제목 불일치 문제 별도 확인 필요

source_id web_05의 등록 제목("The Architecture of LLM-Powered Applications...")과 Source Note 본문의 실제 분석 대상("Gartner Predicts 30%...")이 다르다. 이것이 **동일 source의 두 섹션인지, 다른 source가 혼재된 것인지** 원문 확인 전까지는 불명확하다. 인용 전 반드시 확인 필요.
