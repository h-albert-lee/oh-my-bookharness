### 추론 가능한 핵심 주장 (두 제목을 통합 해석)

**주장 A — 구조화 출력은 파이프라인의 숨겨진 핵심이다**
LLM의 비정형 텍스트 출력을 시스템이 소비할 수 있는 구조화된 형태(JSON, 스키마 등)로 변환하는 것이 프로덕션 AI 시스템의 안정성을 결정한다. "조용한 영웅(Silent Hero)"이라는 표현은 이것이 눈에 잘 띄지 않으면서도 실패의 주요 원인이 된다는 뉘앙스를 담는다.

**주장 B — 출력 처리기(Output Processor)의 독립적 설계 필요성**
구조화 출력은 파이프라인의 특정 구성 요소(출력 처리기)가 전담해야 하며, 이를 LLM 호출 로직에 혼재시키면 실패 지점이 불투명해진다.

**주장 C — TDD 적응론과의 연결**
구조화 출력의 실패는 기존 `assertEqual` 방식의 테스트로 포착되지 않는다. 출력 스키마 검증, 타입 안전성, 파싱 실패율 등 새로운 측정 기준이 필요하다는 논지로 두 주제가 연결된다.

| 출력 구조화 = 평가 가능성의 전제 조건 | ch04 → ch10 연결 고리 | 출력이 구조화되지 않으면 평가 자체가 불가능하다는 논리로, ch04(파이프라인 설계)와 ch10(테스트의 한계), ch11(평가 설계)을 잇는 개념적 다리 역할 |

### 🔵 보조 포인트 (ch02 연계)

- **TDD 한계의 구체적 사례**: 구조화 출력 실패는 기존 단위 테스트가 포착하지 못하는 유형의 실패다. ch02에서 "Assert Equal의 붕괴"를 논할 때, 출력 스키마 불일치를 실패 유형 예시로 사용 가능
- **"조용한 실패(Silent Failure)"**: 구조화 출력 오류는 예외를 발생시키지 않고 잘못된 데이터가 하위 시스템으로 흘러내려가는 방식으로 실패한다. 이것이 왜 전통적 에러 핸들링과 테스트가 무력화되는지의 논거로 활용

### ⚠️ Counterpoint 활용 (ch02)

ch02에서 TDD 적응론의 한계를 논증할 때, 구조화 출력 문제를 끌어오면 논거가 더 구체적이 된다:

```
[적응론의 한계 논증에 삽입 가능한 예시]
"출력 형식을 JSON 스키마로 강제하면 테스트할 수 있지 않는가?"
→ 가능하다. 하지만 스키마 준수 여부는 출력의 품질과 무관하다.
  형식은 맞지만 내용이 잘못된 출력은 스키마 테스트를 통과한다.
  이것이 "검증 가능한 형식"과 "평가 가능한 품질"이
  서로 다른 차원의 문제임을 보여준다.
```

### 🟡 내용적 한계

- **구조화 출력의 범위 모호성**: JSON 강제, 함수 호출(Function Calling), Pydantic 스키마 등 구현 방식이 다양하다. 소스가 어떤 수준의 구조화를 다루는지 불명확하므로, 책에서는 구현 방식 중립적인 개념 수준으로만 활용할 것
- **"Silent Hero"는 수사이지 논증이 아니다**: 강조적 표현으로는 유효하지만, 이것 자체가 출력 처리기의 중요성을 증명하지는 않는다. 반드시 실제 실패 시나리오와 함께 사용해야 한다
- **TDD 연결의 논리 비약 가능성**: 두 제목이 시사하는 두 주제(구조화 출력 + TDD 적응)의 연결이 소스 내에서 명확하게 논증되는지 알 수 없다. 책에서 이 연결을 사용할 때는 저자가 직접 논리를 구성해야 한다
