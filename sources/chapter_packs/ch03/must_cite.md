### web_01 — *The Root Causes of Failure for Artificial Intelligence Projects and How They Can Succeed: Avoiding the Anti-Patterns of AI*

#### 1. 핵심 주장
AI 프로젝트 실패의 근본 원인은 모델 성능이 아니라 조직·프로세스·시스템 설계 수준의 안티패턴에서 비롯된다. "AI 안티패턴"을 범주화함으로써, 실패가 반복 가능한 구조적 패턴임을 보인다.

#### 2. 책에 활용 가능한 포인트
- **귀인 오류 논증의 실증적 근거**: "모델 탓"이 안티패턴이라는 주장을 뒷받침하는 분류 체계로 직접 인용 가능
- **실패 유형의 반복성**: 동일한 실패가 다른 팀·조직에서 반복된다는 구조적 패턴 논증에 사용
- **4-1절(귀인 오류의 구조적 반복)**의 핵심 인용 후보

#### 3. 이 장과의 관련성
장의 핵심 명제("실패는 모델 성능의 문제가 아니라 시스템 설계의 문제다")를 문헌 수준에서 직접 지지하는 가장 강력한 소스. 안티패턴 분류는 **4-2절 실패 지점 지도(Failure Topology)**의 뼈대로도 전용 가능.

#### 4. 주의할 점 / 한계
- 연구 대상이 LLM 이전 세대의 ML 프로젝트 중심일 가능성 있음 → LLM 시대 적용 가능성을 본문에서 명시적으로 연결해야 함
- "안티패턴"이 처방보다 진단에 치우친 경우, PART 2 이후 설계 솔루션으로의 브리지가 필요

#### 5. 인용 시 주의 표현
> "안티패턴 분류는 LLM 기반 시스템에도 구조적으로 유효하지만, 개별 항목의 발현 형태는 다를 수 있다."

### web_02 — *Hidden Technical Debt in Machine Learning Systems*

#### 1. 핵심 주장
ML 시스템의 기술 부채는 모델 코드(전체의 소수)보다 **모델을 둘러싼 파이프라인·인프라·데이터 처리 코드**에 집중된다. 모델은 "작은 섬"이고, 진짜 복잡성은 그 주변에 있다.

#### 2. 책에 활용 가능한 포인트
- **"Only a Small Fraction of Real-World ML Systems is ML Code"** 도식: 4-2절 Failure Topology의 시각적 도입부로 최적. 독자에게 "모델 앞·뒤"의 존재를 직관적으로 납득시킴
- **모델 외부 의존성(entanglement, feedback loop, data dependency)**: 연결 구조 실패 유형을 설명하는 직접 근거
- 1장 하네스 논증과 연결 가능: "하네스 없이 모델만 있으면 기술 부채가 숨겨진다"

#### 3. 이 장과의 관련성
장의 보조 질문 — "AI 시스템에서 실패는 어느 지점에서 발생하는가?" — 에 대한 구조적 해답을 제공하는 가장 고전적·권위 있는 출처. **Failure Topology 전체 절의 이론적 토대**로 기능.

#### 4. 주의할 점 / 한계
- 2015년 Google 논문으로, LLM/에이전트 시스템의 신규 실패 유형(hallucination, tool-calling, multi-turn context)은 포함되지 않음
- 고전적 ML 파이프라인 중심 → web_03, web_04와 병렬 인용하여 "LLM 시대에도 동일한 패턴이 확장·심화된다"는 논리 구성 필요

#### 5. 인용 시 주의 표현
> "기술 부채의 위치가 모델 외부에 집중된다는 이 관찰은 LLM 시스템에서 더욱 증폭된다. 모델의 크기가 커질수록, 모델 주변 시스템의 복잡성 역시 비례해 증가하기 때문이다."

### web_03 — *Failure Modes in LLM Systems: A System-Level Taxonomy for Reliable AI Applications*

#### 1. 핵심 주장
LLM 시스템의 실패는 모델 자체의 오류가 아니라 **시스템 수준의 구조적 실패 모드**로 분류되어야 한다. 입력 처리, 컨텍스트 관리, 출력 검증, 오케스트레이션 등 계층별로 실패 유형이 다르다.

#### 2. 책에 활용 가능한 포인트
- **4-2절 Failure Topology의 핵심 분류 체계**: "모델 앞 / 모델 자체 / 모델 뒤 / 연결 구조"라는 본문의 4분류와 직접 매핑 가능
- 각 계층의 실패 예시: 입력 오염, 프롬프트 인젝션, 컨텍스트 손실, 출력 미검증 등을 사례로 사용
- "시스템 관점으로 전환한다는 것이 무엇을 달리 보는 것인가?"에 대한 조작적 답변 제공

#### 3. 이 장과의 관련성
이 장의 **가장 직접적인 기술 근거 소스**. 챕터의 핵심 질문("진짜 원인은 어디에 있는가?")에 계층별 분류로 응답하며, PART 2 이후 설계 주제(RAG, 가드레일, 파이프라인)가 각 계층의 실패 해법임을 연결하는 브리지 역할.

#### 4. 주의할 점 / 한계
- 분류 체계가 저자마다 다를 수 있으므로, 본문의 4분류와 완전 일치하지 않을 경우 적응적 재해석이 필요
- 실증 데이터보다 분류 체계 중심일 경우, 사례 보강을 web_05에서 가져와야 함

#### 5. 인용 시 주의 표현
> "이 분류는 본문의 Failure Topology와 용어가 다를 수 있으나, 실패가 계층별로 발생한다는 핵심 구조는 동일하다."

### web_04 — *Why Do Multi-Agent LLM Systems Fail?*

#### 1. 핵심 주장
멀티에이전트 LLM 시스템의 실패는 개별 에이전트의 모델 품질이 아니라 **에이전트 간 연결 구조, 오케스트레이션 실패, 피드백 루프 붕괴**에서 발생한다.

#### 2. 책에 활용 가능한 포인트
- **4-2절 Failure Topology 중 "연결 구조" 실패 유형**의 가장 강력한 현대적 사례
- "모델을 바꿔도 문제가 반복된다"는 장의 핵심 질문에 대한 구체적 증거: 각 에이전트 모델이 우수해도 시스템이 실패하는 케이스
- PART 2 이후 에이전트 설계 챕터로의 동기부여 역할

#### 3. 이 장과의 관련성
"연결 구조 실패"를 LLM 시대의 맥락에서 가장 선명하게 보여주는 소스. web_02의 기술 부채 논리를 멀티에이전트 시나리오로 현대화하는 역할. **4-2절의 "연결 구조" 섹션에서 단독 사례 블록으로 사용 가능**.

#### 4. 주의할 점 / 한계
- 멀티에이전트 시스템은 독자에게 아직 고급 토픽일 수 있음 → 3장에서는 "연결 구조 실패의 예시"로만 가볍게 인용하고, 상세 논의는 에이전트 챕터로 미룰 것
- 실험적 연구인 경우 일반화 주의

#### 5. 인용 시 주의 표현
> "멀티에이전트 시스템은 이 장의 논점을 가장 극단적으로 보여주는 사례다. 각 구성 요소가 정상이어도 시스템은 실패할 수 있다."

### web_05 — *Challenges in Deploying Machine Learning: A Survey of Case Studies*

#### 1. 핵심 주장
ML 배포 과정에서 발생하는 실패는 모델 정확도보다 **운영 환경, 데이터 드리프트, 모니터링 부재, 조직적 마찰**에서 비롯된다. 다수의 실제 배포 사례를 메타분석.

#### 2. 책에 활용 가능한 포인트
- **귀인 오류 논증의 실증적 보강**: "현장에서도 모델이 아닌 주변 시스템이 문제였다"는 사례 데이터
- 4-1절의 "새로운 사례 1~2개"를 이 소스에서 추출 가능
- 독자에게 친숙한 배포 문제(모니터링 없음, 데이터 파이프라인 부재)로 추상적 논증을 구체화

#### 3. 이 장과의 관련성
web_01의 안티패턴 분류를 **현장 사례로 뒷받침**하는 역할. 단독으로는 3장 논증을 이끌기 어렵지만, web_01·web_02와 병렬 인용 시 "이론+실증" 구조를 완성함.

#### 4. 주의할 점 / 한계
- 서베이 논문 특성상 특정 주장의 출처가 원 사례로 분산되어 있음 → 재인용 시 원 사례 확인 권장
- LLM 이전 전통 ML 중심일 가능성

#### 5. 인용 시 주의 표현
> "서베이의 사례 다수가 LLM 이전 시스템이므로, LLM 환경에서의 적용은 맥락을 명시해야 한다."

### web_07 — *Building LLM Applications for Production*

#### 1. 핵심 주장
LLM 애플리케이션의 프로덕션 배포는 모델 선택이 아니라 **신뢰성, 지연, 비용, 오케스트레이션**의 시스템 문제다. 실무자 관점에서 시스템 설계의 필요성을 직접 서술.

#### 2. 책에 활용 가능한 포인트
- **"시스템 관점으로 전환한다는 것이 실제로 무엇을 달리 보는 것인가?"** 에 대한 실무적 답변
- 독자층(ML 실무자)의 언어로 쓰인 소스 → 직관적 인용 가능
- PART 2 이후 설계 주제의 필요성을 동기부여하는 마지막 절에서 사용

#### 3. 이 장과의 관련성
3장의 마무리 — "그렇다면 무엇을 달리 설계해야 하는가?" — 로 넘어가는 브리지에서 사용. PART 2 도입의 직접적 동기부여 소스.

#### 4. 주의할 점 / 한계
- 블로그/실무 문서 형태일 경우, 학술 인용으로서의 권위가 낮을 수 있음
- 특정 프레임워크(LangChain 등)에 종속된 논의는 제거하고 원리 수준만 인용

#### 5. 인용 시 주의 표현
> "특정 구현체 의존 내용은 제외하고, 시스템 설계 원칙 수준의 주장만 인용할 것."

### web_06 — *Machine Learning in Production: From Models to Products*

#### 1. 핵심 주장
모델 개발과 프로덕션 운영 사이의 간극(Model-to-Product gap)을 체계화. 프로덕션 ML은 정확도 최적화가 아니라 시스템 신뢰성 관리 문제임을 논증.

#### 2. 책에 활용 가능한 포인트
- 장의 도입부에서 "모델 개발 완료 ≠ 제품 완성"이라는 명제를 지지하는 배경 인용
- web_02의 기술 부채 논의와 연결하여 "왜 프로덕션에서 실패하는가"의 맥락 제공

#### 3. 이 장과의 관련성
web_02, web_05를 보완하는 맥락 소스. 독립 인용보다는 묶음 인용(web_02 + web_06)으로 효율적.

#### 4. 주의할 점 / 한계
- 다른 소스들과 주장이 상당 부분 중복될 수 있음 → 독자적 contribution이 명확한 경우에만 단독 인용

#### 5. 인용 시 주의 표현
> "web_02, web_05와 주장이 유사하므로, 차별화된 포인트가 없다면 묶음 인용 또는 생략을 권장."
