# Must-Cite: Chapter 04 — LLM 파이프라인 기본 구성 요소

---

## 인용 원칙

이 장의 핵심 논지는 **"LLM 호출 ≠ LLM 시스템 설계"**이며, 파이프라인을 역할이 구분된 구성 요소들의 연결로 분해해서 보여주는 것이 목적이다. 아래 must-cite 항목은 그 논지를 뒷받침하거나, 각 구성 요소의 설계적 근거를 제공하는 자료들이다.

---

## TIER 1: 반드시 인용 (논지의 뼈대)

---

### 🔴 web_01 — *Emerging Architectures for LLM Applications*

**인용 이유**
이 장의 출발점이 되는 자료. LLM 기반 애플리케이션이 단일 모델 호출이 아니라, 서로 다른 역할을 가진 컴포넌트들의 조합으로 구성된다는 것을 구조적으로 최초로 정의한 레퍼런스 중 하나다. "입력 처리 → 컨텍스트 구성 → LLM 호출 → 출력 처리"라는 파이프라인 흐름을 아키텍처 다이어그램으로 제시한다.

**활용 포인트**
- 4-1절 "파이프라인으로서의 LLM 시스템" 개념 도입 시 구조적 근거 자료로 인용
- 각 구성 요소의 역할 경계를 정의할 때 개념 참조
- "RAG, 에이전트 등은 이 기본 구조의 특정 컴포넌트가 확장된 것"이라는 맥락 연결에 사용

**주의 표현**
> "LLM 애플리케이션의 아키텍처는 단일 호출이 아닌 계층화된 컴포넌트 구조로 이해해야 한다 (web_01)."

---

### 🔴 web_05 — *The Architecture of LLM-Powered Applications: How It Differs from Conventional Software Architecture*

**인용 이유**
이 장의 핵심 질문("LLM을 호출하는 것과 LLM 시스템을 설계하는 것은 무엇이 다른가?")에 직접 대응하는 자료. 기존 소프트웨어 아키텍처와 LLM 기반 시스템 아키텍처의 구조적 차이를 명시적으로 대비한다.

**활용 포인트**
- 장 서두의 문제 설정: "왜 LLM 시스템은 기존 소프트웨어와 다른 설계 단위가 필요한가"의 논거
- 비결정성(non-determinism), 상태 관리, 오류 전파 방식이 기존 시스템과 다르다는 점을 설명할 때
- 독자가 "익숙한 소프트웨어 개발 관점"에서 "LLM 시스템 관점"으로 전환하는 전환점으로 사용

**주의 표현**
> "LLM 기반 시스템은 기존 소프트웨어의 결정론적 흐름 제어 가정이 성립하지 않는다 (web_05)."

---

### 🔴 web_09 — *How to Develop Modular LLM Pipelines?*

**인용 이유**
"각 구성 요소는 독립적으로 교체하거나 평가할 수 있는가?"라는 부속 질문에 직접 답하는 자료. 모듈화가 LLM 파이프라인 설계에서 갖는 의미, 그리고 각 단계를 독립 단위로 설계해야 하는 실용적 근거를 제공한다.

**활용 포인트**
- 4-2절 각 구성 요소가 "왜 독립된 설계 단위가 되어야 하는가"의 설계 원칙 근거
- 모듈 경계를 어떻게 정의할 것인가에 대한 실무적 관점 제공
- 나중에 5장(RAG), 6장(에이전트)에서 "어느 모듈이 확장되는가"를 설명할 때의 복선으로 활용

**주의 표현**
> "파이프라인의 각 단계를 독립적으로 테스트·교체 가능하게 설계하는 것은 유지보수성과 디버깅 가능성 모두에 영향을 미친다 (web_09)."

## TIER 2: 핵심 구성 요소별 인용 (각 절의 설계적 근거)

---

### 🟠 web_04 — *From Prompts to Templates: A Systematic Prompt Template Analysis for Real-world LLM apps*

**인용 이유**
4-2절의 "프롬프트 구성(Prompt Construction)" 컴포넌트를 설명할 때, 프롬프트가 단순 문자열이 아니라 **템플릿화된 구조적 단위**여야 한다는 근거를 제공하는 자료. "프롬프트 엔지니어링은 파이프라인 전체 중 한 단계에 불과하다"는 이 장의 주장과 정확히 맞물린다.

**활용 포인트**
- 프롬프트 구성 컴포넌트가 왜 단순 문자열 concatenation이 아닌 독립된 설계 단위여야 하는지의 근거
- 템플릿화를 통해 재사용성, 버전 관리, 테스트 가능성이 생긴다는 논점
- "프롬프트는 코드다"라는 설계 원칙을 뒷받침

**주의 표현**
> "프롬프트를 템플릿으로 관리한다는 것은 단순한 편의가 아니라, 파이프라인에서 프롬프트 구성을 독립된 관심사로 분리하는 아키텍처 결정이다 (web_04)."

---

### 🟠 web_08 — *LLM Context Management: How to Improve Performance and Lower Costs*

**인용 이유**
4-2절의 "상태 관리(State Management)" 및 "컨텍스트 구성(Context Assembly)" 컴포넌트에 대한 핵심 참조 자료. 컨텍스트 윈도우 관리가 단순한 토큰 절감 문제가 아니라, **파이프라인의 독립된 설계 단위**여야 하는 이유를 설명한다.

**활용 포인트**
- 컨텍스트 관리 컴포넌트의 역할 정의 (무엇을 얼마나 어떤 순서로 넣을 것인가)
- 상태가 잘못 관리될 때 발생하는 성능·비용·품질 저하의 구체적 사례
- "구성 요소 간의 정보 흐름은 어떻게 설계되어야 하는가"라는 부속 질문의 답변 근거

**주의 표현**
> "컨텍스트 관리는 비용 최적화 전술이 아니라, LLM 파이프라인에서 무엇이 모델에게 전달되는가를 결정하는 독립된 설계 레이어다 (web_08)."

---

### 🟠 web_07 — *Failure Modes in LLM Systems: A System-Level Taxonomy for Reliable AI Applications*

**인용 이유**
"파이프라인의 어느 지점에서 무엇이 실패할 수 있는지를 예측하는 사고 습관"을 독자에게 심어주는 것이 이 장의 학습 목표 중 하나다. 이 자료는 LLM 시스템의 실패 모드를 **시스템 레벨 분류체계**로 정리하며, 각 파이프라인 컴포넌트가 독립된 실패 지점이 될 수 있음을 보여준다.

**활용 포인트**
- 각 구성 요소(입력 처리, 프롬프트 구성, LLM 호출, 출력 처리) 설명 후 "이 컴포넌트에서 발생할 수 있는 실패 모드" 소단락에 인용
- "왜 각 컴포넌트를 독립적으로 설계해야 하는가"의 방어적 근거 (실패 격리)
- 독자가 파이프라인을 설계할 때 실패 지점을 선제적으로 예측하는 사고 프레임 제공

**주의 표현**
> "파이프라인의 각 컴포넌트는 독립된 실패 지점(failure point)이다. 컴포넌트 경계를 명확히 할수록 실패의 원인을 격리하기 쉬워진다 (web_07)."

### 🟡 web_02 — *Building LLM Applications for Production*

**인용 이유**
프로덕션 환경에서 LLM 파이프라인이 실제로 어떤 복잡성을 가지는지를 설명하는 실무 중심 자료. 이 장의 "이것은 튜토리얼이 아니다"라는 톤을 유지하면서, 설계 결정이 프로덕션에서 어떤 결과로 이어지는지를 보여줄 때 사용.

**활용 포인트**
- 각 컴포넌트 설명 후 "이것이 프로덕션에서 실제로 중요한 이유" 맥락에서 인용
- "개념 증명(PoC) 수준의 단일 호출"과 "프로덕션 수준의 파이프라인"의 격차를 보여주는 근거
- 5장(RAG), 6장(에이전트) 복선으로 "파이프라인이 확장될수록 각 컴포넌트의 역할이 더 중요해진다"는 논점 지원

**활용 조건**
> 단독 인용보다는 web_01, web_05와 함께 묶어서 "설계 원칙 → 프로덕션 현실"의 흐름으로 사용하는 것이 효과적이다.

---

### 🟡 web_06 — *Retries, Fallbacks, and Circuit Breakers in LLM Apps: A Production Guide*

**인용 이유**
4-2절에서 "LLM 호출(LLM Call)" 컴포넌트를 설명할 때, 이 컴포넌트가 단순한 API 호출이 아니라 **재시도, 폴백, 회로 차단기**를 포함하는 독립된 설계 레이어여야 한다는 근거 제공.

**활용 포인트**
- LLM 호출 컴포넌트의 책임 범위 정의 (단순 호출 이상의 것들)
- 외부 서비스 의존성이 파이프라인 전체에 미치는 영향을 격리하는 설계 원칙
- web_07의 실패 모드 논의와 연결: "실패를 예측했다면, 이렇게 대응한다"

**활용 조건**
> 이 장에서는 개념 수준으로만 언급하고, 상세 구현은 후속 장(신뢰성, 오류 처리 챕터)으로 넘기는 방식이 구조적으로 적합하다. 과도하게 인용하면 "튜토리얼이 아니다"라는 톤이 무너진다.

---

### 🟡 web_03 — *Building Effective Agents*

**인용 이유**
이 장에서 정의하는 기본 파이프라인 구조가 6장(에이전트)에서 어떻게 확장되는지를 예고할 때 참조. 에이전트 시스템도 결국 이 장에서 정의한 컴포넌트들의 재조합이라는 점을 보여준다.

**활용 포인트**
- 장 말미의 "이 구조는 어디까지 확장되는가" 섹션에서 복선으로 인용
- "에이전트는 새로운 시스템이 아니라, 기본 파이프라인 컴포넌트에 루프와 도구 호출이 추가된 것이다"라는 논점 지원

**활용 조건**
> 이 장의 메인 논지와 직접 연관성이 낮으므로, 본문이 아닌 "다음 장을 예고하는" 맥락에서만 사용. 과도한 인용은 이 장의 집중도를 분산시킨다.

### 🟡 web_10 — *Beyond Prompt Engineering: What Engineers Are Actually Using to Make LLMs Reliable in Production*

**인용 이유**
"프롬프트 엔지니어링이 파이프라인 전체 중 한 단계에 불과하다"는 이 장의 핵심 주장을 실무 엔지니어링 관점에서 뒷받침하는 자료. 실제 현장에서는 프롬프트 이상의 시스템 설계가 이루어지고 있다는 empirical evidence로 기능한다.

**활용 포인트**
- 장 서두의 문제 설정: "프롬프트만 잘 짜면 된다는 오해를 어떻게 깨는가"
- 신뢰성은 프롬프트가 아닌 시스템 구조에서 온다는 논점의 실무적 근거

**활용 조건**
> 자료의 성격상 실무 경험 기반의 비공식 인사이트이므로, 학술적·구조적 주장의 근거보다는 "현장 관찰"로 포지셔닝하여 인용.
