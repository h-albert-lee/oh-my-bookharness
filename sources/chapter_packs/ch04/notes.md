## Source Pack Notes — ch04: LLM 파이프라인 기본 구성 요소

### 1. 이 source pack의 목적

ch04는 LLM 기반 시스템의 각 구성 요소를 역할 중심으로 정의하고, 왜 그것들이 독립된 설계 단위가 되어야 하는지를 구조적으로 설명하는 장이다. 따라서 수집된 자료는 다음 세 가지 기능으로 분류된다.

1. **파이프라인 구조화의 권위 있는 근거** — a16z 레퍼런스 아키텍처, Anthropic 설계 원칙, Chip Huyen 블로그
2. **각 구성 요소의 필요성을 역순 서술하는 사례** — 실패 분류 논문, 프롬프트 드리프트 분석, 컨텍스트 관리 가이드
3. **소프트웨어 설계 원칙과의 연결 고리** — SoC 적용 사례, 아키텍처 비교 분석

### 3. Supporting Sources 활용 방향

**프레임워크 비교 자료 (web_12, web_15, web_16)**
- 프레임워크(LangChain, LlamaIndex)가 파이프라인을 어떻게 분해하는지 비교
- "구성 요소는 특정 도구에 종속되지 않는다"는 ch04 핵심 주장의 실증 사례
- 각 프레임워크가 서로 다른 방식으로 같은 개념 문제를 해결한다는 점 강조

**출력 처리 심화 자료 (web_17, web_18, web_21)**
- 출력 처리기 절(역할 / 실패 시 증상 / 설계 고려사항)에 밀도 추가
- data contract 개념 → 정보 흐름 설계 논의로 자연스러운 연결
- 파싱 오류 처리 계층 구조로 출력 처리기와 오류 처리기의 협력 관계 설명

**아키텍처 원칙 자료 (web_11, web_13, web_14)**
- 구성 요소 분리의 설계적 이유 절(4-3)에서 이론적 배경으로 활용
- P-t-E 패턴(web_14)은 ch06(에이전트) 연결 포석으로도 기능

**운영 실패 사례 자료 (web_19, web_22)**
- ch03("실패 원인이 시스템에 있다") → ch04("파이프라인 어느 단계가 원인인가") 전환에 사용
- 프롬프트 드리프트 사례는 프롬프트 구성기가 독립 단위여야 하는 운영적 근거

**Google ADK 사례 (web_20)**
- 상태 관리자와 컨텍스트 구성의 분리를 Big Tech 실제 결정으로 뒷받침
- ch06(에이전트) 상태 관리 확장 논의의 포석
