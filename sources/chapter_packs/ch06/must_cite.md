## 🔴 TIER 1 — 반드시 인용 (장의 핵심 논지를 직접 지지)

### web_02 | Building Effective Agents
**인용 이유:**
ch06의 핵심 주장인 "에이전트를 단일 모델 호출이 아닌 시스템으로 보라"는 관점의 실천적 근거를 제공한다. 워크플로우(workflow)와 에이전트(agent)의 구분, 컴포넌트 조합 방식에 대한 실용적 프레임은 "시스템을 설계할 때 어디를 고정하고 어디를 교체 가능하게 두어야 하는가"라는 질문에 직접 답한다.

**활용 포인트:**
- 에이전트 복잡도를 높이기 전에 단순 파이프라인부터 시작하라는 권고 → "시스템 경계" 개념 도입 시 인용
- 오케스트레이터-서브에이전트 구조 설명 시 구조 참조
- "언제 에이전트가 필요한가" 기준 → 독자가 시스템 관점을 내면화하는 진입점으로 활용

---

### web_04 | Why Do Multi-Agent LLM Systems Fail? (MAST)
**인용 이유:**
"에이전트가 잘못된 결과를 냈을 때 원인은 모델인가 시스템인가?"라는 ch06의 핵심 질문에 실증적으로 답하는 유일한 소스다. MAST 분류 체계는 실패를 컴포넌트 단위로 귀인(attribution)하는 언어를 제공하며, 이는 독자가 시스템 분해 능력을 갖추도록 돕는 ch06의 구조적 목표와 정확히 일치한다.

**활용 포인트:**
- 실패 분류 taxonomy(태스크 명세 실패, 도구 실패, 에이전트 간 오류 전파 등) → "실패 경로(failure path)" 절의 핵심 근거
- 멀티에이전트 시스템에서 오류가 컴포넌트 경계를 넘어 전파되는 메커니즘 → 컴포넌트 간 의존 절에서 반드시 인용
- "모델 탓인가, 시스템 탓인가"를 판별하는 분석틀로 직접 제시 가능

### web_05 | Chapter 3: Architectures for Building Agentic AI
**인용 이유:**
에이전트 아키텍처를 학술적·구조적으로 정리한 소스로, ch06이 필요로 하는 "컴포넌트 분해 어휘"를 가장 체계적으로 제공한다. 시스템 경계, 컴포넌트 유형, 아키텍처 패턴을 동시에 다루며 신뢰성(reliability) 관점을 포함한다는 점에서 ch06 전체의 이론적 골격으로 기능할 수 있다.

**활용 포인트:**
- 에이전트 구성 컴포넌트 분류(플래너, 실행기, 메모리, 도구 인터페이스 등) → ch06의 "시스템으로 분해하는 어휘" 절에 직접 대응
- 아키텍처 선택이 신뢰성에 미치는 영향 → 설계 기준 제시 절에서 인용
- 독립된 arxiv 논문이므로 학술적 인용 신뢰도 높음

### web_01 | LLM Powered Autonomous Agents
**인용 이유:**
에이전트를 "Planning + Memory + Tools + Action"의 컴포넌트 합성으로 정의한 최초의 영향력 있는 프레임워크 소스다. ch06이 ch01–ch05의 개념을 하나의 시스템으로 묶는 역할을 한다는 점에서, 이 소스의 구조 분해 방식은 그 통합의 출발 지점으로 인용 가치가 높다.

**활용 포인트:**
- 에이전트를 서브시스템 단위(Planning / Memory / Tool Use)로 처음 분해한 시각 → ch06의 "시스템으로 분해하는 어휘" 절 도입부에서 역사적 기준점으로 인용
- 이 분류가 이후 실제 구현에서 어떻게 변형·구체화되었는지를 web_02, web_05와 대비시킬 수 있음

> ⚠️ **주의:** 블로그 포스트 형식으로, 실증 데이터 없이 개념적 정리에 그친다. 이론적 도입부에서만 사용하고, 설계 근거로는 web_02/web_05로 넘어갈 것.

---

### web_03 | ReAct: Synergizing Reasoning and Acting in Language Models
**인용 이유:**
"추론과 행동이 분리된 스텝으로 구성된다"는 ReAct의 구조는 컴포넌트 간 계약(contract)과 실행 순서라는 ch06의 핵심 개념을 설명하는 데 가장 직관적인 실증 사례다. 에이전트 내부 루프(Thought → Act → Observe)를 시스템 흐름으로 시각화하는 데 활용할 수 있다.

**활용 포인트:**
- Thought-Act-Observe 루프 → "에이전트 내부에서 컴포넌트가 어떤 계약으로 연결되는가"를 보여주는 도해 소재
- 단일 에이전트 수준에서의 시스템 경계 명시 → "단순하게 보이는 에이전트의 내부 흐름을 구조적으로 읽는 능력" 목표에 직결
- 논문이므로 인용 신뢰도 높음

---

### web_06 | Harness Engineering for Coding Agent Users
**인용 이유:**
"평가 하네스(harness)가 왜 시스템 외부에 놓여야 하는가"라는 ch06의 명시적 설계 목표에 직접 답하는 유일한 소스다. 이 장에서 평가 하네스의 설계 근거를 논하는 절이 존재한다면 이 소스 없이는 성립하지 않는다.

**활용 포인트:**
- 하네스를 에이전트 시스템 외부에 배치하는 설계 원칙 → "시스템 경계" 절에서 평가 관점 보완
- 코딩 에이전트 사례 → 추상적 논의를 구체적 실무 사례로 착지시킬 때 사용
- ch05(평가)와 ch06(시스템)을 연결하는 가교 소스로 기능 가능

> ⚠️ **주의:** 코딩 에이전트에 특화된 사례이므로 일반화 시 "이 원칙은 도메인에 무관하게 적용된다"는 별도 논증 필요.

### web_07 | Towards a Science of AI Agent Reliability
**인용 이유:**
신뢰성을 과학적으로 정의하려는 시도로, ch06이 "설계 근거"를 넘어 "검증 가능한 시스템 품질"로 논의를 확장할 때 유용하다. 단, 이 장의 주요 목표가 개념 정립과 설계 판단력 제공에 있으므로, 신뢰성의 과학적 측정보다 "신뢰성을 시스템 속성으로 보는 시각"만 선택적으로 발췌한다.

**활용 포인트:**
- 에이전트 신뢰성이 컴포넌트 단위가 아닌 시스템 수준에서 측정되어야 한다는 주장 → "컴포넌트 간 의존" 절 마무리 논점으로 사용
- 반론으로도 활용 가능: "신뢰성을 정의하기도 어렵다면 설계 기준은 어떻게 세울 것인가?"

---

### web_08 | Where LLM Agents Fail and How They Can Learn From Failures
**인용 이유:**
실패 사례를 학습 피드백으로 전환하는 관점은 ch06의 "실패 경로" 논의를 정적 분석에서 동적 개선으로 확장하는 데 활용할 수 있다. 단, 이 장의 범위가 시스템 설계 원칙 수립에 집중되어 있으므로 "학습으로부터의 회복"은 부차적 내용으로만 다룬다.

**활용 포인트:**
- 실패 유형별 학습 전략 → 실패 경로 절의 "어떻게 대응할 것인가" 파트에서 1–2문장 인용
- web_04(MAST)와 쌍으로 인용하면 실패의 분류(web_04)와 대응(web_08)을 함께 제시하는 구조가 됨

> ⚠️ **주의:** 이 소스가 제시하는 자동화된 실패 학습 메커니즘은 ch06의 설계 범위를 넘어선다. 과도하게 인용하면 장의 초점이 분산될 위험이 있음.
